---

체크리스트:

✅ 1. 도입부 훅: 통과 — `#EXT-X-TARGETDURATION`을 "흔히 넘어가는 설정값"으로 제시하며, "그 선택이 CDN 트래픽, 재버퍼링률, 서버 CPU에 어떤 파급을 일으키는지"라는 현실적 문제의식으로 시작. 독자가 공감할 실무 상황이다.

✅ 2. 저자 포지셔닝: 통과 — "FFmpeg + nginx-rtmp + HLS.js 스택으로 직접 세그먼트 크기별 측정 환경을 구성하고, 2초/6초/10초 세 구간에서 재버퍼링 비율, 세그먼트 요청 빈도, 인코딩 오버헤드를 기록했다"고 명시. 직접 측정했다는 맥락이 구체적으로 드러난다.

✅ 3. 구조 흐름: 통과 — HLS 개념 정의 → 측정 환경(스택/파라미터) → 2초/6초/10초 각각 장단점 → 비교 표 → GOP 정렬 심화 → 결론 순서로 점진적 깊이가 있다. 개념 → 셋업 → 수치 결과 흐름이 명확하다.

✅ 4. 결론 회응: 통과 — 도입부에서 제기한 "2초/6초/10초 중 무엇을 고를까"라는 질문에, 결론이 세 케이스별 판단 기준("지연이 곧 돈인 서비스 → 2초", "안정적인 대규모 라이브 → 6초", "DVR/모바일 우선 → 10초")으로 직접 답한다.

✅ 5. 구체적 수치: 통과 — 추상 표현 없이 전 구간 측정값 포함.
- 라이브 지연: 7.2초 / 18초 / 38초 (실측)
- 재버퍼링: 4.1회/분 / 1.8회/분 / 1.1회/분 (0.5% 패킷 손실 조건 명시)
- CDN 요청 수: 300회 / 100회 / 60회 (10분 단일 클라이언트 기준)
- CDN 캐시 히트율: 78% / 92% / 96%
- CPU 부하: +8~12% / 기준 / -9%
측정 조건(환경: nginx-rtmp, HLS.js, tc netem 100Mbps 0.5%, 5회 반복)도 "측정 환경" 섹션에 명시되어 있다.

✅ 6. 코드 완성도: 통과 — FFmpeg 세그먼트 생성 명령과 GOP 정렬 명령 두 블록 모두 실행 가능한 완전한 형태. 핵심 파라미터(`-g`, `-keyint_min`, `-sc_threshold`, `-hls_time`, `-hls_list_size`, `-hls_flags`)에 한 줄 주석 첨부. 플레이리스트 샘플도 주석 포함.

✅ 7. 출처 링크: 통과 — 4개 링크 확인.
1. HLS RFC 8216 (IETF) — 프로토콜 정의
2. Apple HLS Authoring Specification — 6초 권고 출처
3. FFmpeg HLS 공식 문서 — `-hls_time` 동작 설명
4. Twitch 기술 블로그 — 대규모 라이브 플랫폼 사례
