# LL-HLS 수업 내용 (2026-05-30)

> 상태: 강의 완료, 퀴즈 미완료

---

## 한 줄 요약

기존 HLS의 20~30초 지연을 2~5초로 줄인 버전. 세그먼트를 잘게 쪼개고, 폴링 대신 서버가 응답을 기다리는 방식으로 지연을 없앴다.

---

## 기존 HLS 지연의 근본 원인

```
① 세그먼트 완성 대기: 6초 분량 쌓여야 .ts 파일 생성 가능  → 6초
② 플레이어 버퍼:      끊김 방지용 3개 선수집               → 18초
③ 폴링 대기:          "새 거 있어?" N초마다 물어보는 시간   → 3~6초

총합: 약 27~30초
```

LL-HLS는 이 세 군데를 각각 따로 공격한다.

---

## 핵심 기술 3가지

### ① Partial Segment — 세그먼트를 더 잘게 자르기

```
기존 HLS:
[━━━━━━━━━━━━━━━━━━━━━━━━━━━] seg_0.ts (6초짜리 완성돼야 배포)

LL-HLS:
[━━━━━] [━━━━━] [━━━━━] [━━━━━] [━━━━━] [━━━━━]
 part_0  part_1  part_2  part_3  part_4  part_5
 (1초)   (1초)   (1초)   (1초)   (1초)   (1초)
   ↑ 완성 즉시 배포
```

- 크기: 200ms~1초
- .fmp4 필수 (.ts는 파일 중간부터 독립 디코딩 불가)
- Partial Segment = 완전한 파일이 아닌 바이트 범위(EXT-X-BYTERANGE)로 표현

**.fmp4가 필수인 이유:**
`.fmp4`는 `moof`(메타데이터) + `mdat`(영상 데이터) 쌍 구조 → 파일 일부만 잘라도 독립 디코딩 가능.
`.ts`는 이 구조가 없어서 중간부터 디코딩 어려움.

---

### ② Blocking Playlist Reload — 폴링 없애기

```
기존 HLS:
플레이어: "새 세그먼트 있어?" → 서버: "없어" → 3초 대기 → 반복 (비효율)

LL-HLS:
플레이어: "part_5 다음에 뭐 오는지, 생기면 바로 알려줘"
서버: (part_6 완성될 때까지 응답 보류)
서버: part_6 완성 → 즉시 응답
```

쿼리 파라미터로 구현:
```
GET /playlist.m3u8?_HLS_msn=150&_HLS_part=3
```

**HTTP/2 필수인 이유:** HTTP/1.1은 홀드 중인 요청이 다른 요청을 막지만, HTTP/2는 멀티플렉싱으로 홀드 중에도 다른 요청 처리 가능.

---

### ③ Preload Hint — 다음 Partial Segment 선점 요청

```
현재 재생 중: part_3
플레이어:     part_4 아직 없지만 미리 요청
서버:         part_4 완성 즉시 → 대기 중인 플레이어에게 바로 전달
```

m3u8에 명시:
```m3u8
#EXT-X-PRELOAD-HINT:TYPE=PART,URI="seg_151.m4s",BYTERANGE-START=2048000
```

---

## LL-HLS m3u8 실제 구조

```m3u8
#EXTM3U
#EXT-X-VERSION:9
#EXT-X-SERVER-CONTROL:CAN-BLOCK-RELOAD=YES,PART-HOLD-BACK=1.0
#EXT-X-PART-INF:PART-TARGET=1.0

#EXTINF:6.0,
seg_148.m4s

#EXT-X-PART:DURATION=1.0,URI="seg_149.m4s",BYTERANGE="102400@0",INDEPENDENT=YES
#EXT-X-PART:DURATION=1.0,URI="seg_149.m4s",BYTERANGE="102400@102400"
#EXT-X-PART:DURATION=1.0,URI="seg_149.m4s",BYTERANGE="102400@204800"

#EXT-X-PRELOAD-HINT:TYPE=PART,URI="seg_149.m4s",BYTERANGE-START=307200
```

`INDEPENDENT=YES`: 이 Partial이 앞의 것 없이 독립 디코딩 가능 (I-frame 있는 경우).

---

## 지연 계산

```
LL-HLS:
① Partial Segment 완성 대기:  1초   → 1초
② 플레이어 버퍼:              2~3 Partial Segments → 2~3초
③ Blocking Reload:            즉시 수신 → ~0초

총합: 2~5초 (기존 30초 대비 약 10배 감소)
```

---

## 하위 호환성

LL-HLS 플레이리스트는 기존 HLS의 상위 집합.
기존 플레이어는 `#EXT-X-PART`, `#EXT-X-PRELOAD-HINT` 무시 → 완성 세그먼트만 재생 → 자동으로 기존 HLS로 fallback.

---

## 트레이드오프

| | LL-HLS | WebRTC |
|--|--------|--------|
| 지연 | 2~5초 | 0.5초 미만 |
| 동시 시청 | CDN 활용, 수백만 명 | SFU 필요, 확장 어려움 |
| CDN | 가능 (지원 CDN 필요) | 불가능 |
| 구현 난이도 | 중간 | 높음 |

---

## 실무 포인트 — CDN 지원이 핵심

일반 CDN은 Blocking Playlist Reload를 처리 못 함:
- 타임아웃 처리 or 캐시된 구버전 플레이리스트 반환
- → LL-HLS 효과 없음

LL-HLS 지원 CDN 필요 (Fastly, Akamai, AWS CloudFront — 2021~2022년부터 지원).

---

## 치지직 관점

```
스트리머 OBS → 인제스트 → 트랜스코더 (.fmp4 필수)
→ LL-HLS 패키저 (1초 Partial Segment)
→ Origin → LL-HLS 지원 CDN → 시청자
```

LL-HLS가 해결하는 것: 채팅과 방송 화면 싱크 (30초 → 3초)
LL-HLS로 해결 안 되는 것: 실시간 목소리 대화 → WebRTC 필요

## 면접 포인트

**Q. HLS와 LL-HLS의 핵심 차이는?**
> Partial Segment(1초 단위 즉시 배포) + Blocking Playlist Reload(폴링 제거) + Preload Hint(선점 요청). 이 세 가지가 합쳐져서 30초 → 2~5초.

**Q. LL-HLS 도입 시 가장 신경 써야 할 것은?**
> CDN 선택. Blocking Reload를 지원하는 CDN이 따로 있고, .fmp4 사용 필수.

**Q. LL-HLS 대신 WebRTC를 선택해야 하는 경우는?**
> 동시 시청자 수천 명 이하 + 0.5초 미만 지연 필수인 경우. 대규모 라이브엔 LL-HLS, 소규모 인터랙티브엔 WebRTC.
