## 레벨 4-4 / 2강: fMP4 박스 구조 깊이 분석

---

### MP4의 정체 — ISO Base Media File Format

MP4는 ISO 14496-12 (ISOBMFF) 표준 위에 있다.

```
[ISOBMFF 파생 포맷]
.mp4   (MPEG-4 Part 14)
.m4a   (오디오 전용)
.m4v   (비디오)
.mov   (QuickTime, 거의 동일)
.m4s   (CMAF segment)
.heic  (이미지)
```

같은 박스(box) 구조 공유. 박스 = 길이 + 타입 + 데이터.

---

### 박스 구조

모든 데이터가 박스로 감싸짐.

```
[일반 박스]
+--------+--------+----------------+
| size   | type   | data           |
| 4byte  | 4byte  | size - 8       |
+--------+--------+----------------+

예:
00 00 00 18  "ftyp"  isom...
^^^^^^^^^^^  ^^^^^^  ^^^^^^^
size=24      type    data
```

박스 안에 박스 들어갈 수 있다 (재귀 구조).

---

### init.mp4 박스 구조

초기화 세그먼트의 박스 트리.

```
init.mp4
├── ftyp (file type, "cmfc" for CMAF)
├── moov (movie)
│   ├── mvhd (movie header)
│   ├── trak (track) - 비디오
│   │   ├── tkhd (track header)
│   │   ├── mdia
│   │   │   ├── mdhd (media header)
│   │   │   ├── hdlr (handler = "vide")
│   │   │   └── minf
│   │   │       └── stbl
│   │   │           └── stsd
│   │   │               └── avc1 (H.264 sample entry)
│   │   │                   └── avcC (H.264 codec config)
│   │   └── ...
│   └── mvex (movie extends, fragmented 신호)
└── ...
```

`avcC` 안에 H.264 SPS/PPS가 들어있다. 이게 없으면 디코더 초기화 불가.

`mvex`가 있어야 fragmented 파일임을 표시.

---

### CMAF 호환 ftyp

CMAF init.mp4의 ftyp는 특정 브랜드.

```
ftyp:
  major_brand: "cmfc"  (CMAF compatible)
  compatible_brands: ["cmfc", "iso6", "dash"]
```

`cmfc` 브랜드가 있으면 CMAF 호환. 플레이어가 이걸 보고 처리 방식 결정.

```bash
# ftyp 확인
ffprobe -show_format input.mp4 | grep major_brand
```

---

### .m4s 미디어 세그먼트 박스 구조

각 .m4s 파일.

```
seg_1.m4s
├── styp (segment type)
│   └── compatible_brands: "msdh", "msix"
├── sidx (segment index, 선택)
│   └── 세그먼트 내 위치/시간 정보
├── moof (movie fragment)
│   ├── mfhd (movie fragment header, sequence number)
│   └── traf (track fragment)
│       ├── tfhd (track fragment header)
│       ├── tfdt (track fragment decode time)
│       └── trun (track fragment run, 각 샘플 정보)
└── mdat (media data, 실제 H.264 NAL units)
```

`moof`가 메타데이터, `mdat`이 실제 데이터.

---

### tfdt — 시간 정보의 핵심

`tfdt` (Track Fragment Decode Time)가 이 세그먼트의 시작 시간 표시.

```
tfdt:
  baseMediaDecodeTime: 7,200,000  (90000 timescale = 80초)
```

플레이어가 이걸 보고 "이 세그먼트는 80초 시점부터 시작" 인식.

세그먼트 간 연결, 시킹, ABR 전환에 필수.

---

### trun — 샘플별 정보

각 프레임(샘플)의 크기와 시간.

```
trun:
  sample_count: 144  (4초 @ 36fps)
  sample_durations: [2500, 2500, 2500, ...]
  sample_sizes: [12345, 6789, 3456, ...]
  sample_flags: [...]
```

플레이어가 mdat 안의 NAL unit을 trun 정보로 잘라서 디코더에 넘긴다.

---

### mdat — 실제 미디어

H.264 NAL unit이 들어가는 곳.

```
mdat:
  [H.264 NAL units]
  - SPS / PPS (init.mp4에도 있지만 일부)
  - IDR (키프레임)
  - P-frame
  - P-frame
  - ...
```

각 NAL unit의 시작/크기는 trun이 알려줌.

---

### timescale — 시간 단위

박스 안의 시간이 timescale 단위.

```
timescale: 90000 → 1초 = 90000

예:
sample_duration: 3000 → 3000/90000 = 1/30초 (30fps)
sample_duration: 1500 → 1500/90000 = 1/60초 (60fps)
baseMediaDecodeTime: 7200000 → 7200000/90000 = 80초
```

비디오 timescale은 보통 90000 (MPEG 표준). 오디오는 샘플레이트 (48000 등).

서로 다른 timescale을 정확히 동기화하는 게 트랜스코더의 일.

---

### 박스 분석 도구

박스 구조를 봐야 할 때.

```bash
# MP4Box (GPAC)
MP4Box -info input.mp4

# bento4
mp4dump input.mp4
mp4info input.mp4

# ffprobe
ffprobe -show_format -show_streams -of json input.mp4

# 박스 트리 시각화
mp4dump --verbose input.mp4 | head -100
```

mp4dump 출력 예시:
```
[ftyp] size=24
  major_brand = cmfc
  minor_version = 0
[moov] size=2048
  [mvhd] size=108
  [mvex] size=32
  [trak] size=908
    [tkhd] size=92
    [mdia] size=808
      ...
```

라이브 인프라 디버깅할 때 자주 쓴다. "세그먼트가 이상한데?" → mp4dump로 박스 확인.

---

### 깨진 세그먼트 케이스 — 실전

자주 만나는 문제.

**1. moof 없는데 mdat만**
```
styp + mdat
```
moof가 빠진 .m4s. 디코더가 시간 정보 없어 재생 불가.

원인: 인코더 버그 또는 컨테이너 변환 실수.

**2. tfdt 시간 점프**
```
seg_1.m4s: baseMediaDecodeTime=0
seg_2.m4s: baseMediaDecodeTime=540000 (6초)
seg_3.m4s: baseMediaDecodeTime=600000 (1초만 점프) ← 비정상
```

라이브에서 인코더 재시작 시 발생. discontinuity 마커 안 박으면 플레이어가 혼란.

**3. init.mp4와 .m4s의 코덱 정보 불일치**
init.mp4: H.264 profile=high
seg_1.m4s 안의 NAL: H.264 profile=main

디코더 초기화 실패. 인코더 도중 설정 변경하면 발생.

---

### CMAF Constraint — 표준 준수 사항

CMAF로 인정받으려면 지켜야 할 규칙.

```
[CMAF Constraint 주요 항목]
1. 모든 세그먼트가 키프레임(IDR)으로 시작
2. moof + mdat 박스 순서
3. mfhd의 sequence_number 순차 증가
4. tfdt baseMediaDecodeTime 정확
5. timescale 일관성
6. init.mp4의 mvex 박스 필수
7. .m4s의 styp 박스에 'msdh' 또는 'msix' 브랜드
```

이걸 자동 보장하는 게 Shaka Packager. 직접 fMP4 만들면 실수하기 쉬움.

---

### ffprobe로 fMP4 분석

```bash
# 전체 정보
ffprobe -v error -show_format -show_streams \
  -show_packets -show_frames \
  -of json input.m4s

# 프레임별 PTS
ffprobe -select_streams v -show_entries packet=pts_time,size,flags \
  -of csv input.m4s
```

출력:
```
packet,0.000000,12345,K_   ← K_는 키프레임
packet,0.033333,6789,__
packet,0.066667,3456,__
...
```

각 프레임의 PTS와 크기, 키프레임 여부 확인.

라이브 인코딩 결과 검증할 때 자주 쓴다.

---

### fMP4 만들기 — FFmpeg 명령

```bash
ffmpeg -i input.mp4 \
  -c:v libx264 -c:a aac \
  -movflags +frag_keyframe+empty_moov+default_base_moof \
  -f mp4 output.m4s
```

핵심 옵션:
- `frag_keyframe`: 키프레임마다 새 fragment
- `empty_moov`: 처음 moov 비우기 (init만 별도 분리 가능)
- `default_base_moof`: tfdt 사용 (CMAF 권장)

---

### init.mp4 + .m4s 분리 생성

```bash
ffmpeg -i input.mp4 \
  -c:v libx264 -c:a aac \
  -movflags +frag_keyframe+separate_moof+empty_moov \
  -f hls \
  -hls_segment_type fmp4 \
  -hls_fmp4_init_filename init.mp4 \
  -hls_segment_filename 'seg_%05d.m4s' \
  playlist.m3u8
```

결과:
```
init.mp4
seg_00001.m4s
seg_00002.m4s
...
playlist.m3u8
```

플레이어가 init.mp4를 먼저 받고, 이후 .m4s를 받음.

### MP4 vs MPEG-TS 비교

같은 비디오를 두 컨테이너에 담았을 때.

```
[오버헤드]
MPEG-TS: 188바이트 패킷 헤더가 4바이트 = 2% 오버헤드
fMP4:    moof만 박스당 ~100바이트 = 0.5% 오버헤드

[시킹]
MPEG-TS: 패킷 단위 파싱 필요
fMP4:    sidx로 즉시

[메타데이터]
MPEG-TS: 단순 (PID 기반)
fMP4:    풍부 (코덱 정보, 메타데이터)

[오류 복구]
MPEG-TS: 패킷 단위 복구 가능 (방송 표준)
fMP4:    박스 손상 시 fragment 전체 못 씀
```

라이브 송출(인제스트)은 MPEG-TS, CDN/시청자(배포)는 fMP4. 각 단계 맞는 컨테이너.

"fMP4 박스 구조 알아두면 어디서 도움 되나요?" 질문.

```
"라이브 인프라 디버깅에서 직접적입니다.

자주 만나는 문제:
1. 세그먼트가 이상한데 어디서 깨졌는지 → mp4dump로 박스 확인
2. tfdt 시간 점프 → discontinuity 처리 필요
3. init.mp4와 .m4s 불일치 → 코덱 설정 변경 추적

CMAF Constraint 준수 검증:
- 모든 세그먼트가 IDR로 시작? trun 분석
- timescale 일관성? mdhd 확인

직접 fMP4 만들 일은 거의 없고 Shaka Packager가 하지만,
문제 생기면 박스 구조 알아야 디버깅 가능합니다.

ffprobe + mp4dump가 일상 도구입니다."
```

---

### 2강 핵심 3줄

1. fMP4는 init.mp4 (moov) + 여러 .m4s (moof + mdat) 구조 — 각 .m4s가 독립 디코딩 가능
2. tfdt가 세그먼트 시작 시간, trun이 각 샘플 정보 — CMAF Constraint는 키프레임 시작 + 순차 시퀀스 필수
3. 라이브 디버깅에 박스 분석 (mp4dump, ffprobe) 필수 — 깨진 세그먼트, PTS 점프, 코덱 불일치 추적
