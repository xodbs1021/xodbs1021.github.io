## 레벨 4-4 / 4강: CMAF 실전 — DRM, 광고 삽입, 운영

---

### CENC — Common Encryption

CMAF의 강점인 통합 DRM.

```
[CENC 표준]
ISO/IEC 23001-7
하나의 암호화 방식을 여러 DRM이 공유
HLS (FairPlay), DASH (Widevine, PlayReady) 모두 같은 .m4s 사용
```

이전엔 DRM별로 따로 암호화. CENC로 통일.

---

### 두 가지 암호화 모드 — CTR vs CBCS

CENC가 정의한 두 모드.

```
[AES-CTR]
- Counter mode
- 표준 모드
- 빠름
- DASH (Widevine, PlayReady) 표준

[AES-CBCS]
- CBC mode with subsample
- Apple FairPlay 표준
- 약간 무거움
- iOS/Safari 필수
```

플랫폼이 양쪽 다 지원하려면 두 번 암호화. 또는 둘 다 지원하는 모드 선택.

```
[현실적 선택]
- DASH 전용: CTR
- HLS 전용: CBCS
- 통합 (Multi-DRM): 콘텐츠를 두 번 암호화하거나
                    또는 모두 CBCS로 (Widevine도 CBCS 지원)
```

최근 Widevine이 CBCS 지원해서 단일 암호화로 가는 추세.

---

### 키 시스템 — DRM 벤더별

```
[주요 DRM]
Widevine: Google
  - Android, Chrome, ChromeOS, 스마트TV
  - 3단계 보안 레벨 (L1, L2, L3)

FairPlay: Apple
  - iOS, macOS Safari, Apple TV
  - 모바일 시청에 필수

PlayReady: Microsoft
  - Xbox, Windows, Edge, 일부 스마트TV
  - 점차 비중 감소
```

OTT는 보통 셋 다 지원 (Multi-DRM).

---

### Shaka Packager로 암호화

DRM 통합 패키징.

```bash
packager \
  'in=video.mp4,stream=video,init_segment=v_init.mp4,segment_template=v_seg_$Number$.m4s,drm_label=video' \
  'in=video.mp4,stream=audio,init_segment=a_init.mp4,segment_template=a_seg_$Number$.m4s,drm_label=audio' \
  \
  --enable_widevine_encryption \
  --key_server_url https://widevine-license-server/v1/key \
  --content_id 1234567890abcdef \
  --signer "my_signer" \
  --aes_signing_key abc123... \
  --aes_signing_iv def456... \
  \
  --enable_playready_encryption \
  --playready_server_url https://playready-license-server/v1/key \
  \
  --enable_fairplay_encryption \
  --fairplay_key_uri skd://fairplay-server/key \
  \
  --hls_master_playlist_output master.m3u8 \
  --mpd_output manifest.mpd
```

한 명령으로 세 DRM 모두 처리. 같은 .m4s 파일이 여러 DRM으로 보호.

### 라이센스 서버 흐름

DRM의 실제 동작.

```
[시청자가 영상 클릭]
1. 매니페스트 받음 (master.m3u8 or .mpd)
2. 매니페스트에 KID (Key ID) 포함
3. .m4s 받기 시작 → 암호화돼 있어 디코딩 불가
4. 플레이어가 라이센스 서버에 요청:
   - KID
   - 디바이스 정보
   - 사용자 토큰
5. 라이센스 서버 응답:
   - 키 (디바이스에 맞게 암호화)
   - 권한 (재생 가능 횟수, 출력 제한 등)
6. 플레이어가 키로 .m4s 복호화 → 재생
```

라이센스 서버가 콘텐츠 보호의 핵심. 자체 운영 또는 SaaS (BuyDRM, Axinom).

### 광고 삽입 — SCTE-35 마커

라이브에서 광고 끼우기.

```
[SCTE-35]
방송 표준 광고 마커
"여기서 광고 시작" "여기서 광고 끝" 신호
HLS 매니페스트에 #EXT-X-CUE 태그로 표현
```

매니페스트 예시:
```
#EXTINF:6.0,
seg_42.m4s
#EXT-X-CUE-OUT:DURATION=30
#EXTINF:6.0,
seg_43.m4s
#EXTINF:6.0,
seg_44.m4s
...
#EXT-X-CUE-IN
#EXTINF:6.0,
seg_48.m4s
```

`CUE-OUT`: 광고 시작 신호. 플레이어가 광고 매니페스트로 전환.
`CUE-IN`: 광고 끝, 본 콘텐츠 복귀.

---

### SSAI vs CSAI

광고 삽입 방식 두 가지.

**CSAI (Client-Side Ad Insertion)**:
```
[기존 방식]
플레이어가 광고 시점에 광고 영상 별도 요청
광고 차단기로 회피 가능
지연 발생 (광고 영상 다운로드)
```

**SSAI (Server-Side Ad Insertion)**:
```
[현대 방식]
서버가 본 영상에 광고를 미리 끼워서 한 스트림으로 제공
광고 차단 불가 (구분 안 됨)
원활한 전환
```

SSAI가 표준이 됨. YouTube, Hulu, Disney+ 다 SSAI.

---

### SSAI 구현 - 동적 매니페스트

각 시청자마다 다른 광고를 끼우려면 매니페스트가 동적이어야.

```
[일반]
master.m3u8 - 모든 시청자 동일

[SSAI]
master.m3u8?user_id=12345 - 시청자별 다른 매니페스트
- 시청자 12345의 광고 끼움
- 시청자 67890은 다른 광고
```

각 매니페스트가 다르니 CDN 캐시 효율 떨어짐. SSAI 서버가 매니페스트 생성.

광고 세그먼트는 공유 (CDN 캐시 효율).

---

### 라이브 광고 - 인기 스트리머 진입 시점

```
[라이브 광고 트리거]
1. 스트리머가 광고 명령 (OBS 단축키)
2. 시스템이 SCTE-35 신호 인제스트에 삽입
3. 트랜스코딩이 마커 보존
4. 패키저가 매니페스트에 CUE-OUT 박음
5. 시청자 플레이어가 광고로 전환
6. 30초 후 CUE-IN, 라이브 복귀
```

광고 동안 라이브 영상은 어떻게? 보통 30초 지연 버퍼링하면서 광고 끝나면 catch-up.

또는 광고 시간만큼 라이브가 뒤로 처짐 (그 후 라이브 엣지로 점프).

### 광고 분석 - 비콘

각 광고 시청 데이터 수집.

```
[광고 비콘 이벤트]
- impression: 광고 시작
- 25%: 1/4 시청
- 50%: 절반
- 75%: 3/4
- complete: 끝
- skip: 건너뛰기
- click: 클릭
```

각 이벤트마다 광고 서버에 비콘 전송. VAST/VPAID 표준.

광고 매출의 핵심 데이터.

---

### CMAF + 워터마킹

콘텐츠 보호의 또 다른 층.

```
[워터마킹]
영상에 시청자 식별 정보 은닉
- A/B 워터마크: 두 버전 미세 차이로 시청자 식별
- 디지털 fingerprint: 콘텐츠 추출 가능
- 시각 워터마크: 화면에 표시 (사용자 ID, IP)
```

콘텐츠 유출 시 누구한테서 나왔는지 추적.

```bash
# 시각 워터마크 (사용자 ID 표시)
ffmpeg -i video.mp4 \
  -vf "drawtext=text='ID\: 12345':x=W-tw-20:y=H-th-20:fontsize=14:fontcolor=white@0.3" \
  -c:v libx264 watermarked.mp4
```

OTT 무료 체험판이나 유출 위험 콘텐츠에 적용.

### 운영 - 매니페스트 관리

CMAF 라이브의 매니페스트는 매초 변함.

```
[매니페스트 라이프사이클]
t=0:    초기 매니페스트 생성
t=6초:  새 세그먼트 추가, 매니페스트 업데이트
t=12초: 또 업데이트
...
t=180초: 30분 후 사용자 시청 시작
        → 그동안 변한 매니페스트
        → 새로 받음
```

매니페스트가 자주 변하니 CDN 캐시 TTL 짧게 (1~2초).

```
Cache-Control: max-age=2
```

세그먼트(.m4s)는 길게 (한 번 만들어지면 안 변함).

```
Cache-Control: max-age=86400
```

### 라이브 → VOD 자동 변환

방송 종료 후 다시보기.

```
[자동 변환 흐름]
1. 라이브 방송 중: event 타입 매니페스트, ENDLIST 없음
2. 방송 종료 (스트리머가 OBS 중지)
3. 시스템이 ENDLIST 태그 추가
4. 매니페스트가 VOD가 됨
5. 시청자는 다시보기로 처음부터 볼 수 있음
```

세그먼트는 그대로 보존. 매니페스트만 수정.

VOD 변환 후 추가 트랜스코딩 (썸네일, 모바일용 별도 ladder) 작업.

---

### CMAF Quality Check - QC 파이프라인

라이브 인프라의 QC.

```
[QC 파이프라인]
1. 매분 무작위 스트림 샘플링
2. 매니페스트 파싱 검증
   - EXT-X-MAP 존재
   - 세그먼트 순서
   - tfdt 일관성
3. 세그먼트 박스 구조 검증
   - moof + mdat 쌍
   - CMAF Constraint 준수
4. 디코딩 테스트
   - 처음부터 5초 디코딩 성공?
5. VMAF 측정
6. 광고 마커 일관성
```

매분 수십~수백 스트림 자동 검증. 이상 발견 즉시 알람.

### CMAF 운영 - 흔한 장애

**1. init.mp4 변경 누락**
인코딩 설정 변경 시 init.mp4 새로 만들었는데 매니페스트는 이전 init 가리킴. 디코딩 실패.

```
[해결]
init.mp4도 버전 관리: init_v1.mp4, init_v2.mp4
매니페스트의 EXT-X-MAP URI 함께 업데이트
```

**2. 세그먼트 누락**
seg_42 다음에 seg_44 (43 누락). 라이브에서 트랜스코딩 노드 hang.

```
[해결]
discontinuity 마커 추가
#EXT-X-DISCONTINUITY
플레이어가 끊김 감지하고 catch-up
```

**3. PTS 점프**
인코딩 재시작으로 tfdt가 0에서 다시 시작.

```
[해결]
baseMediaDecodeTime을 이전 끝부분 + 1로 강제
또는 discontinuity 마커 + 새 매니페스트 시퀀스
```

### 면접 답변용

"CMAF 라이브 운영 어떻게 하시겠어요?" 질문.

```
"CMAF 인프라의 운영 포인트는 매니페스트와 세그먼트의 일관성입니다.

핵심 운영:
1. CDN 캐시 TTL 분리
   - 매니페스트 1~2초 (자주 변함)
   - 세그먼트 24시간 (변하지 않음)

2. QC 파이프라인 (매분 샘플링)
   - 매니페스트 파싱 + 박스 구조 검증
   - 5초 디코딩 테스트
   - VMAF 회귀 감지

3. DRM 통합 (Shaka로 Widevine + PlayReady + FairPlay)
   CENC로 같은 .m4s에 다중 DRM

4. SSAI 광고 삽입
   SCTE-35 마커 → 매니페스트 CUE-OUT/IN
   세그먼트는 공유로 CDN 캐시 효율

장애 대응:
- init.mp4 변경은 버전 관리로 추적
- 세그먼트 누락은 discontinuity 마커
- PTS 점프는 매니페스트 시퀀스 강제 리셋

라이브 → VOD 자동 변환은 ENDLIST 태그만 추가하는 가벼운 작업으로 유지했습니다."
```

---

### 4강 핵심 3줄

1. CENC로 한 .m4s에 Widevine + PlayReady + FairPlay 통합 — Shaka Packager가 multi-DRM 표준 도구
2. SSAI는 매니페스트 동적 생성으로 시청자별 광고, SCTE-35 마커로 라이브 광고 트리거
3. CMAF 운영의 핵심은 매니페스트(짧은 TTL)와 세그먼트(긴 TTL) 분리, 매분 QC 파이프라인으로 자동 검증
