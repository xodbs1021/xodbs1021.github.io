## 레벨 2-6 / 1강: WebRTC가 왜 만들어졌나 — 브라우저에서 실시간 통신

---

### 2010년대 초 — 브라우저의 한계

웹은 본질적으로 요청-응답 모델이다. 클라이언트가 요청하면 서버가 응답. 단방향.

```
[전통 HTTP]
브라우저 → 서버: "데이터 줘"
서버 → 브라우저: "여기"
연결 종료
```

실시간 양방향 통신이 안 됐다. 화상 채팅, 음성 통화, 실시간 게임 같은 게 브라우저에선 불가능.

이런 기능을 쓰려면 Flash, Java Applet, ActiveX 같은 플러그인이 필요했다. 보안 취약, 설치 번거로움, 모바일 미지원.

### Skype의 독점 — 2010년대 초 화상 통화

화상 통화는 Skype가 사실상 독점. 자체 P2P 프로토콜 사용.

```
[Skype 모델]

- 자체 계정 생성
- 자체 프로토콜 (비공개)

```

웹에 통화 기능을 넣고 싶어도 Skype API는 제한적이고 비싸다.

대체 솔루션:
- Adobe Flash: 양방향 가능하지만 Flash 자체가 사라지는 중
- Java: 무거움, 보안 문제
- 자체 클라이언트: 사용자가 안 깔음

### WebRTC가 뭔가 — 정의

**WebRTC (Web Real-Time Communication)**: 브라우저나 모바일 앱에서 P2P로 음성/영상/데이터를 실시간 교환하는 표준 기술.

핵심 특징:
- **P2P 우선**: 가능하면 클라이언트끼리 직접 통신
- **저지연**: 200ms 이하 목표
- **암호화 기본**: DTLS-SRTP 필수
- **표준 API**: 브라우저 JavaScript API
- **무료/오픈소스**: 누구나 사용

---

### WebRTC의 3가지 핵심 API

브라우저에 추가된 API.

**1. MediaStream (getUserMedia)**
카메라/마이크 접근.

```javascript
const stream = await navigator.mediaDevices.getUserMedia({
  video: true,
  audio: true
});

document.querySelector('video').srcObject = stream;
```

**2. RTCPeerConnection**
P2P 연결 관리.

```javascript
const pc = new RTCPeerConnection({
  iceServers: [{ urls: 'stun:stun.l.google.com:19302' }]
});

// 로컬 미디어 추가
stream.getTracks().forEach(track => pc.addTrack(track, stream));

// 원격 미디어 수신
pc.ontrack = (event) => {
  remoteVideo.srcObject = event.streams[0];
};
```

**3. RTCDataChannel**
임의의 데이터 전송. 게임 상태, 채팅 메시지 등.

```javascript
const dataChannel = pc.createDataChannel('chat');

dataChannel.onopen = () => {
  dataChannel.send('Hello!');
};

dataChannel.onmessage = (event) => {
  console.log('Received:', event.data);
};
```

### WebRTC가 사용하는 프로토콜 스택

WebRTC는 새 프로토콜이 아니다. 기존 표준 프로토콜의 묶음.

```
[WebRTC 프로토콜 스택]
┌────────────────────────────────────────┐
│  WebRTC API (JS)                       │
├────────────────────────────────────────┤
│  영상       │ 오디오    │ 데이터       │
│  VP8/H.264  │ Opus/G.711│ SCTP         │
├────────────────────────────────────────┤
│  SRTP (암호화 RTP)                     │
├────────────────────────────────────────┤
│  DTLS (UDP 위 TLS)                     │
├────────────────────────────────────────┤
│  ICE/STUN/TURN (NAT 통과)              │
├────────────────────────────────────────┤
│  UDP (주) / TCP (폴백)                 │
├────────────────────────────────────────┤
│  IP                                    │
└────────────────────────────────────────┘
```

각 계층이 표준 RFC 프로토콜. WebRTC는 이걸 묶어서 브라우저에 통합.

### P2P가 왜 어려운가 — NAT 문제

인터넷에는 사설 IP와 공용 IP가 있다.

```
[가정 네트워크]
공용 IP: 1.2.3.4 (KT가 할당)
공유기:  공용 IP를 가짐

내부 기기들:
- 노트북: 192.168.0.10
- 폰: 192.168.0.11
- TV: 192.168.0.12

외부에서 "192.168.0.10"이라고 해도 못 찾음
```

내부 기기가 외부로 연결할 때는 NAT가 변환:
```
노트북 → 외부:
  src: 192.168.0.10:54321
       ↓ NAT 변환
  src: 1.2.3.4:60001
```

이런 상황에서 외부에서 노트북으로 직접 연결하려면? 노트북 주소를 모른다.

이게 P2P의 가장 큰 적이다. **양쪽 모두 NAT 뒤에 있으면 어떻게 연결하나?**

---

### ICE — NAT 통과의 표준 답

**ICE (Interactive Connectivity Establishment)**: 양 P2P가 가능한 모든 연결 경로를 시도해서 성공하는 걸 사용.

```
[ICE 흐름]
1. 각자 자기 IP 후보를 수집
   - 자기 사설 IP (집 와이파이 내부)
   - NAT 매핑된 공용 IP (STUN으로 알아냄)
   - 릴레이 서버 IP (TURN 폴백)

2. 후보 목록을 교환 (시그널링 서버 경유)

3. 모든 후보 쌍에 대해 연결 시도

4. 성공한 경로로 P2P 통신
```

복잡하지만 효과적. 95% 이상의 환경에서 P2P 가능.

---

### STUN — 자기 공용 IP 알아내기

**STUN (Session Traversal Utilities for NAT)**: NAT 뒤 클라이언트가 자기 공용 IP를 알아내는 프로토콜.

```
[STUN 흐름]
클라이언트 (192.168.0.10) → STUN 서버: "내 공용 IP가 뭐야?"
                                 ↓ NAT 변환
                            src: 1.2.3.4:60001
                                 ↓
                            STUN 서버: "당신은 1.2.3.4:60001 로 보임"
클라이언트 ← STUN 서버: "1.2.3.4:60001"
```

STUN 서버는 단순한 echo. 받은 패킷의 src IP를 응답으로 알려줌.

Google이 공개 STUN 서버 운영:
```
stun:stun.l.google.com:19302
stun:stun1.l.google.com:19302
```

---

### TURN — STUN으로 안 될 때 릴레이

**TURN (Traversal Using Relays around NAT)**: P2P가 불가능한 환경에서 중계 서버로 우회.

```
[Symmetric NAT 환경 - P2P 불가능]
일부 NAT는 같은 내부 주소라도 외부 주소에 따라 다른 매핑.
A → STUN: "1.2.3.4:60001"
A → B(직접): "1.2.3.4:60002"  ← 다른 포트
→ B가 60001로 연결 시도해도 실패

해결: TURN 서버 거침
A ──RTP──→ TURN ──RTP──→ B
       (릴레이)
```

TURN은 대역폭 비용이 크다. 모든 미디어가 서버를 거치니까.

상용 서비스에서 TURN 서버는 비용의 큰 부분.

### 시그널링 — WebRTC에 빠진 부분

WebRTC 표준에 **시그널링은 없다**.

왜? 시그널링은 회사마다 다르니까. WebSocket, Socket.IO, MQTT, 자체 프로토콜 등.

```
[시그널링이 하는 일]
- Peer 식별 (누구와 연결할지)
- SDP 교환 (코덱, 미디어 정보)
- ICE 후보 교환 (어떻게 연결할지)
- 연결 상태 추적
```

WebRTC 앱 만들 때 시그널링 서버를 직접 구축해야 한다.

```javascript
// 가장 단순한 WebSocket 시그널링
const ws = new WebSocket('wss://signaling.example.com');

ws.send(JSON.stringify({
  type: 'offer',
  sdp: localDescription,
  to: 'peer_user_id'
}));

ws.onmessage = (event) => {
  const message = JSON.parse(event.data);
  if (message.type === 'answer') {
    pc.setRemoteDescription(message.sdp);
  } else if (message.type === 'ice_candidate') {
    pc.addIceCandidate(message.candidate);
  }
};
```

### WebRTC의 미디어 — SRTP

P2P 연결이 수립되면 미디어는 **SRTP (Secure RTP)** 로 흐른다.

```
[SRTP 패킷 구조]
RTP 헤더 + 암호화된 페이로드 + 인증 태그
```

특징:
- **RTP 기반**: 실시간 전송 표준
- **암호화**: AES-128 기본
- **인증**: HMAC-SHA1
- **키 교환**: DTLS 핸드셰이크 활용

DTLS-SRTP가 표준. 키 교환은 DTLS, 미디어 전송은 SRTP.

### WebRTC vs 다른 프로토콜 비교

저지연 라이브 영상 전송 비교.

| | HLS | LL-HLS | SRT | WebRTC |
|---|---|---|---|---|
| 지연 | 25초 | 3초 | 0.5-1초 | 0.2초 |
| 모델 | Pull (HTTP) | Pull (HTTP) | Push | P2P |
| CDN | ✅ | ✅ | ❌ | ❌ |
| 대규모 시청자 | 100만+ | 100만+ | 어려움 | 수십~수백 |
| NAT 통과 | 자연스러움 | 자연스러움 | Rendezvous | ICE |
| 브라우저 지원 | 라이브러리 | 라이브러리 | ❌ | 네이티브 |

WebRTC의 강점: 압도적 저지연, 브라우저 네이티브.
WebRTC의 약점: 확장성 (P2P라 100만 명 못 함).

### 어디에 쓰이나

WebRTC가 활용되는 영역.

**1. 화상 통화**
Google Meet, Zoom (일부), Microsoft Teams, 디스코드.

**2. 실시간 게임**
양방향 데이터 채널로 게임 상태 동기화.

**3. 라이브 인터랙션**
시청자 화면 공유, 라이브 통화 방송.

**4. 파일 전송**
RTCDataChannel로 P2P 파일 전송. WebTorrent.

**5. IoT 모니터링**
CCTV에 WebRTC 내장해서 즉시 시청.

라이브 방송 시청자 측엔 거의 안 쓰임. CDN 못 쓰니까.

### 1강 핵심 3줄

1. WebRTC는 Google이 2011년 GIPS 인수 후 오픈소스화한 브라우저 실시간 통신 표준 — MediaStream, RTCPeerConnection, RTCDataChannel 3가지 API
2. P2P 우선이지만 NAT 통과를 위해 ICE(STUN+TURN) 사용 — STUN은 공용 IP 확인, TURN은 릴레이 폴백
3. 시그널링은 표준에 없어 직접 구축 (WebSocket 등) — 미디어는 SRTP, 키 교환은 DTLS, 지연 200ms로 화상통화에 적합
