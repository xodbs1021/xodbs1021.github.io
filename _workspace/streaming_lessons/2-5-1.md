## 레벨 2-5 / 1강: SRT의 탄생 — 인터넷에서 방송 품질을 전송하는 문제

---

### 2010년대 초 — 방송 업계의 고민

전통 TV 방송은 위성, 광케이블, 마이크로파 같은 전용 회선을 썼다. 안정적이지만 매우 비싸다.

```
[전통 방송 백홀 비용]
서울 스튜디오 → 방송국:
  - 전용 광케이블: 월 수천만 원
  - 위성 회선: 월 수억 원
  - 안정성: 99.999%
```

방송국, 케이블TV, IPTV는 이 비용을 매년 부담했다.

2010년대 들어 인터넷 회선이 빨라지면서 의문이 생겼다.

**"그냥 인터넷으로 보내면 안 되나?"**

문제는 인터넷이 방송 품질을 못 따라간다는 것.

```
[인터넷 회선의 문제]
- 패킷 손실: 0.1~3% (방송은 0.0001% 이하 요구)
- 지터(jitter): 가변적
- 대역폭 변동: 시간대별 차이
- 경로 변경: BGP 라우팅 변동
```

기존 프로토콜로는 안 됐다.

**TCP**: 신뢰성은 좋은데 재전송이 무한정 기다림. 라이브 영상에 부적합.
**UDP**: 빠른데 손실 복구가 없음. 화면 깨짐.
**RTMP**: TCP 기반. 지연 큼. 게다가 이 시점엔 Flash 죽어가는 중.

새로운 프로토콜이 필요했다.

---

### Haivision의 등장 — 2013년 SRT 개발 시작

캐나다 회사 **Haivision**이 문제를 풀려고 나섰다. 방송 장비를 만드는 회사.

목표:
- **인터넷에서 방송 품질 전송**
- **낮은 지연 (1초 미만)**
- **패킷 손실 복구**
- **방화벽 통과 쉬움**
- **암호화 기본 내장**

2013년부터 개발 시작. 2017년 오픈소스로 공개. **Secure Reliable Transport (SRT)** 의 시작.

---

### SRT의 핵심 설계 — UDP 위에 신뢰성 추가

SRT의 가장 큰 결정은 **UDP 기반**.

```
[프로토콜 스택]
┌──────────────────┐
│  Application     │  영상 데이터
│  (MPEG-TS 등)    │
├──────────────────┤
│  SRT             │  신뢰성, 지연 관리, 암호화
├──────────────────┤
│  UDP             │  IP 위 가장 단순한 전송
├──────────────────┤
│  IP              │
└──────────────────┘
```

UDP를 선택한 이유:

**1. TCP의 HoL Blocking 회피**
TCP는 패킷 하나 손실되면 뒤 패킷 전부 대기. 라이브 영상에서 이게 치명적.

**2. 빠른 시작**
TCP는 핸드셰이크 3-way + slow start. SRT는 빠르게 시작.

**3. 직접 제어**
재전송 타임아웃, 윈도우 크기 등을 직접 결정. TCP의 표준 정책 무시.

UDP는 신뢰성이 없다. 그걸 SRT가 위에 얹는다.

---

### ARQ — SRT의 핵심 신뢰성 메커니즘

**ARQ (Automatic Repeat reQuest)**: 패킷 손실 시 재전송 요청.

```
[정상 흐름]
송신: 패킷 1 → 수신: ACK 1
송신: 패킷 2 → 수신: ACK 2
송신: 패킷 3 → 수신: ACK 3

[손실 발생]
송신: 패킷 4 → 수신: 받음
송신: 패킷 5 → [손실됨]
송신: 패킷 6 → 수신: 받음
                      ↓
                수신: "5번이 없네!" → NACK 5 전송
                      ↓
송신: 패킷 5 재전송 → 수신: 받음
```

TCP의 ACK + 재전송과 비슷하지만, SRT는 **NACK 기반**이다.

- TCP: 받은 패킷에 ACK. 안 오면 송신측이 타임아웃 후 재전송.
- SRT: 빠진 패킷을 수신측이 발견하고 NACK 전송. 즉시 재전송.

NACK 방식이 더 빠르다. 손실 감지 후 재전송 요청까지 RTT 1번.

---

### Latency 모드 — SRT의 핵심 트레이드오프

SRT가 일반 ARQ와 다른 점은 **타임아웃을 짧게 잡는다**는 거다.

TCP는 패킷이 늦게 오더라도 무한정 기다린다. 결국 도착하면 처리. 영상에선 이게 문제.

```
[TCP 방식 - 잘못된 흐름]
12:00:00 패킷 1: 정상 도착
12:00:00 패킷 2: 손실
12:00:00 패킷 3: 정상 도착 → 대기
12:00:00 패킷 4: 정상 도착 → 대기
12:00:02 패킷 2: 재전송 도착 → 패킷 3, 4 풀어줌
        ↑
   2초 지연. 시청자 화면 멈춤
```

SRT는 미리 정한 시간(latency) 안에 안 오면 그냥 버린다.

```
[SRT 방식 - latency 500ms 설정]
12:00:00.000 패킷 1: 정상 도착 → 즉시 재생
12:00:00.033 패킷 2: 손실
12:00:00.066 패킷 3: 정상 도착 → 잠깐 대기
12:00:00.100 NACK 전송
12:00:00.200 패킷 2: 재전송 도착 → 패킷 2, 3 풀어줌
        ↑
   재전송 성공. 약 200ms 지연만 발생

[다른 시나리오 - latency 안에 못 옴]
12:00:00.000 패킷 1: 정상 도착
12:00:00.033 패킷 2: 손실
12:00:00.500 latency 만료 → 패킷 2 포기, 패킷 3 그냥 재생
        ↑
   화면 약간 깨지지만 지연은 없음
```

**"지연이 더 큰 문제다. 늦게 오면 버린다."**

이게 라이브 스트리밍에 맞는 설계.

---

### Latency 설정 — 얼마로 잡나

SRT의 latency는 **RTT × 4 정도**가 일반적.

```
[서울 → 서울 (RTT 5ms)]
latency = 20~50ms
지연 = 매우 짧음
손실 복구율 = 거의 100%

[서울 → 도쿄 (RTT 30ms)]
latency = 120~200ms
지연 = 짧음
손실 복구율 = 95%

[서울 → 미국 (RTT 150ms)]
latency = 600ms~1초
지연 = 보통
손실 복구율 = 90%
```

먼 거리일수록 latency를 크게 잡아야 재전송 기회가 있다. 대신 지연도 커진다.

---

### SRT의 4가지 모드

SRT는 같은 프로토콜로 여러 모드를 지원한다.

**1. Caller (호출자)**
연결을 시작하는 쪽. TCP의 client에 해당.

**2. Listener (대기자)**
연결을 기다리는 쪽. TCP의 server에 해당.

```
[일반 동작]
스트리머 (Caller) → SRT 연결 → 서버 (Listener)
```

**3. Rendezvous (랑데부)**
양쪽 다 connect. 방화벽 뒤에 있는 두 호스트가 연결할 때 사용.

```
[Rendezvous]
호스트 A (Rendezvous): srt://host_b:9000
호스트 B (Rendezvous): srt://host_a:9000
→ 양쪽이 동시에 연결 시도 → NAT 통과
```

WebRTC의 ICE와 비슷한 NAT 통과 기법.

**4. Bonding**
여러 네트워크 회선을 묶음. 4G + 와이파이 동시 사용.

```
[Bonding - 모바일 방송 차량]
회선 1: KT LTE
회선 2: SKT LTE
회선 3: LGU+ LTE
회선 4: 위성

→ SRT가 패킷을 4개 회선에 분산 + 합침
→ 한 회선 끊겨도 다른 회선이 커버
```

방송 중계차에서 자주 쓰는 기능.

---

### Stream ID — 한 포트에 여러 스트림

SRT는 한 포트로 여러 스트림을 구분할 수 있다.

```
[일반 RTMP]
포트 1935 → 한 스트림만

[SRT]
srt://server:9000?streamid=channel_a
srt://server:9000?streamid=channel_b
srt://server:9000?streamid=channel_c
→ 같은 포트 9000으로 여러 채널 동시 수신
```

서버가 streamid 보고 어느 채널로 라우팅할지 결정.

방송 콘솔처럼 여러 채널 동시 처리할 때 유용.

---

### 암호화 — 기본 내장

SRT는 처음부터 암호화가 포함됐다. **AES-128, AES-192, AES-256** 지원.

```
[연결 URL에 패스프레이즈 포함]
srt://server:9000?passphrase=MySecretPassword

[옵션 설정]
pbkeylen=16 (128비트)
pbkeylen=24 (192비트)
pbkeylen=32 (256비트)
```

연결 핸드셰이크에서 키 교환. 이후 모든 데이터 암호화.

RTMP는 별도로 RTMPS(TLS)를 써야 하지만, SRT는 기본 옵션.

방송국이 SRT를 좋아하는 이유 중 하나. 콘텐츠 보호가 필수.

---

### MPEG-TS와의 결합

SRT는 컨테이너 포맷이 아니다. **무엇을 전송할지**는 위 계층이 결정.

대부분의 경우 MPEG-TS를 SRT 위에 얹는다.

```
[전형적 사용]
스트리머: H.264 영상 + AAC 오디오 → MPEG-TS 포장 → SRT 전송
수신측: SRT 수신 → MPEG-TS 추출 → 디코딩
```

왜 MPEG-TS인가?

**1. 방송 표준**
방송 장비들이 모두 MPEG-TS를 뱉음. 변환 없이 그대로 SRT로 보냄.

**2. 멀티 트랙**
PID 구조로 여러 오디오 트랙. (PokeClip에서 사용한 그 구조)

**3. 손실 복구**
SRT가 못 받은 패킷이 있어도 MPEG-TS의 sync byte로 다음 패킷 찾기 가능.

SRT + MPEG-TS는 사실상 한 묶음.

---

### SRT vs RTMP vs WebRTC 비교

ingest 프로토콜 비교:

| | RTMP | SRT | WebRTC |
|---|---|---|---|
| 기반 | TCP | UDP + ARQ | UDP + RTP |
| 지연 | 2-3초 | 0.5-1초 | 200ms |
| 멀티 오디오 | ❌ | ✅ (MPEG-TS) | ⚠️ (복잡) |
| 암호화 | RTMPS 옵션 | 기본 내장 | 기본 내장 (DTLS) |
| NAT 통과 | 어려움 | Rendezvous 가능 | ICE/STUN |
| OBS 지원 | 기본 | 가능 (별도 설정) | OBS 30+ |
| 방송 장비 | ⚠️ | ✅ | ❌ |
| 인터넷 안정성 | TCP 의존 | 강함 | 강함 |

SRT는 RTMP보다 지연 짧고 멀티 트랙 가능. WebRTC보단 안정적이고 방송 장비 호환.

방송 백홀과 분석용 ingest 사이 영역에 적합.
