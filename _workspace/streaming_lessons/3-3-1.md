## 레벨 3-3 / 1강: H.265가 왜 등장했나 — 4K 시대의 필요

---

### H.264로 4K를 송출해보면

문제 상황부터.

```
[해상도별 H.264 비트레이트]
720p: 3 Mbps
1080p: 6 Mbps
4K (3840×2160): 25~40 Mbps
8K: 80~120 Mbps
```

4K가 1080p의 4배 픽셀. 비트레이트도 4배 이상.

```
[4K 라이브 시청자 100만명]
한 시청자: 40 Mbps
총 트래픽: 40 Tbps
한 달 CDN 비용: 수십~수백억
```

이 비용을 어떻게든 줄여야 한다.

---

### H.265의 등장 — 2013년

같은 화질에 절반 비트레이트 목표.

```
[H.265 압축률]
같은 화질에 H.264 대비 50% 비트레이트 절감

4K H.264: 25 Mbps
4K H.265: 12~15 Mbps
```

대역폭 비용 절반. 대형 OTT에 결정적.

---

### H.265의 이름들

```
정식 명칭:
- H.265 (ITU-T)
- HEVC (High Efficiency Video Coding)
- MPEG-H Part 2 (ISO/IEC 23008-2)
```

H.264 → H.265, AVC → HEVC. 같은 코덱의 다른 이름.

스트리밍 산업에선 보통 HEVC라고 부름.

---

### 어디서 H.265 쓰나 — 실무 현황

```
[H.265 채택 영역]
✅ 4K UHD 블루레이
✅ Netflix 4K 콘텐츠
✅ Apple TV+ 4K
✅ Disney+ 4K
✅ iPhone 카메라 (4K 녹화)
✅ 일부 IPTV 4K 채널

❌ YouTube (VP9/AV1 위주)
❌ 치지직, 트위치 (호환성 우선, H.264)
❌ 광고 기반 무료 서비스 (라이센스 비용)
❌ 저가폰 시청자 (디코딩 못 함)
```

4K VOD가 H.265의 주 영역. 일반 라이브는 여전히 H.264.

---

### H.265의 큰 장벽 — 라이센스

H.264와 가장 큰 차이.

```
[H.264 라이센스]
한 곳: MPEG LA
비교적 단순
비용 예측 가능

[H.265 라이센스]
여러 풀:
- HEVC Advance
- MPEG LA
- Velos Media
- 개별 특허권자

비용:
- 디바이스당 $0.20~0.40
- 콘텐츠당 추가 비용
- 누적 시 매우 큰 비용
```

이게 H.265 보급이 느린 이유.

작은 회사나 무료 서비스는 도입 부담.

이래서 AV1 (무료 대안) 등장.

---

### 인코딩 비용 — H.264 대비

```
[인코딩 시간 - 같은 화질 기준]
H.264 medium: 1x
H.265 medium: 약 10x 느림
H.265 slow: 약 20x 느림
```

H.265는 H.264보다 10배 느림.

라이브 트랜스코딩에 부담. 같은 서버에 동시 채널 수 1/10.

---

### NVENC로 해결 — H.265 GPU 인코딩

NVIDIA Pascal (2016) 이후 H.265 NVENC 지원.

```bash
ffmpeg -i input.mp4 \
  -c:v hevc_nvenc \
  -preset p4 \
  -tune ll \
  -rc cbr \
  -b:v 3000k \
  output.mp4
```

GPU 사용 시 H.264와 비슷한 속도. 라이브도 가능.

```
[같은 서버 동시 1080p 트랜스코딩]
H.264 x264 medium (CPU): 4 채널
H.265 x265 medium (CPU): 0.5 채널
H.265 NVENC (GPU): 50+ 채널
```

GPU 없으면 H.265 라이브 사실상 불가능.

---

### 디코딩 부담 — 시청자 측 문제

인코딩만큼은 아니지만 디코딩도 무거움.

```
[디코딩 CPU 부하 - 1080p60]
H.264: 매우 낮음 (저가폰도 가능)
H.265: 보통 (미드레인지 이상)

[디코딩 - 4K]
H.264: 고가폰만
H.265: 미드레인지 이상 (HW 가속 시)
```

저가폰은 H.265 못 봄. 이게 라이브 스트리밍에서 H.264 유지하는 이유.

---

### 하드웨어 디코더 — 거의 모든 현대 기기

```
[H.265 하드웨어 디코더]
iPhone: 6 이후 (2014)
Samsung Galaxy: S6 이후 (2015)
모든 4K TV
PS5, Xbox Series X/S
최근 5년 내 PC GPU
```

하드웨어 가속 있으면 CPU 부하 거의 없음.

문제는 5년 이상 된 기기.

---

### 코덱 문자열 — MIME 표기

```
[H.265 MIME]
hev1.1.6.L93.B0

hev1 = H.265
1 = profile_space
6 = profile_idc (Main Profile)
L93 = Level 3.1
B0 = compatibility flags
```

브라우저 지원 확인:
```javascript
MediaSource.isTypeSupported('video/mp4; codecs="hev1.1.6.L93.B0"')
```

브라우저 지원 현황 (2024):
- Safari: 처음부터 지원
- Chrome: 2023년 11월부터 (PC)
- Firefox: 아직 미지원
- Edge: 지원

---

### Profile — 사용 시나리오별

```
[H.265 주요 Profile]
Main: 8비트, 4:2:0 (일반 4K)
Main 10: 10비트 (HDR)
Main 4:2:2: 4:2:2 색공간
Main Still: 정지 이미지
```

대부분의 4K 스트리밍은 **Main** 또는 **Main 10**.

HDR은 Main 10 필수.

```bash
# Main 8비트
ffmpeg -i input.mp4 \
  -c:v libx265 -profile:v main -pix_fmt yuv420p \
  output.mp4

# Main 10 (HDR)
ffmpeg -i input.mp4 \
  -c:v libx265 -profile:v main10 -pix_fmt yuv420p10le \
  output.mp4
```

---

### Level — 처리 능력 표시

```
[H.265 주요 Level]
Level 3.0: 720p30
Level 3.1: 720p60, 1080p30
Level 4.0: 1080p60
Level 5.0: 2K
Level 5.1: 4K30
Level 5.2: 4K60
Level 6.0: 8K
```

디코더가 처리 가능한 최대 사양.

스트리밍에서 명시:
```bash
ffmpeg -i input.mp4 \
  -c:v libx265 -profile:v main -level 5.1 \
  output.mp4
```

### Netflix 4K HDR 인코딩 예시

```bash
ffmpeg -i input_master.mov \
  -c:v libx265 \
  -preset slow \
  -profile:v main10 \
  -pix_fmt yuv420p10le \
  -x265-params "colorprim=bt2020:transfer=smpte2084:colormatrix=bt2020nc" \
  -b:v 12000k \
  -c:a eac3 -b:a 384k -ac 6 \
  output_4k_hdr.mp4
```

VOD라 preset slow OK. 화질 우선.

`-x265-params`로 HDR10 메타데이터 설정.

---

### 라이브 H.265 인코딩 예시

```bash
ffmpeg -i input \
  -c:v hevc_nvenc \
  -preset p4 \
  -tune ll \
  -profile:v main \
  -rc cbr \
  -b:v 6000k -maxrate 6000k -bufsize 12000k \
  -g 60 \
  -bf 0 \
  -c:a aac -b:a 128k \
  -f flv rtmp://server/live/key
```

라이브는 GPU 사용 + zerolatency 설정.

---

### H.264 + H.265 다중 트랙 운영

OTT 플랫폼이 자주 하는 방식.

```
[ABR Ladder]
같은 콘텐츠를 H.264와 H.265 둘 다:

H.264 트랙 (호환성):
- 1080p 6 Mbps
- 720p 3 Mbps
- 480p 1.5 Mbps

H.265 트랙 (효율):
- 4K 12 Mbps
- 1080p 3 Mbps
- 720p 1.5 Mbps
```

시청자 디바이스 따라 자동 선택.

H.265 디코딩 가능: H.265 받음 (대역폭 절약).
불가능: H.264 받음.

```
[iPhone 카메라 설정]
```

### 실무 결정 가이드

H.265 도입 여부.

```
[H.265 도입 OK]
✅ 4K VOD 콘텐츠 (Netflix급)
✅ 대규모 OTT (라이센스 협상 가능)
✅ 자체 인프라 + GPU 서버
✅ 시청자 디바이스 신형 위주
✅ 프리미엄 콘텐츠 (수익으로 라이센스 충당)

[H.264 유지 OK]

❌ 무료/광고 모델

```

치지직 같은 한국 게임 방송 → H.264 유지가 맞다.

### 면접 답변용

"왜 H.265 안 쓰세요?" 같은 질문.

```
"라이센스 비용과 시청자 디바이스 호환성 때문입니다.
H.265는 H.264 대비 50% 대역폭 절감하지만:
1. 여러 라이센스 풀에 비용 지불해야 함
2. 5년 이상 된 기기는 디코딩 못 함
3. 일반 라이브 방송은 1080p가 주력이라 H.265 절감 효과 작음

저희 시청자 70%가 모바일이고 그 중 저가폰 비율이 높아
H.264 호환성을 유지하는 게 비즈니스적으로 맞다고 판단했습니다.
4K VOD나 프리미엄 콘텐츠 진입 시 H.265 도입을 검토할 예정입니다."
```

---

### 1강 핵심 3줄

1. H.265 (HEVC)는 2013년 등장, H.264 대비 50% 비트레이트 절감 — 4K VOD (Netflix, Apple TV+, iPhone 카메라)에서 표준
2. 도입 장벽은 복잡한 라이센스 (HEVC Advance + MPEG LA + Velos Media), 인코딩 10배 비용, 저가폰 디코딩 불가 — 일반 라이브 (치지직)는 H.264 유지
3. 실무는 NVENC GPU 인코딩이 필수, H.264 + H.265 다중 트랙으로 호환성과 효율 동시 달성 — iPhone은 4K 녹화에 H.265 강제
