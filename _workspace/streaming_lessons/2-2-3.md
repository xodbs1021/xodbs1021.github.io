## 레벨 2-2 / 3강: HLS 세그먼트 내부 — .ts 파일 안에 뭐가 들었나

---

### .ts 파일을 hexdump로 열어보면

HLS 세그먼트 하나를 바이너리로 열면 이렇게 생겼다.

```
00000000: 4747 0040 1000 b00d 0001 c100 0000 01f0
00000010: 002a b104 b2ff ffff ffff ffff ffff ffff
00000020: 4740 4011 0042 f025 0001 c100 0000 ffff
00000030: f021 0001 f041 f00c 0a04 6b6f 7200 5004
00000040: 656e 6700 0fe0 4ff0 0681 0e00 4e3f 0000
...
```

첫 바이트 `47`. 이게 sync byte다. 1-2강에서 본 MPEG-TS의 그 0x47.

188바이트씩 끊어서 패킷 단위로 분석할 수 있다.

```
[패킷 1: 0x47로 시작, 188 byte]
[패킷 2: 0x47로 시작, 188 byte]
[패킷 3: 0x47로 시작, 188 byte]
...
```

6초 분량 1080p 세그먼트면 약 4.5MB ÷ 188 = 약 24,000개 패킷.

---

### 첫 번째 패킷은 항상 PAT

MPEG-TS는 자기 안에 뭐가 들어있는지 설명하는 목차를 가진다.

**PAT (Program Association Table)**: PID 0번 패킷. "이 스트림에 어떤 프로그램이 있는지" 목록.

```
PAT 내용:
- Program 1 → PMT는 PID 4096에 있음
```

**PMT (Program Map Table)**: PAT가 가리키는 PID. "이 프로그램에 어떤 트랙이 있는지".

```
PMT 내용:
- Program 1:
  - Video Stream → PID 256, codec H.264
  - Audio Stream → PID 257, codec AAC
```

플레이어는 세그먼트 받자마자 PAT → PMT 순서로 읽어서 어디에 영상이 있고 어디에 오디오가 있는지 파악한다.

```
[세그먼트 첫 부분]
[패킷 0 - PAT]     ← 목차의 목차
[패킷 1 - PMT]     ← 트랙 정보
[패킷 2 - Video]   ← 실제 영상
[패킷 3 - Video]
[패킷 4 - Audio]
[패킷 5 - Video]
...
```

PAT/PMT는 세그먼트마다 반복된다. 시청자가 중간 세그먼트부터 시청 시작해도 트랙 정보를 알 수 있도록.

---

### 영상 데이터의 계층 구조

세그먼트 안에서 영상은 여러 계층으로 감싸져 있다.

```
[Level 1: MPEG-TS 패킷] (188 byte)
   ↓ 헤더 제거
[Level 2: PES (Packetized Elementary Stream)]
   ↓ 페이로드 추출
[Level 3: NAL Unit (H.264 단위)]
   ↓ 디코딩
[Level 4: 픽셀 데이터]
```

각 계층의 역할:

**PES**: MPEG-TS 위에서 영상/오디오를 묶는 단위. 타임스탬프(PTS, DTS) 정보가 여기 들어있다.

**NAL Unit**: H.264 코덱이 정의하는 단위. 한 NAL Unit이 한 슬라이스(이미지의 일부) 또는 메타데이터.

```
[NAL Unit 종류]
- SPS (Sequence Parameter Set): 영상 전체 설정 (해상도, 프로파일)
- PPS (Picture Parameter Set): 픽처별 설정
- IDR Slice: I-frame
- Non-IDR Slice: P/B-frame
- SEI: 보조 정보 (HDR, 워터마크)
```

세그먼트 시작에 SPS/PPS가 있어야 디코더가 영상을 해석할 수 있다.

---

### IDR Frame이 세그먼트 시작에 있어야 하는 이유

**IDR (Instantaneous Decoder Refresh)**: 디코더 상태를 초기화하는 키프레임.

```
일반 I-frame: 이전 프레임 참조 없이 그릴 수 있음. 하지만 다음 P-frame이 이전 GOP 참조 가능
IDR I-frame: 이전 모든 프레임 참조 금지. 완전한 초기화
```

세그먼트 경계에서는 IDR이 필수. 이유:

```
[시나리오: 시청자가 중간부터 시청]
1. .m3u8 받음
2. seg100.ts부터 다운로드 시작
3. 디코더 초기화
4. seg100.ts의 첫 프레임 디코딩 시도
   → IDR이 아니면 "이전 프레임 정보 없음" 오류
   → 화면 깨짐
```

IDR이 있으면 디코더가 처음부터 새로 시작할 수 있다.

FFmpeg에서 `-force_key_frames`로 IDR을 강제 삽입.

```bash
# 6초마다 IDR 강제 (HLS 세그먼트 경계와 일치)
ffmpeg -i input.mp4 \
  -c:v libx264 \
  -force_key_frames "expr:gte(t,n_forced*6)" \
  -hls_time 6 \
  output.m3u8
```

### PTS와 DTS — 두 종류의 타임스탬프

영상 데이터에는 두 가지 시간 정보가 있다.

**PTS (Presentation Timestamp)**: 언제 화면에 표시할지.

**DTS (Decoding Timestamp)**: 언제 디코딩할지.

왜 두 개가 필요하냐?

```
[B-frame 때문]

표시 순서: I → B → B → P → B → B → P
디코딩 순서: I → P → B → B → P → B → B
```

B-frame은 이전 프레임과 이후 프레임을 모두 참조한다. 그러니 디코딩하려면 미래 프레임을 먼저 받아야 한다.

```
[예시: 30fps 영상]
프레임 1 (I):  PTS=0.000, DTS=0.000
프레임 4 (P):  PTS=0.100, DTS=0.033   ← 디코딩 먼저
프레임 2 (B):  PTS=0.033, DTS=0.066   ← 그 다음
프레임 3 (B):  PTS=0.066, DTS=0.100
프레임 7 (P):  PTS=0.200, DTS=0.133
```

라이브 스트리밍은 보통 B-frame을 안 쓴다. 지연 줄이려고. 그래서 DTS = PTS인 경우가 많다.

### 오디오와 영상의 동기화

세그먼트 안에 영상과 오디오가 같이 들어있다. 어떻게 동기화하나?

```
[MPEG-TS 패킷 흐름]
Video PTS=0.000
Video PTS=0.033
Audio PTS=0.000     ← 영상과 같은 시점
Video PTS=0.066
Audio PTS=0.021
Video PTS=0.100
Audio PTS=0.042
...
```

PTS만 보면 된다. 같은 PTS의 영상과 오디오는 동시에 재생.

근데 인코딩 시 영상과 오디오가 정확히 같은 시점에 시작 안 할 수 있다.

```
세그먼트 시작점
영상: PTS=6.000 부터 시작
오디오: PTS=5.987 부터 시작 (13ms 더 빠름)
```

이런 약간의 어긋남을 **AV Sync Drift**라고 한다. 13ms 정도는 사람 귀로 못 느끼지만, 누적되면 입 모양과 소리가 안 맞아 보인다.

플레이어가 알아서 보정한다. 대부분 영상 PTS 기준으로 오디오를 살짝 빠르게/느리게 재생.

### 세그먼트 끝 처리 — Continuity

연속된 세그먼트 사이에는 자연스러운 흐름이 보장돼야 한다.

```
[seg100.ts 마지막 프레임]
PTS=5.967 (마지막 영상 프레임)
PTS=5.991 (마지막 오디오 프레임)

[seg101.ts 첫 프레임]
PTS=6.000 (영상 시작)
PTS=5.991 (오디오 시작)
```

PTS가 연속되어 있어야 한다. 끊기면 플레이어가 흠칫.

**Continuity Counter**: MPEG-TS 헤더에 있는 4비트 카운터. 0~15 순환. 패킷이 빠지면 카운터 점프해서 감지 가능.

```
[정상]
패킷 PID=256, CC=0
패킷 PID=256, CC=1
패킷 PID=256, CC=2
패킷 PID=256, CC=3

[손실 발생]
패킷 PID=256, CC=0
패킷 PID=256, CC=1
(패킷 손실)
패킷 PID=256, CC=4    ← CC가 2, 3을 건너뜀
                       플레이어가 손실 감지
```

플레이어는 손실 감지 시 디코더에 알려서 시각적 글리치를 최소화한다.

---

### 세그먼트 크기 = 시간 × 비트레이트

세그먼트 파일 크기는 거의 정확히 계산된다.

```
1080p60 6 Mbps × 6초 = 36 Mbit = 4.5 MB

720p60 4.5 Mbps × 6초 = 27 Mbit = 3.4 MB

480p30 1.5 Mbps × 6초 = 9 Mbit = 1.1 MB
```

VBR이면 약간 변동. CBR은 거의 정확.

이 크기가 CDN 캐시 효율과 네트워크 비용에 직결된다. 작은 세그먼트 = 많은 파일 = CDN 메타데이터 부담. 큰 세그먼트 = 적은 파일 = 효율 좋음. 단 라이브 지연 증가.

---

### MPEG-TS의 오버헤드

188바이트 패킷마다 4바이트 헤더. 약 2% 오버헤드.

```
6 MB 페이로드 + 2% 오버헤드 = 6.12 MB 실제 전송
```

작은 오버헤드 같지만 대규모 서비스에선 누적된다. 일일 10 PB 트래픽이면 200 TB가 헤더 오버헤드. CDN 비용 무시 못 함.

이게 LL-HLS가 fMP4로 바꾼 이유 중 하나. fMP4가 오버헤드 더 적다.

---

### 3강 핵심 3줄

1. .ts 세그먼트는 188바이트 MPEG-TS 패킷의 연속체 — PAT/PMT 목차로 트랙 정보 제공, IDR 프레임으로 시작해야 디코딩 가능
2. PTS(표시 시간)와 DTS(디코딩 시간)로 영상/오디오 동기화 — B-frame 때문에 두 시간이 다를 수 있음
3. MPEG-TS 헤더 오버헤드 약 2%, 대규모 서비스에선 CDN 비용 무시 못 함 — fMP4로 전환하는 이유 중 하나
