## 레벨 1-4 / 1강: CDN의 기본 구조와 동작 원리

---

### 왜 CDN이 필요한가 — 다시 짚기

1-1에서 봤지만 깊이 들어간다.

서버가 서울 한 곳에 있으면 안 되는 이유:

**1. 지연 (Latency)**
서울 → LA 물리적 거리 9,500km. 빛의 속도로도 32ms 걸린다. 실제 라우팅 거치면 150~200ms.

**2. 대역폭 한계**
서울 데이터센터 한 대의 네트워크 카드 = 100 Gbps 정도. 100만 명이 6 Mbps 영상 본다 = 6 Tbps 필요. 60대 서버로도 안 된다.

**3. 비용**
한국에서 미국으로 트래픽 보내는 비용 비싸다. 국제 트래픽보다 같은 국가 내 트래픽이 훨씬 싸다.

→ 시청자 가까운 곳에 미리 복사해두자. 이게 CDN.

---

### CDN의 계층 구조

CDN은 단순히 "엣지 서버 분산"이 아니다. 여러 계층이 있다.

```
[Origin Server]
  - 콘텐츠 원본 1곳 (또는 소수)
  - 모든 데이터의 최종 출처
        ↓
[Origin Shield / Mid-tier]
  - 5~10개 정도
  - Origin 부하 줄이는 중간 캐시
        ↓
[Edge Server]
  - 수백~수천 개
  - 시청자가 실제 연결되는 서버
  - 전세계 주요 도시에 위치
        ↓
[시청자]
```

이게 1-3 마지막에 본 Cascading Cache 구조다.

---

### POP — Point of Presence

CDN 회사가 서버를 두는 물리적 위치를 **POP**라고 한다.

```
Cloudflare POP 분포 (2024 기준)
- 전세계 300개 도시
- 한국: 서울, 부산
- 일본: 도쿄, 오사카, 나하
- 미국: 뉴욕, LA, 시카고, ...
```

각 POP에	는 여러 대의 서버가 있다. 각 서버가 캐시를 들고 있다.

```
[Cloudflare 서울 POP]
  - 서버 1: 캐시 10TB
  - 서버 2: 캐시 10TB
  - 서버 3: 캐시 10TB
  ...
  - 서버 100: 캐시 10TB

  총 캐시 1PB
```

---

### 시청자가 가까운 POP로 가는 방법 — Anycast

시청자가 어떻게 자기랑 가까운 POP를 찾을까? 직접 찾는 게 아니다.

**Anycast**: 여러 서버가 같은 IP를 공유. 라우터가 가장 가까운 서버로 자동 라우팅.

```
Cloudflare IP: 104.16.0.1
  → 이 IP를 서울 POP, 도쿄 POP, LA POP 모두 사용

서울 시청자가 104.16.0.1 요청 → 라우터가 서울 POP로 보냄
LA 시청자가 104.16.0.1 요청 → 라우터가 LA POP로 보냄
```

DNS가 다른 IP를 주는 게 아니다. 같은 IP를 줘도 BGP 라우팅이 알아서 가까운 곳으로 보낸다.

---

### DNS 기반 라우팅 — 또 다른 방식

CloudFront, Akamai 같은 곳은 DNS로 라우팅한다.

```
시청자: "cdn.chzzk.naver.com 어디야?"
        ↓
DNS: "당신 위치는 서울이군요. 13.225.X.X (서울 POP)로 가세요"

다른 시청자: "cdn.chzzk.naver.com 어디야?"
        ↓
DNS: "당신 위치는 LA군요. 54.230.Y.Y (LA POP)로 가세요"
```

같은 도메인이 위치에 따라 다른 IP를 반환한다. **GeoDNS** 방식.

Anycast vs GeoDNS 비교:

| | Anycast | GeoDNS |
|---|---|---|
| 라우팅 방식 | BGP | DNS 응답 |
| 정밀도 | ISP 단위 | 국가/지역 단위 |
| 장애 대응 | 자동 (라우팅 변경) | DNS TTL 만료 후 |
| 대표 회사 | Cloudflare | Akamai, CloudFront |

### CDN이 받는 요청의 종류

CDN에 오는 트래픽은 다 같지 않다. 캐시 정책이 달라야 한다.

**1. 정적 콘텐츠 (Static)**
- 이미지, JS, CSS, 영상 세그먼트
- 한 번 만들어지면 안 변함
- 캐시 길게 (1일~30일)

**2. 동적 콘텐츠 (Dynamic)**
- 사용자별 다른 HTML
- 캐시 불가능. 매번 Origin 호출

**3. 준동적 (Semi-dynamic)**
- 라이브 플레이리스트 (`.m3u8`)
- 자주 바뀌지만 캐시는 함
- 캐시 짧게 (1~5초)

치지직 라이브 페이지 한 번 열 때:
- 페이지 HTML: 동적 (사용자 로그인 정보)
- 플레이어 JS: 정적 (장기 캐시)
- 채팅 메시지: 동적 (실시간)
- 세그먼트 `.ts`: 정적 (장기 캐시)
- 플레이리스트 `.m3u8`: 준동적 (2초 캐시)

---

### Cache-Control 헤더 — 캐시 정책 표현

서버가 응답에 Cache-Control 헤더로 캐시 정책을 알린다.

```http
HTTP/1.1 200 OK
Cache-Control: public, max-age=86400, s-maxage=86400
ETag: "abc123"
```

주요 디렉티브:

| 디렉티브 | 의미 |
|---------|------|
| `public` | CDN 캐시 가능 |
| `private` | 브라우저만 캐시, CDN 불가 |
| `no-store` | 캐시 금지 |
| `max-age=N` | 브라우저 캐시 N초 |
| `s-maxage=N` | CDN 캐시 N초 |
| `must-revalidate` | 만료 시 반드시 재검증 |

라이브 스트리밍 응답 예시:

```http
# seg100.ts (영원불변)
Cache-Control: public, max-age=86400, immutable

# playlist.m3u8 (계속 업데이트)
Cache-Control: public, max-age=2
```

### CDN 캐시 키 — 무엇이 "같은 요청"인가

CDN은 URL을 기준으로 캐시한다. 근데 같은 URL이라도 다르게 처리해야 할 때가 있다.

```
GET /seg100.ts
  → 캐시 키: "/seg100.ts"

GET /seg100.ts (with cookie: user=A)
  → 캐시 키: "/seg100.ts"  (쿠키 무시)
  → 같은 캐시 사용
```

쿠키를 캐시 키에 포함하면 사용자마다 다른 캐시가 생긴다. 100만 명이 본다 = 100만 개 캐시. 의미 없다.

라이브 스트리밍에선 **쿠키와 인증 토큰을 캐시 키에서 제외**한다. 모든 사용자가 같은 세그먼트를 받으니까.

대신 인증은 별도 메커니즘으로 처리한다. **Signed URL**.

```
GET /seg100.ts?token=abc123&expires=1700000000
  → CDN: 토큰 검증만 하고 캐시는 /seg100.ts로 통일
```

---

### 1강 핵심 3줄

1. CDN은 Origin → Mid-tier → Edge의 다층 캐시 구조 — 시청자는 가까운 Edge에 연결
2. Anycast(Cloudflare)와 GeoDNS(Akamai/CloudFront)는 시청자를 가까운 POP로 보내는 두 방식
3. Cache-Control 헤더로 캐시 정책 표현 — 라이브에선 세그먼트는 길게, 플레이리스트는 짧게
