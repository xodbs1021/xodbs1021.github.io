## 레벨 2-2 / 2강: M3U8 플레이리스트 깊이 분석

---

### M3U의 역사 — 영상이 아니라 음악에서 시작

`.m3u`는 1990년대 Winamp 음악 플레이어가 만든 플레이리스트 포맷이다.

```
# Winamp .m3u 파일 (1995년경)
song1.mp3
song2.mp3
song3.mp3
```

단순히 파일 경로 나열. Apple이 이걸 그대로 가져와서 영상 스트리밍에 활용했다. UTF-8 인코딩 명시를 위해 **M3U8** (M3U + UTF-8).

기존 도구들이 이미 `.m3u`를 알고 있어서 호환성 확보가 쉬웠다. "있는 거 재활용" 전략.

---

### M3U8 두 가지 종류

HLS는 M3U8을 두 가지 목적으로 쓴다.

**1. Master Playlist (마스터)**
화질별 sub playlist 목록.

```
#EXTM3U
#EXT-X-VERSION:3

#EXT-X-STREAM-INF:BANDWIDTH=6500000,RESOLUTION=1920x1080,CODECS="avc1.640028,mp4a.40.2"
1080p/playlist.m3u8

#EXT-X-STREAM-INF:BANDWIDTH=4800000,RESOLUTION=1280x720,CODECS="avc1.4d401f,mp4a.40.2"
720p/playlist.m3u8

#EXT-X-STREAM-INF:BANDWIDTH=1600000,RESOLUTION=854x480,CODECS="avc1.4d401e,mp4a.40.2"
480p/playlist.m3u8
```

**2. Media Playlist (미디어)**
실제 세그먼트 목록.

```
#EXTM3U
#EXT-X-VERSION:3
#EXT-X-TARGETDURATION:6
#EXT-X-MEDIA-SEQUENCE:100
#EXTINF:6.000,
seg100.ts
#EXTINF:6.000,
seg101.ts
#EXTINF:5.987,
seg102.ts
```

플레이어는 먼저 Master를 받고, 적절한 Media Playlist를 골라서 그 안의 세그먼트들을 다운로드.

---

### 필수 태그 분석

M3U8 파일 안의 모든 `#EXT-X-*` 태그가 의미를 가진다.

**`#EXTM3U`**: 파일 시작. 반드시 첫 줄.

**`#EXT-X-VERSION:N`**: HLS 버전. 새 기능 쓰면 버전 올려야 함.

```
버전 1: 기본 (.ts만)
버전 3: 부동소수점 duration 허용
버전 4: byte range 요청 가능
버전 6: I-frame only playlist
버전 7: fMP4 지원 (LL-HLS 기반)
```

치지직은 보통 버전 3~4 사용.

**`#EXT-X-TARGETDURATION:6`**: 세그먼트 최대 길이(초). 정수만. 실제 세그먼트가 이보다 길면 안 됨.

**`#EXT-X-MEDIA-SEQUENCE:100`**: 플레이리스트 첫 세그먼트의 순번. 라이브에서 새 세그먼트 추가될 때마다 증가.

**`#EXTINF:6.000,`**: 다음 세그먼트의 길이. 부동소수점 가능 (버전 3+).

---

### EXTINF의 정확한 의미

각 세그먼트마다 정확히 어떻게 적나.

```
#EXTINF:6.000,
seg100.ts
```

`6.000`이 길이. 콤마 뒤에는 제목을 넣을 수 있는데, 라이브 영상에선 보통 비어 있음.

```
# 길이가 정확히 안 맞는 경우
#EXTINF:6.000,
seg100.ts
#EXTINF:5.987,    ← 마지막 세그먼트는 짧을 수 있음
seg101.ts
```

GOP 경계에 맞추다 보면 정확히 6초가 안 될 수 있다. 5.987초같이 표시.

---

### 라이브 플레이리스트의 동적 갱신

라이브는 플레이리스트가 계속 바뀐다.

```
[12:00:00]
#EXT-X-MEDIA-SEQUENCE:100
#EXTINF:6.000, seg100.ts
#EXTINF:6.000, seg101.ts
#EXTINF:6.000, seg102.ts

[12:00:06] - 새 세그먼트 추가
#EXT-X-MEDIA-SEQUENCE:101
#EXTINF:6.000, seg101.ts
#EXTINF:6.000, seg102.ts
#EXTINF:6.000, seg103.ts    ← 새로 추가
                              ← seg100은 빠짐

[12:00:12] - 또 새 세그먼트
#EXT-X-MEDIA-SEQUENCE:102
#EXTINF:6.000, seg102.ts
#EXTINF:6.000, seg103.ts
#EXTINF:6.000, seg104.ts    ← 새로 추가
                              ← seg101은 빠짐
```

플레이리스트 안에 항상 3개 세그먼트만 유지. 새 게 들어오면 오래된 게 빠진다.

**Sliding Window**라고 한다. 이미 공부한 거다.

---

### 플레이어의 폴링 주기 결정

플레이어는 새 세그먼트를 어떻게 찾나? `.m3u8`을 주기적으로 다시 받는다.

HLS 스펙 권장:
- **정상 갱신 시**: `TARGETDURATION` × 0.5 ~ 1.5 간격으로 폴링
- **변화 없으면**: `TARGETDURATION` × 0.5 ~ 1.5 후 재시도

```
TARGETDURATION:6 인 경우
플레이어가 3~9초마다 .m3u8 재요청
```

너무 자주 폴링하면 서버 부하. 너무 늦으면 라이브 엣지에서 멀어짐. 균형점이 ~6초.

### VOD 플레이리스트는 어떻게 다른가

VOD는 플레이리스트가 한 번 만들어지면 안 바뀐다.

```
#EXTM3U
#EXT-X-VERSION:3
#EXT-X-PLAYLIST-TYPE:VOD       ← VOD 명시
#EXT-X-TARGETDURATION:6
#EXT-X-MEDIA-SEQUENCE:0
#EXTINF:6.000, seg000.ts
#EXTINF:6.000, seg001.ts
#EXTINF:6.000, seg002.ts
...
#EXTINF:6.000, seg599.ts
#EXT-X-ENDLIST                  ← 끝 표시
```

핵심 차이:
- **`#EXT-X-PLAYLIST-TYPE:VOD`**: VOD라고 명시
- **`#EXT-X-ENDLIST`**: 더 이상 추가 안 됨
- **모든 세그먼트 포함**: Sliding Window 없음
- **`MEDIA-SEQUENCE:0`**: 처음부터

플레이어는 `#EXT-X-ENDLIST`를 보면 폴링을 멈춘다. 모든 세그먼트를 알고 있으니 시킹도 자유.

### Discontinuity — 광고 삽입의 핵심

방송 중간에 광고를 끼워 넣고 싶을 때. 영상의 인코딩 파라미터(코덱, 해상도, 타임스탬프)가 달라질 수 있다.

```
#EXTM3U
#EXTINF:6.000, content_seg100.ts     ← 본방송 (H.264 1080p)
#EXTINF:6.000, content_seg101.ts
#EXT-X-DISCONTINUITY                  ← 여기서 끊김
#EXTINF:15.000, ad_seg001.ts          ← 광고 (다른 인코딩)
#EXTINF:15.000, ad_seg002.ts
#EXT-X-DISCONTINUITY                  ← 다시 끊김
#EXTINF:6.000, content_seg102.ts     ← 본방송 복귀
```

`#EXT-X-DISCONTINUITY` 태그는 "여기서 인코딩 파라미터가 달라진다"를 플레이어에 알린다. 플레이어가 디코더를 리셋한다.

광고 삽입(SSAI, Server-Side Ad Insertion)의 핵심 메커니즘. CDN 광고가 이런 식으로 동작한다.

### Byte Range — 한 파일을 여러 세그먼트로

여러 작은 파일 대신 하나의 큰 파일을 부분적으로 받을 수 있다.

```
#EXTM3U
#EXT-X-VERSION:4
#EXT-X-TARGETDURATION:6
#EXTINF:6.000,
#EXT-X-BYTERANGE:1000000@0       ← 0번째부터 1MB
big_segments.ts
#EXTINF:6.000,
#EXT-X-BYTERANGE:1100000@1000000 ← 1MB 지점부터 1.1MB
big_segments.ts
#EXTINF:6.000,
#EXT-X-BYTERANGE:950000@2100000  ← 2.1MB 지점부터 950KB
big_segments.ts
```

플레이어는 `Range: bytes=0-999999` 같은 HTTP 헤더로 부분 요청.

장점:
- **파일 개수 감소**: 6초 세그먼트 600개 대신 큰 파일 하나
- **CDN 캐시 효율**: 큰 파일이 캐시 친화적
- **저장 비용 절약**: 파일 시스템 메타데이터 줄어듦

LL-HLS의 부분 세그먼트(Partial Segment)가 이 Byte Range 방식을 활용한다.

### 자막과 오디오 트랙 — Media Group

HLS는 영상, 자막, 오디오를 분리해서 제공할 수 있다.

```
#EXTM3U

# 오디오 트랙들 (별도 m3u8)
#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="audio",NAME="Korean",LANGUAGE="ko",URI="audio_ko.m3u8",DEFAULT=YES
#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="audio",NAME="English",LANGUAGE="en",URI="audio_en.m3u8"

# 자막 트랙들
#EXT-X-MEDIA:TYPE=SUBTITLES,GROUP-ID="subs",NAME="Korean",LANGUAGE="ko",URI="subs_ko.m3u8"
#EXT-X-MEDIA:TYPE=SUBTITLES,GROUP-ID="subs",NAME="English",LANGUAGE="en",URI="subs_en.m3u8"

# 영상 + 묶음 참조
#EXT-X-STREAM-INF:BANDWIDTH=6500000,AUDIO="audio",SUBTITLES="subs"
video_1080p.m3u8
```

영상은 한 번만 받고, 오디오와 자막은 선택해서 받는다. 넷플릭스 같은 곳에서 언어 바꿔도 영상이 재시작 안 되는 이유.

---

### 2강 핵심 3줄

1. M3U8은 음악 플레이리스트 포맷 재활용 — Master(화질 목록)와 Media(세그먼트 목록) 두 종류로 사용
2. 라이브는 Sliding Window로 플레이리스트 갱신, VOD는 `#EXT-X-ENDLIST`로 끝 표시 — 플레이어가 폴링 주기를 다르게 함
3. `#EXT-X-DISCONTINUITY`로 광고 삽입, Byte Range로 파일 통합, Media Group으로 다국어 — 다양한 확장 기능
