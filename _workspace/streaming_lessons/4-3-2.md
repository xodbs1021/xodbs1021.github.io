## 레벨 4-3 / 2강: NVENC 옵션 깊이 — 화질을 짜내는 튜닝

---

### NVENC 옵션 매핑 — x264 → NVENC

x264 옵션과 NVENC 옵션의 대응.

```
[x264 → NVENC 매핑]
-preset veryfast       → -preset p4
-tune zerolatency      → -tune ll
-crf 23                → -cq 23 (constqp)
-b:v 6000k             → -b:v 6000k (같음)
-maxrate -bufsize      → -maxrate -bufsize (같음)
-sc_threshold 0        → -no-scenecut 1
-g 120                 → -g 120 (같음)
-bf 0                  → -bf 0 (같음)
-profile:v high        → -profile:v high (같음)
-level 4.0             → -level 4.0 (같음)
```

대부분 그대로. preset 체계만 다름.

---

### CBR vs VBR vs CQP — 비트레이트 제어

NVENC의 rc 모드 상세.

**CBR** (Constant Bit Rate):
```bash
-rc cbr -b:v 6000k -maxrate 6000k -bufsize 12000k
```
라이브 표준. 비트 일정.

**VBR** (Variable Bit Rate):
```bash
-rc vbr -b:v 6000k -maxrate 8000k -bufsize 12000k
```
화질 일정, 비트 변동. VOD용.

**CQP** (Constant Quantization Parameter):
```bash
-rc constqp -qp 23
```
양자화 파라미터 고정. 단순하지만 비트 예측 불가.

**CBR_HQ** (CBR with high quality):
```bash
-rc cbr -tune hq
# 또는
-rc:v cbr -cbr 1 -2pass 1
```
CBR이면서 lookahead로 화질 짜냄. 라이브에서 가능한 최고 화질.

---

### 2-Pass NVENC — 라이브에서 가능한 2-pass

CPU x264 2-pass는 전체 영상 2번. 라이브엔 불가.

NVENC는 짧은 lookahead로 즉시 2-pass 효과.

```bash
ffmpeg ... \
  -c:v h264_nvenc \
  -2pass 1 \
  -rc-lookahead 32 \
  ...
```

`rc-lookahead 32`: 32프레임 미리 보고 비트 배분. 약 0.5초 지연 증가, VMAF 2~3점 향상.

라이브 표준 zerolatency에서는 lookahead 0이지만, 약간 지연 허용하면 lookahead 32 권장.

---

### Spatial AQ / Temporal AQ — 적응적 양자화

NVENC만의 화질 향상 기법.

```bash
-spatial-aq 1   # 공간적 적응 양자화
-temporal-aq 1  # 시간적 적응 양자화
-aq-strength 8  # 강도 1~15
```

**Spatial AQ**: 같은 프레임 안의 평탄한 영역과 텍스처 영역을 다르게 양자화.
- 평탄 영역: 비트 적게
- 텍스처/엣지: 비트 많이

**Temporal AQ**: 시간축으로 안정적인 영역과 변화 많은 영역을 다르게.
- 정적 배경: 비트 적게
- 움직이는 객체: 비트 많이

같은 비트레이트에서 VMAF 1~2점 향상. CPU 부담 거의 없음.

라이브 표준에 추가 권장.

---

### Weighted Prediction

P-frame의 압축 효율 향상.

```bash
-weighted_pred 1
```

페이드 인/아웃, 밝기 변화에서 효과 큼. 게임의 폭발 장면 같은 곳.

```
[효과]
일반 콘텐츠: VMAF 0~1점 향상
밝기 변화 많음: VMAF 2~3점 향상
```

켜는 게 표준. 약간의 인코딩 시간 증가.

---

### B-frame 사용 (zerolatency 아닐 때)

지연 좀 허용하면 B-frame이 화질 짜는 데 효과적.

```bash
-bf 3                # B-frame 3개
-b_ref_mode middle   # B-frame을 참조 프레임으로
```

`b_ref_mode middle`: B-frame을 참조로 쓰는 모드. NVENC만의 기법.

```
[같은 비트레이트 화질]
bf 0:              VMAF 87
bf 3:              VMAF 89
bf 3 + b_ref_mode: VMAF 91
```

라이브엔 부담. VOD/오프라인엔 필수.

---

### NVENC HEVC (H.265)

같은 NVENC가 H.265도 지원. 코덱만 바꾸면 됨.

```bash
ffmpeg ... \
  -c:v hevc_nvenc \
  -preset p4 \
  -tune ll \
  -rc cbr_hq \
  -b:v 3000k -maxrate 3000k -bufsize 6000k \
  ...
```

같은 비트레이트에서 H.264 대비 30% 화질 향상. 또는 같은 화질에서 30% 비트 절감.

라이브 인프라에서 H.265 안 쓰는 이유는 인코더 문제가 아니라 디코더 호환성. 저가폰이 못 디코딩.

특수 케이스만 사용:
- 4K HDR 콘텐츠
- 디바이스 한정 서비스 (Apple TV 앱 같은)

---

### NVENC AV1 (Ada부터)

RTX 4000 시리즈의 새 기능.

```bash
ffmpeg ... \
  -c:v av1_nvenc \
  -preset p4 \
  -tune ll \
  -rc cbr_hq \
  -b:v 2500k -maxrate 2500k -bufsize 5000k \
  ...
```

H.264 대비 50% 비트 절감, H.265 대비 30%. 라이브에서 실시간 AV1 인코딩 가능.

문제는 시청자 디코더. AV1 디코더 보급된 디바이스가 30%. 점진 도입.

Twitch가 RTX 4000 보유 스트리머 대상 AV1 베타. 호환성 자동 폴백.

---

### Look-Ahead와 Adaptive B-frame

```bash
-rc-lookahead 32          # 32프레임 미리 분석
-no-scenecut 0            # 씬 컷 자동 감지 켬 (-1로 끄기)
```

NVENC가 미래 프레임을 보고:
- 비트 배분 최적화
- B-frame 개수 적응적 결정 (장면 따라 0~3)
- 씬 컷 위치에 키프레임

라이브 zerolatency에서는 lookahead 0. 약간 지연 허용 시 lookahead 32 권장.

---

### NVENC 디버그 — 실제 적용 옵션 확인

NVENC가 옵션을 받았는데 다 적용하는 건 아님.

```bash
ffmpeg ... -c:v h264_nvenc ... -loglevel debug 2>&1 | grep nvenc
```

출력에서 실제 적용된 설정 확인.

```
[nvenc @ 0x...] 
  preset: p4
  tuning info: low latency
  rc: cbr
  multipass: qres
  ...
```

`multipass: qres`가 보이면 2-pass 활성화된 것.

### NVENC 한계와 회피

**1. 일부 옵션 무시**
```bash
# 무시되는 옵션
-bf 4              # NVENC는 최대 3 (Ada는 4 가능)
-aq-strength 20    # NVENC는 최대 15
```

문서 확인 필수.

**2. 화질 한계**
같은 비트레이트에서 x264 slow에 약 5점 VMAF 부족. VOD 최고 화질엔 부적합.

**3. 세션 한계 (소비자 GPU)**
GeForce는 8 세션. 데이터센터 GPU로 회피.

**4. 메모리 할당 실패**
GPU 메모리 부족하면 인코더 초기화 실패. 채널 패킹 잘 계산.

### 화질 벤치마크 — 실제 측정

같은 입력으로 옵션 조합 테스트.

```bash
#!/bin/bash
# benchmark_nvenc.sh

INPUT="test_1080p60.mp4"

# 옵션 조합
configs=(
  "p4 ll cbr"
  "p4 ll cbr_hq"
  "p4 ll cbr_hq -spatial-aq 1 -temporal-aq 1"
  "p4 hq vbr_hq -2pass 1 -rc-lookahead 32"
  "p7 hq vbr_hq -2pass 1 -rc-lookahead 32"
)

for config in "${configs[@]}"; do
  out="out_${config// /_}.mp4"
  ffmpeg -y -i $INPUT \
    -c:v h264_nvenc \
    -b:v 6000k -maxrate 6000k -bufsize 12000k \
    -g 120 $config \
    $out
  
  # VMAF 측정
  ffmpeg -i $INPUT -i $out \
    -lavfi libvmaf=log_path=vmaf_${config// /_}.json:log_fmt=json \
    -f null - 2>&1 | grep "VMAF score"
done
```

결과 비교로 최적 옵션 결정.

### NVENC 옵션 — 완전한 표준 명령

라이브에서 화질 최대한 짜는 명령.

```bash
ffmpeg \
  -hwaccel cuda \
  -hwaccel_output_format cuda \
  -re -i rtmp://input \
  \
  -c:v h264_nvenc \
  -preset p4 \
  -tune ll \
  -rc cbr_hq \
  -2pass 1 \
  -rc-lookahead 32 \
  \
  -b:v 6000k \
  -maxrate 6000k \
  -bufsize 12000k \
  \
  -g 120 \
  -keyint_min 120 \
  -no-scenecut 1 \
  \
  -bf 0 \
  \
  -spatial-aq 1 \
  -temporal-aq 1 \
  -aq-strength 8 \
  \
  -weighted_pred 1 \
  \
  -profile:v high \
  -level 4.0 \
  -pix_fmt yuv420p \
  \
  -c:a aac -b:a 128k -ar 48000 \
  -f flv rtmp://output
```

VMAF 88~90 수준. x264 veryfast급. 라이브 실용 최선.

### 라이브 지연 vs 화질 트레이드오프

NVENC에서 옵션별 지연 영향.

```
[지연 추가]
zerolatency only:   기준 (~150ms)
+ lookahead 32:     +500ms (~650ms)
+ B-frame 3:        +100ms (~750ms)
+ 2-pass:           +50ms

[화질 향상]
zerolatency only:   VMAF 85
+ lookahead 32:     VMAF 88
+ B-frame 3:        VMAF 90
+ 2-pass:           VMAF 91
```

LL-HLS 같은 저지연 서비스: lookahead 0, B-frame 0.  
일반 HLS: lookahead 32, B-frame 0~1.  
지연 무관 라이브: lookahead 32, B-frame 3.

### 2강 핵심 3줄

1. NVENC의 화질은 옵션 조합으로 VMAF 5점 이상 향상 가능 — cbr_hq + 2pass + lookahead + AQ가 핵심
2. 라이브 지연 vs 화질 트레이드오프 — LL-HLS는 lookahead/B-frame 0, 일반 HLS는 lookahead 32 권장
3. Ada부터 AV1 NVENC 지원 (50% 비트 절감), 디코더 호환성 때문에 점진 도입
