## 레벨 4-2 / 4강: ABR 운영 — 비용 최적화와 화질 모니터링

---

### 트랜스코딩 비용 구조

플랫폼 인프라 비용에서 트랜스코딩이 차지하는 비중.

```
[라이브 플랫폼 인프라 비용 비중]
CDN 트래픽:     40~50%
트랜스코딩:     20~30%
인제스트:       5~10%
스토리지:       5~10%
관리/모니터링:  5~10%
```

트랜스코딩이 두 번째로 큰 비용. 1080p60 사다리 하나가 32코어 서버 1대를 점유. 동시 방송 1000개면 서버 1000대.

---

### CPU vs GPU 인코딩 비용 비교

```
[1080p60 H.264 인코딩 한 채널 비용]

CPU (x264 veryfast):
  32코어 CPU 서버: 1대 ($500/월)
  채널당 4코어 점유 → 8채널/대
  채널당 비용: $62/월

GPU (NVENC, RTX 4000):
  GPU 서버: 1대 ($2000/월)
  RTX 4000으로 30채널 동시
  채널당 비용: $67/월

GPU (T4, 클라우드):
  AWS g4dn.xlarge: 1대 ($380/월)
  T4로 15채널
  채널당 비용: $25/월
```

GPU가 일반적으로 저렴. 그래서 대형 플랫폼이 다 GPU로 갔다.

근데 GPU 화질이 CPU보다 약간 떨어짐 (같은 비트레이트에서 VMAF 1~2점 차이). 화질 vs 비용 트레이드오프. 다음 레벨에서 깊이 다룸.

---

### 사다리 단계 수의 비용 영향

3단계 vs 5단계 사다리.

```
[3단계: 1080p, 720p, 480p]
인코딩 부하: x3 (1080p 기준)
스토리지: 600 MB/시간 × 3 = 1.8 GB/시간

[5단계: 1080p, 720p, 540p, 360p, 240p]
인코딩 부하: x5
스토리지: 600 + 350 + 250 + 150 + 80 = 1.43 GB/시간

(하위 사다리는 비트레이트 낮아 절대 크기 작음)
```

스토리지는 거의 비슷하지만 인코딩은 정직하게 늘어남.

5단계가 시청 경험 더 좋지만 인코딩 비용 67% 증가. 트레이드오프.

---

### 사다리 최적화 — 시청자 분포 분석

플랫폼 통계로 각 화질의 시청 비율을 본다.

```
[치지직 추정 (예시)]
1080p60: 35% (PC 와이파이)
720p60:  15% (PC 일반)
720p30:  20% (모바일 와이파이)
480p30:  20% (모바일 LTE)
360p30:  8%  (모바일 약한 신호)
160p30:  2%  (데이터 절약)
```

상위 두 단계(1080p60, 720p60)가 시청자 50%. 인코딩 비용이 50% 단계에서 80% 발생.

근데 1080p60 시청 비중을 줄이는 게 답이 아님. 그 50%가 매출의 80%일 수 있음 (광고 단가가 높음).

---

### 하위 단계 통합 — 비용 절감

가장 흔한 최적화. 시청자 적은 단계 통합.

```
[기존]
240p, 180p, 120p (각 1% 미만)

[최적화]
240p 하나로 통합 (180p, 120p는 240p로 다운스케일)
```

3개 → 1개. 인코딩 비용 절감. 시청자 경험 거의 동일.

대신 가장 낮은 단계의 비트레이트는 약간 올림 (디바이스 호환성).

---

### 비트레이트 조정 — 콘텐츠 카테고리별

전체 사다리를 카테고리별로 다르게.

```
[발로란트 카테고리]
1080p60 @ 8000k  ← 빠른 액션
720p60  @ 4500k
480p30  @ 1800k

[저스트 채팅 카테고리]
1080p30 @ 3000k  ← 정적
720p30  @ 1800k
480p30  @ 1000k

[그림 그리기]
1080p30 @ 4500k  ← 색 변화 많음
720p30  @ 2500k
480p30  @ 1200k
```

카테고리에 따라 평균 30% 비트레이트 절감 가능. 시청자가 못 느낌.

### 모니터링 — 핵심 지표

운영하며 봐야 하는 지표들.

**1. 트랜스코딩 속도**
```
speed=1.95x  ← 실시간보다 1.95배 빠름 (정상)
speed=0.85x  ← 실시간 못 따라감 (위험)
```

라이브에서 speed < 1.0이면 인코딩이 뒤처져서 결국 송출 중단. 알람 필수.

**2. 인코딩 에러율**
```
[지표]
- 입력 PTS 불일치
- 디코딩 실패
- 인코더 크래시
```

스트리머의 OBS 설정 문제로 자주 발생. 재시도 + 로그 분석.

**3. ABR 전환 빈도**
```
[정상]: 시청자당 5분에 1회 미만
[비정상]: 시청자당 1분에 1회 이상

→ 너무 잦으면 사다리 단계 사이 갭이 너무 큼
→ 중간 단계 추가 필요
```

**4. 화질 분포**
```
[목표]: 70% 이상이 최상위 두 단계 (1080p/720p)
[현실]: 50% 이상이 480p
  → ABR 알고리즘이 보수적, 또는 비트레이트가 너무 큼
```

### Prometheus 메트릭 설정

FFmpeg 진행 상황을 Prometheus로.

```bash
# FFmpeg에서 진행 출력
ffmpeg -i input.mp4 \
  -progress unix:/tmp/ffmpeg.sock \
  -nostats \
  ... output.mp4
```

이 sock을 읽어서 Prometheus exporter로 변환.

```python
# ffmpeg_exporter.py
import socket
from prometheus_client import Gauge, start_http_server

speed_gauge = Gauge('ffmpeg_speed', 'Encoding speed', ['stream_id'])
fps_gauge = Gauge('ffmpeg_fps', 'Encoding fps', ['stream_id'])
bitrate_gauge = Gauge('ffmpeg_bitrate_kbps', 'Output bitrate', ['stream_id'])

def parse_progress(line, stream_id):
    if line.startswith('speed='):
        speed = float(line.split('=')[1].rstrip('x'))
        speed_gauge.labels(stream_id=stream_id).set(speed)
    elif line.startswith('fps='):
        fps = float(line.split('=')[1])
        fps_gauge.labels(stream_id=stream_id).set(fps)

# 각 스트림마다 별도 sock 읽기
```

Grafana 대시보드에서:
```
panel: Encoding Speed
query: avg(ffmpeg_speed{job="transcoder"}) by (stream_id)
threshold: < 1.0 → red
```

### Per-Stream 헬스체크

라이브 트랜스코딩 노드가 좀비 상태로 빠지는 경우가 있다. CPU는 살아있는데 출력은 안 나오는.

```python
# 출력 .ts 파일 mtime 체크
import os
import time

def check_stream_health(stream_dir):
    """최근 세그먼트 파일이 6초 이내 생성됐는지"""
    files = sorted(os.listdir(stream_dir), key=lambda f: os.path.getmtime(os.path.join(stream_dir, f)))
    if not files:
        return False
    
    latest_mtime = os.path.getmtime(os.path.join(stream_dir, files[-1]))
    age = time.time() - latest_mtime
    return age < 10  # 10초 이내면 정상
```

10초 동안 새 세그먼트 안 나오면 FFmpeg 재시작.

### VMAF 자동 측정 파이프라인

화질 회귀 감지.

```
[배치 파이프라인]
1. 매시간 무작위 라이브 스트림 10개 샘플
2. 원본 1분 + 트랜스코딩 1분 보존
3. VMAF 측정
4. Prometheus에 저장
```

```bash
  -lavfi libvmaf=log_path=vmaf.json:log_fmt=json \
  -t 30 -f null -
```

화질 회귀가 인코딩 설정 변경 이후 자주 발생. CI 파이프라인에 VMAF 측정 포함.


### 장애 시나리오 — Region Failover

트랜스코딩 데이터센터 장애.

```
[정상]
스트리머 → 서울 인제스트 → 서울 트랜스코딩 → 서울 패키저 → CDN

[서울 트랜스코딩 장애]
서울 인제스트 → 도쿄 트랜스코딩 → 서울 패키저 → CDN
                ↑
            잠시 5초 지연
```

인제스트와 트랜스코딩 사이를 SRT로 연결하면 region 간 전송 안정성 확보. 장애 시 즉시 다른 region으로.

라이브는 실시간이라 5초 끊김도 문제. 일부 플랫폼은 active-active로 두 region에서 동시 트랜스코딩 (비용 2배).

### A/B 테스트 — 사다리 변경

새 사다리 도입 시 검증.

```
[실험 설계]
A: 기존 사다리 (1080p @ 6000k, 720p @ 3000k, ...)
B: 새 사다리 (1080p @ 5000k, 720p @ 2500k, ...)

[측정 지표]
1. Rebuffering Ratio
2. Average Bitrate
3. ABR Switch Frequency
4. VMAF
5. CDN 비용

[결정]
- Rebuffering 변화 없음 → 비트레이트 감소 OK
- Rebuffering 0.5% 증가 → 시청 시간 감소 추정 → 매출 비교
```

화질 절감으로 비용 줄이려다 시청 시간 떨어지면 손해. 데이터 기반 결정.

### 라이브 트랜스코딩 인프라 전체 그림

지금까지 본 걸 합치면.

```
[스트리머] → RTMP/SRT → [인제스트 노드]
                              ↓
                         [트랜스코딩 클러스터 (GPU)]
                              ↓
                         [패키저 (Shaka)]
                              ↓
                         [Origin Storage]
                              ↓
                         [Origin Shield]
                              ↓
                         [CDN Edges]
                              ↓
                         [시청자]

[모니터링]
- Prometheus: 인코딩 속도, 에러율, GPU 사용률
- Grafana: 대시보드, 알람
- VMAF batch: 매시간 샘플 측정
```

치지직 백엔드 트랜스코딩 팀의 일은 이 시스템 전체를 안정적으로, 저렴하게 굴리는 것.

### 4강 핵심 3줄

1. 사다리 단계 수와 비트레이트는 시청자 분포 + 카테고리 + A/B 테스트로 데이터 기반 결정
2. 모니터링 필수 지표: 인코딩 speed (< 1.0 알람), ABR 전환 빈도, 화질 분포, VMAF 회귀
3. 비용 최적화는 GPU 사용률 80% 채널 패킹 + 카테고리별 비트레이트 + 하위 단계 통합
