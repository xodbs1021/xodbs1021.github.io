## 레벨 4-3 / 1강: NVENC가 왜 필요한가 — x264로는 안 되는 이유

---

### x264의 한계 — 라이브 인프라의 벽

1080p60 H.264 인코딩을 x264 veryfast로 했을 때.

```
[32코어 서버 1대 처리량]
1080p60 채널: 약 8개 동시 (CPU 사용률 90%)
720p60 채널:  약 20개
480p30 채널:  약 40개
```

치지직 동시 방송 1000개라면 1080p60 채널만 처리하는데 서버 125대 필요. ABR ladder 5단계까지 합치면 200~300대.

서버당 월 500달러 잡으면 트랜스코딩만 월 1.5억원. 클라우드면 더 비쌈.

---

### GPU 인코딩 등장

NVIDIA가 GPU에 전용 인코딩 칩을 박았다. **NVENC (NVIDIA Encoder)**.

```
[NVENC의 정체]
NVIDIA GPU 안의 별도 ASIC (Application-Specific Integrated Circuit)
- CUDA 코어와 다른 전용 회로
- GPU 사용률에 거의 영향 없음
- 게임하면서 인코딩 가능 (이래서 OBS NVENC가 인기)
```

같은 GPU 안에 CUDA 코어, 텐서 코어, NVENC, NVDEC가 다 따로 있다.

---

### 같은 GPU에서 처리량 비교

RTX 4080 한 대 vs 32코어 CPU 한 대.

```
[1080p60 H.264 동시 인코딩 채널]
CPU (32코어 x264 veryfast):  8채널
GPU (RTX 4080 NVENC):        15~20채널

서버 가격:
CPU: $500/월
GPU: $2000/월

채널당 비용:
CPU: $62/월
GPU: $100~133/월
```

어? GPU가 더 비싸 보임. 채널당 비용이 일반 GPU에선 그렇다.

근데 클라우드에서:

```
AWS g4dn.xlarge (NVIDIA T4): $380/월
T4 NVENC 동시 채널: 15~20
채널당: $19~25
```

T4 같은 데이터센터급 GPU는 가성비가 좋다. 그래서 대형 플랫폼이 T4/A10 GPU 인프라로 갔다.

---

### NVENC 세대별 차이

NVIDIA GPU 세대마다 NVENC 칩이 다르다.

```
[NVENC 세대]
Kepler (GTX 600, 2012):   1세대 - 매우 낮은 화질
Maxwell (GTX 900, 2014):  2세대
Pascal (GTX 1000, 2016):  3세대
Turing (RTX 2000, 2018):  4세대 - x264 veryfast 화질 도달
Ampere (RTX 3000, 2020):  4세대 (개선)
Ada (RTX 4000, 2022):     5세대 + AV1 인코딩
```

Turing 이전 NVENC는 화질이 안 좋아서 안 썼다. Turing부터 라이브 인프라 표준.

**Ada (RTX 4000)부터 AV1 인코딩**. 이게 큰 변화. 차세대 코덱을 실시간으로.

---

### NVENC vs CPU 화질 — 진짜 차이

같은 비트레이트, 같은 콘텐츠.

```
[1080p60 게임 영상, 6000 kbps]
x264 medium:    VMAF 92
x264 veryfast:  VMAF 88
NVENC (Turing): VMAF 87
NVENC (Ada):    VMAF 89
```

NVENC가 약간 떨어지지만 veryfast와 비슷. 라이브 실용 수준.

```
[화질 우선 비교 - 같은 비트레이트]
x264 slow:      VMAF 95 (느림)
x264 medium:    VMAF 92
x264 veryfast:  VMAF 88
NVENC P7:       VMAF 87
NVENC P4:       VMAF 85
```

VOD에서 화질 짜야 한다면 x264 slow가 좋다. 라이브에서는 NVENC가 정답.

---

### NVENC 기본 명령

FFmpeg에서 NVENC 사용.

```bash
ffmpeg -hwaccel cuda -i input.mp4 \
  -c:v h264_nvenc \
  -preset p4 \
  -tune ll \
  -rc cbr \
  -b:v 6000k -maxrate 6000k -bufsize 12000k \
  -g 120 \
  -c:a copy \
  -f flv rtmp://output
```

옵션 차이:
- `-hwaccel cuda`: 디코딩도 GPU로 (선택)
- `-c:v h264_nvenc`: 인코더 NVENC
- `-preset p4`: 1(빠름) ~ 7(화질)
- `-tune ll`: low latency
- `-rc cbr`: 비트레이트 제어 모드

---

### preset — p1 ~ p7

x264와 다른 체계. 숫자가 클수록 화질 우선.

```
[NVENC preset]
p1: fastest, lowest quality
p2:
p3:
p4: balanced ← 라이브 표준
p5:
p6:
p7: slowest, highest quality
```

라이브: p3~p4. VOD: p6~p7.

x264와 비교:
```
NVENC p1 ≈ x264 ultrafast
NVENC p4 ≈ x264 veryfast
NVENC p7 ≈ x264 medium
```

NVENC가 같은 화질을 훨씬 빠르게 (10x 이상).

---

### tune — 사용 사례별

```
-tune hq         # high quality (VOD)
-tune ll         # low latency (라이브)
-tune ull        # ultra low latency (WebRTC급)
-tune lossless   # 무손실
```

라이브 스트리밍: `ll`. 화상회의: `ull`.

`ull`은 B-frame 비활성, lookahead 0. 지연 100ms 이하.

---

### -rc — 비트레이트 제어 모드

```
-rc cbr           # CBR (라이브)
-rc vbr           # VBR
-rc constqp       # Constant QP
-rc cbr_hq        # CBR with HQ improvements
-rc vbr_hq        # VBR with HQ
```

라이브: `cbr` 또는 `cbr_hq`.

`cbr_hq`가 약간 더 화질 좋음. 같은 비트레이트에서 VMAF 1~2점.

---

### NVENC 라이브 표준 명령

CPU x264 라이브 명령 → NVENC 변환.

```bash
ffmpeg \
  -hwaccel cuda \
  -hwaccel_output_format cuda \
  -re -i rtmp://input/live/streamkey \
  \
  -c:v h264_nvenc \
  -preset p4 \
  -tune ll \
  -rc cbr_hq \
  -b:v 6000k -maxrate 6000k -bufsize 12000k \
  -g 120 -keyint_min 120 -no-scenecut 1 \
  -profile:v high -level 4.0 \
  -pix_fmt yuv420p \
  \
  -c:a aac -b:a 128k -ar 48000 \
  -f flv rtmp://output/live/streamkey
```

x264의 `-sc_threshold 0`이 NVENC에선 `-no-scenecut 1`. 옵션명이 다름.

---

### GPU 디코딩까지 — 전체 GPU 파이프라인

NVENC만 쓰면 디코딩은 여전히 CPU. 전체 파이프라인을 GPU로.

```bash
ffmpeg \
  -hwaccel cuda \                  # 디코딩 GPU
  -hwaccel_output_format cuda \    # raw 프레임도 GPU 메모리에
  -i input.mp4 \
  -vf "scale_cuda=1280:720" \      # 스케일도 GPU
  -c:v h264_nvenc -preset p4 \     # 인코딩 GPU
  output.mp4
```

`scale_cuda`가 GPU 스케일 필터. CPU `scale`보다 빠르고 GPU↔CPU 메모리 복사 없음.

전체 GPU 파이프라인이면 CPU 사용률 5% 이하. 채널당 처리량 30~50% 향상.

---

### 한 GPU에 여러 채널 — 세션 한계

NVENC 세션 수 제한.

```
[일반 NVIDIA 드라이버 제한]
GeForce (RTX 4080): 8 세션 (소비자용)
Quadro/Tesla:       무제한
데이터센터 GPU (T4): 무제한
```

GeForce로 8개 이상 띄우려면 NVIDIA 드라이버 패치 (비공식). 데이터센터에서는 T4/A10/H100 같은 데이터센터급 GPU.

```
[T4 동시 채널 처리]
1080p60: 15~20채널
720p60:  30~40채널
480p30:  50채널
```

---

### CUDA 메모리 — GPU의 RAM

각 인코딩 세션이 GPU 메모리 점유.

```
[채널당 GPU 메모리]
1080p60: 약 200 MB
720p60:  약 100 MB
480p30:  약 50 MB

T4 16GB: 1080p60 약 60채널 메모리 가능
하지만 NVENC 처리량 한계로 15~20채널
```

병목은 메모리 아닌 NVENC 처리량.

---

### GPU 인코딩의 함정

**1. preset 잘못 선택**
p1으로 인코딩 → 화질이 너무 떨어짐 → 시청자 불만.

p4가 라이브 기본. p6는 화질 좋지만 처리량 절반.

**2. CBR 강제 안 됨**
```bash
# 약한 CBR
-rc cbr -b:v 6000k

# 강한 CBR (필요)
-rc cbr -b:v 6000k -maxrate 6000k -bufsize 12000k
```

`maxrate`, `bufsize` 안 주면 NVENC가 자체 판단으로 약간 변동.

**3. B-frame 사용**
NVENC가 B-frame 잘 처리. 화질 향상.

```bash
-bf 3              # B-frame 3개
-b_ref_mode middle # B-frame을 참조 프레임으로
```

근데 라이브 zerolatency에선 B-frame 끄는 게 정석.

```bash
-bf 0              # 라이브
```

**4. 메모리 누수**
NVENC 세션을 제대로 정리 안 하면 GPU 메모리 누수. 자주 인코딩 시작/종료하면 결국 OOM.

FFmpeg 자체 종료가 정상 종료해야 메모리 회수.

---

### NVDEC — GPU 디코딩

NVENC의 짝. GPU 디코더.

```bash
ffmpeg -c:v h264_cuvid -i input.mp4 ...
# 또는
ffmpeg -hwaccel cuda -i input.mp4 ...
```

`cuvid`는 명시적 NVDEC. `hwaccel cuda`는 자동 선택.

```
[NVDEC 처리량]
H.264 1080p60: ~250 채널 동시
H.265 1080p60: ~150 채널
AV1 1080p60:   ~80 채널 (Ampere+)
```

디코딩이 인코딩보다 가벼움. NVDEC는 거의 안 막힘.

---

### 트랜스코딩 전체 — GPU 파이프라인 한 채널

```bash
ffmpeg \
  -hwaccel cuda -hwaccel_output_format cuda \
  -i rtmp://input/live/key \
  \
  -filter_complex "\
    [0:v]split=3[v1][v2][v3]; \
    [v1]scale_cuda=1920:1080[v1080]; \
    [v2]scale_cuda=1280:720[v720]; \
    [v3]scale_cuda=854:480[v480]" \
  \
  -map "[v1080]" -map 0:a \
    -c:v h264_nvenc -preset p4 -tune ll -rc cbr_hq \
    -b:v 6000k -maxrate 6000k -bufsize 12000k \
    -g 120 -no-scenecut 1 \
    -c:a aac -b:a 128k -f flv rtmp://out/1080 \
  \
  -map "[v720]" -map 0:a \
    -c:v h264_nvenc -preset p4 -tune ll -rc cbr_hq \
    -b:v 3000k -maxrate 3000k -bufsize 6000k \
    -g 120 -no-scenecut 1 \
    -c:a aac -b:a 128k -f flv rtmp://out/720 \
  \
  -map "[v480]" -map 0:a \
    -c:v h264_nvenc -preset p4 -tune ll -rc cbr_hq \
    -b:v 1500k -maxrate 1500k -bufsize 3000k \
    -g 120 -no-scenecut 1 \
    -c:a aac -b:a 96k -f flv rtmp://out/480
```

디코딩 1번 + GPU 스케일 3번 + GPU 인코딩 3번. 전체 GPU.

T4 한 대에서 이런 채널을 5~7개 동시 처리.

### 1강 핵심 3줄

1. NVENC는 GPU 내 별도 ASIC, CUDA와 독립적 — Turing(RTX 2000)부터 x264 veryfast 화질 도달
2. 라이브 표준 NVENC 옵션: `preset p4` + `tune ll` + `rc cbr_hq` + `no-scenecut 1` + B-frame 0
3. 전체 GPU 파이프라인(hwaccel cuda + scale_cuda + h264_nvenc)으로 CPU 5% 이하, 데이터센터 T4가 가성비 최고
