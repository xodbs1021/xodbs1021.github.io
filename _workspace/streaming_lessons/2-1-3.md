## 레벨 2-1 / 3강: RTMP의 한계와 대체 후보들 — RTMP가 죽지 못하는 진짜 이유

---

### RTMP의 6가지 한계

20년 동안 쓰면서 드러난 RTMP의 문제점들.

**1. 보안 취약**
기본 RTMP는 평문 전송. 누구나 패킷 가로채면 영상 데이터 볼 수 있다. RTMPS(TLS)가 있지만 표준 채택이 느리다.

**2. H.265/AV1 미지원**
Adobe가 스펙을 더 이상 업데이트 안 한다. 새 코덱이 나와도 RTMP에 공식 추가가 없다.

```
공식 지원: H.264, AAC, MP3
비표준 확장: H.265 (일부 서버만)
미지원: AV1, VP9, Opus
```

**3. 멀티 오디오 트랙 불가**
2강에서 본 것. 이게 PokeClip이 SRT 선택한 결정타.

**4. CDN 캐시 불가**
TCP 지속 연결이라 HTTP CDN에 못 올린다. CDN 통한 대규모 배포 불가능. 시청자 측에선 진작에 폐기됐다.

**5. 방화벽 통과 어려움**
포트 1935가 회사 방화벽에서 막히는 경우 많다. RTMPT로 80번 포트 터널링 가능하지만 성능 저하.

**6. 모바일 ingest 약함**
모바일에서 OBS 같은 게 없다. 모바일로 송출하려면 별도 SDK가 필요한데, RTMP가 모바일 친화적이지 않다.

---

### 대체 후보 1: SRT — 신뢰성 있는 UDP

이미 공부한 거다. 간단히 비교만.

| | RTMP | SRT |
|---|---|---|
| 기반 | TCP | UDP + ARQ |
| 지연 | 2-3초 | 0.5-1초 |
| 멀티 트랙 | ❌ | ✅ (MPEG-TS로) |
| 암호화 | RTMPS만 | 기본 내장 (AES) |
| 신생/성숙 | 성숙 (20년) | 신생 (2017년 오픈) |
| OBS 지원 | 기본 | 가능 (별도 설정) |

SRT가 더 좋아 보인다. 그런데 왜 안 갈아탔을까?

**1. OBS의 SRT 설정이 RTMP보다 복잡**
RTMP는 URL + 스트림 키 두 개만 입력. SRT는 모드, 패스프레이즈, latency, maxbw 등 옵션 많다.

**2. 플랫폼이 SRT 받을 준비 안 됨**
치지직, 트위치 등 대부분 RTMP 인제스트만 지원. SRT 받으려면 별도 서버 구축 필요.

**3. 스트리머가 안 옮긴다**
RTMP로 잘 되는데 굳이 SRT로 갈 이유가 없다.

생태계 관성. 1강에서 본 것과 같은 패턴.

---

### 대체 후보 2: WHIP — WebRTC 기반 표준

WebRTC가 양방향 통화용으로 만들어진 건 알 거다. 거기서 "송출" 방향만 표준화한 게 **WHIP (WebRTC-HTTP Ingestion Protocol)**.

```
2023년 IETF RFC 9725로 표준화

WHIP 흐름:
1. OBS → 서버: HTTP POST /whip
   Body: SDP Offer (자기 코덱/네트워크 정보)
2. 서버 → OBS: HTTP 201 Created
   Body: SDP Answer
3. WebRTC 연결 수립 (ICE, DTLS)
4. RTP로 영상 송출 시작
```

WHIP의 장점:

**1. 초저지연**
WebRTC 자체가 200ms 이하 지연을 목표. RTMP의 1/10.

**2. 표준화됨**
IETF RFC. 누구나 같은 방식으로 구현 가능.

**3. NAT/방화벽 통과 강함**
ICE, STUN, TURN 메커니즘 활용.

**4. 최신 코덱 지원**
H.264, H.265, VP8, VP9, AV1 다 됨.

OBS 30 버전부터 WHIP 출력 기본 지원. 유튜브, 트위치가 베타 테스트 중.

```
[OBS 30 설정 화면]
서비스: WHIP
서버: https://whip.youtube.com/...
Bearer Token: ...
```

근데 왜 아직 안 퍼졌나?

**1. 플랫폼 인프라가 RTMP 중심**
기존 RTMP 인제스트 서버를 WHIP으로 다 바꾸려면 비용 큼.

**2. 일부 스트리머 환경 호환성 문제**
WebRTC가 UDP라 통신사/지역에 따라 불안정한 경우 있음.

**3. 트랜스코딩 파이프라인 변경 필요**
RTMP 받아서 MPEG-TS로 변환하는 파이프라인이 이미 검증됨. WHIP 들어가면 RTP 처리부터 다시 짜야 함.

---

### 대체 후보 3: RIST — 방송 업계 표준

방송 업계(BBC, NBC 등)에서 사용하는 신뢰성 전송 프로토콜.

```
RIST (Reliable Internet Stream Transport)
- 2017년부터 개발, 2020년 1.0 발표
- 방송 업계 컨소시엄 주도
- UDP + 재전송 + FEC (Forward Error Correction)
```

SRT와 비슷하지만 방송 표준 위주.

장점:
- **방송 장비 호환성 우수**
- **FEC로 손실 미리 대비** (재전송 안 기다림)
- **MPEG-TS 네이티브** (방송 신호 그대로)

단점:
- **인터넷 스트리밍에 익숙치 않음**
- **OBS 미지원** (방송 장비 시장 위주)
- **일반 개발자에겐 낯섦**

SRT와 RIST는 비슷한 시기에 나와서 경쟁 중. 인터넷 스트리밍은 SRT, 방송 백홀(스튜디오→방송국)은 RIST가 우세.

---

### 대체 후보 4: HLS Ingestion (Apple LL-HLS)

Apple이 만든 LL-HLS 표준에 인제스트도 포함돼 있다.

```
스트리머 → HTTP PUT으로 세그먼트 업로드 → 서버
```

장점:
- **HTTP 기반이라 CDN 활용 가능**
- **시청자 측 LL-HLS와 자연스럽게 연결**

단점:
- **세그먼트 단위라 지연이 RTMP보다 큼**
- **소량 보급** (아직 OBS 미지원)

---

### 대체 후보 비교표

| 프로토콜 | 지연 | 멀티 트랙 | CDN | OBS 지원 | 성숙도 |
|---------|------|---------|-----|---------|------|
| RTMP | 2-3초 | ❌ | ❌ | ⭐⭐⭐⭐⭐ | 매우 성숙 |
| SRT | 0.5-1초 | ✅ | ❌ | ⭐⭐⭐ | 성숙 |
| WHIP | 0.2초 | ✅ | ⭐ | ⭐⭐⭐⭐ | 신생 |
| RIST | 0.5초 | ✅ | ❌ | ❌ | 방송계 한정 |
| HLS Ingestion | 5초+ | ✅ | ✅ | ❌ | 실험적 |

종합 점수가 가장 높은 게 없다. 각각 다른 영역에서 강점.

---

### 왜 RTMP가 살아남는가 — 종합

20년 된 프로토콜이 안 죽는 이유를 정리하면:

**1. 도입 비용 = 0**
스트리머가 OBS로 그냥 송출. 추가 설치 없음. 가장 큰 진입 장벽이 없다.

**2. 호환성 100%**
모든 플랫폼이 받음. 어디로 송출해도 됨.

**3. 트러블슈팅 노하우 축적**
20년치 문제 해결 자료. 새 프로토콜은 문제 생기면 도움받을 데 없음.

**4. ingest 지연이 결정적 문제가 아님**
플랫폼 측에서 어차피 HLS로 트랜스코딩하면서 추가 지연 발생. RTMP의 2-3초 지연이 전체에서 차지하는 비중 작음.

**5. 플랫폼 수익과 무관**
시청자 경험에 영향 없음. 플랫폼이 RTMP를 바꿀 비즈니스 동기 없음.

---

### 그럼 언제 바뀔까

RTMP를 진짜 대체할 시점은:

**시나리오 1: WHIP이 OBS 기본값이 되는 순간**
OBS 새 버전이 WHIP을 메인으로 밀고, 플랫폼들이 받으면 빠르게 전환. 5~10년 예상.

**시나리오 2: 신규 플랫폼이 RTMP 안 받음**
새로 생기는 라이브 플랫폼이 처음부터 WHIP/SRT만 받으면 스트리머가 옮겨감. 단, 한국에서 새 라이브 플랫폼 나올 가능성이 낮음.

**시나리오 3: 정부/표준 단체 압박**
보안 이슈로 RTMP가 규제 대상이 되면 강제 전환. 가능성 낮음.

현실적으로는 **RTMP가 계속 살아있으면서 WHIP이 보조로 들어가는** 양상이 될 가능성이 높다.

---

### 3강 핵심 3줄

1. RTMP의 한계는 명확함 — 보안, 멀티 트랙, CDN, 코덱 모두 약함. 대체 후보는 SRT, WHIP, RIST 셋
2. WHIP이 가장 유력한 후계자 — IETF 표준, 초저지연, OBS 30 지원. 하지만 플랫폼 인프라 전환이 느림
3. RTMP가 안 죽는 이유는 기술이 아닌 생태계 관성 — 도입 비용 0, 호환성 100%, ingest 지연이 결정적 문제 아님
