## 레벨 2-4 / 2강: Partial Segment — 6초를 0.2초로 쪼개는 방법

---

### Partial Segment가 뭔가

일반 HLS의 세그먼트는 6초 단위. 6초가 다 모여야 파일이 만들어진다.

LL-HLS는 그 6초를 더 작은 조각으로 나눈다. 보통 0.2초 단위. 이 조각 하나를 **Partial Segment (또는 Part)** 라고 부른다.

```
[일반 HLS]
seg100.ts (6초 완성된 파일)

[LL-HLS]
seg100.m4s 만들어지는 동안:
  ├── part0 (0.0~0.2초)  ← 만들어지자마자 즉시 배포
  ├── part1 (0.2~0.4초)
  ├── part2 (0.4~0.6초)
  ├── part3 (0.6~0.8초)
  ...
  └── part29 (5.8~6.0초)  ← 마지막. seg100 완성
```

각 part가 만들어지자마자 시청자에게 흘러간다. 시청자는 6초를 기다리지 않고 200ms 만에 새 영상을 본다.

---

### 매니페스트에서 partial이 어떻게 보이나

LL-HLS `.m3u8` 안에 partial이 추가된다.

```
#EXTM3U
#EXT-X-VERSION:9
#EXT-X-TARGETDURATION:6
#EXT-X-PART-INF:PART-TARGET=0.2
#EXT-X-MEDIA-SEQUENCE:98

#EXTINF:6.000,
seg98.m4s

#EXTINF:6.000,
seg99.m4s

#EXTINF:6.000,
seg100.m4s

# seg101은 아직 완성 안 됨. partial로 나옴
#EXT-X-PART:DURATION=0.2,URI="seg101_part0.m4s",INDEPENDENT=YES
#EXT-X-PART:DURATION=0.2,URI="seg101_part1.m4s"
#EXT-X-PART:DURATION=0.2,URI="seg101_part2.m4s"
#EXT-X-PART:DURATION=0.2,URI="seg101_part3.m4s"
```

핵심 태그:

**`#EXT-X-PART-INF:PART-TARGET=0.2`**
모든 partial의 목표 길이. 0.2초.

**`#EXT-X-PART:DURATION=0.2,URI=...`**
개별 partial 정보. 길이와 URL.

**`INDEPENDENT=YES`**
이 partial이 IDR 프레임으로 시작한다는 표시. 시청자가 여기부터 재생 시작 가능.

---

### INDEPENDENT 플래그가 왜 중요한가

모든 partial이 IDR로 시작할 수는 없다. IDR을 너무 자주 박으면 압축 효율이 떨어진다.

그래서 partial 중 일부만 INDEPENDENT다.

```
[GOP 1초 기준 partial 구조 - 30fps]
part0 (IDR)            ← INDEPENDENT=YES
part1
part2
part3
part4 (IDR)            ← INDEPENDENT=YES
part5
part6
part7
part8 (IDR)            ← INDEPENDENT=YES
...
```

1초마다 IDR. 그 사이 partial들은 INDEPENDENT 아님.

시청자가 처음 재생 시작할 때:
- INDEPENDENT 마크된 partial부터 받기 시작
- INDEPENDENT 아닌 partial부터 받으면 화면이 안 그려짐

이게 첫 재생 시 지연을 결정한다. 운 나쁘면 1초 기다림.

---

### Partial 전송 방식 — Chunked Transfer Encoding

서버가 partial을 클라이언트에 어떻게 보내나? HTTP의 **Chunked Transfer Encoding** 활용.

일반 HTTP 응답:
```
HTTP/1.1 200 OK
Content-Length: 50000

[50000 byte 데이터 한 번에 전송]
```

Content-Length를 알아야 보낼 수 있다. 파일이 완성돼야 길이를 안다.

Chunked Transfer Encoding:
```
HTTP/1.1 200 OK
Transfer-Encoding: chunked

[chunk 1: 5000 byte]
[chunk 2: 5000 byte]
[chunk 3: 5000 byte]
...
[chunk 0: 끝]
```

응답을 chunk 단위로 흘려보낸다. 파일이 만들어지는 동안에도 보낼 수 있다.

LL-HLS 서버 동작:
```
[서버]
인코더가 part0 만듦 → chunk로 전송
인코더가 part1 만듦 → chunk로 전송
인코더가 part2 만듦 → chunk로 전송
...
세그먼트 완성 → 응답 종료
```

클라이언트는 응답이 끝날 때까지 기다리지 않고, chunk가 도착할 때마다 partial을 처리한다.

---

### HTTP/2 Push — 더 빠른 partial 전송

Chunked Transfer Encoding의 한계: 클라이언트가 일단 요청을 보내야 응답이 시작된다. 요청 보내고 받는 시간 = RTT 1번.

HTTP/2 Push는 요청 없이도 서버가 데이터를 보낼 수 있다.

```
[HTTP/2 Push 흐름]
클라이언트: GET /seg100_part0.m4s

서버 응답: part0 데이터 + 
        "다음 part1도 푸시할게" + 
        part1 데이터 +
        "그 다음 part2도 푸시할게" +
        part2 데이터
```

한 요청으로 여러 partial을 받는다. RTT 비용 0.

근데 HTTP/2 Push는 2022년에 Chrome에서 deprecated 됐다. 대부분의 LL-HLS는 Chunked Transfer만 사용. HTTP/2 Push는 옵션.

---

### 인코더의 변화 — 실시간 partial 출력

LL-HLS를 구현하려면 인코더가 달라져야 한다.

**일반 HLS 인코더**:
- 영상 30fps를 받는다
- 6초 = 180프레임 모으면 세그먼트 완성
- 세그먼트 파일을 저장한다
- 메니페스트에 추가한다

**LL-HLS 인코더**:
- 영상 30fps를 받는다
- 0.2초 = 6프레임 모으면 partial 완성
- partial을 HTTP 서버에 즉시 흘려보낸다
- 메니페스트도 partial 추가될 때마다 갱신
- 6초가 모이면 partial들을 합쳐서 완성 세그먼트 만듦

FFmpeg에서:
```bash
# 일반 HLS
ffmpeg -i rtmp://input \
  -c:v libx264 \
  -hls_time 6 \
  -hls_segment_type mpegts \
  -f hls output.m3u8

# LL-HLS
ffmpeg -i rtmp://input \
  -c:v libx264 \
  -hls_time 6 \
  -hls_segment_type fmp4 \
  -hls_flags lhls+independent_segments+program_date_time \
  -hls_fmp4_init_filename "init.mp4" \
  -hls_playlist_type event \
  -hls_list_size 6 \
  -f hls output.m3u8
```

FFmpeg는 LL-HLS 지원이 제한적. 전용 LL-HLS 인코더들이 따로 있다 (Wowza, MistServer, OvenMediaEngine).

### Partial 크기의 트레이드오프

PART-TARGET을 얼마로 잡을지가 중요한 결정.

```
[PART-TARGET 0.1초]
- 지연: 매우 짧음
- 파일 수: 1시간에 36000개
- CDN 부하: 매우 큼
- 인코더 오버헤드: 큼

[PART-TARGET 0.2초]
- 지연: 짧음
- 파일 수: 1시간에 18000개
- 권장 기본값

[PART-TARGET 0.5초]
- 지연: 보통
- 파일 수: 1시간에 7200개
- CDN 친화적
- LL-HLS의 장점 일부 손실

[PART-TARGET 1초]
- 지연: LL-HLS 의미 약해짐
```

대부분의 구현에서 0.2~0.3초 사용. Apple 권장값도 0.2초.

### Partial과 일반 세그먼트의 관계

LL-HLS는 partial과 일반 세그먼트가 같이 존재한다.

```
시간 흐름:
─────────────────────────────────────►
[seg98: 완성]  [seg99: 완성]  [seg100: 완성]  [seg101: 생성 중]
                                              part0 part1 part2 part3 ...
```

오래된 세그먼트는 partial 없이 6초 완성 파일.
가장 최근 세그먼트는 partial로 흘러나옴.

플레이어 동작:
- 처음 재생: 가까운 과거 세그먼트 빠르게 받아서 시작
- 현재 시점에 도달하면: partial 받기 시작

매니페스트에는 둘 다 표시된다.

```
#EXTINF:6.000,
seg98.m4s          ← 일반 세그먼트

#EXTINF:6.000,
seg99.m4s

#EXTINF:6.000,
seg100.m4s

# seg101 진행 중 - partial로
#EXT-X-PART:DURATION=0.2,URI="seg101_part0.m4s",INDEPENDENT=YES
#EXT-X-PART:DURATION=0.2,URI="seg101_part1.m4s"
...
```

### Partial이 완성된 후 처리

seg101의 모든 partial이 다 만들어지면, 서버는 어떻게 하나?

```
[seg101 완성 시점]

방법 1: Partial 그대로 두기
- 매니페스트에 partial 30개 + 완성 세그먼트 1개 둘 다 표시
- CDN 캐시에 둘 다 저장
- 시청자에 따라 다른 걸 받음

방법 2: Partial 정리
- 완성된 후 partial 매니페스트에서 제거
- 새 시청자는 완성 세그먼트만 받음
- 캐시 정리
```

대부분 방법 2. Partial은 "현재 진행 중인" 세그먼트에만 필요. 완성된 과거 세그먼트는 일반 HLS처럼 처리.

PART-HOLD-BACK 값으로 결정한다.

```
#EXT-X-SERVER-CONTROL:PART-HOLD-BACK=1.5
```

라이브 엣지에서 1.5초 이내의 partial만 유지. 그 이전은 일반 세그먼트로만 제공.

### Partial과 화질 전환 — 까다로움

ABR로 화질 바꿀 때 partial이 복잡해진다.

```
[1080p 매니페스트]
seg100.m4s 완성
seg101_part0 (0.0~0.2초)
seg101_part1 (0.2~0.4초)
seg101_part2 (0.4~0.6초)

[720p 매니페스트]
seg100.m4s 완성
seg101_part0 (0.0~0.2초)
seg101_part1 (0.2~0.4초)
seg101_part2 (0.4~0.6초)
```

플레이어가 1080p에서 720p로 바꾸려면:
- 1080p partial 다 받음
- 720p의 같은 시간대 partial부터 이어 받음

이때 두 화질의 partial 경계가 시간상 정확히 일치해야 한다. 안 그러면 영상에 끊김이 생긴다.

해결책: 모든 화질이 같은 partial 경계 사용.
```
모든 화질의 partial은 항상 0.0, 0.2, 0.4, 0.6, ... 초에 시작
INDEPENDENT 위치도 같음 (1초 단위 IDR)
```

이걸 강제하려면 인코더가 정확히 동기화돼야 한다. 트랜스코딩 파이프라인 복잡도 증가.

---

### 실측 지연 — Partial 도입 효과

Apple LL-HLS 데모 영상에서 측정한 지연:

```
[전통 HLS - 6초 세그먼트]
시청 지연: 25~30초

[LL-HLS - 0.2초 partial]
시청 지연: 3~5초

[LL-HLS + HTTP/2 Push]
시청 지연: 2~3초
```

10배 가까이 줄었다. 채팅과 영상의 차이가 거의 없는 수준.

---

### 2강 핵심 3줄

1. Partial Segment는 6초 세그먼트를 0.2초 단위로 쪼갠 것 — 인코더가 만들어지는 즉시 Chunked Transfer로 클라이언트에 흘려보냄
2. INDEPENDENT 플래그는 IDR로 시작하는 partial 표시 — 시청자가 그 지점부터 재생 시작 가능
3. PART-TARGET 0.2초가 표준, 너무 작으면 CDN 부하 폭증 — 모든 화질의 partial 경계가 정렬돼야 ABR 전환 부드러움
