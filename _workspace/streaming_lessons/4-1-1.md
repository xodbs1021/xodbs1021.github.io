## 레벨 4-1 / 1강: FFmpeg의 정체 — 파이프라인으로 이해하기

---

### FFmpeg가 뭔가

라이브 스트리밍 인프라 어디서든 등장하는 도구.

```
[스트리밍에서 FFmpeg가 쓰이는 곳]
- 스트리머: OBS 내부에서 인코딩 (libavcodec 사용)
- 인제스트 서버: RTMP → HLS 변환
- 트랜스코딩 서버: 1080p → 720p/480p ABR ladder 생성
- 분석 파이프라인: 멀티 오디오 분리
- 썸네일 생성: 영상에서 정지 이미지 추출
- 라이브 → VOD 변환
```

FFmpeg 모르면 라이브 인프라 일을 못 한다.

---

### FFmpeg는 하나의 도구가 아니다

`ffmpeg` 명령어 뒤에 거대한 생태계가 있다.

```
[FFmpeg 프로젝트 구성]

라이브러리:
- libavcodec: 코덱 (H.264, AAC, ...)
- libavformat: 컨테이너 (MP4, MPEG-TS, ...)
- libavfilter: 필터 (스케일, 크롭, 워터마크)
- libswscale: 이미지 변환
- libswresample: 오디오 변환

CLI 도구:
- ffmpeg: 트랜스코딩
- ffprobe: 메타데이터 분석
- ffplay: 재생
```

OBS, VLC, Chrome 다 내부적으로 FFmpeg 라이브러리를 쓴다. `ffmpeg` CLI는 그 위에 얹힌 한 사용 방식일 뿐.

---

### FFmpeg 파이프라인 — 5단계

`ffmpeg` 명령어 하나는 내부적으로 5단계 파이프라인을 거친다.

```
[입력 파일] 
   ↓
1. Demuxer (디먹서): 컨테이너에서 스트림 분리
   ↓ (압축된 비디오 / 오디오 / 자막 스트림)
2. Decoder (디코더): 압축 해제 → raw 프레임
   ↓ (YUV 픽셀 / PCM 샘플)
3. Filter (필터): 가공
   ↓ (스케일/크롭/워터마크 적용된 raw)
4. Encoder (인코더): 다시 압축
   ↓ (압축된 스트림)
5. Muxer (먹서): 컨테이너에 담기
   ↓
[출력 파일]
```

각 단계가 독립적이다. 이걸 알아야 옵션을 어디에 붙일지 안다.

---

### Demuxer/Muxer — 컨테이너 처리

영상 파일은 컨테이너 + 코덱의 조합이다.

```
input.mp4 (컨테이너: MP4)
├── 비디오 스트림 (코덱: H.264)
├── 오디오 스트림 (코덱: AAC)
└── 자막 스트림 (코덱: SRT)
```

Demuxer가 이걸 풀어서 각 스트림을 분리한다. Muxer가 다시 컨테이너에 묶는다.

```bash
# 재인코딩 없이 컨테이너만 바꾸기 (MP4 → MKV)
ffmpeg -i input.mp4 -c copy output.mkv
```

`-c copy`가 핵심. **디코딩/인코딩 안 함**. demux → 그대로 → mux. CPU 거의 안 씀. 영상 1시간을 1분에 처리 가능.

라이브 인제스트에서 RTMP 받아서 HLS로 패키징할 때 자주 쓴다. 코덱은 그대로 두고 컨테이너만 MPEG-TS로 바꾸는 게 전부.

---

### Decoder/Encoder — 실제 트랜스코딩

화질이나 해상도를 바꾸려면 디코딩 → 인코딩이 필요하다.

```bash
# H.264 → H.265 트랜스코딩
ffmpeg -i input.mp4 -c:v libx265 -c:a copy output.mp4
```

- `-c:v libx265`: 비디오 인코더를 x265로
- `-c:a copy`: 오디오는 그대로 복사

비디오만 트랜스코딩하고 오디오는 안 건드린다. CPU/시간 절약.

---

### Filter — 중간 가공

스케일, 크롭, 워터마크, 색보정 같은 작업.

```bash
# 1080p → 720p 리스케일
ffmpeg -i input.mp4 -vf scale=1280:720 -c:v libx264 output.mp4
```

`-vf` = video filter. `-af` = audio filter.

필터는 decode → filter → encode 사이에 들어간다. raw 프레임을 가공하는 거니까 디코딩이 반드시 선행돼야 한다. `-c copy`랑 같이 못 쓴다.

---

### 옵션 위치가 의미를 바꾼다

FFmpeg의 가장 큰 함정.

```bash
# 옳음: 입력 옵션은 -i 앞에
ffmpeg -ss 60 -i input.mp4 -c copy output.mp4

# 다름: 입력 옵션을 출력 옵션으로 쓰면 의미 바뀜
ffmpeg -i input.mp4 -ss 60 -c copy output.mp4
```

- `-ss 60` 위치가 `-i` **앞**: 입력 시작 60초로 점프 후 디코딩 시작 (빠름, 키프레임 부정확)
- `-ss 60` 위치가 `-i` **뒤**: 전체 디코딩 후 60초까지 버림 (느림, 정확함)

이거 모르면 트랜스코딩 시간이 10배 차이 난다.

---

### -map — 스트림 명시적으로 고르기

기본 동작은 "비디오 1개, 오디오 1개"만 자동 선택. 여러 오디오 트랙 있을 때 문제.

```bash
# 모든 스트림 출력에 포함
ffmpeg -i input.mkv -map 0 -c copy output.mp4

# 비디오 + 첫 번째 오디오만
ffmpeg -i input.mkv -map 0:v:0 -map 0:a:0 -c copy output.mp4

# 비디오 + 모든 오디오
ffmpeg -i input.mkv -map 0:v -map 0:a -c copy output.mp4
```

`-map 0:v:0` = 입력 0번 파일의 비디오 스트림 0번.

PokeClip처럼 멀티 오디오 분석 파이프라인에서는 `-map`을 정확히 알아야 한다. 게임 소리, 마이크, 디스코드를 각각 다른 파일로 빼야 하니까.

---

### -progress — 진행 상황 모니터링

스트리밍 인프라에서 FFmpeg 트랜스코딩이 잘 돌고 있는지 확인해야 한다.

```bash
# 진행 상황을 stdout에 키=값으로 출력
ffmpeg -i input.mp4 \
  -c:v libx264 \
  -progress pipe:1 \
  -nostats \
  output.mp4
```

출력 예시:
```
frame=1234
fps=58.5
bitrate=2500.3kbits/s
total_size=12345678
out_time_ms=20566000
speed=1.95x
```

`speed=1.95x`가 핵심 지표. 1.0x보다 크면 실시간보다 빠름 (라이브 트랜스코딩 OK). 0.5x면 실시간의 절반 → 라이브로는 못 씀.

서버에서 FFmpeg 띄울 때 이 stdout을 파싱해서 Prometheus에 노출시킨다.

---

### ffprobe — 메타데이터 분석

`ffmpeg`로 변환하기 전에 입력 파일의 정체를 알아야 한다.

```bash
# JSON으로 전체 정보 출력
ffprobe -v error -show_format -show_streams -of json input.mp4
```

출력:
```json
{
  "format": {
    "filename": "input.mp4",
    "duration": "3600.000000",
    "bit_rate": "5000000"
  },
  "streams": [
    {
      "codec_type": "video",
      "codec_name": "h264",
      "width": 1920,
      "height": 1080,
      "r_frame_rate": "60/1"
    },
    {
      "codec_type": "audio",
      "codec_name": "aac",
      "sample_rate": "48000",
      "channels": 2
    }
  ]
}
```

라이브 인제스트 서버는 RTMP 받자마자 ffprobe로 코덱 확인하고, 트랜스코딩이 필요한지 판단한다.

### 라이브 vs VOD 트랜스코딩의 차이

같은 FFmpeg지만 옵션이 다르다.

```bash
# VOD: 화질 최우선, 속도 무관
ffmpeg -i input.mp4 \
  -c:v libx264 -preset slow -crf 18 \
  -c:a aac -b:a 192k \
  output.mp4

# 라이브: 실시간 처리, 화질 일정
ffmpeg -i rtmp://input \
  -c:v libx264 -preset veryfast -tune zerolatency \
  -b:v 6000k -maxrate 6000k -bufsize 12000k \
  -c:a aac -b:a 128k \
  -f flv rtmp://output
```

차이점:
- preset: VOD `slow` (화질 우선), 라이브 `veryfast` (속도 우선)
- 비트레이트: VOD CRF (가변), 라이브 CBR (고정)
- tune: 라이브는 `zerolatency` (B-frame 안 씀, 지연 최소)

이 옵션들은 다음 강의에서 깊이 본다.

### FFmpeg 빌드 옵션 확인

같은 `ffmpeg`라도 빌드에 따라 지원 코덱이 다르다.

```bash
# 빌드 정보
ffmpeg -version

# 지원 인코더 확인
ffmpeg -encoders | grep h264

# 출력 예시:
# V..... libx264              libx264 H.264 / AVC / MPEG-4 AVC
# V..... h264_nvenc           NVIDIA NVENC H.264
# V..... h264_qsv             Intel Quick Sync H.264
# V..... h264_videotoolbox    VideoToolbox H.264 (macOS)
```

서버에서 NVENC 쓰려면 NVIDIA 드라이버 + CUDA + 적절히 빌드된 FFmpeg가 필요. 도커 이미지로 관리하는 게 일반적.

### Docker로 FFmpeg 운영

프로덕션에서 FFmpeg는 거의 항상 컨테이너로 돈다.

```dockerfile
# 공식 이미지 (CPU 인코딩만)
FROM jrottenberg/ffmpeg:6.0-ubuntu

# NVIDIA GPU 인코딩
FROM jrottenberg/ffmpeg:6.0-nvidia
```

```bash
# NVENC 컨테이너 실행
docker run --gpus all \
  -v $(pwd):/data \
  jrottenberg/ffmpeg:6.0-nvidia \
  -i /data/input.mp4 \
  -c:v h264_nvenc \
  /data/output.mp4
```

`--gpus all`로 GPU 노출. 이게 NVENC 트랜스코딩 인프라의 기본 패턴.

### 1강 핵심 3줄

1. FFmpeg는 demuxer → decoder → filter → encoder → muxer 5단계 파이프라인 — `-c copy`는 디코더/인코더 건너뛰기
2. 옵션이 `-i` 앞이냐 뒤냐에 따라 입력/출력 옵션으로 의미가 바뀜 (`-ss` 위치가 대표적)
3. 라이브 트랜스코딩은 `-progress pipe:1`로 speed 지표 뽑아 Prometheus 모니터링이 필수
