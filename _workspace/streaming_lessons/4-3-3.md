## 레벨 4-3 / 3강: NVENC 외 하드웨어 인코더 — QSV, AMF, VideoToolbox

---

### 하드웨어 인코더 시장

NVENC만 있는 게 아니다.

```
[하드웨어 인코더 진영]
NVIDIA NVENC:    GPU (GeForce, Quadro, T4, A10, H100)
Intel Quick Sync: CPU 내장 GPU (iGPU)
AMD AMF:          GPU + APU
Apple VideoToolbox: M 시리즈, iPhone, iPad
Google Cloud TPU: 자체 VPU
AWS Nitro:        커스텀 칩
```

플랫폼별 선택 기준이 다르다.

---

### Intel Quick Sync (QSV) — 의외의 다크호스

Intel CPU 내장 GPU의 인코더.

```bash
ffmpeg -hwaccel qsv \
  -c:v h264_qsv \
  -preset veryfast \
  -b:v 6000k \
  ...
```

QSV의 가치.

```
[QSV의 장점]
1. CPU 추가 비용 없음 (CPU에 포함)
2. 12세대 이후 매우 효율적
3. AV1 지원 (Arc GPU, 13세대+)
4. NVENC 대비 약간 떨어지지만 가격 대비 우수

[QSV의 단점]
1. 클라우드 인스턴스 선택지 적음
2. NVENC만큼 채널 처리량 안 나옴 (1080p60 5~8 채널)
```

소규모 인프라엔 QSV가 좋다. 대형은 GPU.

---

### Intel Arc — 새 흐름

2022년 Intel이 디스크리트 GPU 시장 진입.

```
[Intel Arc A770]
NVENC와 비슷한 채널 처리량
AV1 인코딩 지원 (NVIDIA RTX 4000 이전)
가격 절반
```

Twitch가 Intel Arc로 AV1 인코딩 인프라 일부 전환. 비용 압박이 큰 곳에서 도입 중.

```bash
# Arc는 QSV 인터페이스 사용
ffmpeg -hwaccel qsv \
  -c:v av1_qsv \
  -preset 4 \
  -b:v 2500k \
  ...
```

### AMD AMF

AMD GPU의 인코더.

```bash
ffmpeg \
  -c:v h264_amf \
  -quality balanced \
  -b:v 6000k \
  ...
```

AMD AMF 평가.

```
[AMD AMF 위치]
- NVENC 대비 화질 약간 떨어짐
- 처리량 비슷
- 가격 GPU 자체가 NVIDIA보다 쌈
- 클라우드 인스턴스 거의 없음 (소비자 시장 위주)
```

라이브 인프라엔 거의 안 쓴다. 인디 스트리머가 RDNA 3+ GPU로 AV1 송출하는 정도.

---

### Apple VideoToolbox

macOS / iOS의 시스템 인코더.

```bash
ffmpeg \
  -c:v h264_videotoolbox \
  -b:v 6000k \
  ...
```

서버 인프라에는 안 쓰지만, 클라이언트 측에서 중요.

```
[VideoToolbox 활용처]
- OBS Mac 버전 (Apple Silicon)
- iOS 라이브 송출 앱
- macOS 데스크톱 캡처
```

특히 Apple Silicon M 시리즈가 매우 효율적. M2 Pro/Max의 하드웨어 인코더는 NVENC RTX 4080급.

이걸 활용한 사례: Backblaze가 M1 Mac mini 클러스터로 트랜스코딩. NVIDIA보다 와트당 효율 우수.

---

### 클라우드 트랜스코딩 서비스

직접 GPU 인프라 안 굴리고 서비스 쓰기.

```
[클라우드 트랜스코딩]
AWS Elemental MediaConvert: VOD 트랜스코딩
AWS Elemental MediaLive:    라이브 트랜스코딩
AWS IVS:                    풀 라이브 스택
Google Live Stream API:     라이브
Azure Media Services:       풀 미디어
Mux:                        VOD/라이브 통합
Bitmovin:                   고급 인코딩
```

가격은 비싸지만 운영 부담 0. 작은 서비스나 트래픽 변동 큰 곳에 적합.

---

### AWS Elemental MediaLive 가격

대략적 가격 (us-east-1, 2024).

```
[입력 채널]
HD (1080p): $0.20/시간
SD:         $0.10/시간

[출력 채널]
HD HEVC:    $0.40/시간
HD H.264:   $0.20/시간
SD:         $0.10/시간

[ABR ladder 1080p + 720p + 480p, 1시간 라이브]
입력 1 HD + 출력 3 (HD + 2 SD) = $0.20 + $0.20 + $0.10 + $0.10 = $0.60/시간
```

월 720시간 풀로 굴리면 $432/채널.

직접 GPU로 같은 채널 처리하면 채널당 $25~50. 8~17배 차이.

근데 1000채널 GPU 인프라 운영 인건비 + 장애 처리 비용 생각하면 작은 규모는 클라우드가 답.

### 손익분기 — 자체 인프라 vs 클라우드

```
[자체 인프라]
초기 투자: 서버 + 네트워크 + 엔지니어 $100,000
월 운영: $20,000 (서버 + 엔지니어 2명)
처리: 1000 채널

[클라우드 (AWS MediaLive)]
초기 투자: 0
월 운영: 1000 × $432 = $432,000

손익분기: 1000채널 기준 약 2개월
500채널 기준 약 4개월
100채널 기준 1년+
```

이래서 채널 수 많은 플랫폼은 자체. 작은 곳은 클라우드.

치지직 정도면 자체. 작은 OTT는 클라우드.

### 다중 벤더 전략

큰 곳은 한 벤더 의존 안 함.

```
[치지직 트랜스코딩 추정 구조]
- 자체 GPU 클러스터 (메인, 80%)
- AWS MediaLive 백업 (장애 시, 20%)
- 일부 인기 스트리머만 NVIDIA T4 → A10 업그레이드
```

벤더 락인 회피 + 장애 분산. 대신 두 시스템 유지 비용.

### 컨테이너 이미지 - 도커로 GPU 인코더

NVENC 컨테이너.

```dockerfile
FROM nvidia/cuda:12.2.0-runtime-ubuntu22.04

# FFmpeg with NVENC
RUN apt-get update && apt-get install -y \
  ffmpeg \
  && rm -rf /var/lib/apt/lists/*

CMD ["ffmpeg", "-version"]
```

```bash
docker run --gpus all my-nvenc-ffmpeg \
  ffmpeg -hwaccel cuda -i input.mp4 \
  -c:v h264_nvenc output.mp4
```

QSV 컨테이너.

```dockerfile
FROM intel/oneapi-basekit:latest

RUN apt-get install -y \
  intel-media-va-driver-non-free \
  ffmpeg
```

```bash
docker run --device /dev/dri:/dev/dri my-qsv-ffmpeg \
  ffmpeg -hwaccel qsv -i input.mp4 \
  -c:v h264_qsv output.mp4
```

`--device /dev/dri`로 iGPU 노출.

### Kubernetes에서 GPU 인코더

```yaml
# nvenc-transcoder.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: transcoder
spec:
  template:
    spec:
      containers:
      - name: ffmpeg
        image: my-nvenc-ffmpeg
        resources:
          limits:
            nvidia.com/gpu: 1
        command:
          - ffmpeg
          - -hwaccel
          - cuda
          - -i
          - rtmp://input/live/key
          - -c:v
          - h264_nvenc
          - ...
```

NVIDIA device plugin 설치 필요. GPU 자원이 K8s에 노출.

### GPU 스케줄링 — 채널당 GPU 할당

여러 채널을 한 GPU에 패킹.

```yaml
# 한 GPU에 5채널 (NVENC 처리량 25~30%)
nvidia.com/gpu: 0.2

# 정확히 분할은 안 되고 fractional GPU 플러그인 필요
# - NVIDIA MIG (Multi-Instance GPU, A100+)
# - virtualization 솔루션
```

A100 MIG로 GPU를 7조각으로 분할 가능. 각 조각에 채널 할당.

```
[A100 MIG]
1g.5gb: 1/7 슬라이스 (1080p60 2채널)
3g.20gb: 3/7 (1080p60 8채널)
7g.40gb: 전체 (1080p60 20채널)
```

### Multi-Vendor FFmpeg 빌드

같은 FFmpeg에 모든 인코더 포함.

```bash
# 모든 하드웨어 인코더 지원 확인
ffmpeg -encoders | grep -E "nvenc|qsv|amf|videotoolbox"

V..... h264_nvenc
V..... hevc_nvenc
V..... av1_nvenc
V..... h264_qsv
V..... hevc_qsv
V..... av1_qsv
V..... h264_amf
V..... hevc_amf
V..... h264_videotoolbox
V..... hevc_videotoolbox
```

런타임에 가용한 인코더 선택.

```python
def select_encoder():
    if nvidia_available():
        return "h264_nvenc"
    elif intel_qsv_available():
        return "h264_qsv"
    elif apple_silicon():
        return "h264_videotoolbox"
    else:
        return "libx264"  # 폴백
```

### 3강 핵심 3줄

1. NVENC가 메인이지만 Intel QSV (소규모/AV1), Apple Silicon (클라이언트), AWS MediaLive (변동 큰 곳) 선택지 다양
2. 자체 GPU 인프라 vs 클라우드 손익분기는 1000채널 기준 약 2개월 — 작은 규모는 클라우드 유리
3. NVIDIA MIG로 GPU 분할, Kubernetes nvidia.com/gpu로 채널 패킹, 다중 벤더로 락인 회피가 운영 정석
