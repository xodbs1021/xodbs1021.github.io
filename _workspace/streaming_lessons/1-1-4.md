## 레벨 1-1 / 4강: CDN — 전 세계에 동영상을 뿌리는 방법

---

### 문제: 서버가 한 곳에 있으면 어떻게 되나

치지직 서버가 서울에 있고 전 세계 시청자가 본다고 가정하자.

```
서울 시청자:    지연 5ms    → 문제 없음
도쿄 시청자:    지연 30ms   → 괜찮음
LA 시청자:      지연 150ms  → 느림
브라질 시청자:  지연 300ms  → 세그먼트 하나 받는 데만 오래 걸림
```

6초짜리 세그먼트를 300ms 지연 환경에서 요청하면, 요청-응답 왕복만 600ms가 걸린다. 그 사이에 네트워크가 불안정하면 버퍼링이 난다.

그리고 더 큰 문제가 있다. 동시 접속자 100만 명이 서울 서버 한 곳에 붙으면?

```
100만 명 × 1080p 6초 세그먼트(약 24MB) = 한 번에 24TB 전송
서울 서버 대역폭: 아무리 빵빵해도 한계 있음
```

서버가 죽는다.

---

### CDN의 아이디어 — 캐시 서버를 전 세계에 뿌린다

**CDN (Content Delivery Network)**: 콘텐츠를 시청자 가까운 곳에 미리 복사해두는 분산 네트워크.

```
[원본 서버: 서울]
      │
      ├──→ [엣지 서버: 도쿄]
      ├──→ [엣지 서버: LA]
      ├──→ [엣지 서버: 런던]
      └──→ [엣지 서버: 상파울루]
```

LA 시청자가 세그먼트를 요청하면 LA 엣지 서버에서 준다. 서울까지 안 간다.

---

### Cache Hit vs Cache Miss

엣지 서버에 요청한 파일이 있으면 **Cache Hit**, 없으면 **Cache Miss**.

```
[첫 번째 LA 시청자가 seg001.ts 요청]
  → LA 엣지에 없음 (Cache Miss)
  → 서울 원본 서버에서 가져옴
  → LA 엣지에 저장

[두 번째 LA 시청자가 seg001.ts 요청]
  → LA 엣지에 있음 (Cache Hit)
  → LA에서 바로 전달 → 서울 서버 부하 없음
```

시청자가 많을수록 Cache Hit률이 높아진다. 100만 명이 같은 세그먼트를 보면, 서울 원본 서버는 딱 한 번만 요청을 받고 나머지 99만 9999명은 엣지에서 처리된다.

---

### HLS가 CDN과 완벽하게 맞는 이유

CDN의 핵심은 **HTTP 캐시**다. HTTP 응답에 캐시 정책이 있다.

```
HTTP/1.1 200 OK
Cache-Control: max-age=3600   ← 1시간 동안 캐시해도 됨
Content-Type: video/MP2T

[seg001.ts 데이터]
```

`.ts` 세그먼트는 한 번 만들어지면 절대 변하지 않는다. 그러니 캐시 시간을 길게 줘도 안전하다.

반면 `.m3u8` 플레이리스트는 라이브 중에 계속 업데이트된다. 새 세그먼트가 추가되니까. 그래서 캐시를 짧게 줘야 한다.

```
seg001.ts     → Cache-Control: max-age=86400  (24시간)
playlist.m3u8 → Cache-Control: max-age=2      (2초)
```

RTMP나 SRT는 이게 불가능하다. 지속 연결(persistent connection) 방식이라 중간에 캐시 서버를 끼울 수 없다. 직접 연결이어야 한다.

**이게 HLS가 라이브 방송 배포 표준이 된 핵심 이유다.**

---

### 세그먼트 크기와 CDN 캐시 효율

3강에서 세그먼트 크기 트레이드오프를 봤는데, CDN 입장이 추가된다.

```
2초 세그먼트:
  → 1시간 방송 = 1800개 파일
  → 시청자가 분산되면 각 파일 캐시 히트 기회 적음
  → CDN 효율 낮음

10초 세그먼트:
  → 1시간 방송 = 360개 파일
  → 같은 파일 요청 집중 → 캐시 히트 높음
  → CDN 효율 높음
```

라이브 지연(짧은 세그먼트)과 CDN 효율(긴 세그먼트) 사이 트레이드오프가 CDN 레이어에서도 나온다.

---

### 치지직의 실제 구조 (추정)

```
[OBS 스트리머]
      │ RTMP (ingest)
      ↓
[치지직 인제스트 서버: 서울]
      │ 트랜스코딩 (1080p/720p/480p ABR 래더 생성)
      │ HLS 패키징 (.ts + .m3u8)
      ↓
[치지직 원본 서버: S3 or 자체]
      │
      ↓
[CDN 엣지: 전국/전세계]
      │ Cache Hit 처리
      ↓
[시청자 플레이어]
      ↓ .m3u8 2초마다 폴링
      ↓ .ts 세그먼트 요청
```

네가 PokeClip에서 SRT+MPEG-TS로 받는 건 인제스트 서버 단계다. 시청자가 보는 HLS 배포와는 완전히 다른 레이어다.

---

### 4강 핵심 3줄

1. CDN은 원본 서버 부하를 줄이고 전세계 시청자에게 빠르게 전달하기 위한 분산 캐시 네트워크
2. HLS가 CDN과 잘 맞는 이유: `.ts` 세그먼트는 변하지 않으니 오래 캐시 가능, `.m3u8`은 짧게
3. RTMP/SRT는 CDN 불가 — 지속 연결이라 중간에 캐시를 끼울 수 없음
