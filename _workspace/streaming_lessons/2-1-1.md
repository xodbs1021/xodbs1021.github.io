## 레벨 2-1 / 1강: RTMP가 왜 만들어졌나 — Flash 시대의 유산

---

### 시대 배경 — 1996년의 인터넷 영상

RTMP를 이해하려면 1990년대 후반 인터넷이 어땠는지 알아야 한다.

```
1996년 인터넷 환경:
- 평균 대역폭: 56 Kbps 다이얼업 모뎀
- 브라우저: 넷스케이프 네비게이터
- HTML5? 없음
- JavaScript? 매우 제한적
- 영상 코덱? 없음
```

웹에서 영상을 보여줄 방법이 사실상 없었다. 영상 파일을 다운로드 받아서 RealPlayer나 Windows Media Player 같은 별도 프로그램으로 봤다.

이때 등장한 게 **Macromedia Flash** (1996년).

---

### Flash가 인터넷 영상을 점령한 이유

Flash는 원래 애니메이션 도구였다. 그런데 2002년 Flash MX부터 **영상 재생 기능**이 들어갔다.

```
[Flash 영상 재생 흐름]
1. 사이트에 <embed src="player.swf"> 태그 삽입
2. 브라우저가 Flash Player 플러그인 호출
3. Flash가 영상 스트림 받아서 재생
4. 운영체제와 브라우저에 독립적
```

장점:
- **모든 브라우저 동일 동작** (Flash Player만 설치하면 끝)
- **운영체제 무관** (Windows, Mac, Linux 다 됨)
- **저작권 보호 (DRM)** 가능

2005년 유튜브가 출범하면서 Flash가 사실상 인터넷 영상 표준이 됐다.

---

### Flash에 영상을 보내려면 — RTMP의 탄생

Flash Player가 영상을 받으려면 프로토콜이 필요했다. Macromedia(나중에 Adobe 인수)가 만든 게 **RTMP (Real-Time Messaging Protocol)**.

```
2002년 RTMP 1.0 출시
- 목적: Flash Player와 Flash Media Server 간 통신
- 전송 단위: 메시지 (Message)
- 기반: TCP 포트 1935
```

처음엔 폐쇄적 프로토콜이었다. Adobe Flash Media Server를 사야만 쓸 수 있었다.

2009년에 Adobe가 RTMP 스펙을 공개했다. 그때부터 오픈소스 RTMP 서버들이 나왔다 (nginx-rtmp-module, SRS, OvenMediaEngine 등).

---

### RTMP의 핵심 설계 — 왜 TCP 1935 포트인가

**TCP 선택 이유**: 1990년대 후반 인터넷은 불안정했다. UDP를 쓰면 패킷 손실에 대한 처리를 직접 해야 했다. TCP의 자동 재전송이 매력적이었다.

**포트 1935**: Adobe가 IANA에 공식 등록한 RTMP 전용 포트.

```
RTMP 변형들:
- RTMP    : TCP 1935 (기본)
- RTMPS   : TLS over TCP 443 (암호화)
- RTMPT   : HTTP 80 터널링 (방화벽 우회)
- RTMPE   : 약한 자체 암호화 (이제 안 씀)
- RTMPTE  : RTMPT + RTMPE
```

RTMPT는 방화벽이 RTMP 1935를 막을 때 HTTP로 위장해서 통과한다. 옛날 회사 방화벽 환경 대비용.

---

### RTMP의 메시지 구조 — Stream이 아니라 Message

여기가 RTMP가 HLS와 다른 핵심이다.

HLS는 파일 기반. RTMP는 **메시지 기반**.

```
RTMP 메시지 종류:
- Set Chunk Size (청크 크기 설정)
- Acknowledgement (확인 응답)
- User Control (재생 제어 - play, pause, seek)
- Window Acknowledgement Size
- Set Peer Bandwidth
- Audio Message (오디오 데이터)
- Video Message (비디오 데이터)
- Data Message (메타데이터)
- Command Message (RPC - 함수 호출)
```

영상/오디오 데이터도 메시지 중 하나일 뿐이다. RTMP는 본질적으로 **양방향 RPC 프로토콜**이다.

플레이어가 서버에 명령을 보낼 수 있다:

```
플레이어 → 서버: createStream()
서버 → 플레이어: streamId = 1
플레이어 → 서버: play("liveStream")
서버 → 플레이어: Video Message (계속 전송)
플레이어 → 서버: pause(true)
서버 → 플레이어: Video Message 중단
```

이 양방향성이 라이브 인터랙션에 강점이었다. HLS는 단방향(시청자가 폴링)이라 이게 안 된다.

---

### Chunk Stream — RTMP가 작은 데이터를 효율적으로 보내는 방법

RTMP 메시지는 길이가 다양하다. 4바이트짜리 ACK도 있고, 100KB짜리 비디오 프레임도 있다.

큰 메시지를 작은 **청크(Chunk)** 로 쪼개서 보낸다. 기본 청크 크기는 128바이트.

```
[Video Message 100KB]
  → Chunk 1: 128 byte
  → Chunk 2: 128 byte
  → Chunk 3: 128 byte
  ...
  → Chunk 800: 128 byte
```

청크 사이사이에 작은 ACK 메시지를 끼워 넣을 수 있다. 큰 비디오가 작은 제어 메시지를 막지 않게 하는 설계.

이게 HoL Blocking의 RTMP식 해결책이다. 한 TCP 연결 안에서 메시지를 인터리빙(섞어 짜기) 한다.

```
[전송 순서]
Video Chunk 1 → ACK → Video Chunk 2 → Video Chunk 3 → ACK → Video Chunk 4 ...
```

---

### RTMP Handshake — 연결 협상

TCP 연결 후 RTMP는 별도의 핸드셰이크를 수행한다.

```
[Handshake 3단계]

Client → Server: C0 + C1
  C0: 1바이트 (버전 정보)
  C1: 1536바이트 (타임스탬프 + 랜덤 데이터)

Server → Client: S0 + S1 + S2
  S0: 1바이트 (버전)
  S1: 1536바이트 (서버 타임스탬프 + 랜덤)
  S2: 1536바이트 (C1 에코)

Client → Server: C2
  C2: 1536바이트 (S1 에코)
```

총 9 KB의 데이터를 주고받는다. RTT 1번 ~ 1.5번이 걸린다.

핸드셰이크의 목적:
1. **버전 호환성 확인**
2. **타임스탬프 동기화** (영상-오디오 동기화 기준)
3. **랜덤 데이터로 간단한 신원 확인**

이 단계가 끝나야 실제 영상 전송이 시작된다. RTMP 연결 시작에 보통 1~2초가 걸린다.

---

### Flash가 죽었는데 RTMP가 살아남은 이유

2010년 스티브 잡스의 "Thoughts on Flash" 발표. iOS가 Flash를 영원히 거부했다.

2017년 Adobe가 2020년 말로 Flash 지원 종료 발표.

Flash는 죽었다. 그런데 RTMP는 살아남았다. 두 가지 이유.

**1. 인제스트(스트리머 → 플랫폼) 표준이 됐다**
시청자 측은 Flash → HTML5로 빠르게 전환됐다. 하지만 스트리머 측은 RTMP가 그대로 남았다. OBS, XSplit 같은 모든 방송 소프트웨어가 RTMP를 지원한다.

**2. 대안이 없었다**
WebRTC는 너무 복잡, SRT는 너무 새것, HLS는 인제스트에 안 맞음.

```
시청자 측:
2005: Flash + RTMP
2015: HTML5 + HLS
2025: HTML5 + HLS/LL-HLS

스트리머 측:
2005: OBS → RTMP → Flash Media Server
2015: OBS → RTMP → 트위치/유튜브
2025: OBS → RTMP → 치지직/트위치/유튜브 (여전히)
```

20년이 지나도 인제스트는 RTMP. 이게 "생태계 관성"의 대표 사례다.

---

### 1강 핵심 3줄

1. RTMP는 2002년 Flash와 함께 등장한 영상 전송 프로토콜 — Adobe가 만들고 2009년 공개
2. TCP 1935 기반, 메시지 + 청크 구조로 양방향 RPC 가능 — 영상 데이터도 메시지 중 하나
3. Flash는 죽었지만 인제스트(스트리머 → 플랫폼) 표준으로 살아남음 — 대안 부재 + 생태계 관성
