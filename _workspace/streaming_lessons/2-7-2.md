## 레벨 2-7 / 2강: WHEP — WebRTC 시청 표준

---

### WHIP만으로는 부족하다

WHIP은 송출(ingestion) 표준. 그럼 시청(playback)은?

```
[WHIP만 있으면]
스트리머 → WHIP → 플랫폼 서버
                  │
                  └── 시청자는 어떻게 봐?
                      ├── HLS? (지연 25초)
                      ├── LL-HLS? (지연 3초)
                      └── WebRTC? (어떻게 표준화?)
```

시청자 측도 WebRTC로 받으면 200ms 지연. 근데 시청자 측 WebRTC 시그널링도 회사마다 다름.

송출과 같은 문제. 표준이 필요했다.

---

### WHEP의 발표 — WHIP의 짝

**WHEP (WebRTC-HTTP Egress Protocol)**: WebRTC를 사용한 라이브 시청 표준.

WHIP을 거울처럼 뒤집은 구조.

```
[WHEP 흐름]
1. 시청자 → 서버: HTTP POST /whep
   Body: SDP Offer

2. 서버 → 시청자: HTTP 201 Created
   Body: SDP Answer
   Location: /whep/resource_id

3. WebRTC 연결 수립 (ICE, DTLS)

4. 미디어 수신 (SRTP)

5. 종료: HTTP DELETE /whep/resource_id
```

거의 똑같다. 방향만 반대. 시청자가 수신 전용.

WHIP과 WHEP은 IETF에서 별도 RFC. WHEP은 2024년 표준화.

---

### WHIP과 WHEP 비교

| | WHIP | WHEP |
|---|---|---|
| 역할 | 송출 | 시청 |
| HTTP 메서드 | POST | POST |
| SDP 방향 | sendonly | recvonly |
| 인증 | Bearer Token | Bearer Token or 공개 |
| 미디어 | 송신 | 수신 |
| 사용처 | OBS, 카메라 | 브라우저, 앱 |

---

### WHEP의 사용 시나리오

WHEP이 진가를 발휘하는 경우.

**1. 저지연 라이브 시청**
스포츠 중계, 경매, 게임 스트리밍.

**2. 인터랙티브 방송**
시청자 채팅에 즉시 반응.

**3. 모니터링**
CCTV, 드론, 원격 작업 화면.

**4. 라이브 협업**
화면 공유, 원격 도움.

대규모 시청자가 아니거나, 지연이 결정적인 경우.

---

### WHEP 클라이언트 구현

브라우저에서 WHEP으로 시청.

```javascript
async function watchWhep(whepUrl) {
  const pc = new RTCPeerConnection({
    iceServers: [{ urls: 'stun:stun.l.google.com:19302' }]
  });
  
  // 수신 트랜시버 추가
  pc.addTransceiver('audio', { direction: 'recvonly' });
  pc.addTransceiver('video', { direction: 'recvonly' });
  
  // 미디어 수신 처리
  pc.ontrack = (event) => {
    const video = document.getElementById('video');
    video.srcObject = event.streams[0];
  };
  
  // SDP Offer 생성
  const offer = await pc.createOffer();
  await pc.setLocalDescription(offer);
  
  // HTTP POST로 Offer 전송
  const response = await fetch(whepUrl, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/sdp',
    },
    body: offer.sdp,
  });
  
  // Answer 받기
  const answerSdp = await response.text();
  const location = response.headers.get('Location');
  
  // Answer 적용
  await pc.setRemoteDescription({
    type: 'answer',
    sdp: answerSdp,
  });
  
  // 종료 함수
  return async () => {
    await fetch(location, { method: 'DELETE' });
    pc.close();
  };
}

// 사용
const stop = await watchWhep('https://whep.example.com/live/channel123');

// 종료 시
// stop();
```

50줄 미만으로 저지연 시청 구현.

### 시청자 확장성 문제 — WHEP의 한계

WHEP은 WebRTC라 시청자가 늘면 서버 부하가 폭증.

```
[WHEP 단일 서버]
시청자 1: 1.5 Mbps 송신
시청자 2: 1.5 Mbps 송신
...
시청자 1000: 1.5 Mbps 송신

서버 송신: 1000 × 1.5 Mbps = 1.5 Gbps
```

10,000명, 100,000명이면 한 서버로 불가능.

해결: SFU 계층화 + 지역 분산.

```
[대규모 WHEP 인프라]
[Origin SFU - 1개]
       │
   ┌───┼───┐
[지역 SFU][지역 SFU][지역 SFU]
   │       │       │
시청자  시청자  시청자
1만명   1만명   1만명
```

각 지역 SFU가 1만 명 정도. 전세계 100만 명 처리.

근데 이게 CDN보다 비싸다. HTTP CDN은 거의 무한 확장 가능하지만 WebRTC SFU는 비싼 서버 필요.

---

### WHEP vs HLS — 시청자 측 비교

라이브 시청 옵션 비교.

| | HLS | LL-HLS | WHEP |
|---|---|---|---|
| 지연 | 25초 | 3초 | 0.5초 |
| 시청자 확장성 | CDN 100만+ | CDN 100만+ | SFU 1만 정도 |
| 비용 | 매우 낮음 | 낮음 | 높음 |
| 브라우저 호환 | iOS만 네이티브 | iOS만 | 모든 브라우저 |
| 모바일 | 매우 안정 | 안정 | 변동 |
| 적합 | 일반 방송 | 채팅 동기 | 인터랙티브 |

치지직 같은 대규모 라이브: HLS + LL-HLS 조합.
인터랙티브 또는 소규모 라이브: WHEP.

---

### 하이브리드 구조 — 시청자가 선택

플랫폼이 여러 옵션 제공.

```
[다중 프로토콜 라이브 플랫폼]
시청자 옵션:
- HLS: 일반 시청 (지연 25초)
- LL-HLS: 채팅 동기 (지연 3초)
- WHEP: 인터랙티브 (지연 0.5초, 추가 비용)

시청자가 선택하거나 자동 결정
```

자동 결정 로직:
```
if 인기 스트리머 & 평범한 시청 → HLS
elif 채팅 활발 → LL-HLS
elif 게임/스포츠 인터랙션 → WHEP
```

### Cloudflare Stream + WHEP

Cloudflare Stream이 WHEP을 간단히 활용 가능.

```javascript
// Cloudflare Stream Live에서 WHEP URL 받기
const liveInputId = 'abc123';
const whepUrl = `https://customer-${customerId}.cloudflarestream.com/${liveInputId}/webRTC/play`;

// 위의 WHEP 클라이언트 코드 사용
const stop = await watchWhep(whepUrl);
```

플랫폼이 WHEP 인프라를 운영하니 개발자는 URL만 알면 됨.

요금: 일반 HLS 대비 비싸지만 자체 SFU 운영보다 훨씬 저렴.

### Twitch의 WHIP/WHEP 도입 (2024)

Twitch가 베타에서 안정 단계로 전환.

```
[Twitch WHIP]
서버: https://ingest.twitch.tv/whip
Bearer Token: Stream Key 활용

[Twitch WHEP - 시청자 측]
URL: https://video-edge-XXXXX.twitch.tv/playlist.whep
```

기존 RTMP/HLS와 병행. 시청자가 화질 설정에서 "Low Latency" 선택 시 WHEP 활성화.

지연 비교 (Twitch 발표):
- 일반 HLS: 12~15초
- LL-HLS: 3~4초
- WHEP: 0.5~1초

### WHEP과 P2P 결합

WebRTC라 P2P 보조 가능.

```
[하이브리드 WHEP + P2P]
시청자 A (조기 시청): 서버에서 WHEP으로 받음
시청자 B (조금 늦게): 시청자 A에게서 P2P로 받음
시청자 C: 시청자 B에게서 P2P로 받음
```

서버 부하 분산. 비용 절감.

WebTorrent, Streamroot 같은 솔루션이 이걸 자동화.

### WHEP의 라이브 채팅 통합

WebRTC DataChannel로 채팅도 같이.

```javascript
const pc = new RTCPeerConnection(config);

// 영상 수신
pc.addTransceiver('video', { direction: 'recvonly' });

// 채팅 데이터 채널
pc.addTransceiver('audio', { direction: 'recvonly' });

pc.ondatachannel = (event) => {
  const channel = event.channel;
  
  channel.onmessage = (msg) => {
    const data = JSON.parse(msg.data);
    if (data.type === 'chat') {
      displayChat(data.user, data.message);
    }
  };
};
```

영상과 채팅이 같은 WebRTC 연결로. 동기성 보장 (RTP와 DataChannel 모두 같은 SCTP/DTLS).

채팅이 영상보다 빠른 문제 해결.

### 시청 통계 — WHEP의 도전

HLS는 CDN 액세스 로그로 시청 통계 쉬움. WHEP은 다르다.

```
[HLS 통계 수집]
시청자가 .ts 파일 요청 → CDN 로그 → 시청 분석

[WHEP 통계 수집]
시청자가 WebRTC 연결 시작 → 시그널링 서버 추적
시청자가 끊기면? → DELETE 안 보낼 수도 있음
시청 시간 정확히 측정 어려움
```

해결책:
- 시청자 클라이언트가 주기적으로 PATCH 요청 (heartbeat)
- 일정 시간 응답 없으면 자동 종료 처리
- WebRTC 통계 API 활용

---

### WHEP의 미래 — 산업 변화 예측

WHEP이 보급되면.

**1. 라이브 방송 표준 변화**
시청자가 25초 지연을 못 견디게 될 것.

**2. 채팅과 영상 동기화**
스트리머와 시청자 간 인터랙션 자연스러움.

**3. 새로운 비즈니스 모델**
저지연 기반 게임/투표/경매 라이브.

**4. 모바일 라이브의 부상**
폰으로 송출/시청이 모두 저지연.

**5. CDN 시장 변화**
HTTP CDN + WebRTC SFU 통합 서비스.

### 2강 핵심 3줄

1. WHEP은 WHIP의 시청자 측 짝 — HTTP POST로 SDP Offer 보내고 Answer 받아 WebRTC 시청, 지연 0.5초
2. 시청자 확장성이 HLS의 약점 — SFU 한 대로 1만 명 정도, 대규모는 지역 분산 SFU 또는 P2P 보조
3. Twitch/Cloudflare Stream이 도입, DataChannel로 채팅 동기 가능 — 일반 시청 HLS + 인터랙티브 WHEP 하이브리드가 미래
