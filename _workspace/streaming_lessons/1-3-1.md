## 레벨 1-3 / 1강: 버퍼링이란 정확히 뭔가

---

### 버퍼란 단어부터 정리

"버퍼링"이라는 말이 두 가지로 쓰여서 혼란스럽다.

**버퍼 (Buffer)**: 데이터를 미리 받아두는 임시 저장 공간. 좋은 것.

**버퍼링 (Buffering)**: 그 버퍼가 비어서 재생을 멈추고 다시 채우는 상태. 나쁜 것.

영어로는 둘을 구분한다 (buffer vs rebuffering). 한국어에선 둘 다 "버퍼링"이라 부르니 헷갈린다.

---

### 플레이어 버퍼의 정체

플레이어는 항상 **지금 재생 중인 위치보다 앞선 데이터**를 미리 받아둔다.

```
[현재 재생 위치]              [버퍼 끝]
       ↓                          ↓
───────●━━━━━━━━━━━━━━━━━━━━━━━━━○───────────
       재생중    버퍼링된 구간 (30초)    아직 안 받음
```

이 미리 받은 구간을 **재생 버퍼**라고 한다.

왜 미리 받느냐? 네트워크는 불안정하기 때문이다.

```
0초: 세그먼트 다운로드 5초 걸림 (네트워크 느림)
6초: 세그먼트 다운로드 1초 걸림 (정상)
12초: 세그먼트 다운로드 8초 걸림 (네트워크 매우 느림)
```

만약 버퍼가 0이면 0초 시점에 5초간 멈춰 있어야 한다. 버퍼가 30초 차 있으면 5초 다운로드 지연이 와도 버퍼에서 빼서 계속 재생할 수 있다.

**버퍼는 네트워크 변동성을 흡수하는 완충 장치다.**

---

### 버퍼 길이의 트레이드오프

버퍼를 길게 잡으면 안 끊긴다. 근데 단점이 있다.

**버퍼 길이 = 라이브 지연**

라이브 방송에서 플레이어 버퍼가 30초면, 실제 방송 시점보다 시청자는 30초 늦게 본다. 스트리머가 "지금!" 외친 게 30초 후에 들린다.

```
[버퍼 30초]
  → 안정성: ⭐⭐⭐⭐⭐ (네트워크 변동에 강함)
  → 지연:   ⭐ (시청자 경험 나쁨)

[버퍼 5초]
  → 안정성: ⭐ (네트워크 살짝 흔들리면 버퍼링)
  → 지연:   ⭐⭐⭐⭐⭐ (실시간에 가까움)
```

LL-HLS가 부분 세그먼트(Partial Segment)를 쓰는 이유가 이거다. 작은 단위로 받아서 버퍼를 짧게 유지하면서도 안정성을 확보하려는 시도.

---

### 버퍼링(=재버퍼링)이 발생하는 4가지 원인

버퍼가 0으로 떨어지는 상황을 분석해보면:

**1. 네트워크 대역폭 부족**
세그먼트 다운로드 속도 < 재생 속도. 가장 흔한 원인.

```
6초짜리 세그먼트 = 4.5 MB
다운로드 속도 = 0.5 MB/s
→ 1개 받는 데 9초 걸림. 재생 속도(6초)보다 느림 → 버퍼 고갈
```

**2. 네트워크 지연(RTT) 증가**
대역폭은 충분한데 RTT가 길어진 경우. TCP 재전송이 한 번 발생할 때마다 RTT 만큼 멈춘다.

**3. 서버/CDN 응답 지연**
세그먼트 요청에 서버가 늦게 응답. 이상한 인기 영상에 100만 명이 몰리면 발생.

**4. 플레이어 자체 처리 지연**
디코딩이 느린 경우. 4K 영상을 저사양 폰에서 보면 CPU/GPU가 못 따라간다.

### TCP의 Head-of-Line Blocking

2강에서 봤던 TCP 재전송 문제가 버퍼링의 근본 원인 중 하나다. 정식 명칭이 있다.

**Head-of-Line Blocking (HoL Blocking)**: 줄 맨 앞 손님이 못 가면 뒤 손님 전부 대기.

```
[브라우저]                     [서버]
  │── seq:1 받음                 │
  │── seq:2 받음                 │
  │── seq:3 ❌ 손실               │
  │── seq:4 받음 → 대기 ←────────│
  │── seq:5 받음 → 대기 ←────────│
                ↑
        seq:3 재전송 올 때까지
        애플리케이션에 전달 안 함
```

seq:4, 5가 이미 도착했어도 운영체제가 애플리케이션(브라우저)에 안 넘긴다. 순서가 어긋난 채로 주면 안 되니까.

이러면 RTT 150ms 환경에서 패킷 하나 손실 = 최소 300ms 멈춤 (재전송 요청 + 도착).

---

### QUIC — HoL Blocking을 우회한 새 프로토콜

이 문제를 해결하려고 구글이 만든 게 **QUIC**다.

```
TCP: 모든 데이터를 한 연결로 → seq 순서 보장 → HoL Blocking
QUIC: 여러 스트림을 병렬로 → 한 스트림 막혀도 다른 스트림 진행
```

UDP 기반이지만 TCP의 신뢰성과 암호화를 모두 가진다. HTTP/3가 QUIC 위에서 동작한다.

라이브 스트리밍에서 QUIC을 쓰면:
- 세그먼트 1개 다운로드 중 패킷 손실 → 다른 세그먼트 다운로드는 영향 없음
- TLS 핸드셰이크 시간 단축 (0-RTT)

치지직 같은 플랫폼이 점진적으로 HTTP/3를 도입하는 중이다.

### 1강 핵심 3줄

1. 버퍼는 네트워크 변동성을 흡수하는 미리 받은 데이터, 버퍼링(재버퍼링)은 그 버퍼가 0이 돼서 멈추는 상태
2. 버퍼 길이는 안정성 vs 지연의 트레이드오프 — LL-HLS는 부분 세그먼트로 양쪽 잡으려는 시도
3. TCP의 HoL Blocking이 재버퍼링의 근본 원인 — QUIC/HTTP/3가 이걸 해결하는 방향
