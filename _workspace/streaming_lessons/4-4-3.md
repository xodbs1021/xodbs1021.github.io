## 레벨 4-4 / 3강: Chunked CMAF — 저지연 라이브의 기반

---

### 일반 CMAF의 지연 문제

CMAF가 fMP4 통합으로 비용 절감했지만, 지연은 여전히 일반 HLS와 같다.

```
[일반 CMAF (6초 세그먼트)]
인코더가 6초어치 인코딩 끝나야 .m4s 생성
플레이어가 .m4s 받기 시작 → 6초 + 다운로드 시간
총 지연: 18~25초
```

저지연 라이브 (5초 이하) 구현 불가.

---

### Chunked CMAF의 아이디어 — 미완성 세그먼트 보내기

세그먼트를 만드는 도중에 조각씩 보낸다.

```
[Chunked CMAF (6초 세그먼트, 200ms 청크)]

시간 t=0:    인코더가 첫 200ms 청크 생성 → 즉시 전송 시작
시간 t=0.2:  두 번째 200ms 청크 생성 → 전송
시간 t=0.4:  세 번째 청크 → 전송
...
시간 t=6.0:  마지막 청크 → 세그먼트 완성

플레이어:
시간 t=0.2:  첫 청크 받자마자 디코딩 시작
지연: 200~500ms + 네트워크 지연
```

세그먼트 완성을 기다리지 않고 청크 단위로 받는다.

---

### Chunked Transfer Encoding — HTTP 1.1 기능

청크 단위 전송이 어떻게 가능한가? HTTP/1.1의 chunked encoding.

```
[일반 HTTP 응답]
Content-Length: 6000000
[전체 6MB 받고 끝]

[Chunked HTTP 응답]
Transfer-Encoding: chunked

200000 (청크 1 크기)
[200KB 데이터]
200000 (청크 2)
[200KB 데이터]
...
0 (끝)
```

서버가 다음 청크를 만드는 동안 클라이언트는 이미 받은 청크를 처리.

이 HTTP 1.1 기능 + CMAF의 박스 구조 = chunked CMAF.

---

### fMP4 박스의 chunked 분할

CMAF chunked의 핵심 구조.

```
[일반 .m4s]
moof + mdat (전체)

[Chunked .m4s]
moof + mdat (청크 1)
moof + mdat (청크 2)
moof + mdat (청크 3)
...
```

한 .m4s 안에 여러 (moof + mdat) 쌍. 각 쌍이 독립 디코딩 가능.

플레이어가 청크 받자마자 디코더에 넘김. 모든 청크 다 받을 필요 없음.

---

### Chunked CMAF 인코딩 — Shaka Packager

```bash
packager \
  'in=video.mp4,stream=video,init_segment=init.mp4,segment_template=seg_$Number$.m4s' \
  \
  --hls_master_playlist_output master.m3u8 \
  --hls_playlist_type LIVE \
  --segment_duration 6 \
  --fragment_duration 0.2 \      # 200ms 청크
  --low_latency_dash_mode
```

- `segment_duration 6`: 세그먼트 6초
- `fragment_duration 0.2`: 청크 200ms (6초 안에 30 청크)
- `low_latency_dash_mode`: chunked 전송 활성

---

### LL-HLS 프로토콜 - 청크를 어떻게 시청자에게 전달하나

LL-HLS가 CMAF chunked를 시청자에게 노출하는 방식.

매니페스트에 청크 정보 추가:

```
#EXTM3U
#EXT-X-VERSION:9
#EXT-X-TARGETDURATION:6
#EXT-X-SERVER-CONTROL:CAN-BLOCK-RELOAD=YES,PART-HOLD-BACK=1.0
#EXT-X-PART-INF:PART-TARGET=0.2

#EXT-X-MAP:URI="init.mp4"

#EXTINF:6.0,
seg_1.m4s

#EXTINF:6.0,
seg_2.m4s

# 현재 만들어지는 중인 세그먼트 (청크들)
#EXT-X-PART:DURATION=0.2,URI="seg_3.m4s",BYTERANGE="50000@0"
#EXT-X-PART:DURATION=0.2,URI="seg_3.m4s",BYTERANGE="42000@50000"
#EXT-X-PART:DURATION=0.2,URI="seg_3.m4s",BYTERANGE="47000@92000"
...
```

`#EXT-X-PART`가 청크 (part). URI와 byterange로 위치 지정.

플레이어가 완성된 세그먼트 + 진행 중인 청크를 모두 받음.

---

### Blocking Playlist Reload

매니페스트 폴링도 청크 속도에 맞춰야.

```
[일반 HLS]
플레이어가 6초마다 매니페스트 폴링
새 세그먼트 있나 확인

[LL-HLS Blocking Reload]
플레이어가 매니페스트 요청 시 "다음 시퀀스가 나올 때까지 기다려"
서버가 새 청크 준비되면 즉시 응답
```

서버가 응답을 보류했다가 새 청크 생기면 즉시 회신. 폴링 없이 push 효과.

매니페스트 요청 URL:
```
GET /playlist.m3u8?_HLS_msn=42&_HLS_part=15
```
"시퀀스 42의 파트 15보다 새 게 나올 때까지 대기"

---

### Preload Hint

다음에 받을 청크를 미리 알림.

```
#EXT-X-PRELOAD-HINT:TYPE=PART,URI="seg_4.m4s",BYTERANGE-START=0
```

플레이어가 이 URL을 미리 GET 요청. 서버는 청크 준비되면 즉시 응답.

연결 재사용 + 청크 즉시 수신.

---

### Rendition Report

다른 화질의 진행 상황도 매니페스트에 포함.

```
#EXT-X-RENDITION-REPORT:URI="720p/playlist.m3u8",LAST-MSN=42,LAST-PART=14
#EXT-X-RENDITION-REPORT:URI="480p/playlist.m3u8",LAST-MSN=42,LAST-PART=15
```

ABR 전환 시 플레이어가 다른 화질이 현재 어디까지 만들어졌는지 알 수 있음.

전환 즉시 가장 최신 청크로 점프.

### Chunked CMAF + LL-HLS 시청자 경험

```
[실제 지연 예시 - LL-HLS 잘 튜닝]

스트리머 OBS 인코딩: 50ms
RTMP/SRT 전송: 100ms
인제스트 → 트랜스코딩: 200ms
NVENC 인코딩: 200ms
청크 패키징: 100ms
CDN 전송: 200ms
플레이어 버퍼: 1000ms
디코딩: 50ms
─────────────────────────
총 지연: ~1.9초
```

WebRTC급 지연은 아니지만 일반 HLS의 18~25초 대비 10배 개선.

---

### LL-HLS 구현 - 서버 측 요구사항

청크 단위 전송을 지원하려면 서버가:

```
1. HTTP/1.1 chunked encoding 지원
2. Long polling (Blocking Reload)
3. 청크 생성 즉시 응답
4. 매니페스트 동적 업데이트
```

일반 Nginx로는 부족. 전용 LL-HLS 서버 필요.

```
[LL-HLS 서버 옵션]
- Shaka Packager + 커스텀 Origin
- AWS Elemental MediaPackage
- Wowza Streaming Engine
- Akamai Media Services Live (CDN 내장)
- 자체 Go/Rust 구현
```

---

### CDN 호환성

LL-HLS는 CDN과 약간 까다로움.

```
[CDN 요구사항]
- HTTP/2 또는 HTTP/3 (멀티플렉싱)
- chunked 응답 캐시 가능
- 매니페스트 짧은 TTL (1초)
- byterange 요청 효율 처리
```

CloudFront, Akamai, Fastly가 LL-HLS 공식 지원. 일부 작은 CDN은 미지원.

---

### DASH의 저지연 — LL-DASH

DASH도 같은 chunked CMAF 위에 저지연 구현.

```xml
<MPD type="dynamic" 
     minimumUpdatePeriod="PT2S"
     suggestedPresentationDelay="PT2S"
     availabilityStartTime="2024-01-01T00:00:00Z">
  <Period>
    <AdaptationSet>
      <Representation>
        <ProducerReferenceTime id="0" type="encoder" wallClockTime="2024-01-01T00:00:00Z"/>
        <SegmentTemplate 
          initialization="init.mp4"
          media="seg_$Number$.m4s"
          availabilityTimeOffset="5.8"   ← 청크 단위 가용
          duration="6"/>
      </Representation>
    </AdaptationSet>
  </Period>
</MPD>
```

`availabilityTimeOffset`: 세그먼트가 만들어지는 동안 (6초) 청크 단위로 일찍 받을 수 있음.

같은 CMAF chunked 세그먼트를 HLS/DASH가 공유. 한 인코딩으로 양쪽 저지연.

---

### LL-HLS 도입 시 트레이드오프

```
[장점]
- 지연 1~3초 (일반 18~25초 대비)
- 채팅과 영상 동기화 좋음
- 라이브 경매, 스포츠 중계에 적합

[단점]
- CDN 비용 증가 (작은 청크 요청 많음)
- 모바일 배터리 소모 증가
- 약간의 안정성 손해 (작은 청크가 손실되면 영향 큼)
- 구현/운영 복잡
```

치지직 같은 라이브 플랫폼이 LL-HLS 도입하는 이유:
- 게임 방송 채팅 동기화
- 경쟁사 (Twitch)도 도입

도입 안 하는 이유:
- 비용 증가
- 기존 HLS도 시청자 만족

---

### 진짜 저지연 — WebRTC

LL-HLS의 한계: 1~3초. 그 이하는 WebRTC.

```
[지연 비교]
일반 HLS:    18~25초
LL-HLS:      1~3초
WebRTC:      100~500ms
```

WebRTC는 UDP 기반, 다른 프로토콜. 다음 레벨에서.

라이브 플랫폼의 미래는 WebRTC + LL-HLS 하이브리드. 인터랙티브는 WebRTC, 일반 시청은 LL-HLS.

### 면접 답변용

"LL-HLS 어떻게 구현하시겠어요?" 질문.

```
"Chunked CMAF가 핵심입니다.

기술 스택:
- 인코딩: NVENC + tune ll, lookahead 0
- 패키징: Shaka Packager (fragment_duration 0.2)
- Origin: 자체 Go 서버 (chunked transfer + blocking reload)
- CDN: CloudFront 또는 Akamai (HTTP/2)

핵심 옵션:
- segment_duration 6, fragment_duration 0.2 (200ms 청크)
- 매니페스트에 EXT-X-PART, EXT-X-PRELOAD-HINT
- Blocking reload로 폴링 제거

CDN 비용 30% 증가 트레이드오프 있지만,
게임 방송 채팅 동기화 효과로 시청 시간 증가.

WebRTC는 별도 트랙으로,
인터랙티브 (경매, 퀴즈)는 WebRTC,
일반 시청은 LL-HLS로 혼합 운영합니다."
```

### 3강 핵심 3줄

1. Chunked CMAF는 6초 세그먼트를 200ms 청크로 분할 전송 — HTTP/1.1 chunked encoding 활용
2. LL-HLS = chunked CMAF + EXT-X-PART + Blocking Reload + Preload Hint로 지연 1~3초 달성
3. CDN 비용 30% 증가 트레이드오프, WebRTC (100~500ms)와 LL-HLS 하이브리드가 미래 트렌드
