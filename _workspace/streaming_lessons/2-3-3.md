## 레벨 2-3 / 3강: DASH vs HLS 비교 — 어떤 차이가 실전에 영향을 미치나

---

### 산업 점유율 현실

먼저 시장 상황부터.

```
[2024년 영상 스트리밍 프로토콜 점유율 추정]
HLS:  60~65% (iOS, Safari, 호환성)
DASH: 30~35% (Chrome, Android, 표준화)
기타: 5% (자체 프로토콜, 옛날 RTMP 등)
```

DASH가 기술적으로 우수한데도 HLS가 압도적. 이유:

**1. iOS 강제**
iOS Safari가 DASH 네이티브 지원 안 함. iPhone 시청자는 무조건 HLS.

**2. 단순함**
HLS의 M3U8이 MPD보다 작고 단순. 개발자 진입 장벽 낮음.

**3. 도구 생태계**
FFmpeg, hls.js, OBS 등이 HLS 위주로 개발됨.

DASH는 정답인데 시장이 안 따라가는 케이스.

---

### 기술적 비교 — 상세 표

| 항목 | HLS | DASH |
|------|-----|------|
| 발표 | 2009 Apple | 2012 ISO/IEC |
| 매니페스트 포맷 | M3U8 (텍스트) | MPD (XML) |
| 매니페스트 크기 | 작음 (~5 KB) | 큼 (~50 KB) |
| 컨테이너 | TS, fMP4 | fMP4, WebM, TS |
| 코덱 | 제한적 (H.264, H.265 일부) | 자유 (AV1, VP9 다) |
| iOS 네이티브 | ✅ | ❌ |
| Android 네이티브 | ⭕ (제한적) | ✅ (ExoPlayer) |
| 브라우저 네이티브 | Safari만 | 없음 (JS 필요) |
| ABR 알고리즘 | 플레이어 자율 | 플레이어 자율 |
| 자막 | WebVTT | TTML, WebVTT |
| 광고 삽입 | DISCONTINUITY | Period 분리 |
| DRM | FairPlay (Apple), AES-128 | CENC (공통 암호화) |
| 라이브 지연 | 18~25초 (LL-HLS 2~5초) | 6~15초 |
| 표준화 기관 | IETF (RFC 8216) | ISO/IEC 23009-1 |

### 라이브 지연 차이의 진짜 이유

DASH가 HLS보다 라이브 지연이 짧다. 왜?

**HLS (전통)**:
```
플레이어는 .m3u8을 폴링
세그먼트가 완성된 후 .m3u8에 추가됨
플레이어가 다음 폴링 때 발견 → 다운로드
→ 평균 지연: TARGETDURATION × 3
```

**DASH**:
```
SegmentTemplate으로 URL 예측 가능
플레이어가 직접 다음 세그먼트 URL 계산
세그먼트가 가용해지자마자 요청
→ 평균 지연: TARGETDURATION × 1~2
```

DASH의 SegmentTemplate이 결정적. URL을 미리 알 수 있어서 플레이리스트 폴링 대기가 없다.

```javascript
// DASH 플레이어 의사 코드
const startTime = mpd.availabilityStartTime;
const elapsed = (Date.now() - startTime) / 1000;
const segmentNumber = Math.floor(elapsed / segmentDuration);
const nextSegmentUrl = `seg_${segmentNumber + 1}.m4s`;

// 다음 세그먼트가 가용해질 시간 계산
const nextAvailableTime = startTime + (segmentNumber + 1) * segmentDuration;
const waitTime = nextAvailableTime - Date.now();

setTimeout(() => {
  fetch(nextSegmentUrl);
}, waitTime);
```

플레이어가 시간 계산만으로 세그먼트 가용 시점 예측.

근데 LL-HLS가 나오면서 격차가 사라졌다. LL-HLS는 partial segment로 DASH보다도 빠른 지연.

### MPEG-DASH의 장점 — 광고 삽입

DASH의 Period 구조가 광고 처리에 우수.

```xml
<MPD>
  <Period id="content_part1" duration="PT15M">
    <BaseURL>https://cdn.example.com/content/</BaseURL>
    <AdaptationSet>...</AdaptationSet>
  </Period>
  
  <Period id="ad_personalized_user12345" duration="PT30S">
    <BaseURL>https://ad-server.example.com/user12345/</BaseURL>
    <AdaptationSet>...</AdaptationSet>
  </Period>
  
  <Period id="content_part2" duration="PT15M">
    <BaseURL>https://cdn.example.com/content/</BaseURL>
    <AdaptationSet>...</AdaptationSet>
  </Period>
</MPD>
```

광고 Period에 사용자별 다른 BaseURL을 줄 수 있다. 같은 콘텐츠 시청자가 각자 다른 광고를 본다.

```
시청자 A: user12345 광고 (자동차 광고)
시청자 B: user67890 광고 (게임 광고)
```

이걸 **Server-Side Ad Insertion (SSAI)** 또는 **Dynamic Ad Insertion (DAI)** 라고 한다.

HLS도 가능하지만 DISCONTINUITY 태그 + 플레이리스트 조작이 필요해 복잡.

YouTube가 DASH를 강하게 미는 이유 중 하나. 광고 비즈니스 모델에 유리.

---

### DRM 통합 — CENC

DASH는 **CENC (Common Encryption)** 표준 사용.

```xml
<ContentProtection
  schemeIdUri="urn:mpeg:dash:mp4protection:2011"
  value="cenc"
  cenc:default_KID="abc-123-..."/>

<ContentProtection
  schemeIdUri="urn:uuid:edef8ba9-79d6-4ace-a3c8-27dcd51d21ed"
  value="Widevine">
  <cenc:pssh>...</cenc:pssh>
</ContentProtection>

<ContentProtection
  schemeIdUri="urn:uuid:9a04f079-9840-4286-ab92-e65be0885f95"
  value="PlayReady">
  <cenc:pssh>...</cenc:pssh>
</ContentProtection>
```

**CENC의 핵심**: 한 번 암호화한 콘텐츠를 여러 DRM 시스템(Widevine, PlayReady)이 공유.

```
[CENC 없이]
콘텐츠를 Widevine 키로 암호화 → Chrome용
콘텐츠를 PlayReady 키로 암호화 → Edge용
→ 같은 영상을 두 번 인코딩, 저장 2배

[CENC]
콘텐츠를 한 번 암호화
Widevine 라이센스 서버: 그 키 제공
PlayReady 라이센스 서버: 그 키 제공
→ 인코딩/저장 1배
```

HLS는 FairPlay 따로, AES-128 따로. CENC 같은 통합 표준 없음. 멀티 DRM이 더 복잡.

OTT 플랫폼(넷플릭스 등)이 DASH를 선호하는 큰 이유.

### 매니페스트 크기 — 비용 차이

라이브 채널이 많을수록 매니페스트 크기가 누적된다.

```
[치지직 동시 라이브 채널 1만 개]
[각 시청자가 2초마다 매니페스트 폴링]

HLS:
- M3U8 평균 3 KB
- 시청자 100만 명 × 0.5 req/s × 3 KB = 1.5 GB/s 트래픽

DASH:
- MPD 평균 30 KB
- 같은 조건: 15 GB/s 트래픽
```

매니페스트만으로 10배 트래픽 차이. CDN 비용에 영향.

DASH는 ETag + 304 Not Modified로 갱신 트래픽 줄임. 그래도 초기 다운로드가 크다.

### 파서 복잡도 — 개발 비용

M3U8 파서:
```javascript
function parseM3U8(text) {
  const lines = text.split('\n');
  const segments = [];
  let currentDuration = null;
  
  for (const line of lines) {
    if (line.startsWith('#EXTINF:')) {
      currentDuration = parseFloat(line.split(':')[1]);
    } else if (!line.startsWith('#') && line.trim()) {
      segments.push({
        url: line.trim(),
        duration: currentDuration
      });
    }
  }
  return segments;
}
```

수십 줄로 끝. 라인 단위 파싱.

MPD 파서:
```javascript
function parseMPD(xmlText) {
  const parser = new DOMParser();
  const doc = parser.parseFromString(xmlText, 'text/xml');
  
  const mpd = doc.documentElement;
  const periods = Array.from(mpd.getElementsByTagName('Period')).map(period => {
    const adaptationSets = Array.from(period.getElementsByTagName('AdaptationSet')).map(set => {
      const representations = Array.from(set.getElementsByTagName('Representation')).map(rep => {
        // SegmentTemplate, SegmentList, SegmentBase, SegmentTimeline 각각 처리
        // BaseURL 해석
        // codecs 파싱
        // duration/timescale 처리
        // ...
      });
      return { representations, ... };
    });
    return { adaptationSets, ... };
  });
  
  return { periods, ... };
}
```

수백 줄. XML 파서 + 계층 처리 + 다양한 분기.

이게 hls.js가 dash.js보다 가벼운 이유.

### dash.js — DASH 플레이어

브라우저에서 DASH를 재생하는 표준 JS 라이브러리.

```javascript
import dashjs from 'dashjs';

const player = dashjs.MediaPlayer().create();
player.initialize(
  document.querySelector("#videoPlayer"),
  "https://example.com/manifest.mpd",
  true  // autoplay
);

// ABR 설정
player.updateSettings({
  streaming: {
    abr: {
      autoSwitchBitrate: { video: true },
      initialBitrate: { video: 3000 },
    },
    buffer: {
      bufferTimeAtTopQuality: 60,
    }
  }
});
```

내부 동작은 hls.js와 비슷.
1. MPD 다운로드 & 파싱
2. 화질 선택
3. 세그먼트 다운로드
4. MSE에 fMP4 주입

DASH는 처음부터 fMP4 기반이라 hls.js처럼 트랜스먹싱 불필요. 그 점에선 더 단순.

### Shaka Player — 양쪽 다 지원

Google이 만든 플레이어. HLS와 DASH 모두 처리.

```javascript
import shaka from 'shaka-player';

const video = document.getElementById('video');
const player = new shaka.Player(video);

// HLS도 됨
await player.load('https://example.com/master.m3u8');

// DASH도 됨
await player.load('https://example.com/manifest.mpd');
```

대형 플랫폼이 Shaka를 선호하는 이유: 코드 하나로 양쪽 처리.

YouTube가 Shaka의 주요 사용자. 자체 확장으로 광고, DRM 등 처리.

### 어떤 걸 쓸지 결정 가이드

**HLS만 쓰면 되는 경우:**
- 한국 라이브 방송 (시청자 단일 시장)
- 단순 콘텐츠 배포
- iOS 비중 높음
- 개발 리소스 적음

**DASH도 같이 쓰는 경우:**
- 글로벌 OTT 서비스
- 광고 비즈니스 (SSAI 활용)
- 멀티 DRM 필수 (Widevine + PlayReady + FairPlay)
- 다양한 디바이스 (스마트TV, 게임 콘솔)

치지직 같은 한국 라이브 플랫폼은 HLS만으로 충분. 넷플릭스, 디즈니플러스 같은 글로벌 OTT는 DASH + HLS 병행.

### CMAF가 만드는 미래

CMAF로 둘의 격차가 줄어들고 있다.

```
[전통]
HLS 세그먼트: .ts (MPEG-TS)
DASH 세그먼트: .m4s (fMP4)
→ 같은 영상을 두 번 인코딩

[CMAF 시대]
공통 세그먼트: .cmaf (fMP4)
HLS 매니페스트가 참조
DASH 매니페스트가 참조
→ 한 번 인코딩, 두 매니페스트
```

미래에는 인코더가 콘텐츠를 한 번만 만들고, 시청자 기기에 따라 매니페스트만 다르게 제공.

```
iOS 시청자 → master.m3u8 → 같은 세그먼트
Android 시청자 → manifest.mpd → 같은 세그먼트
```

저장 비용 절감, CDN 캐시 효율 향상.

---

### 3강 핵심 3줄

1. DASH가 기술적으로 우수 (코덱 자유, 광고 SSAI, CENC DRM)하지만 시장 점유율은 HLS가 60% — iOS 강제와 단순함의 승리
2. DASH의 SegmentTemplate으로 라이브 지연이 HLS보다 짧았으나, LL-HLS 등장으로 격차 사라짐
3. CMAF로 양쪽이 같은 세그먼트 파일 공유 — 매니페스트만 다르게 제공하는 미래로 수렴 중
