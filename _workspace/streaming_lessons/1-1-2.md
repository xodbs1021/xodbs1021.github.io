### HTTP 요청 하나가 어떻게 동작하는가

브라우저에서 `.ts` 세그먼트 파일 하나를 요청할 때 실제로 무슨 일이 벌어지는지부터.

```
[브라우저]                    [서버]
    │                           │
    │── GET /seg001.ts ────────→│
    │                           │ (파일 읽기)
    │←──────── 200 OK ──────────│
    │←── [데이터 1] ────────────│
    │←── [데이터 2] ────────────│
    │←── [데이터 3] ────────────│
    │         ...               │
    │←── [데이터 N] ────────────│
```

여기서 "데이터"는 하나의 덩어리로 오는 게 아니다. TCP는 데이터를 **패킷**이라는 작은 조각으로 쪼개서 보낸다. 패킷 하나의 크기는 보통 **1460바이트** (MTU 1500바이트에서 헤더 뺀 것).

1MB짜리 `.ts` 파일 하나를 전송하면:
```
1MB ÷ 1460바이트 ≈ 700개 패킷
```

이 700개 패킷이 인터넷을 통해 이동한다.

---

### 패킷이 인터넷을 여행하는 방법

서울에서 미국 서버로 패킷 하나를 보낼 때, 패킷은 한 번에 목적지에 가지 않는다. **라우터**를 거쳐서 간다.

```
내 PC → 공유기 → KT 라우터 → 해저케이블 → Equinix LA → AWS 서버
```

각 라우터는 패킷을 받아서 "다음에 어디로 보낼지" 결정한다. 이 결정이 매번 같지 않을 수 있다. 패킷 1번은 A 경로로, 패킷 2번은 B 경로로 갈 수 있다.

그러면 **도착 순서가 뒤바뀔 수 있다**.

```
보낸 순서:  [1] [2] [3] [4] [5]
도착 순서:  [1] [3] [2] [5] [4]  ← 순서 다름
```

TCP가 하는 일 중 하나가 이걸 원래 순서대로 재조립하는 거다.

### TCP의 신뢰성 보장 메커니즘

TCP가 "신뢰성 있는 전송"을 보장하는 방법:

**1. 순서 번호 (Sequence Number)**
모든 바이트에 번호를 매긴다. 수신 측은 순서 번호로 데이터를 올바른 순서로 재조립한다.

**2. ACK (확인 응답)**
데이터를 받으면 "잘 받았어"라고 응답한다.

**3. 재전송**
일정 시간 안에 ACK가 안 오면 같은 패킷을 다시 보낸다.

```
[송신]                     [수신]
  │── seq:1 ───────────────→│
  │── seq:2 ───────────────→│
  │── seq:3 ──── ❌ 손실    │
  │── seq:4 ───────────────→│
  │                          │ (seq:4 받았지만 seq:3 없으니 대기)
  │←── ACK:2 ───────────────│ (2번까지만 확인)
  │── seq:3 재전송 ─────────→│
  │                          │ (seq:3, seq:4 재조립 완료)
  │←── ACK:4 ───────────────│
```

**이게 버퍼링의 근본 원인이다.** seq:3이 손실되면 seq:4가 이미 도착해 있어도 애플리케이션에 넘겨주지 않는다. 재전송이 올 때까지 기다린다.

### 대역폭 vs 지연 — 다른 개념이다

많이 혼동하는 두 가지:

**대역폭 (Bandwidth)**: 1초에 얼마나 많은 데이터를 보낼 수 있는가
- 단위: Mbps (초당 메가비트)
- 고속도로의 차선 수

**지연 (Latency)**: 데이터가 출발지에서 목적지까지 가는 데 걸리는 시간
- 단위: ms (밀리초)
- 고속도로의 거리

```
서울 ↔ 서울:  대역폭 1Gbps, 지연 1ms
서울 ↔ 미국:  대역폭 1Gbps, 지연 150ms
```

라이브 스트리밍에서 **지연이 더 중요하다**.

대역폭이 아무리 커도 지연이 150ms면, TCP 재전송 한 번 발생할 때 최소 150ms × 2(왕복) = 300ms를 기다려야 한다.

---

### 이게 RTMP, HLS, SRT 설계에 어떻게 영향을 미쳤나

| 프로토콜 | 전송 방식 | 설계 선택 이유 |
|---------|---------|-------------|
| RTMP | TCP | 안정성 우선. 스트리머 → 플랫폼 인제스트라 지연보다 신뢰성 중요 |
| HLS | TCP(HTTP) | CDN 캐시 활용 위해 HTTP 필수. 시청자 배포용 |
| SRT | UDP + ARQ | TCP 재전송 대기 대신 짧은 타임아웃(120ms)으로 제한. 라이브 분석용 |

SRT가 "UDP인데 신뢰성이 있다"는 게 결국 이 뜻이다. TCP의 재전송을 그대로 쓰면 지연이 길어지니, 똑같은 재전송 메커니즘을 UDP 위에 얹되 타임아웃을 120ms로 짧게 잡아서 "늦게 오는 패킷은 그냥 버린다"고 결정한 거다.

---

### 오늘 2강 핵심 3줄

1. TCP는 패킷 손실 시 재전송을 기다리는데, 이 대기 시간이 버퍼링의 근본 원인이다
2. 대역폭(얼마나 많이)과 지연(얼마나 빨리)은 다른 개념 — 라이브 스트리밍은 지연이 더 중요하다
3. SRT가 UDP를 선택한 이유: TCP 재전송의 무한 대기 대신 120ms 안에 안 오면 버리는 전략
