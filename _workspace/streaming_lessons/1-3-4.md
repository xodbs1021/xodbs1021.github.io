## 레벨 1-3 / 4강: 플랫폼이 버퍼링을 측정하고 줄이는 방법

---

### 측정해야 줄일 수 있다 — QoE 지표

플랫폼 엔지니어가 보는 핵심 지표를 **QoE (Quality of Experience)** 라고 한다. 시청자가 실제로 어떤 경험을 했는지.

**1. Rebuffering Ratio (재버퍼링 비율)**
전체 시청 시간 중 멈춘 시간의 비율.

```
1시간 시청 중 버퍼링 30초 → Rebuffering Ratio = 0.83%
```

업계 표준: 1% 미만이 좋은 서비스, 3% 이상이면 문제.

**2. Startup Time (시작 지연)**
플레이 버튼 누르고 첫 프레임 나올 때까지 시간.

```
좋음: 1초 미만
보통: 2~3초
나쁨: 5초 이상 → 시청자 이탈
```

**3. Video Start Failure**
재생 시도했는데 아예 시작 못한 비율.

**4. Average Bitrate**
시청자가 실제로 받은 평균 화질. ABR이 잘 동작하는지 보는 지표.

**5. Bitrate Switch Frequency**
화질 전환 빈도. 너무 잦으면 시청자에게 거슬린다.

### 시청자 이탈과 직결되는 지표

Conviva 같은 영상 분석 회사 연구에서 나온 수치:

```
Startup Time 2초 → 시청자 이탈률 6%
Startup Time 5초 → 시청자 이탈률 25%
Rebuffering 1% 발생 → 시청 시간 5% 감소
Rebuffering 3% 발생 → 시청 시간 25% 감소
```

버퍼링은 단순히 불편한 게 아니라 **매출 직결 지표**다. 광고 매출이 시청 시간에 비례하니 버퍼링 1% 줄이는 게 수십억 매출과 직결된다.

### 클라이언트 지표 수집 — Beacon

각 플레이어가 자기 상태를 서버에 보고한다. 보고 데이터를 **Beacon**이라고 한다.

```javascript
// hls.js에서 이벤트 후킹
hls.on(Hls.Events.BUFFER_STALLED, () => {
  sendBeacon({
    event: 'rebuffer_start',
    timestamp: Date.now(),
    currentQuality: '720p',
    bufferLength: 0,
    bandwidth: estimatedBandwidth
  });
});

hls.on(Hls.Events.BUFFER_APPENDED, () => {
  sendBeacon({
    event: 'rebuffer_end',
    timestamp: Date.now(),
    bufferLength: currentBufferLength
  });
});
```

이 Beacon들이 모이면 플랫폼 엔지니어가 대시보드에서 본다.

```
[QoE 대시보드 - 치지직 예시]
지역별 Rebuffering Ratio
  서울:    0.4% ✅
  부산:    0.6% ✅
  제주:    2.1% ⚠️ ← 해저케이블 문제?
  광주:    0.5% ✅

ISP별 Startup Time
  KT:      1.2초 ✅
  SKT:     1.4초 ✅
  LGU+:    2.8초 ⚠️ ← Peering 이슈?
```


### CDN과 Origin 간 문제 분리

버퍼링이 일어나도 어디 책임인지가 다르다.

```
[시청자 → CDN 엣지]
  문제 발생 시: 시청자 ISP나 모바일 환경 문제
  플랫폼이 할 수 있는 게 적음

[CDN 엣지 → Origin]
  문제 발생 시: 플랫폼 잘못
  - CDN 캐시 미스율 높음
  - Origin 서버 응답 느림
  - 인코딩 지연
```

플랫폼은 두 구간을 **분리해서 측정**한다.

```
[CDN Hit Rate]
  95% 이상: 정상
  80% 이하: 인기 콘텐츠가 CDN에 없음 → 캐시 정책 조정

[Origin Response Time]
  100ms 이하: 정상
  500ms 이상: Origin 부하 → 스케일 아웃
```

### 사전 대응 — Pre-positioning

인기 스트리머 방송 시작 전에 CDN을 미리 워밍업하는 기법.

```
[방송 시작 5분 전]
  CDN 엣지 서버들이 빈 세그먼트 미리 요청
  → 캐시 히트율 향상
  → 방송 시작 직후 Thundering Herd 방지
```

치지직 인기 스트리머는 방송 시간이 예측 가능하니, 그 시간 5분 전부터 CDN 캐시를 준비한다.

---

### A/B 테스트 — 정책 변경 효과 측정

ABR 알고리즘이나 버퍼 정책을 바꾸려면 A/B 테스트가 필수다.

```
[실험 설계]
사용자 A 그룹 (50%): 버퍼 길이 18초
사용자 B 그룹 (50%): 버퍼 길이 12초

7일간 측정:
  A: Rebuffering 0.5%, 라이브 지연 22초
  B: Rebuffering 0.8%, 라이브 지연 16초

결정: A 유지 (Rebuffering이 더 중요)
```

라이브 지연을 줄이고 싶지만 버퍼링이 같이 늘면 손해다. 두 지표의 균형점을 데이터로 찾는다.

---

### Origin Shield — 중간 캐시 레이어

CDN 엣지 → Origin 사이에 한 단계 더 둔다.

```
시청자 → 엣지 (지역별 100개) → Shield (5개) → Origin (1개)
```

Shield 없을 때: 새 세그먼트 요청이 100개 엣지에서 동시에 Origin으로 → 100배 부하.
Shield 있을 때: 100개 엣지가 5개 Shield로 → 5개 Shield가 Origin으로 → 5배 부하.

**Cascading Cache**. 라이브 인프라에서 필수.

---

### 4강 핵심 3줄

1. QoE 지표 (Rebuffering Ratio, Startup Time)는 매출 직결 — 1% 버퍼링 = 시청시간 5% 감소
2. 플레이어 Beacon → Prometheus + Grafana 대시보드가 백엔드 라이브 인프라의 핵심 업무
3. Pre-positioning, Origin Shield, A/B 테스트 같은 운영 기법으로 버퍼링을 조직적으로 줄임
