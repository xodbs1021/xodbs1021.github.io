## 레벨 2-6 / 3강: 미디어 서버 — WebRTC를 다인 통화로 확장

---

### P2P의 한계 — 인원이 많아지면

WebRTC는 기본적으로 P2P. 2명 통화는 완벽하다. 하지만 10명, 100명은?

**Mesh 방식 (순수 P2P)**:
모든 참가자가 서로와 직접 연결.

```
[5명 Mesh 통화]
A ↔ B, A ↔ C, A ↔ D, A ↔ E
B ↔ C, B ↔ D, B ↔ E
C ↔ D, C ↔ E
D ↔ E

연결 수: 5 × 4 / 2 = 10개
각자 4명에게 송신, 4명에게서 수신
```

각 클라이언트가 자기 영상을 4번 인코딩해서 보낸다. 업로드 대역폭 4배. 모바일에선 즉시 한계.

100명이면? 99번 송신, 99번 수신. 불가능.

---

### 서버 기반 솔루션 — 두 가지 방식

확장하려면 서버가 필요하다. 두 가지 아키텍처.

**MCU (Multipoint Control Unit)**:
서버가 모든 영상을 받아서 **하나로 합쳐서** 보냄.

```
[MCU]
A → 서버 (영상)
B → 서버 (영상)
C → 서버 (영상)

서버: 3개 영상을 합성 → 모자이크 1개
서버 → A: 모자이크
서버 → B: 모자이크
서버 → C: 모자이크
```

장점: 클라이언트는 1 send, 1 receive. 모바일 친화적.
단점: 서버에서 디코딩 + 합성 + 인코딩. CPU/GPU 매우 큼.

**SFU (Selective Forwarding Unit)**:
서버가 영상을 받아서 **그대로 다른 참가자들에게 전달**.

```
[SFU]
A → 서버 (영상)
B → 서버 (영상)
C → 서버 (영상)

서버 → A: B 영상 + C 영상 (2개 따로)
서버 → B: A 영상 + C 영상
서버 → C: A 영상 + B 영상

서버: 라우팅만, 디코딩 안 함
```

장점: 서버 부하 적음 (트랜스코딩 없음).
단점: 클라이언트가 N-1개 영상 수신. 다운로드 대역폭 큼.

현대 대부분의 솔루션은 **SFU**. 서버 비용이 결정적.

### SFU의 동작 흐름

SFU 내부에서 일어나는 일.

```
[SFU 내부]
참가자 A 송신:
  WebRTC 연결 → SRTP 패킷 → SFU 수신
  
SFU 라우팅:
  패킷 받음 → 헤더만 보고 어느 참가자 영상인지 식별
  → 다른 참가자들의 송신 큐에 복사
  → 각자에게 SRTP로 전송

서버 자원:
  - CPU: 거의 안 씀 (라우팅만)
  - 메모리: 참가자별 큐
  - 네트워크: 송신 + 수신 트래픽 모두
```

서버는 미디어 페이로드를 **건드리지 않는다**. SRTP 암호화된 채로 그대로 전달.

근데 진짜 그대로면 안 된다. RTP 헤더의 SSRC, sequence number는 SFU가 재작성해야 함.

```
[SFU의 RTP 헤더 처리]
원본:    SSRC=A_camera, seq=100
SFU 처리: SSRC=A_camera_to_B (새로 할당), seq=100
SFU → B:  SSRC=A_camera_to_B, seq=100

→ B는 A의 SSRC를 보고 "A 영상이구나" 인식
```

### 유명 SFU 오픈소스

**Janus Gateway**:
가장 오래된 WebRTC SFU. C로 구현. 플러그인 구조.

**Jitsi Videobridge**:
오픈소스 화상 회의의 백본. Jitsi Meet에서 사용. Java.

**mediasoup**:
Node.js 기반 SFU 라이브러리. 임베드해서 직접 회의 시스템 만들 때 사용.

**LiveKit**:
Go로 만든 WebRTC SFU + SDK. 클라우드 서비스도 제공.

**Pion**:
Go WebRTC 라이브러리. SFU 만들 때 빌딩 블록으로 사용.

### LiveKit으로 SFU 사용 예시

LiveKit이 가장 접근하기 쉽다.

서버 설치:
```yaml
# docker-compose.yml
services:
  livekit:
    image: livekit/livekit-server:latest
    command: --dev --bind 0.0.0.0
    ports:
      - "7880:7880"
      - "7881:7881"
      - "7882:7882/udp"
```

클라이언트:
```javascript
import { Room, RoomEvent } from 'livekit-client';

const room = new Room();

await room.connect('wss://livekit.example.com', token);

// 카메라/마이크 활성화
await room.localParticipant.enableCameraAndMicrophone();

// 다른 참가자 입장
room.on(RoomEvent.ParticipantConnected, (participant) => {
  console.log('Joined:', participant.identity);
});

// 다른 참가자 트랙 수신
room.on(RoomEvent.TrackSubscribed, (track, publication, participant) => {
  if (track.kind === 'video') {
    const element = track.attach();
    document.body.appendChild(element);
  }
});
```

P2P 코드를 직접 안 짜도 된다. LiveKit이 SFU 처리.

### Simulcast — 화질을 여러 개로

다인 통화에서 각자 네트워크 환경이 다르다.

```
[참가자 환경]
- A: 1Gbps 광랜, 4K 모니터
- B: 100Mbps, 1080p 모니터
- C: 5Mbps LTE, 폰

A의 1080p 영상을:
- A → SFU 1080p 송신
- SFU → B: 1080p 전달 (OK)
- SFU → C: 1080p 전달 → C 버퍼링!
```

해결: **Simulcast**. A가 여러 화질을 동시에 보냄.

```
[Simulcast]
A → SFU: 1080p + 720p + 360p (동시에 3개)

SFU 분배:
- SFU → B: 1080p (B 네트워크 좋음)
- SFU → C: 360p (C 네트워크 나쁨)
```

A의 업로드 대역폭은 늘지만 다른 참가자 경험이 좋아진다.

`a=ssrc-group:SIM` SDP 속성으로 표현.

### SVC — Scalable Video Coding

Simulcast의 진화형. 한 비트스트림에 여러 화질 계층 내장.

```
[SVC 비트스트림 구조]
Layer 0 (Base):     360p 30fps  (기본)
Layer 1 (Enhanced): + 720p 데이터 (Layer 0과 합치면 720p)
Layer 2 (Top):      + 1080p 데이터 (모두 합치면 1080p)
```

SFU가 Layer 골라서 전달:
```
- B에게: Layer 0 + 1 + 2 → 1080p 재구성
- C에게: Layer 0만 → 360p
```

Simulcast보다 효율적. 하지만 인코더/디코더 복잡도 증가. VP9, AV1에 SVC 내장.

### Bandwidth Estimation — 실시간 대역폭 추정

WebRTC는 네트워크 상태를 실시간 추정한다.

**REMB (Receiver Estimated Maximum Bitrate)**:
수신측이 "이만큼만 보내" 알림.

**Transport-CC**:
모든 패킷 도착 시각을 송신측에 보고. 송신측이 정확한 네트워크 상태 추정.

```
[Transport-CC 흐름]
A → B: 패킷 100, 101, 102
B → A: "100: 도착 12:00:00.010, 101: 12:00:00.025, 102: 12:00:00.050"

A 분석:
- 100과 101 사이: 15ms (정상)
- 101과 102 사이: 25ms (지연 증가!)
- 네트워크 혼잡 시작 → 비트레이트 줄임
```

송신측이 알아서 비트레이트 조절. 시뮬캐스트 화질 선택도 같이.

### Selective Forwarding 알고리즘

SFU가 어떤 화질을 누구에게 보낼지 결정.

```
[Subscriber 측 신호]
B: "내 다운로드 대역폭 5 Mbps, 화면 작음"
C: "내 다운로드 대역폭 1 Mbps, 폰 화면"

[SFU 결정]
A의 영상을:
- B에게 → 1080p 1.5Mbps 전달 (마진 충분)
- C에게 → 360p 0.5Mbps 전달

매번 재평가:
B의 대역폭 떨어지면 → 720p로 자동 전환
```

이게 화상 회의의 핵심 알고리즘. 좋은 SFU는 이걸 잘한다.

### Audio 처리 — 다인 통화의 어려움

영상은 selective forwarding이 쉽지만 오디오는 다르다.

10명이 동시에 말하면? 모든 오디오를 합쳐서 들려줘야 한다.

**해결책 1: 클라이언트 측 믹싱**
모든 참가자 오디오를 받아서 클라이언트가 합침.
- 10명 = 10개 오디오 스트림 다운로드
- 대역폭 증가

**해결책 2: 서버 측 믹싱 (MCU)**
서버가 오디오만 믹싱.
- 영상은 SFU, 오디오는 MCU 하이브리드
- 대부분의 큰 회의 시스템

**해결책 3: Loudest N**
가장 큰 소리 N개만 전송.
- 10명 중 2명만 실제로 말하면 그 2개만
- 자연스럽고 효율적

대부분의 SFU가 Loudest N 또는 하이브리드.

### Audio Level Detection

누가 말하고 있는지 감지.

```
[Audio Level RTP 헤더 확장]
extension: ssrc-audio-level
값: 0~127 (큰 소리일수록 작은 값)
```

각 RTP 패킷에 오디오 크기 정보. SFU가 이걸 보고 화면 우선순위 결정.

```
[화상 회의 화면]
가장 큰 소리: 메인 화면 (대형)
다음 큰 소리: 사이드 화면
조용한 사람: 작은 썸네일
```

Zoom의 "발언자 보기"가 이 메커니즘.

### SFU 확장성 — 대규모 처리

큰 회의 (수백 명)나 라이브 방송에 SFU 적용 시 문제.

```
[대규모 SFU]
1000명 통화:
- 각자 송신: 1000 × 1.5Mbps = 1.5 Gbps 수신
- 각자 수신: 1000 × 999 × 1.5Mbps = 1.5 Tbps 송신
- → 서버 하나로 불가능
```

해결: **Cascading SFU**. 서버 여러 대 묶음.

```
[Cascading]
지역별 SFU:
- 서울 SFU (한국 참가자 100명)
- 도쿄 SFU (일본 참가자 50명)
- LA SFU (미국 참가자 30명)

SFU 간 연결:
서울 ↔ 도쿄 ↔ LA

한국 참가자가 미국 참가자 영상 보기:
LA SFU → 서울 SFU → 한국 참가자
```

이게 Google Meet, Zoom의 대규모 회의 구조.

### Recording — 회의 녹화

SFU가 모든 미디어를 받으니 녹화도 가능.

```
[SFU 녹화]
모든 참가자 영상 수신
→ MCU처럼 합성 (모자이크) 또는
→ 개별 영상 + 메타데이터로 저장 (선택적 편집 가능)
→ MP4 또는 HLS로 저장
```

LiveKit, Jitsi 모두 녹화 기능 내장.

### WebRTC를 라이브 방송에 쓰려면

WebRTC 자체로 100만 시청자에 라이브 송출은 불가능.

조합 솔루션:

```
[스트리머]
  WebRTC → 미디어 서버
              ↓
       MCU/SFU가 받음
              ↓
       내부에서 HLS로 트랜스코딩
              ↓
       CDN으로 시청자에 배포
```

스트리머 측만 WebRTC. 시청자 측은 HLS. 스트리머의 저지연 송출 + CDN 확장성.

이게 다음 강의에서 다룰 **WHIP/WHEP**의 아이디어.

### 3강 핵심 3줄

1. Mesh P2P는 5명 이상 불가능 — MCU(서버 합성, CPU 큼)와 SFU(서버 라우팅만, 표준) 두 가지 서버 아키텍처
2. Simulcast/SVC로 화질 여러 개 동시 송신, SFU가 수신자별 적절한 화질 선택 — Transport-CC로 대역폭 추정
3. LiveKit, Janus, mediasoup 같은 SFU 오픈소스 활용, 1000명 이상 회의는 Cascading SFU로 지역 분산
