## 레벨 1-2 / 3강: ABR — 화질을 자동으로 바꾸는 메커니즘

---

### 문제: 시청자의 네트워크는 항상 변한다

집 와이파이로 시작 → 지하철 진입 → 4G → 다시 와이파이.

비트레이트 6 Mbps 영상을 받고 있는데 네트워크가 1 Mbps로 떨어지면? 6 Mbps 영상을 받으려고 계속 시도하다가 버퍼링이 난다.

**ABR (Adaptive Bitrate Streaming)**: 네트워크 상황에 따라 자동으로 화질을 바꾸는 메커니즘.

---

### 인코딩 단계: ABR Ladder

플랫폼이 같은 영상을 여러 화질로 미리 인코딩해둔다.

```
원본: 1080p60 6 Mbps
        ↓ 트랜스코딩
├── 1080p60 6,000 Kbps
├── 720p60  4,500 Kbps
├── 720p30  3,000 Kbps
├── 480p30  1,500 Kbps
├── 360p30  800 Kbps
└── 240p30  400 Kbps
```

이 화질 목록이 **ABR Ladder (사다리)**다. 사다리를 어떻게 설계하느냐가 플랫폼 엔지니어의 핵심 역량이다.

너무 촘촘하면 인코딩 비용이 폭증한다 (각 화질마다 인코더 한 대씩 필요). 너무 듬성하면 시청자 경험이 떨어진다 (네트워크 1.5 Mbps인데 1,500 Kbps와 3,000 Kbps 중간이 없으면 1,500을 받게 됨).

치지직 같은 한국 플랫폼은 1080p / 720p / 480p / 360p 4단계가 일반적이다.

---

### Master Playlist — 사다리의 목차

HLS에서 ABR을 어떻게 표현하나? `.m3u8` 파일 하나가 더 있다.

```
master.m3u8 (마스터 플레이리스트)
├── playlist_1080p.m3u8  (BANDWIDTH=6500000)
├── playlist_720p.m3u8   (BANDWIDTH=4800000)
├── playlist_480p.m3u8   (BANDWIDTH=1600000)
└── playlist_360p.m3u8   (BANDWIDTH=900000)
```

각 화질은 자기만의 세그먼트 목록을 가진다.

```
playlist_720p.m3u8:
├── seg001_720p.ts
├── seg002_720p.ts
├── seg003_720p.ts
...

playlist_480p.m3u8:
├── seg001_480p.ts
├── seg002_480p.ts
├── seg003_480p.ts
...
```

플레이어는 처음에 master.m3u8을 받고, 현재 적합한 화질의 sub playlist를 골라서 본다.

---

### 화질 전환 결정 — 플레이어가 한다

여기가 중요하다. **화질을 바꿀지 말지는 서버가 아니라 플레이어가 결정한다.**

플레이어가 측정하는 것:

1. **다운로드 속도**: 직전 세그먼트를 몇 초만에 받았는지
2. **버퍼 상태**: 현재 버퍼에 몇 초치 데이터가 있는지

```
[직전 세그먼트 다운로드]
  - 크기: 6 MB
  - 받는 데 걸린 시간: 2초
  → 추정 대역폭: 6 MB / 2초 = 24 Mbps (충분)

[버퍼 상태]
  - 현재 버퍼: 25초 (충분)
  
→ 다음 세그먼트는 1080p 유지
```

```
[직전 세그먼트 다운로드]
  - 크기: 4.5 MB
  - 받는 데 걸린 시간: 8초
  → 추정 대역폭: 4.5 Mbps → 6 Mbps 못 받음

[버퍼 상태]
  - 현재 버퍼: 5초 (위험)
  
→ 즉시 480p로 다운그레이드
```

---

### ABR 알고리즘 — 단순 대역폭만으로 부족하다

순진하게 대역폭만 보고 결정하면 문제가 생긴다.

**Throughput-based 알고리즘**: 대역폭만 본다.
- 문제: 와이파이 신호가 잠깐 튀어서 30 Mbps 측정 → 1080p로 올림 → 다시 떨어짐 → 다운그레이드. **반복적인 화질 변동**이 생긴다.

**Buffer-based 알고리즘**: 버퍼 상태도 본다.
- 버퍼가 30초 차 있으면 → 잠깐 대역폭 떨어져도 화질 유지 (재전송 기회 있음)
- 버퍼가 5초만 차 있으면 → 무조건 안전한 낮은 화질로

**Hybrid (BOLA, MPC 등)**: 둘 다 본다. 현대 플레이어들은 모두 하이브리드를 쓴다.

hls.js, Shaka Player 같은 라이브러리들이 자체 ABR 알고리즘을 구현한다. 라이브러리를 고르는 것 자체가 ABR 정책을 고르는 거다.

---

### 화질 전환 시 끊김 — 왜 안 끊기나

플레이어가 1080p에서 480p로 바꿀 때 영상이 안 끊긴다. 비밀이 있다.

**모든 화질 사다리의 세그먼트 경계가 같은 위치에 있다.**

```
1080p: seg001(0~6초) | seg002(6~12초) | seg003(12~18초)
720p:  seg001(0~6초) | seg002(6~12초) | seg003(12~18초)
480p:  seg001(0~6초) | seg002(6~12초) | seg003(12~18초)
```

6초 시점에 화질을 바꾸면, 1080p seg001을 다 본 다음 480p seg002를 받아서 이어 붙인다. 시점은 같으니 끊김 없이 자연스럽게 이어진다.

조건: **모든 화질의 GOP 경계가 정렬되어 있어야 한다.** I-frame이 같은 시점에 찍혀야 한다.

FFmpeg로 ABR ladder를 만들 때 `-g`(GOP 크기)와 `-force_key_frames`를 정확히 맞추는 이유가 이거다. 안 맞으면 화질 전환 시 깨진다.

```bash
# 1080p
ffmpeg -i input -c:v libx264 -b:v 6000k -g 60 -keyint_min 60 \
  -force_key_frames "expr:gte(t,n_forced*2)" 1080p.m3u8

# 720p — GOP 설정 동일하게
ffmpeg -i input -c:v libx264 -b:v 4500k -g 60 -keyint_min 60 \
  -force_key_frames "expr:gte(t,n_forced*2)" 720p.m3u8
```

### CMAF — ABR 효율을 한 단계 끌어올린 포맷

LL-HLS 공부할 때 fMP4 봤지? 그게 CMAF다.

기존 ABR의 비효율: HLS는 `.ts`, DASH는 `.mp4`. 같은 영상을 두 포맷으로 따로 인코딩해야 하면 인코더와 스토리지가 2배 든다.

CMAF: HLS와 DASH 모두 같은 fMP4 세그먼트를 쓸 수 있게 표준화한 컨테이너.

```
이전:
├── HLS용:  seg001.ts, seg002.ts, ...
└── DASH용: seg001.m4s, seg002.m4s, ...   ← 별도 인코딩

CMAF:
└── 공통:   seg001.cmaf, seg002.cmaf, ...  ← 하나로 통합
    ├── HLS 플레이리스트가 이걸 가리킴
    └── DASH 매니페스트가 이걸 가리킴
```

스토리지 절약, 인코딩 비용 절감, CDN 캐시 효율 향상. 멘토 피드백에 "CMAF 패키징"이 있던 게 이거다.

---

### 3강 핵심 3줄

1. ABR Ladder는 여러 화질을 미리 인코딩해두고 마스터 플레이리스트로 묶는 구조 — 사다리 설계가 비용과 시청자 경험의 트레이드오프
2. 화질 전환은 플레이어가 결정 — 대역폭과 버퍼 상태를 함께 보는 하이브리드 알고리즘이 표준
3. 모든 화질의 GOP 경계가 정렬되어야 화질 전환 시 끊김이 없다 — CMAF는 HLS/DASH 공통 포맷으로 비용을 절감하는 표준
