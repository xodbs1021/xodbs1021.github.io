## 레벨 4-1 / 2강: 입력/출력 옵션 — 컨테이너와 스트림 다루기

---

### 입력 옵션이 따로 있는 이유

FFmpeg는 입력 파일을 처리하기 전에 컨테이너를 어떻게 읽을지 미리 알아야 한다.

```bash
# 입력 옵션 (-i 앞)
ffmpeg \
  -re \                        # 실시간 속도로 읽기
  -fflags +genpts \            # PTS 자동 생성
  -i input.mp4 \
  -c copy output.flv
```

입력 옵션은 demuxer 동작을 바꾼다. 출력 옵션은 muxer/encoder를 바꾼다.

---

### -re — 실시간 속도로 입력

기본 FFmpeg는 입력을 최대 속도로 읽는다. 1시간 영상을 5분에 다 읽는다. 라이브 송출 테스트할 때 문제가 된다.

```bash
# 파일을 라이브처럼 보내기
ffmpeg -re -i sample.mp4 \
  -c copy \
  -f flv rtmp://localhost/live/test
```

`-re` 없으면 1시간 영상을 5분 만에 RTMP 서버에 다 밀어넣는다. 서버가 죽거나 비정상 동작.

`-re` 붙이면 1초어치 데이터를 1초마다 보낸다. 진짜 라이브 송출처럼.

---

### -stream_loop — 무한 반복 입력

테스트용 24/7 라이브 채널 만들 때.

```bash
# 영상을 무한 반복해서 RTMP로 송출
ffmpeg -re -stream_loop -1 \
  -i loop.mp4 \
  -c copy \
  -f flv rtmp://localhost/live/test
```

`-stream_loop -1` = 무한. `-stream_loop 3` = 3번 더 반복 (총 4번).

스테이징 환경에서 24시간 라이브 부하 테스트할 때 자주 쓴다.

---

### -ss와 -t / -to — 자르기

영상 일부만 처리.

```bash
# 60초부터 30초 동안만
ffmpeg -ss 60 -i input.mp4 -t 30 -c copy clip.mp4

# 60초부터 120초까지
ffmpeg -ss 60 -i input.mp4 -to 120 -c copy clip.mp4
```

`-t`는 지속 시간. `-to`는 끝나는 시점.

- `-ss`를 `-i` 앞: 빠른 시킹 (키프레임 단위)
- `-ss`를 `-i` 뒤: 정확한 시킹 (전체 디코딩)

```bash
# 빠른 + 정확한 시킹 조합
ffmpeg -ss 55 -i input.mp4 -ss 5 -t 30 -c:v libx264 clip.mp4
# → 55초로 빠르게 시킹 후, 정확히 5초 더 (총 60초 지점)
```

VOD 클립 뽑기, 썸네일 추출 등에 쓴다.

---

### 컨테이너 강제 지정 — -f

확장자가 없거나 잘못된 경우.

```bash
# 출력 컨테이너 명시
ffmpeg -i input.mp4 -c copy -f mpegts output.ts

# 표준 출력으로 (파일 안 만들고 파이프로)
ffmpeg -i input.mp4 -c copy -f mpegts - | ...

# RTMP로 보내기
ffmpeg -re -i input.mp4 -c copy -f flv rtmp://server/live/key
```

RTMP는 컨테이너 이름이 `flv`다. 헷갈리는 부분.

| 프로토콜 | -f 값 |
|---------|------|
| RTMP | flv |
| HLS | hls |
| DASH | dash |
| MPEG-TS over UDP | mpegts |
| SRT | mpegts (SRT는 컨테이너로 ts 사용) |

---

### 스트림 ID 명명 규칙

`-map 0:v:0`처럼 스트림을 지정할 때 ID 체계.

```
0       = 첫 번째 입력 파일
0:v     = 그 파일의 모든 비디오 스트림
0:v:0   = 그 파일의 비디오 스트림 0번
0:a     = 모든 오디오 스트림
0:a:1   = 오디오 스트림 1번 (두 번째)
0:s     = 자막 스트림
0:d     = 데이터 스트림
```

다중 입력 파일.

```bash
# 두 파일 합성: 첫 번째에서 비디오, 두 번째에서 오디오
ffmpeg \
  -i video.mp4 \
  -i audio.aac \
  -map 0:v:0 -map 1:a:0 \
  -c copy output.mp4
```

음원 교체, 다국어 더빙 합성 등에 쓴다.

---

### -map_metadata — 메타데이터 복사

기본은 입력 메타데이터를 그대로 출력에 복사한다. 끄거나 다른 파일에서 가져올 수 있다.

```bash
# 메타데이터 다 버리기
ffmpeg -i input.mp4 -map_metadata -1 -c copy clean.mp4

# 다른 파일에서 메타데이터 가져오기
ffmpeg -i video.mp4 -i meta.mp4 -map_metadata 1 -c copy output.mp4
```

라이브 인제스트에서 보통 메타데이터 정리하고 다시 박는다. 스트리머가 OBS에 박은 비표준 태그가 CDN에서 문제 될 수 있어서.

---

### Duration / Timestamp 조작

PTS(Presentation Timestamp) 문제는 라이브에서 자주 만난다.

```bash
# PTS 강제 재생성 (TS 파일이 깨졌을 때)
ffmpeg -fflags +genpts -i broken.ts -c copy fixed.ts

# 첫 PTS를 0으로 리셋
ffmpeg -i input.ts -reset_timestamps 1 -c copy reset.ts

# DTS 누락 무시
ffmpeg -fflags +genpts+igndts -i input.ts -c copy output.ts
```

RTMP/MPEG-TS 받다가 가끔 PTS가 음수거나 점프하는 경우가 있다. 인제스트 단에서 정리해주지 않으면 CDN에서 깨진다.

---

### 무손실 트랜스먹싱 — 라이브 인제스트 패턴

RTMP로 들어온 스트림을 HLS로 그대로 패키징.

```bash
ffmpeg \
  -fflags +genpts \
  -i rtmp://localhost/live/streamkey \
  -c copy \
  -f hls \
  -hls_time 6 \
  -hls_list_size 5 \
  -hls_flags delete_segments+independent_segments \
  -hls_segment_filename 'seg%05d.ts' \
  /var/www/hls/playlist.m3u8
```

- `-c copy`: 재인코딩 없음. CPU 거의 안 씀
- `-hls_time 6`: 6초 세그먼트
- `-hls_list_size 5`: 플레이리스트에 5개 유지 (sliding window)
- `delete_segments`: 오래된 .ts 자동 삭제
- `independent_segments`: 각 세그먼트가 키프레임으로 시작

라이브 인제스트의 가장 기본 명령. 인코딩 부하 없이 RTMP → HLS만 변환한다.

---

### 한 입력에서 여러 출력 동시 생성

ABR ladder의 핵심 패턴.

```bash
ffmpeg -i input.mp4 \
  -c:v libx264 -preset veryfast \
  \
  -map 0:v -map 0:a -s 1920x1080 -b:v 6000k -f hls hls_1080.m3u8 \
  -map 0:v -map 0:a -s 1280x720  -b:v 3500k -f hls hls_720.m3u8 \
  -map 0:v -map 0:a -s 854x480   -b:v 1500k -f hls hls_480.m3u8
```

한 번 디코딩 → 세 번 인코딩. 디코딩은 한 번이라 CPU 절약.

근데 이게 라이브에서 진짜 빠른 게 아니다. 디코딩 한 번이 절약되긴 하지만 인코딩 3번이 직렬로 돌면 결국 1080p보다 3배 느림. NVENC 같은 GPU 인코더 쓸 때 진가 발휘.

---

### -shortest — 가장 짧은 입력에 맞춰 종료

```bash
# 음악(긴 것) + 정지 이미지(짧은 것) 합성
ffmpeg \
  -loop 1 -i thumbnail.jpg \
  -i music.mp3 \
  -shortest \
  -c:v libx264 -c:a copy \
  podcast.mp4
```

이미지를 `-loop 1`로 무한 반복하게 하고, 음악 끝나면 `-shortest`로 같이 끝낸다.

팟캐스트 영상화, 음원 게시판용 영상 만들 때 표준 패턴.

---

### 표준 입출력 파이프

FFmpeg를 다른 도구와 연결할 때.

```bash
# 입력을 stdin에서 받기
cat input.mp4 | ffmpeg -i pipe:0 -c copy output.mp4

# 출력을 stdout으로
ffmpeg -i input.mp4 -c copy -f mpegts pipe:1 | nc server 9000

# 두 ffmpeg 연결 (트랜스코딩 분산)
ffmpeg -i input.mp4 -c:v libx264 -f mpegts - | \
  ffmpeg -i - -c copy output.ts
```

분산 트랜스코딩 파이프라인에서 자주 쓴다. 한 노드에서 디코딩하고 다른 노드에서 인코딩 같은 패턴.

---

### 라이브에서 자주 만나는 입력 문제

**1. 비표준 코덱 들어옴**
스트리머가 H.264 아닌 걸로 송출.

```bash
# 들어온 코덱 확인 후 재인코딩 결정
ffprobe -v error -select_streams v:0 \
  -show_entries stream=codec_name input.flv
# h264 → -c copy, 그 외 → libx264
```

**2. 오디오 샘플레이트 비표준**
44100이 아닌 48000 같은 거. HLS 표준은 48000.

```bash
# 오디오 강제 리샘플
ffmpeg -i input.flv -c:v copy -ar 48000 -c:a aac output.ts
```

**3. PTS 비정상**
첫 패킷 PTS가 음수거나 너무 큰 값.

```bash
ffmpeg -fflags +genpts -i input.flv ...
```

라이브 인제스트 서버는 이 케이스들을 자동 감지해서 처리한다.

### 2강 핵심 3줄

1. 입력 옵션은 `-i` 앞, 출력 옵션은 뒤 — `-re` `-stream_loop` `-ss` 같은 핵심 옵션 위치가 동작을 바꿈
2. `-c copy`로 재인코딩 없이 컨테이너만 변환하는 게 라이브 인제스트의 기본 패턴 (HLS 패키징)
3. 한 명령으로 여러 출력 (ABR ladder)이 가능하고, ffprobe로 입력 분석 후 분기 처리가 인제스트 서버의 표준 구조
