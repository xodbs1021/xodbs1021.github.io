## 레벨 2-2 / 1강: HLS의 탄생 배경과 핵심 아이디어

---

### 2009년 — Apple의 고민

iPhone OS 3.0 출시 시점. Apple은 큰 문제를 안고 있었다.

```
[2009년 모바일 영상 환경]
- iPhone 3G/3GS가 막 보급
- 모바일 네트워크: 3G (1~3 Mbps, 매우 불안정)
- 기존 영상 표준: RTMP (Flash 기반)
- 그런데 Apple은 iOS에서 Flash를 안 씀
```

iOS에서 영상을 어떻게 보여줄 것인가? 두 가지 문제가 동시에 있었다.

**1. Flash 없이 영상 재생**
HTML5 video 태그가 있긴 했지만 단순 다운로드 방식. 라이브 불가능.

**2. 불안정한 모바일 네트워크**
3G가 자주 끊긴다. 지하철 들어가면 신호 없어진다. 기존 RTMP는 끊기면 처음부터 다시.

Apple이 자체 표준을 만들기로 결정. 2009년 HLS 발표.

---

### HLS의 발상 — "그냥 HTTP로 작은 파일들 다운받자"

기존 스트리밍 프로토콜들이 복잡한 이유는 **연결을 유지**하기 때문이었다.

```
[RTMP 방식]
서버 ──지속 연결── 클라이언트
       (영상 데이터 흐름)
```

연결을 유지하려면 서버가 클라이언트 상태를 알아야 한다. 끊기면 재연결 절차 필요. 서버 비용 비싸다.

HLS의 아이디어:

```
[HLS 방식]
서버 ────HTTP GET────▶ 클라이언트 (6초 파일)
        (응답 후 끊김)
서버 ────HTTP GET────▶ 클라이언트 (다음 6초 파일)
        (응답 후 끊김)
서버 ────HTTP GET────▶ 클라이언트 (다음 6초 파일)
        ...
```

영상을 작은 파일들로 잘라서 HTTP로 하나씩 요청. 서버는 그냥 정적 파일 호스팅 서버면 된다.

미친 발상이었다. "라이브 영상도 그냥 파일이다."

---

### HLS가 해결한 문제들

이 단순한 발상이 동시에 여러 문제를 해결했다.

**1. CDN 자연스럽게 활용**
HTTP 정적 파일이니 모든 CDN이 그대로 캐시 가능. RTMP는 못 했던 게 공짜로 됨.

**2. 방화벽 통과**
80/443 포트 사용. RTMP의 1935 차단 문제 없음.

**3. 화질 자동 전환**
파일이 작게 잘라져 있으니 다음 파일부터 다른 화질로 요청 가능. ABR이 쉬워짐.

**4. 라이브와 VOD 동일 구조**
라이브는 파일이 계속 추가되고, VOD는 끝이 정해진 것. 같은 메커니즘.

**5. 끊김 복구**
네트워크 끊겨도 HTTP 재요청만 하면 됨. 복잡한 재연결 프로토콜 불필요.

---

### HLS의 두 가지 파일 — .m3u8과 .ts

HLS는 두 종류 파일로 구성된다.

**`.m3u8` (Playlist)**: 목차 역할. 어떤 세그먼트가 있는지 나열.

```
#EXTM3U
#EXT-X-VERSION:3
#EXT-X-TARGETDURATION:6
#EXT-X-MEDIA-SEQUENCE:0
#EXTINF:6.000,
seg000.ts
#EXTINF:6.000,
seg001.ts
#EXTINF:6.000,
seg002.ts
#EXT-X-ENDLIST
```

**`.ts` (Transport Stream)**: 실제 영상 데이터. 6초 분량.

`.m3u8`는 텍스트, `.ts`는 바이너리. 텍스트 목차 + 바이너리 데이터 조합.

---

### 왜 하필 MPEG-TS 컨테이너?

HLS가 등장했을 때 영상 컨테이너는 여러 개가 있었다. MP4, FLV, MPEG-TS 등.

Apple이 MPEG-TS를 선택한 이유:

**1. 손실 복구**
이미 공부한 188바이트 패킷 + sync byte. 모바일 네트워크에 강함.

**2. 방송 표준**
방송 장비들이 이미 MPEG-TS를 뱉음. 변환 없이 그대로 활용.

**3. 멀티 트랙**
PID 구조로 여러 트랙 담기 가능 (다국어 자막 등).

**4. 라이브 친화적**
파일 끝을 모르고도 스트리밍 시작 가능. MP4는 메타데이터가 끝에 있어야 해서 라이브에 불리.

근데 LL-HLS에 와서는 MP4 기반 fMP4로 바뀌었다. 이건 LL-HLS 공부할 때 본 내용.

### HLS가 산업 표준이 된 흐름

```
2009: Apple HLS 발표 (iOS 3.0)
2010: Safari 데스크탑 지원
2011: hls.js 같은 JS 플레이어 등장
2014: 트위치가 RTMP 시청자 측을 HLS로 전환
2015: 유튜브 라이브 HLS 채택
2017: RFC 8216으로 IETF 표준화
2019: LL-HLS 발표
현재: 라이브 시청자 측 사실상 100% HLS
```

Apple이 만든 폐쇄적 프로토콜이 어떻게 산업 표준이 됐냐:

**1. iOS의 시장 점유율**
모바일 영상의 30~40%가 iOS. iOS가 HLS만 지원하니 모든 플랫폼이 HLS 만들 수밖에.

**2. 단순함**
HTTP + 텍스트 플레이리스트. 누구나 이해 가능. 진입 장벽 낮음.

**3. CDN 호환**
CDN 회사들이 환영. 기존 인프라 그대로 활용 가능.

**4. 오픈소스 생태계**
hls.js, FFmpeg, nginx-rtmp 등 무료 도구들이 잘 갖춰짐.

### HLS의 기본 동작 흐름

시청자가 라이브 방송을 클릭할 때:

```
[1단계: 마스터 플레이리스트 다운로드]
GET /live/master.m3u8
응답: 화질별 sub playlist 목록

[2단계: 화질 선택]
플레이어가 현재 대역폭 기준 적절한 화질 선택
(첫 화질은 보통 중간 화질 720p)

[3단계: Sub Playlist 다운로드]
GET /live/720p/playlist.m3u8
응답: 세그먼트 목록 (최근 5개)

[4단계: 세그먼트 다운로드 시작]
GET /live/720p/seg100.ts
GET /live/720p/seg101.ts
GET /live/720p/seg102.ts
...

[5단계: Playlist 폴링]
2~6초마다 Sub Playlist 재요청
새 세그먼트 있으면 추가 다운로드
```

이 흐름이 끝없이 반복된다. 라이브 방송이 끝날 때까지.

---

### HLS의 본질적 한계 — 지연

HLS의 가장 큰 약점은 지연이다.

```
[지연 발생 구간]
1. 인코더가 6초 세그먼트 생성: 6초
2. 서버 업로드: 0.5초
3. CDN 캐시: 1초
4. 플레이어가 폴링으로 발견: 평균 3초
5. 다운로드: 1초
6. 플레이어 버퍼 채움 (보통 3 세그먼트): 18초

총 지연: 약 25~30초
```

스트리머가 "지금!" 외친 게 시청자에게 25초 뒤 도달. 채팅이 영상보다 빠른 이유.

이게 LL-HLS가 나온 배경. 이미 공부한 거다.

---

### 1강 핵심 3줄

1. HLS는 2009년 Apple이 iPhone용으로 만든 영상 프로토콜 — "라이브도 그냥 HTTP 파일"이라는 발상의 전환
2. `.m3u8` 텍스트 목차 + `.ts` MPEG-TS 세그먼트 조합 — CDN 캐시, 방화벽 통과, ABR이 자연스럽게 해결
3. iOS 점유율과 단순함, CDN 호환성 덕에 산업 표준이 됐지만 본질적으로 지연이 25초 — LL-HLS로 보완
