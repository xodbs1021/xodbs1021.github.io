## 레벨 3-5 / 1강: VP9이 왜 만들어졌나 — Google의 무료 코덱 전략

---

### Google의 코덱 고민 — 2010년대 초

Google이 H.264 라이센스에 매년 큰 돈을 내고 있었다.

```
[Google의 H.264 부담]
YouTube 트래픽: 전세계 1위
H.264 인코딩 라이센스: 콘텐츠당 비용
H.264 디코딩 라이센스: Android 디바이스마다 비용

연간 누적: 수억~수십억 달러
```

H.265가 나오자 라이센스가 더 복잡해지고 비싸짐.

```
[H.265 라이센스 풀]
- HEVC Advance
- MPEG LA
- Velos Media
- 개별 특허권자

→ Google 입장에서 받아들이기 어려움
```

Google이 결정: 직접 무료 코덱 만들자.

---

### Google의 코덱 인수 — On2 Technologies

2010년 Google이 영상 코덱 회사 On2를 1.34억 달러에 인수.

```
[On2의 코덱 계보]
VP3 → VP6 → VP7 → VP8 (오픈소스화)
```

VP8을 받자마자 오픈소스 + 무료로 공개. 2010년 WebM 프로젝트 시작.

이후 자체 개발:
- VP9 (2012년) - H.265 경쟁
- VP10 (개발 중단) - AV1로 흡수

---

### VP9의 등장 — 2012년

VP8 후속작.

```
[VP9 목표]
H.264 대비 50% 비트레이트 절감
H.265와 비슷한 압축률
완전 무료, 로열티 없음
오픈소스
```

H.265 발표 (2013년)와 비슷한 시기. Google이 H.265 대안으로.

---

### VP9 vs H.265 — 압축률 비교

```
[같은 화질 기준 비트레이트]
H.264: 100% (기준)
VP9: 60~70%
H.265: 50~60%
```

VP9이 H.265보다 약간 떨어지지만 비슷한 수준.

핵심은 라이센스 비용:
- H.265: 디바이스당 + 콘텐츠당 비용
- VP9: 0원

YouTube 같은 거대 무료 서비스에 결정적.

---

### YouTube의 VP9 채택

YouTube가 VP9의 가장 큰 사용처.

```
[YouTube 영상 코덱]
일반 화질 (1080p 이하): H.264
4K, HDR: VP9
일부 콘텐츠: AV1 (2018년 이후)

시청자 디바이스 따라 자동 선택
```

YouTube 영상의 대부분이 VP9. 특히 고화질.

---

### VP9의 한계 — 보급률 낮음

기술적으론 우수했지만 시장 확산은 제한적.

```
[VP9 보급 한계]
1. Apple 안 씀 (iOS, Safari)
   → iPhone 시청자가 VP9 못 봄
   → H.264 폴백 필수

2. 모바일 하드웨어 디코더 부족
   → 2018년 이전 폰엔 거의 없음
   → CPU 디코딩은 배터리 부담

3. 다른 기업 채택 미온적
   → Netflix, Disney+ 등은 H.265 위주
   → YouTube + 일부만 VP9

4. 인코딩 느림
   → libvpx-vp9이 x264보다 5~10배 느림
```

이게 AV1 (더 강력한 동맹) 으로 가게 된 배경.

---

### VP8 — VP9의 전 단계

VP9의 형 격. WebRTC 영상 표준.

```
[VP8 특징]
- 2010년 오픈소스화
- H.264와 비슷한 압축률
- WebRTC 의무 코덱
- 무료, 로열티 없음
```

WebRTC에서 VP8이 표준인 이유:
- WebRTC 발표 시점 (2011년)에 무료 코덱이 VP8 외 없었음
- H.264도 의무 코덱 추가됐지만 VP8 먼저

```
[WebRTC SDP - 영상]
m=video 9 UDP/TLS/RTP/SAVPF 96 97 100
a=rtpmap:96 VP8/90000
a=rtpmap:97 VP9/90000
a=rtpmap:100 H264/90000
```

VP8, VP9, H.264 모두 협상 가능.

---

### 어디서 VP9 만나나 — 실무 현황

```
[VP9 사용처]
✅ YouTube (4K, HDR)
✅ WebRTC (Chrome, Firefox)
✅ Google Meet
✅ Twitch (백엔드 일부)

❌ Apple 생태계 (iOS Safari)
❌ Netflix, Disney+ (H.265 사용)
❌ 한국 라이브 (치지직 등 - H.264)
❌ 일반 영상 파일 (MP4보다 WebM 적음)
```

YouTube가 거의 유일한 대규모 채택처.

---

### WebM 컨테이너 — VP9의 짝

VP9은 보통 WebM 컨테이너에 담김.

```
[WebM 구조]
영상: VP8 또는 VP9
오디오: Opus 또는 Vorbis
컨테이너: Matroska 기반
```

```bash
# WebM 인코딩
ffmpeg -i input.mp4 \
  -c:v libvpx-vp9 -b:v 2000k \
  -c:a libopus -b:a 128k \
  output.webm
```

YouTube가 WebM 적극 활용. MP4 (H.264) + WebM (VP9) 동시 제공.

---

### VP9 FFmpeg 인코딩

```bash
# 기본 VP9
ffmpeg -i input.mp4 \
  -c:v libvpx-vp9 \
  -b:v 2000k \
  -minrate 1000k -maxrate 3000k \
  -tile-columns 6 -frame-parallel 1 \
  -threads 8 \
  -speed 1 \
  -auto-alt-ref 1 -lag-in-frames 25 \
  -c:a libopus -b:a 128k \
  output.webm

# 2-pass (VOD 고화질)
ffmpeg -i input.mp4 \
  -c:v libvpx-vp9 -b:v 2000k \
  -pass 1 -f null /dev/null

ffmpeg -i input.mp4 \
  -c:v libvpx-vp9 -b:v 2000k \
  -pass 2 -c:a libopus -b:a 128k \
  output.webm
```

옵션:
- `-tile-columns 6`: 병렬 처리
- `-speed 1`: 화질 우선 (0=최고화질, 4=속도우선)
- `-auto-alt-ref 1`: 참조 프레임 활용

### VP9 NVENC가 없는 문제

```
[하드웨어 인코더 지원]
H.264: NVENC, Quick Sync, AMD AMF, Apple ✅
H.265: NVENC, Quick Sync, AMD AMF, Apple ✅
VP9: 거의 없음 ❌
AV1: NVENC (RTX 4000+), Intel Arc, AMD RDNA 3+
```

VP9 하드웨어 인코더 보급이 미미.

→ VP9 라이브 인코딩 부담 큼.

→ VOD 위주로만 활용 (YouTube가 대표).

---

### VP9 디코딩

다행히 디코딩은 보급됨.

```
[VP9 하드웨어 디코더]
Android: 대부분 폰 (2016년 이후)
PC GPU: NVIDIA GTX 1000+ (2016년 이후)
스마트TV: 대부분 (2017년 이후)
Chromecast: 모든 모델

```

iOS만 빼면 대부분 시청 가능.

### YouTube ABR Ladder 예시 - VP9 활용

```
[YouTube 영상 ABR]
4K HDR: VP9 + Opus
1440p: VP9 또는 H.264
1080p: H.264 + AAC
720p: H.264 + AAC
480p: H.264 + AAC
360p: H.264 + AAC

시청자 디바이스:
- Chrome/Android: VP9 가능하면 사용
- Safari/iOS: H.264 강제
- 옛 디바이스: H.264 폴백
```

VP9은 고화질에서 빛남. 저화질은 H.264로 호환성.

---

### Twitch와 VP9

Twitch도 일부 VP9 활용 시도.

```
[Twitch VP9 - 2017년 시도]
- 백엔드에서 VP9 트랜스코딩 테스트
- 결과: 비용 절감 효과 있지만 시청자 측 호환성 문제
- 결국 H.264 주력 유지

[현재 - 2024년]
H.264 위주
일부 AV1 베타 (H.265 건너뜀)
```

라이브 스트리밍에 VP9이 안 자리잡은 이유 = 호환성.

언제 어느 거?

```
[VP9 선택]
✅ YouTube 류 거대 무료 서비스
```

### VP9 코덱 문자열 - MIME

```
vp09.00.50.08

vp09 = VP9
00 = Profile (Profile 0, 8-bit 4:2:0)
50 = Level (5.0)
08 = bit depth (8-bit)
```

브라우저 지원 확인:
```javascript
MediaSource.isTypeSupported('video/webm; codecs="vp09.00.50.08"')
MediaSource.isTypeSupported('video/mp4; codecs="vp09.00.50.08"')
```

VP9은 WebM과 MP4 둘 다 가능. 보통 WebM.

---

### VP9의 미래 - AV1로 흡수

```
[Google의 코덱 전략 변화]
2010: VP8 (오픈소스화)
2012: VP9 (H.265 대안)
2015: AOMedia 설립 → AV1 개발
2018: AV1 표준 완성
2020+: AV1 위주 전환

VP9은 점진적으로 AV1에 자리 넘김
```

VP9은 과도기 코덱이 됨.

YouTube가 점점 AV1으로 전환 중. VP9은 여전히 유지지만 신규 콘텐츠는 AV1.

### Mozilla, Cisco의 영향

VP8/VP9 보급에 다른 회사들도 기여.

```
[지원 기업]
Mozilla: Firefox에서 VP8/VP9 적극 지원
Cisco: WebRTC에 VP8 의무화 푸시
Amazon: Fire TV에서 VP9 지원
Samsung: TV에서 VP9 디코더 내장
```

Google 혼자가 아니라 동맹.

이게 H.265에 대항하는 진영의 시작.

### 면접 답변용

"VP9 어떻게 활용하세요?" 같은 질문.

```
"저희 같은 일반 라이브 플랫폼은 VP9 안 씁니다. 이유는:

1. iOS Safari 미지원 → 시청자 30%+ 제외
2. VP9 하드웨어 인코더 거의 없음 → 라이브 트랜스코딩 부담
3. H.264로 충분히 운영 가능

VP9이 의미 있는 곳은:
- YouTube (4K, 거대 무료 서비스)
- Google Meet (WebRTC 환경)
- 자체 디바이스 가진 회사 (Chromecast 등)

저희가 4K HDR 콘텐츠 진입한다면 H.265 또는 AV1을 검토하지
VP9은 건너뛸 가능성이 큽니다. AV1이 압축률도 더 좋고 차세대 표준이라."
```

### 1강 핵심 3줄

1. VP9은 Google이 2012년 만든 무료 영상 코덱 — H.265와 비슷한 압축률에 라이센스 비용 0, YouTube의 4K/HDR 표준
2. iOS Safari 미지원 + 하드웨어 인코더 부족으로 보급 제한적 — Apple/Netflix 진영은 H.265 선택
3. VP9은 과도기 코덱, Google이 AV1으로 옮겨가는 중 — 한국 라이브는 호환성 때문에 H.264 유지가 정답
