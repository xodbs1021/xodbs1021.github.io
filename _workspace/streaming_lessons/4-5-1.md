### 라이브 트랜스코딩 인프라의 7층

지금까지 본 모든 컴포넌트.

```
┌─────────────────────────────────┐
│ 1. 스트리머 (OBS, XSplit)        │
└─────────┬───────────────────────┘
          │ RTMP / SRT
          ▼
┌─────────────────────────────────┐
│ 2. 인제스트 클러스터              │
│    - 인증, 검증, 포워딩          │
└─────────┬───────────────────────┘
          │ 내부 SRT
          ▼
┌─────────────────────────────────┐
│ 3. 트랜스코딩 클러스터 (GPU)      │
│    - ABR ladder 생성             │
└─────────┬───────────────────────┘
          │ 인코딩 결과
          ▼
┌─────────────────────────────────┐
│ 4. 패키저 클러스터                │
│    - HLS/DASH/CMAF 매니페스트     │
└─────────┬───────────────────────┘
          │
          ▼
┌─────────────────────────────────┐
│ 5. Origin Storage                │
│    - 매니페스트 + 세그먼트 저장   │
└─────────┬───────────────────────┘
          │
          ▼
┌─────────────────────────────────┐
│ 6. Origin Shield + CDN           │
│    - 글로벌 분배                  │
└─────────┬───────────────────────┘
          │
          ▼
┌─────────────────────────────────┐
│ 7. 시청자 (Player)               │
└─────────────────────────────────┘

[관제 (모든 층)]
- Prometheus + Grafana
- 로그 (Loki/ELK)
- 알람 (PagerDuty)
- VMAF QC 파이프라인
```

---

### 각 층의 책임 분리 - 마이크로서비스

왜 한 노드에 다 안 하고 분리하나.

```
[각 층의 자원 패턴]
인제스트:    네트워크 바운드 (대용량 NIC)
트랜스코딩:  GPU 바운드 (NVENC)
패키저:      CPU 바운드 (가벼움)
Origin:      I/O 바운드 (저장)
CDN:         지리적 분산

[같은 인스턴스에 모으면]
GPU 강한 서버에 모음 → 인제스트가 NIC 못 살림
CPU 강한 서버에 모음 → GPU 부족
→ 자원 효율 낮음
```

분리하면 각 층에 맞는 인스턴스 타입 사용. 비용 30% 절감.

```
[t=0.1초]

입력 검증: H.264 + AAC, 1080p60, 8Mbps

[t=0.2초]
인제스트 → 트랜스코딩 스케줄러: "방송 시작" 알림
스케줄러: GPU 노드 부하 확인 → 트랜스코딩 노드 (서울 #12) 할당
인제스트 노드: SRT 연결로 트랜스코딩 노드에 포워딩 시작

[t=0.5초]

[t=1초]

[t=6초]
패키저: 첫 .m4s 생성
Origin Storage에 PUT (Redis + S3)
매니페스트 업데이트

[t=6.1초]
CDN Origin Shield: 새 매니페스트 감지
인기 스트리머면 pre-positioning으로 엣지에 미리 푸시

[t=6.2초~]
시청자 A (서울): CDN 엣지에서 매니페스트 받음
시청자 B (LA): LA 엣지로부터 받음 (origin 안 침)
```

### 인제스트 클러스터 상세

스트리머가 직접 접속하는 곳.

```
[구성]
- Anycast IP (지리적으로 가까운 노드에 자동 라우팅)
- Nginx-RTMP 또는 SRS
- 인증 마이크로서비스 (Go/Java)
- 검증 마이크로서비스 (ffprobe 호출)
```

스펙:
```
인스턴스: c6i.4xlarge (16 vCPU, 32GB)
NIC: 12.5 Gbps
가격: $0.70/시간
```

한 노드에 동시 인제스트:
```
1080p60 8Mbps × 100 = 800Mbps
8Mbps 동시 1000 채널 가능 (이론) / 실제 200~300
```

부하는 네트워크보다 RTMP 연결 관리. TCP 연결 수가 한계.

### Origin Storage 상세

세그먼트와 매니페스트 저장.

```
[구조]
1. Redis Cluster (최근 1분 캐시)
   - 매니페스트 빠른 업데이트
   - 라이브 시청에 즉시 응답

2. S3 (영구 저장)
   - 다시보기용
   - 비동기 업로드
   - Lifecycle: 24시간 후 IA, 30일 후 Glacier

3. CDN Pull Origin
   - Redis + S3 통합 라우팅
   - 최근 데이터 → Redis
   - 오래된 데이터 → S3
```

```python
# Pull Origin 로직
def get_segment(path):
    # 최근 데이터부터 확인
    if redis.exists(path):
        return redis.get(path)
    if s3.exists(path):
        return s3.get(path)
    return 404
```

### CDN 통합 - Multi-CDN

큰 곳은 여러 CDN 동시 사용.

```
[Multi-CDN 이유]
1. 벤더 락인 방지
2. 지역별 강점 차이 (CloudFront 미국, Akamai 글로벌, Fastly 빠름)
3. 한 CDN 장애 시 fallback
4. 비용 협상력
```

```
[Multi-CDN 라우팅]
시청자 → DNS 라우터 (자체 또는 NS1)
       → 지역/시간/가격 기준 CDN 선택
       → CDN 엣지

예:
한국 시청자: 주로 Akamai (국내 PoP 많음)
미국 서부: CloudFront
모바일: Cloudflare (HTTP/3 효율 좋음)
```

### Multi-region 배포

지리적 분산.

```
[Region 구성]
서울:        국내 메인 (인제스트 + 트랜스코딩)
도쿄:        일본/대만/홍콩 + 한국 백업
싱가포르:    동남아
미국 동부:   북미 백업
유럽:        스파라딕 콘텐츠
```

각 region에 완전한 트랜스코딩 클러스터. CDN은 글로벌.

```
[Region 간 fallback]
서울 트랜스코딩 장애:
  인제스트가 도쿄 트랜스코딩으로 SRT 포워딩 전환
  지연 +30ms (서울-도쿄 RTT)
  시청자는 거의 못 느낌
```

### 인프라 코드 - Terraform 예시

```hcl
# transcoding-cluster.tf
resource "aws_eks_cluster" "transcoding" {
  name = "transcoding-${var.region}"
  
  vpc_config {
    subnet_ids = aws_subnet.transcoding[*].id
  }
}

resource "aws_eks_node_group" "gpu" {
  cluster_name = aws_eks_cluster.transcoding.name
  node_group_name = "gpu-nodes"
  
  instance_types = ["g5.xlarge"]  # A10 GPU
  
  scaling_config {
    desired_size = 10
    max_size = 100
    min_size = 5
  }
  
  labels = {
    "gpu-type" = "nvidia-a10"
  }
}
```

전체 인프라를 코드로 관리. region 추가 시 같은 코드 다른 region에.
