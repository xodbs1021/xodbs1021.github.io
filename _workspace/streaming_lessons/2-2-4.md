## 레벨 2-2 / 4강: HLS Player의 동작 — hls.js 내부 들여다보기

---

### 브라우저는 HLS를 어떻게 재생하나

iOS Safari와 macOS Safari는 HLS를 네이티브로 지원한다. `<video src="...m3u8">` 한 줄이면 끝.

```html
<video controls>
  <source src="https://cdn.chzzk.naver.com/live/master.m3u8" type="application/vnd.apple.mpegurl">
</video>
```

근데 Chrome, Firefox, Edge는 HLS 네이티브 지원이 없다. 이유:

```
Apple: HLS 만든 회사 → 당연히 자기 브라우저(Safari)에 넣음
Google: 자기들이 만든 DASH 밀고 싶음 → Chrome에 HLS 안 넣음
Mozilla: 둘 다 안 넣음 (라이센스 이슈)
```

그래서 JavaScript로 HLS를 처리하는 라이브러리가 필요하다. **hls.js**.

---

### hls.js의 핵심 — Media Source Extensions (MSE)

브라우저는 HLS는 모르지만 **MP4 fragment**는 안다. MSE API로 이걸 활용.

```javascript
// MSE 기본 흐름
const video = document.querySelector('video');
const mediaSource = new MediaSource();
video.src = URL.createObjectURL(mediaSource);

mediaSource.addEventListener('sourceopen', () => {
  const sourceBuffer = mediaSource.addSourceBuffer('video/mp4; codecs="avc1.640028"');
  
  // 이 sourceBuffer에 MP4 fragment 데이터 넣으면 재생됨
  sourceBuffer.appendBuffer(mp4Data);
});
```

브라우저 비디오 엔진은 `sourceBuffer`에 들어오는 MP4 데이터를 디코딩해서 재생한다.

근데 HLS는 `.ts` 파일이다. MP4가 아니다. 어떻게 변환하나?

**hls.js의 핵심 기능: .ts → fMP4 변환을 JavaScript로 실시간 수행.**

---

### hls.js의 처리 파이프라인

```
[1단계: M3U8 다운로드]
fetch('master.m3u8')
  ↓ 파싱
[2단계: 화질 선택 + Media Playlist 다운로드]
fetch('720p/playlist.m3u8')
  ↓ 세그먼트 목록 파악
[3단계: TS 세그먼트 다운로드]
fetch('seg100.ts')
  ↓ 188 byte 패킷 파싱
[4단계: 트랜스먹싱 (transmuxing)]
TS → fMP4 변환 (코덱 데이터는 그대로, 컨테이너만 바꿈)
  ↓
[5단계: MSE에 주입]
sourceBuffer.appendBuffer(fMp4Data)
  ↓
[6단계: 브라우저가 재생]
```

핵심은 4단계 **트랜스먹싱**. JavaScript로 H.264 데이터를 .ts 컨테이너에서 빼서 fMP4 컨테이너로 다시 포장한다. 디코딩은 안 한다. 컨테이너 껍데기만 갈아치우는 것.

---

### 트랜스먹싱 — 디코딩 없이 포맷 변환

핵심 아이디어: H.264 비트스트림은 그대로, MPEG-TS 헤더 → fMP4 헤더로 바꿈.

```
[원본 .ts 패킷]
[TS 헤더 4byte][PES 헤더][NAL Unit 1][NAL Unit 2]...

[트랜스먹싱 후 fMP4]
[moof box][mdat box: NAL Unit 1, NAL Unit 2]
```

`moof`는 fragment 헤더, `mdat`는 실제 데이터. NAL Unit(영상 비트스트림)은 그대로 보존.

JavaScript로 충분히 처리 가능. CPU 부하 1% 미만.

---

### ABR Controller — 화질 자동 선택

hls.js 내부의 ABR 로직을 보면.

```javascript
// hls.js의 ABR 의사 코드
class AbrController {
  estimateBandwidth(lastFragment) {
    // 직전 세그먼트 다운로드 속도 측정
    const downloadTime = lastFragment.endTime - lastFragment.startTime;
    const bytes = lastFragment.totalBytes;
    return bytes * 8 / downloadTime;  // bps
  }
  
  chooseNextLevel(estimatedBandwidth, currentBufferLength) {
    // 안전 마진 적용 (대역폭 80%만 사용)
    const safeBandwidth = estimatedBandwidth * 0.8;
    
    // 버퍼가 적으면 더 보수적으로
    if (currentBufferLength < 10) {
      // 위험. 낮은 화질로 빠르게
      return this.findLevelBelow(safeBandwidth * 0.5);
    }
    
    // 버퍼가 충분하면 적정 화질
    return this.findLevelBelow(safeBandwidth);
  }
}
```

추정 대역폭 × 안전 마진(0.8) + 버퍼 상태 = 다음 화질.

설정 가능한 옵션:

```javascript
const hls = new Hls({
  abrEwmaDefaultEstimate: 500000,  // 초기 대역폭 추정 500 Kbps
  abrBandWidthFactor: 0.95,         // 대역폭 95% 사용
  abrBandWidthUpFactor: 0.7,        // 화질 올릴 땐 70%만 (보수적)
  abrMaxWithRealBitrate: false,     // 실측 비트레이트 활용
});
```

`abrBandWidthUpFactor`가 흥미롭다. **화질 올릴 땐 보수적, 내릴 땐 빠르게**. 이유:

올렸다가 다시 내리면 시청자가 흠칫. 안 올리는 게 낫다. 내릴 땐 버퍼링 막으려 빠르게.

### Stream Controller — 세그먼트 로딩 전략

플레이어가 어떤 순서로 세그먼트를 다운로드하나.

```javascript
// 의사 코드
class StreamController {
  loadNextFragment() {
    const currentTime = video.currentTime;
    const bufferedEnd = video.buffered.end(video.buffered.length - 1);
    
    // 이미 버퍼 충분하면 안 받음
    if (bufferedEnd - currentTime > maxBufferLength) {
      setTimeout(this.loadNextFragment, 1000);
      return;
    }
    
    // 다음 세그먼트 결정
    const nextFragment = this.findFragmentAt(bufferedEnd);
    
    // 다운로드
    fetch(nextFragment.url)
      .then(response => response.arrayBuffer())
      .then(data => {
        this.transmux(data);  // TS → fMP4
        this.appendToSourceBuffer();
      });
  }
}
```

핵심 로직:
- 현재 재생 위치 + 버퍼 길이 = 다음에 필요한 세그먼트
- 버퍼가 충분하면 다운로드 중단 (대역폭 절약)
- 버퍼가 부족하면 즉시 다운로드

### Recovery Strategy — 에러 복구

네트워크 끊김, 세그먼트 손실 시 복구 방법.

```javascript
hls.on(Hls.Events.ERROR, (event, data) => {
  if (data.fatal) {
    switch (data.type) {
      case Hls.ErrorTypes.NETWORK_ERROR:
        // 네트워크 에러: 재시도
        hls.startLoad();
        break;
      case Hls.ErrorTypes.MEDIA_ERROR:
        // 미디어 에러: 디코더 리셋
        hls.recoverMediaError();
        break;
      default:
        // 복구 불가: 플레이어 파괴
        hls.destroy();
        break;
    }
  }
});
```

복구 시도 순서:
1. **재시도** (1초, 2초, 4초 백오프)
2. **레벨 강등** (낮은 화질로)
3. **미디어 리셋** (MSE 다시 만듦)
4. **포기** (사용자에게 에러 표시)

라이브 방송에서 에러 복구 후 라이브 엣지로 점프하는 코드도 hls.js에 내장.

### Buffer 관리 — 메모리 한계

브라우저 메모리는 무한이 아니다. 너무 많이 버퍼링하면 다른 탭에 영향.

```javascript
const hls = new Hls({
  maxBufferLength: 30,         // 30초만 버퍼링
  maxMaxBufferLength: 600,     // 최대 10분 (대역폭 충분할 때)
  maxBufferSize: 60 * 1000 * 1000,  // 60 MB 메모리 제한
  maxBufferHole: 0.5,          // 버퍼 갭 0.5초 이상이면 점프
});
```

**maxBufferHole**이 흥미롭다. 세그먼트가 부분 손실되면 영상에 갭이 생긴다. 이 갭이 0.5초 이상이면 그냥 건너뛴다. 버퍼링하지 않고.

라이브 방송에서 핵심 기능. 작은 손실 때문에 멈추면 안 된다.

### Live Sync — 라이브 엣지 따라가기

라이브에서 시청자가 가능한 한 최신을 보도록.

```javascript
const hls = new Hls({
  liveSyncDuration: 15,           // 라이브 엣지에서 15초 뒤
  liveMaxLatencyDuration: 30,     // 30초 이상 처지면 점프
  liveDurationInfinity: true,     // 라이브는 무한 길이
});

hls.on(Hls.Events.LEVEL_UPDATED, () => {
  // 매 플레이리스트 갱신마다 라이브 엣지 거리 확인
  const liveEdgeDistance = currentLiveEdge - video.currentTime;
  
  if (liveEdgeDistance > 30) {
    // 너무 처짐: 라이브 엣지로 점프
    video.currentTime = currentLiveEdge - liveSyncDuration;
  }
});
```

처음 재생 시작할 때도 라이브 엣지에서 15초 뒤 위치로 시작. 너무 엣지에 붙으면 버퍼링 위험.

### LL-HLS 지원 — 부분 세그먼트 처리

LL-HLS에서는 `.ts` 파일이 완성되기 전에 부분(Partial Segment)을 받는다.

```javascript
const hls = new Hls({
  lowLatencyMode: true,
  liveSyncDurationCount: 1,  // 라이브 엣지 매우 가까이
});
```

내부적으로:
- HTTP/2 Push로 다음 partial 미리 받음
- Chunked Transfer Encoding으로 스트림 받음
- 작은 partial들을 누적해서 fMP4 fragment 구성

복잡도가 일반 HLS의 3배 정도. 그래서 hls.js의 LL-HLS 지원이 비교적 늦었다 (1.x 후반).

---

### hls.js를 안 쓰고 직접 만들 수 있나

이론적으로 가능. 실제로 큰 회사들은 자체 플레이어를 만든다.

- **Twitch**: 자체 hls 플레이어 (광고 차단 우회 방지)
- **YouTube**: Shaka Player + 자체 확장
- **Netflix**: 자체 플레이어 (DRM 통합)

직접 만들 때 들어가는 일:
1. M3U8 파서
2. MPEG-TS 디먹서
3. H.264/AAC 비트스트림 처리
4. fMP4 빌더
5. MSE 통합
6. ABR 알고리즘
7. 에러 복구
8. 라이브 동기화
9. 자막 처리
10. 광고 처리

작은 회사는 hls.js 그대로 쓰는 게 답. 대형 회사는 핵심 기능 직접 구현하고 hls.js를 참고.
