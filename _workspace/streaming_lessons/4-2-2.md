## 레벨 4-2 / 2강: Per-Title Encoding — Netflix의 비용 혁명

---

### Static 사다리의 비효율 — Netflix의 발견

2015년 Netflix가 데이터 분석 후 충격적 결과 발표.

```
[기존 Static Ladder의 문제]
"피카츄 만화" 영상 1080p @ 5800 kbps
  → 실제로는 1800 kbps로도 같은 화질
  → 4000 kbps가 낭비

"폭발 액션" 영상 1080p @ 5800 kbps
  → 실제로는 8000 kbps 필요
  → 모자라서 화질 깨짐
```

Netflix 데이터센터에서 매년 수십 PB의 콘텐츠가 흐른다. 평균 30%의 비트레이트 낭비 = 수십억 달러의 CDN 비용.

이걸 해결하려고 만든 게 **Per-Title Encoding**.

---

### Per-Title의 기본 아이디어

콘텐츠마다 사다리를 다르게 짠다.

```
[A 영상: 카페 잡담]
복잡도: 낮음
사다리:
  1080p @ 2500k  ← 일반 사다리는 6000
  720p  @ 1500k
  480p  @ 800k

[B 영상: 슈퍼히어로 액션]
복잡도: 매우 높음
사다리:
  1080p @ 8000k  ← 일반 사다리는 6000
  720p  @ 4500k
  480p  @ 2200k
```

복잡도 낮으면 비트 줄여서 절약. 복잡도 높으면 비트 늘려서 화질 보장.

---

### 복잡도를 어떻게 측정하나

영상 전체를 다양한 비트레이트로 인코딩한 후 화질 측정.

```
[1단계: 다양한 비트레이트로 인코딩]
1080p @ 500k
1080p @ 1000k
1080p @ 2000k
1080p @ 4000k
1080p @ 8000k
720p @ 500k
720p @ 1000k
...

[2단계: 각각 VMAF 측정]
1080p @ 500k → VMAF 45
1080p @ 1000k → VMAF 65
1080p @ 2000k → VMAF 82
1080p @ 4000k → VMAF 92
1080p @ 8000k → VMAF 96
```

VMAF 92가 시청자 만족 임계점. 이 영상은 1080p @ 4000k가 최적.

다른 영상은 1080p @ 4000k에서 VMAF 78. 이 영상은 6000k가 필요.

---

### Per-Title의 비용

인코딩이 N배 증가. 50개 조합 인코딩 × VMAF 측정 = 영상 1시간당 트랜스코딩 50시간.

```
[기존]
영상 1시간 → 트랜스코딩 30분 × 5단계 = 2.5시간

[Per-Title]
영상 1시간 → 분석 5시간 + 트랜스코딩 2.5시간 = 7.5시간
```

근데 CDN 비용 절약이 더 크다. 분석은 한 번, 시청은 수천만 번. 분석 비용 << 전송 비용.

---

### Netflix 결과 — 2015년 발표

```
[Per-Title 적용 결과]
평균 비트레이트: 20% 감소
동일 비트레이트 화질: 20% 향상
CDN 비용: 수억 달러 절감 (추정)
```

이 발표 이후 산업 표준이 됨. YouTube, Amazon Prime, Disney+ 다 도입.

### Per-Title 실전 — FFmpeg + VMAF

직접 만든다면 이런 흐름.

```bash
# 1. 다양한 비트레이트로 인코딩
for bitrate in 500 1000 2000 4000 8000; do
  ffmpeg -i original.mp4 \
    -c:v libx264 -b:v ${bitrate}k -maxrate ${bitrate}k \
    -y test_${bitrate}.mp4
done

# 2. 각각 VMAF 측정
for bitrate in 500 1000 2000 4000 8000; do
  ffmpeg -i original.mp4 -i test_${bitrate}.mp4 \
    -lavfi libvmaf=log_path=vmaf_${bitrate}.json:log_fmt=json \
    -f null -
done

# 3. 결과 분석 → 사다리 결정
python analyze.py vmaf_*.json
```

분석 스크립트:
```python
results = []
for bitrate in [500, 1000, 2000, 4000, 8000]:
    with open(f'vmaf_{bitrate}.json') as f:
        data = json.load(f)
    vmaf = data['pooled_metrics']['vmaf']['mean']
    results.append((bitrate, vmaf))

# VMAF 92를 처음 넘는 비트레이트 찾기
target_vmaf = 92
for bitrate, vmaf in results:
    if vmaf >= target_vmaf:
        print(f"Optimal: {bitrate}k")
        break
```

### Per-Scene Encoding — 다음 단계

Per-Title이 영상 전체를 분석한다면, Per-Scene은 장면별로 분석.

```
[영상 내부에서도 복잡도 다름]
0~2분:   타이틀 시퀀스 (정적) → 비트 적게
2~10분:  대화 장면 → 비트 보통
10~15분: 액션 → 비트 많이
15~20분: 엔딩 (정적) → 비트 적게
```

장면별로 비트레이트를 다르게 인코딩. CRF가 비슷한 원리지만, Per-Scene은 사전 분석 + 명시적 비트 배분.

```bash
# x264의 CRF가 이걸 자동으로 함
ffmpeg -i input.mp4 \
  -c:v libx264 -crf 23 -maxrate 6000k -bufsize 12000k \
  output.mp4
```

CRF가 보장하는 것 = "장면 복잡도에 맞춰 자동 비트 배분, 평균이 아닌 화질 일정".

### Shot-Based Encoding — Netflix Dynamic Optimizer

2018년 Netflix 발표. Per-Scene의 진화.

```
[Shot-Based]
1. 영상을 "shot" 단위로 분할 (카메라 컷 단위)
2. 각 shot마다 비트레이트 + 해상도 독립 결정
3. 시청자가 보는 건 부드럽게 전환
```

예:
```
Shot 1 (0~5초, 타이틀): 480p @ 200k로 충분
Shot 2 (5~10초, 대화): 720p @ 1500k
Shot 3 (10~15초, 폭발): 1080p @ 8000k
```

CMAF + DASH의 표현력으로 가능. 같은 사다리 안에서 shot마다 해상도가 바뀐다.

이게 Netflix의 최첨단. 산업계 전체가 따라가는 중.

### Per-Title은 라이브에서 가능한가

기본적으로 어렵다. 전체 콘텐츠 분석이 필요한데 라이브는 미래를 모르니까.

그래도 변형이 있다.

**1. 적응형 라이브 인코딩**
처음엔 표준 사다리로 시작. 첫 1분 분석 후 조정.

```
[0~60초]
표준: 1080p @ 6000k

[60초~]
분석: 복잡도 낮음 → 1080p @ 3500k로 동적 전환
```

방송 도중 비트레이트 바뀜. 시청자는 SEGMENT 단위로 자연스럽게 전환.

**2. 카테고리 기반 사다리**
방송 카테고리에 따라 다른 사다리 사전 결정.

```
[저스트 채팅]: 일반 사다리 (4000k)
[FPS 게임]:    빠른 액션 사다리 (6500k)
[그림 그리기]: 정적 사다리 (2500k)
```

치지직이 카테고리별 비트레이트 권장값을 가지고 있는 게 이거의 흔적.

**3. 사전 학습 기반 예측**
스트리머의 과거 방송 분석 → 비트레이트 예측 → 실시간 적용.

```
스트리머 A의 이전 100개 방송:
평균 복잡도: 0.8 (높음)
→ 다음 방송도 비트레이트 높게 시작
```

ML 기반. 대형 플랫폼이 도입 중.

### Per-Title의 함정 — VMAF의 한계

VMAF가 만능이 아니다.

```
[VMAF가 못 잡는 것]
- 깜빡임 (특정 프레임만 잘못)
- HDR 색상 정확도
- 빠른 모션 블러
- 코덱별 특성 (AV1 점수가 H.264보다 부풀려짐)
```

Netflix는 VMAF에 추가 지표 결합.

```
[멀티 메트릭]
VMAF 95 + PSNR 42 + SSIM 0.95 + 시각적 검수
```

Per-Title 도입 시 단일 메트릭 의존 금지. 여러 지표 조합 + 샘플 영상 시각 검수.

### 실전 비용 분석 — 도입 가치 계산

언제 Per-Title 도입할지 정량 판단.

```
[현재 상황]
월 시청 시간: 1억 시간
평균 비트레이트: 4000 kbps
월 CDN 트래픽: 1.8 PB
CDN 비용: $20,000/PB → $36,000/월

[Per-Title 도입 후 30% 절감]
월 트래픽: 1.26 PB
CDN 비용: $25,200/월
월 절약: $10,800

[도입 비용]
인코딩 클라우드: $2,000/월
엔지니어 1명: $10,000/월
초기 시스템 구축: $50,000 (1회)

[ROI]
월 순절감: -$1,200 (적자)
1년차: -$64,400
```

이 규모에선 도입 안 함. 도입은 보통 월 트래픽 10 PB 이상부터.

치지직 정도 규모면 Per-Title 도입 가치 있음. Twitch는 이미 도입.

---

### Twitch의 결정 — 부분 적용

Twitch도 Per-Title 일부 도입 (2020년 발표).

```
[Twitch 결정]
인기 스트리머 (Top 10%): Per-Title 적용
일반 스트리머: 표준 사다리

근거:
- 인기 스트리머가 트래픽 80% 차지
- 일반 스트리머는 분석 비용 vs 절감 안 맞음
```

치지직도 비슷한 전략 가능. 인기 스트리머만 Per-Title.

### 면접 답변용

"Per-Title 도입 가치를 어떻게 판단하시겠어요?" 질문.

```
"규모 + 비용 구조로 판단합니다.

월 CDN 트래픽 10 PB 이상이면 도입 가치 있고,
그 이하면 분석 인프라 비용이 절감액 초과합니다.

라이브의 경우 전통적 Per-Title 어려우니
- 카테고리 기반 사다리 사전 결정
- 첫 1분 분석 후 동적 조정
- 인기 스트리머만 적용

이런 변형으로 도입합니다.

VMAF만으론 부족하니 PSNR, SSIM, 시각 검수 병행하고,
사다리 결정은 Convex Hull로 효율적 점만 선택합니다."
```

### 2강 핵심 3줄

1. Per-Title Encoding은 콘텐츠별 복잡도 분석 → 최적 비트레이트 결정 (Netflix 30% 절감)
2. 라이브는 전통 Per-Title 어려워서 카테고리 사다리 / 적응형 조정 / 인기 스트리머만 적용 같은 변형
3. VMAF 단일 의존 금지 — PSNR/SSIM/시각 검수 병행 + Convex Hull 분석으로 비효율 단계 제거
