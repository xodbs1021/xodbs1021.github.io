### Netflix 4K 인코딩 파이프라인

대표적인 H.265 운영 사례.

```
[Netflix 4K HDR 흐름]
1. 제작사에서 4K HDR 마스터 받음 (ProRes 또는 EXR)
2. Per-Title 분석 (영상 복잡도 측정)
3. ABR Ladder 결정 (영화별 다름)

5. CMAF 패키징
6. CDN 배포
```

각 영화마다 다른 ABR Ladder. Per-Title Encoding.

### Per-Title Encoding 핵심

```
[전통 방식]
모든 영화: 4K 12 Mbps
- 정적인 토크쇼: 비트 낭비
- 액션 영화: 비트 부족

[Per-Title]
영화별 분석:
- 정적 토크쇼: 4K 6 Mbps (50% 절감)
- 액션 영화: 4K 18 Mbps (정밀)
- 평균: 8 Mbps (33% 절감)
```

대규모 운영에서 효과 큼.

CDN 비용 절감 + 시청자 경험 유지.

### iPhone HDR → HDR10 변환

Dolby Vision을 HDR10으로.

```bash
ffmpeg -i iphone_input.mov \
  -c:v libx265 \
  -preset slow \
  -profile:v main10 \
  -pix_fmt yuv420p10le \
  -x265-params "colorprim=bt2020:transfer=smpte2084:colormatrix=bt2020nc:hdr-opt=1:repeat-headers=1" \
  -b:v 12000k \
  -c:a aac -b:a 192k \
  output_hdr10.mp4
```

Dolby Vision 메타데이터 제거. HDR10 메타데이터 추가.

색공간/transfer/매트릭스는 유지 (BT.2020 + PQ).

---

### iPhone HDR → SDR 변환

호환성 우선 (대부분 시청자).

```bash
ffmpeg -i iphone_input.mov \
  -vf "zscale=t=linear:npl=100,format=gbrpf32le,zscale=p=bt709,tonemap=tonemap=hable:desat=0,zscale=t=bt709:m=bt709:r=tv,format=yuv420p" \
  -c:v libx264 \
  -preset slow \
  -b:v 6000k \
  -c:a aac -b:a 128k \
  output_sdr.mp4
```

복잡한 zscale 필터로 톤 매핑. HDR 1000 nits → SDR 100 nits.

### YouTube의 H.265 회피

YouTube는 일관되게 H.265 미사용.

```
[YouTube 전략]
H.264: 호환성
VP9: 4K + 무료 (Google이 만듦)
AV1: 차세대 무료

[H.265 안 쓰는 이유]
1. 라이센스 비용 = YouTube 시청자 수 × 비용
2. 직접 만든 VP9/AV1 활용
3. AV1이 H.265보다 효율 좋음 + 무료
```

대규모 무료 서비스가 H.265 피하는 패턴.

### 라이브 4K HDR HLG 인코딩 — 올림픽급

스포츠 중계 같은 경우.

```bash
ffmpeg -i input \
  -c:v hevc_nvenc \
  -preset p4 \
  -tune ll \
  -profile:v main10 \
  -pix_fmt p010le \
  -x265-params "colorprim=bt2020:transfer=arib-std-b67:colormatrix=bt2020nc" \
  -b:v 25000k -maxrate 25000k -bufsize 50000k \
  -g 60 \
  -c:a ac3 -b:a 384k -ac 6 \
  -f flv rtmp://server/live/key
```

4K 60fps HLG HDR. 25 Mbps.

NVENC GPU 필수.

### 실전 트러블슈팅

H.265 도입 시 자주 만나는 문제.

```
[문제 1: 시청자가 못 봄]
원인: 디바이스 H.265 미지원
해결: H.264 트랙 함께 제공, 자동 폴백

[문제 2: 화면 깨짐]
원인: 컨테이너 호환성 문제
해결: -tag:v hvc1 (Apple) 또는 hev1 (일반)

[문제 3: HDR 색상 이상]
원인: 색공간 메타데이터 누락
해결: -x265-params에 colorprim/transfer/matrix 명시

[문제 4: GPU 메모리 부족]
원인: 4K 인코딩에 GPU 메모리 큼
해결: 채널 수 줄이거나 GPU 추가

[문제 5: 비트레이트 들쭉날쭉]
원인: VBR 사용 중
해결: -rc cbr로 CBR 강제
```

### iPhone 호환 태그 — hvc1 vs hev1

Apple 디바이스 호환성.

```bash
# Apple 디바이스 호환 (iPhone, Mac)
ffmpeg -i input.mp4 \
  -c:v libx265 \
  -tag:v hvc1 \
  output.mp4

# 일반 호환 (Android, Windows)
ffmpeg -i input.mp4 \
  -c:v libx265 \
  -tag:v hev1 \
  output.mp4
```

`hvc1`은 init segment에 모든 정보 포함.
`hev1`은 stream 안에 정보.

iOS Safari는 `hvc1`만 지원. 헷갈리지만 중요.

### 한국 라이브 시장의 H.265

치지직, SOOP 같은 한국 플랫폼.

```
[한국 라이브 시장 특징]
- 시청자 70% 모바일
- 안드로이드 비율 큼
- 저가폰 시청자 많음
- 5G 보급률 세계 1위 (대역폭 부담 적음)

→ H.265 도입 동기 약함
→ 5년 후도 H.264 위주 예상
```

대역폭 부담이 적어서 H.265의 효율 절감 동기가 약함.

미국/유럽보다 한국이 H.265 도입 느림.

### 3강 핵심 3줄

1. Netflix/Apple TV+/Disney+ 4K HDR은 H.265 + Per-Title Encoding 표준 — Twitch는 RTX 4000 베타, YouTube는 VP9/AV1로 회피
2. iPhone HDR 영상은 Dolby Vision → HDR10 또는 SDR 변환, Apple 디바이스 호환은 `-tag:v hvc1` 필수
3. 한국 라이브 시장은 시청자 디바이스 + 5G 보급률 때문에 H.265 도입 동기 약함 — 일반 라이브는 H.264 유지가 비즈니스적 정답
