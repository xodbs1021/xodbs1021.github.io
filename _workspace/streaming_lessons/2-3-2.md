## 레벨 2-3 / 2강: MPD 파헤치기 — DASH 매니페스트 실전 분석

---

### MPD의 최상위 속성

MPD 루트 엘리먼트의 핵심 속성들.

```xml
<MPD xmlns="urn:mpeg:dash:schema:mpd:2011"
     type="dynamic"
     minimumUpdatePeriod="PT2S"
     timeShiftBufferDepth="PT5M"
     suggestedPresentationDelay="PT12S"
     availabilityStartTime="2026-06-13T10:00:00Z"
     publishTime="2026-06-13T10:15:00Z"
     minBufferTime="PT4S"
     profiles="urn:mpeg:dash:profile:isoff-live:2011">
```

각 속성의 의미:

**`type="dynamic"`** vs **`type="static"`**
- dynamic: 라이브 (M3U8의 라이브 플레이리스트)
- static: VOD (M3U8의 `EXT-X-ENDLIST` 있는 것)

**`minimumUpdatePeriod="PT2S"`**
플레이어가 MPD를 얼마나 자주 다시 받을지. 2초마다 갱신. HLS의 폴링 주기와 같은 개념.

**`availabilityStartTime`**
방송 시작 시각. ISO 8601 형식. 모든 세그먼트 타임라인의 기준점.

**`timeShiftBufferDepth="PT5M"`**
얼마나 과거 영상을 볼 수 있는지. 5분이면 시청자가 5분 전까지 되돌려볼 수 있음. DVR 기능.

**`suggestedPresentationDelay="PT12S"`**
권장 라이브 지연. 12초. 너무 엣지에 가까이 붙으면 버퍼링 위험.

**`minBufferTime="PT4S"`**
최소 버퍼링 시간. 4초 이하 버퍼는 위험.

### Period — 시간 구간

MPD는 여러 Period를 가질 수 있다.

```xml
<MPD type="static" mediaPresentationDuration="PT30M">
  <Period id="intro" duration="PT5S">
    <!-- 인트로 영상 -->
    <AdaptationSet>...</AdaptationSet>
  </Period>
  
  <Period id="main" start="PT5S" duration="PT25M">
    <!-- 본 영상 -->
    <AdaptationSet>...</AdaptationSet>
  </Period>
  
  <Period id="ad" start="PT25M5S" duration="PT30S">
    <!-- 광고 (다른 코덱 가능) -->
    <AdaptationSet>...</AdaptationSet>
  </Period>
  
  <Period id="outro" start="PT25M35S" duration="PT4M25S">
    <!-- 본방송 복귀 -->
    <AdaptationSet>...</AdaptationSet>
  </Period>
</MPD>
```

라이브에서는 보통 Period 하나. VOD에서 광고 삽입 시 여러 Period.

Period가 끝나면 플레이어가 디코더를 리셋한다. HLS의 Discontinuity와 같은 효과.

---

### AdaptationSet — 트랙 그룹화

같은 종류의 트랙을 묶음.

```xml
<AdaptationSet
  id="1"
  mimeType="video/mp4"
  segmentAlignment="true"
  startWithSAP="1"
  maxWidth="1920"
  maxHeight="1080"
  maxFrameRate="60">
  
  <Representation id="1080p60" bandwidth="6000000" .../>
  <Representation id="720p60" bandwidth="4500000" .../>
  <Representation id="480p30" bandwidth="1500000" .../>
</AdaptationSet>
```

**`segmentAlignment="true"`**: 모든 Representation의 세그먼트가 시간상 정렬됨. ABR 전환 시 끊김 없음 보장.

**`startWithSAP="1"`**: 모든 세그먼트가 SAP Type 1으로 시작. SAP은 Stream Access Point의 줄임말. 한마디로 IDR 프레임이 시작에 있다는 뜻.

오디오 AdaptationSet 분리:

```xml
<AdaptationSet
  id="2"
  mimeType="audio/mp4"
  lang="ko"
  audioSamplingRate="48000">
  
  <Representation id="ko_aac_128k" bandwidth="128000" codecs="mp4a.40.2"/>
  <Representation id="ko_aac_64k" bandwidth="64000" codecs="mp4a.40.2"/>
</AdaptationSet>

<AdaptationSet
  id="3"
  mimeType="audio/mp4"
  lang="en"
  audioSamplingRate="48000">
  
  <Representation id="en_aac_128k" bandwidth="128000" codecs="mp4a.40.2"/>
</AdaptationSet>
```

언어별로 AdaptationSet 분리. 플레이어가 한국어/영어 중 선택.

---

### Representation — 실제 트랙 하나

가장 안쪽 단위. 한 화질, 한 비트레이트, 한 코덱.

```xml
<Representation
  id="1080p60"
  bandwidth="6000000"
  width="1920"
  height="1080"
  frameRate="60"
  codecs="avc1.640028"
  sar="1:1">
  
  <SegmentTemplate
    media="$RepresentationID$/seg_$Number%05d$.m4s"
    initialization="$RepresentationID$/init.mp4"
    startNumber="1"
    duration="6000"
    timescale="1000"/>
</Representation>
```

**`codecs="avc1.640028"`**: 코덱과 프로파일 명시.

```
avc1 = H.264 (AVC)
640028 = High Profile, Level 4.0
```

브라우저는 이 값으로 자기가 디코딩 가능한지 미리 안다. MSE의 `MediaSource.isTypeSupported('video/mp4; codecs="avc1.640028"')` 호출에 사용.

**`sar="1:1"`**: Sample Aspect Ratio. 픽셀 비율 1:1 (정사각 픽셀). 옛날 4:3 콘텐츠는 1.33:1 같은 값.

### SegmentTemplate — URL 패턴

위 예시의 SegmentTemplate 자세히.

```xml
<SegmentTemplate
  media="$RepresentationID$/seg_$Number%05d$.m4s"
  initialization="$RepresentationID$/init.mp4"
  startNumber="1"
  duration="6000"
  timescale="1000"/>
```

**`media`**: 세그먼트 URL 패턴. `$RepresentationID$`는 부모 Representation의 id로 치환. `$Number%05d$`는 1, 2, 3, ...을 5자리로 패딩.

```
1080p60/seg_00001.m4s
1080p60/seg_00002.m4s
1080p60/seg_00003.m4s
```

**`initialization`**: 초기화 세그먼트. 첫 다운로드. 코덱 정보, 트랙 메타데이터 포함.

```
1080p60/init.mp4
```

**`duration="6000"` + `timescale="1000"`**: 세그먼트 길이 = duration / timescale = 6000/1000 = 6초.

왜 6초를 그냥 안 쓰고 6000/1000? **정확한 표현**을 위해.

```
duration=6006, timescale=1001 → 5.999...초 (29.97 fps와 동기화)
duration=6000, timescale=1000 → 정확히 6초
```

영상 프레임레이트와 타임스케일이 맞아야 정확한 동기화.

### Number 기반 vs Time 기반 SegmentTemplate

세그먼트 식별 두 가지 방식.

**Number 기반** (위의 예시):
```xml
<SegmentTemplate
  media="seg_$Number$.m4s"
  startNumber="1"
  duration="6000"
  timescale="1000"/>
```

URL: `seg_1.m4s`, `seg_2.m4s`, `seg_3.m4s`...

플레이어 계산: 현재 시간 / duration = 세그먼트 번호.

**Time 기반**:
```xml
<SegmentTemplate
  media="seg_$Time$.m4s"
  duration="6000"
  timescale="1000"/>
```

URL: `seg_0.m4s`, `seg_6000.m4s`, `seg_12000.m4s`...

URL에 직접 타임스탬프. 더 명확하지만 URL이 길어짐.

라이브에서는 Time 기반이 유리. 정확한 시간 추적 가능.

### SegmentTimeline — 가변 길이 세그먼트

세그먼트 길이가 일정하지 않을 때.

```xml
<SegmentTemplate media="seg_$Time$.m4s" timescale="1000">
  <SegmentTimeline>
    <S t="0" d="6000"/>          <!-- 0초부터 6초 -->
    <S d="6000" r="9"/>            <!-- 6초 세그먼트 10개 더 -->
    <S d="5500"/>                  <!-- 5.5초 (광고 경계) -->
    <S d="6000" r="4"/>            <!-- 6초 세그먼트 5개 -->
  </SegmentTimeline>
</SegmentTemplate>
```

`r="9"`는 "9번 반복"을 의미.

라이브에서 인코더가 정확히 6초마다 세그먼트를 못 만들 때 사용. GOP 경계 맞추다 보면 살짝 빗나갈 수 있음.

### MPD 갱신 — 라이브의 시간 추적

라이브 MPD는 주기적으로 갱신된다.

```
[10:00:00 - 방송 시작]
MPD 발행
- availabilityStartTime: 10:00:00
- 첫 세그먼트: 1번 (10:00:00 ~ 10:00:06)

[10:00:06 - 세그먼트 1 가용]
플레이어는 MPD 보고 "지금 1번 세그먼트가 가용하구나" 판단

[10:00:12 - 세그먼트 2 가용]
플레이어가 MPD 다시 받음 (minimumUpdatePeriod 2초)
새 세그먼트가 가용한지 확인

[10:00:18 - 세그먼트 3 가용]
...
```

MPD에 `publishTime` 속성이 있다.

```xml
<MPD publishTime="2026-06-13T10:00:12Z">
```

플레이어가 이걸로 MPD가 얼마나 신선한지 안다. 5분 전 MPD라면 정보가 오래된 것.

### Time-based vs Wall-clock 라이브

DASH 라이브의 시간 처리는 두 방식.

**Time-based**:
```xml
<MPD type="dynamic"
     availabilityStartTime="2026-06-13T10:00:00Z">
  <Period>
    <SegmentTemplate
      startNumber="1"
      duration="6000"
      timescale="1000"/>
  </Period>
</MPD>
```

플레이어 동작:
1. 현재 UTC 시간 확인
2. availabilityStartTime 기준 경과 시간 계산
3. 경과 시간 / duration = 현재 가용한 세그먼트 번호

이게 동작하려면 **플레이어와 서버 시간이 동기화**돼야 함. NTP 필수.

**Wall-clock 기반**:
플레이어가 MPD의 publishTime을 신뢰. 시간 동기화 덜 중요.

대부분 Time-based. NTP 동기화 잘 되니까.

### MPD 크기 문제 — 라이브 장시간 방송

VOD MPD는 한 번 만들고 끝. 크기 신경 안 써도 됨.

라이브는 매니페스트가 계속 커진다.

```
[방송 1시간 후]
MPD에 세그먼트 600개 정보
크기: 약 100 KB

[방송 6시간 후]
세그먼트 3600개
크기: 약 600 KB

[방송 24시간 후 - 24시간 채널]
세그먼트 14,400개
크기: 약 2.4 MB
```

플레이어가 2초마다 2.4 MB MPD 받으면 부담.

해결책:

**1. SegmentTemplate + 짧은 TSBD**
```xml
<MPD timeShiftBufferDepth="PT5M">
  <SegmentTemplate ... />  <!-- 패턴만 -->
</MPD>
```

세그먼트 목록 안 적고 패턴만. 5분 이내만 유지.

**2. Patch MPD (DASH 2nd Edition)**
바뀐 부분만 보내는 패치 방식. 아직 보급 중.

대부분 1번 방식. 그래서 MPD 크기는 일정하게 유지됨.

### 2강 핵심 3줄

1. MPD는 type(dynamic/static), minimumUpdatePeriod, timeShiftBufferDepth로 라이브/VOD 동작 제어
2. Period > AdaptationSet > Representation 계층, SegmentTemplate으로 URL 패턴 표현 — Number/Time 기반 두 방식
3. 라이브 MPD는 SegmentTemplate + 짧은 TSBD로 크기 일정 유지 — 매니페스트 폭증 방지
