## 레벨 2-4 / 4강: Preload Hint와 Rendition Report — 마지막 지연 요소 제거

---

### 다음 partial을 미리 알 수 있나

지금까지 본 LL-HLS 동작:

```
[현재 시점]
플레이어가 매니페스트 받음:
  - partial 5번까지 존재함
  - 다음 partial 6번이 만들어지면 알려줘 (blocking)

서버: partial 6번 만들어지면 즉시 응답
```

근데 여기 또 지연이 있다.

플레이어가 매니페스트로 partial 6번 URL을 받은 후에 그제서야 partial 6번을 요청한다. 매니페스트 받기 + partial 요청 = RTT 2번 추가.

```
[12:00:00.000] partial 5번 받기 완료
[12:00:00.200] 서버에서 partial 6번 만들어짐
[12:00:00.250] 서버가 blocking reload 응답 (매니페스트)
[12:00:00.300] 플레이어가 매니페스트 받음
[12:00:00.300] 플레이어가 partial 6번 요청
[12:00:00.350] partial 6번 받기 시작
```

매니페스트 받고 partial 요청까지 50~100ms 추가 지연.

---

### Preload Hint의 아이디어

**다음에 올 partial의 URL을 미리 알려준다.**

```
#EXTINF:6.000,
seg100.m4s

#EXT-X-PART:DURATION=0.2,URI="seg101_part5.m4s"

# 아직 만들어지지 않은 partial의 URL 힌트
#EXT-X-PRELOAD-HINT:TYPE=PART,URI="seg101_part6.m4s"
```

플레이어는 partial 6번이 나오기를 기다리지 않고, **미리 partial 6번 URL을 GET 요청**한다.

```
[플레이어가 partial 6번 미리 요청]
GET /seg101_part6.m4s

[서버 응답]
- 아직 partial 6번 없음
- 응답을 보류함 (Chunked Transfer)
- 인코더가 partial 6번 만들면 즉시 chunk로 흘려보냄
```

서버가 partial이 생기는 순간 chunk로 흘려보낸다. 클라이언트는 즉시 받기 시작.

---

### Preload Hint와 Blocking Reload의 결합

두 기능이 합쳐지면 이런 흐름이 된다.

```
[12:00:00.000] partial 5번 받기 완료

[12:00:00.000] 동시에 두 요청 발사:
  요청 A: blocking reload (매니페스트 갱신 대기)
  요청 B: GET /seg101_part6.m4s (preload hint)
         → 서버가 응답 보류

[12:00:00.200] 서버에서 partial 6번 만들어짐
  요청 A 응답: 새 매니페스트
  요청 B 응답: partial 6번 데이터 (chunk로)
              ↑
            동시에 두 응답이 옴

[12:00:00.250] 플레이어가 partial 6번 받기 시작
                매니페스트도 갱신
                다음 partial 7번 preload hint 받음
                → 즉시 요청 발사
```

매번 미리 다음 거 받기 요청을 걸어두니까, partial이 만들어지자마자 즉시 데이터가 흐른다.

추가 RTT 0. 진정한 의미의 실시간.

---

### Preload Hint 종류

세 가지 타입이 있다.

```
#EXT-X-PRELOAD-HINT:TYPE=PART,URI="seg101_part6.m4s"
#EXT-X-PRELOAD-HINT:TYPE=MAP,URI="init.mp4"
```

**TYPE=PART**: 다음 partial 미리 요청.

**TYPE=MAP**: 초기화 세그먼트 미리 요청. fMP4는 init 세그먼트가 따로 있다 (코덱 정보, 트랙 메타데이터).

처음 재생 시작할 때 init 세그먼트와 첫 partial을 동시에 받을 수 있어서 시작 지연 줄임.

### Rendition Report — 다른 화질 정보

ABR로 화질 바꿀 때 새로운 지연이 생긴다.

```
[1080p 시청 중]
- partial 5번까지 받음
- 네트워크 느려짐 감지

[720p로 전환 결정]
- 720p 매니페스트 받기 위해 새 요청
GET /720p/playlist.m3u8

- 응답 받음
- 720p에 partial 7번까지 있다는 것 알게 됨
- 720p partial 7번 요청
```

매니페스트 한 번 더 받기. 시간 손해.

**Rendition Report**는 다른 화질의 현재 상태를 마스터 매니페스트에 포함.

```
[1080p 매니페스트 끝부분]
#EXT-X-PART:DURATION=0.2,URI="1080p_seg101_part5.m4s"

# 다른 화질의 현재 상태 보고
#EXT-X-RENDITION-REPORT:URI="720p/playlist.m3u8",LAST-MSN=101,LAST-PART=6
#EXT-X-RENDITION-REPORT:URI="480p/playlist.m3u8",LAST-MSN=101,LAST-PART=6
```

**`#EXT-X-RENDITION-REPORT`**: 다른 화질이 어디까지 갔는지.

```
LAST-MSN=101: 720p는 seg101까지 진행 중
LAST-PART=6: 그 안에 partial 6번까지 있음
```

플레이어는 720p 매니페스트 받지 않고도, 어디부터 받아야 할지 안다.

```
[전환 흐름]
720p partial 7번 직접 요청 → 받기 시작
720p 매니페스트는 백그라운드에서 받음
```

매니페스트 받기 지연 사라짐.

---

### 모든 화질의 partial 경계가 맞아야 한다

Rendition Report가 의미를 가지려면 화질들의 partial 경계가 정확히 같아야 한다.

```
1080p 매니페스트:
  seg101_part0 (0.0~0.2초)
  seg101_part1 (0.2~0.4초)
  seg101_part2 (0.4~0.6초)
  ...

720p 매니페스트:
  seg101_part0 (0.0~0.2초)   ← 같은 시간 구간
  seg101_part1 (0.2~0.4초)
  seg101_part2 (0.4~0.6초)
  ...
```

플레이어가 1080p partial 5번 받은 후 720p partial 6번을 받으면 연속 재생.

경계 안 맞으면:
```
1080p partial 5번 끝: 1.0초
720p partial 6번 시작: 1.05초
→ 0.05초 갭. 화면 끊김
```

이걸 보장하려면 모든 화질의 인코더가 정확히 동기화돼야 한다. 같은 클럭, 같은 GOP 구조.

### Rendition Report의 비용

화질이 4개면 매니페스트마다 3개의 Rendition Report가 들어간다.

```
[1080p 매니페스트]
... 1080p 정보 ...
#EXT-X-RENDITION-REPORT:URI="720p/...",LAST-MSN=101,LAST-PART=6
#EXT-X-RENDITION-REPORT:URI="480p/...",LAST-MSN=101,LAST-PART=6
#EXT-X-RENDITION-REPORT:URI="360p/...",LAST-MSN=101,LAST-PART=6
```

매번 갱신할 때마다 다른 화질 정보도 같이 갱신해야 한다. 서버 부담 증가.

### LL-HLS의 전체 지연 흐름 — 종합

지금까지 본 4가지 기술이 어떻게 합쳐지는지.

```
[처음 재생 시작]

T=0.000: 플레이어가 마스터 매니페스트 요청
T=0.050: 응답 받음 (Rendition Report로 모든 화질 정보)
T=0.050: 720p 매니페스트 요청 + init 세그먼트 preload
T=0.100: 매니페스트 응답 (partial 5번까지 + preload hint)
T=0.100: 가장 최근 INDEPENDENT partial (3번) 요청
T=0.150: partial 3번 받기 시작 (chunk로)
T=0.350: partial 3번 받기 완료, 디코딩 시작
T=0.350: 첫 프레임 표시
```

처음 재생 시작에 약 350ms.

```
[정상 라이브 시청 중]

매 partial마다:
- Preload Hint로 다음 partial 미리 요청 중
- Blocking Reload로 매니페스트 갱신 대기 중

T=0.000: partial N번 받기 완료
T=0.000: 서버에서 partial N+1번 만들어짐
T=0.050: chunk로 partial N+1번 받기 시작 (preload 덕분)
T=0.250: partial N+1번 받기 완료
T=0.250: 재생
```

partial 사이 지연 약 50ms.

라이브 엣지에서 약 2~3초 지연으로 시청 가능.

### LL-HLS 도입 비용 — 종합

LL-HLS는 좋은 기술이지만 도입 부담이 크다.

**1. 인코더 교체**
일반 HLS 인코더로 안 됨. LL-HLS 지원 인코더 필요 (Wowza, MistServer 등). 라이센스 비용 추가.

**2. CDN 비용 증가**
- 파일 수 30배 (6초 → 0.2초)
- 동시 connection 증가 (Blocking Reload)
- 일부 CDN은 미지원 또는 별도 요금

**3. Origin 서버 부하**
- Long polling 처리
- 효율적인 비동기 서버 필요

**4. 모바일 호환성 검증**
- 백그라운드 처리
- 네트워크 전환

**5. 플레이어 업그레이드**
- hls.js 등 라이브러리 LL-HLS 지원 버전 사용
- 자체 플레이어면 코드 추가

총 비용이 일반 HLS의 2~3배. 지연이 정말 필요한 서비스만 도입.

---

### 어떤 서비스가 LL-HLS를 쓰나

지연이 중요한 라이브 서비스:

**스포츠 중계**: 5초 차이로 결과 알면 시청 동기 떨어짐.
**경매**: 입찰 동기성 필요.
**TV 방송 동시 송출**: 케이블/IPTV와 동기.
**인터랙티브 라이브 쇼**: 시청자 투표, 채팅 동기.

지연이 덜 중요한 서비스:
- 일반 게임 방송 (스트리머가 채팅 보고 반응하니 어차피 지연)
- VOD 형식 방송
- 토크쇼

치지직은 부분 도입. 인기 채널만 LL-HLS, 나머지는 일반 HLS.

### LL-HLS vs DASH의 저지연 모드

DASH도 저지연 모드(LL-DASH)가 있다. 비교.

| | LL-HLS | LL-DASH |
|---|---|---|
| 발표 | 2019 Apple | 2017 DASH-IF |
| 지연 목표 | 2-5초 | 2-5초 |
| 매니페스트 | M3U8 (확장) | MPD (확장) |
| Partial 방식 | partial segment | chunked CMAF |
| Polling 제거 | Blocking Reload | SegmentTemplate 시간 계산 |
| iOS 지원 | ✅ | ❌ |
| 표준화 | RFC 8216bis (개정 중) | ISO/IEC 23009 |

핵심 차이: iOS Safari는 LL-HLS만 됨. 그래서 글로벌 라이브 서비스는 LL-HLS + LL-DASH 둘 다 운영.

CMAF 공통 세그먼트로 저장은 통합 가능. 매니페스트만 두 개.

---

### LL-HLS의 미래 — WHIP/WHEP과의 경쟁

지연 < 1초가 필요한 영역은 WebRTC 기반 WHEP이 들어온다.

```
[지연 스펙트럼]
일반 HLS: 25초
LL-HLS:   3초
LL-DASH:  3초
WHEP:     0.5초
WebRTC:   0.2초
```

LL-HLS는 3초 영역에서 최적화. 그 이하는 WHEP이 점령 중.

대형 라이브 플랫폼은 사용 시나리오에 따라 셋 다 운영:
- 일반 시청자: HLS (저비용)
- 인기 채널: LL-HLS (저지연)
- 상호작용 필요: WHEP (초저지연)

### 4강 핵심 3줄

1. Preload Hint로 다음 partial URL을 미리 알려 플레이어가 즉시 요청 → 서버가 chunk로 흘려보냄으로 RTT 추가 제거
2. Rendition Report로 다른 화질의 현재 상태를 마스터에 포함 → 화질 전환 시 매니페스트 추가 받기 지연 제거
3. LL-HLS는 인코더/CDN/서버 비용 2-3배 증가 — 스포츠/경매/방송 동기 등 지연 중요 서비스만 도입
