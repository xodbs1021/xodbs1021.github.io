## 레벨 2-4 / 1강: LL-HLS가 왜 만들어졌나 — 지연 시간과의 싸움

---

### 라이브 방송에서 지연이 왜 문제인가

스트리머가 폭발 장면을 보면서 "지금!"이라고 외친다. 시청자는 25초 후에 그 장면을 본다. 그 사이 채팅창에는 이미 "와 미쳤다", "지금 죽었어" 같은 반응이 올라온다.

영상보다 채팅이 빠르다. 시청자는 결과를 알고 영상을 본다. 라이브의 의미가 깨진다.

```
[12:00:00.000] 스트리머: 폭발 장면 발생
[12:00:00.500] 다른 시청자 채팅: "와!"
[12:00:00.800] 다른 시청자 채팅: "미쳤다"
[12:00:25.000] 내가 영상에서 폭발 장면을 봄
```

이 지연이 어디서 발생하는지부터 보자.

---

### HLS의 지연 구성 — 25초가 어디에서 나오나

HLS는 영상을 6초짜리 세그먼트 파일로 잘라서 보낸다. 그러면 자연스럽게 단계마다 지연이 쌓인다.

**1. 인코딩 대기 (6초)**
세그먼트는 6초가 모여야 만들어진다. 0초에 일어난 사건이 6초 세그먼트에 들어가려면 5.99초까지 기다려야 한다.

**2. 서버 업로드 (0.5초)**
완성된 세그먼트를 인코더에서 Origin 서버로 보낸다.

**3. CDN 캐시 전파 (1초)**
Origin에서 시청자 근처 CDN 엣지로 콘텐츠가 가야 한다.

**4. 플레이어 폴링 발견 (평균 3초)**
시청자 플레이어는 `.m3u8`을 6초 간격으로 다시 받아본다. 새 세그먼트가 추가됐는지 확인하는 거다. 운 나쁘면 6초, 운 좋으면 0초. 평균 3초 기다린다.

**5. 다운로드 (1초)**
세그먼트 파일을 받는 시간.

**6. 플레이어 버퍼 (18초)**
시청자가 영상을 끊김 없이 보려면 미리 받아둔 영상이 있어야 한다. HLS 플레이어는 보통 3개 세그먼트(18초)를 미리 들고 시작한다.

```
6 + 0.5 + 1 + 3 + 1 + 18 = 29.5초
```

대략 25~30초 지연. 이게 HLS의 본질적 한계다.

---

### 왜 그냥 세그먼트를 짧게 만들면 안 되나

"6초 세그먼트가 문제면 2초로 만들면 되잖아?"

가능하긴 하다. 근데 다른 문제가 생긴다.

**문제 1: CDN 캐시 효율 떨어짐**
1시간 방송이 세그먼트 1800개로 늘어난다. 파일 개수가 늘면 CDN 캐시 관리 부담이 커진다. 작은 파일들은 HTTP 오버헤드(헤더) 비율이 늘어나 효율이 떨어진다.

**문제 2: 재버퍼링 위험**
세그먼트가 작으면 한 번에 받는 데이터도 작다. 플레이어 버퍼가 채워지는 속도가 느려져서 재버퍼링이 자주 발생한다.

**문제 3: 인코더 부하**
세그먼트 경계마다 IDR 프레임이 들어가야 한다. 자주 IDR을 박으면 압축 효율이 떨어져서 같은 화질에 비트레이트가 늘어난다.

```
[세그먼트 6초 vs 2초 비교]
6초 세그먼트:
- 1시간 = 600 파일
- 1080p 평균 비트레이트 6 Mbps

2초 세그먼트:
- 1시간 = 1800 파일
- 1080p 평균 비트레이트 7.5 Mbps (IDR 더 많이 들어가서)
```

세그먼트 길이를 줄이는 단순한 해법으론 안 된다.

### Apple의 해결책 발표 — 2019년 WWDC

Apple이 2019년 WWDC에서 LL-HLS를 발표했다. 정식 명칭은 **Apple Low-Latency HLS**.

목표:
- **3~5초 지연** (기존 25초의 1/5)
- **기존 HLS와 하위 호환**
- **CDN 인프라 그대로 활용**

이게 핵심이다. 새 프로토콜을 만들면 모든 인프라를 다시 만들어야 한다. LL-HLS는 기존 HLS 위에 확장 기능을 얹는 방식.

LL-HLS를 지원하지 않는 옛날 플레이어도 같은 콘텐츠를 그냥 일반 HLS로 재생할 수 있다.

---

### LL-HLS의 4가지 핵심 기술

LL-HLS가 지연을 줄이는 방법은 네 가지다.

**1. Partial Segment (부분 세그먼트)**
6초 세그먼트가 완성되기 전에, 0.2초 단위 부분(Partial)을 먼저 배포.

**2. Blocking Playlist Reload (블로킹 플레이리스트 갱신)**
시청자가 새 정보를 폴링하지 않고, 서버가 새 정보 생기면 응답해주는 방식.

**3. Preload Hint (미리 로드 힌트)**
다음에 올 partial의 URL을 미리 알려줌. 시청자가 즉시 요청할 수 있게.

**4. Rendition Report (다른 화질 정보)**
화질 전환 시 새 화질의 최신 상태를 마스터에서 알려줌. 폴링 없이 즉시 전환.

각 기술은 다음 강의에서 자세히 본다.

---

### LL-HLS의 컨테이너 변화 — fMP4 강제

전통 HLS는 `.ts` (MPEG-TS) 세그먼트를 썼다. LL-HLS는 다르다.

**LL-HLS는 fMP4 (CMAF) 컨테이너 사용 권장.**

왜 바뀌었나?

**1. Partial Segment 표현이 fMP4가 자연스러움**
fMP4는 작은 fragment 단위로 나뉘어 있어서 부분 추출이 쉽다. MPEG-TS는 188바이트 패킷 단위라 partial 경계 만들기 복잡.

**2. CMAF로 DASH와 통합**
fMP4 기반이라 DASH도 같은 파일 쓸 수 있다. 저장 비용 절감.

**3. 효율적 헤더**
MPEG-TS는 약 2% 헤더 오버헤드. fMP4는 1% 미만.

기존 `.ts` 기반 HLS 인프라를 가진 회사들이 LL-HLS 도입할 때 컨테이너부터 바꿔야 한다. 작지 않은 변경.

### LL-HLS 매니페스트 예시

기존 HLS와 LL-HLS의 차이를 매니페스트에서 보면:

```
[전통 HLS .m3u8]
#EXTM3U
#EXT-X-VERSION:3
#EXT-X-TARGETDURATION:6
#EXT-X-MEDIA-SEQUENCE:100
#EXTINF:6.000,
seg100.ts
#EXTINF:6.000,
seg101.ts
#EXTINF:6.000,
seg102.ts


[LL-HLS .m3u8]
#EXTM3U
#EXT-X-VERSION:9
#EXT-X-TARGETDURATION:6
#EXT-X-SERVER-CONTROL:CAN-BLOCK-RELOAD=YES,PART-HOLD-BACK=1.5
#EXT-X-PART-INF:PART-TARGET=0.2
#EXT-X-MEDIA-SEQUENCE:100
#EXTINF:6.000,
seg100.m4s
#EXT-X-PART:DURATION=0.2,URI="seg101_part0.m4s"
#EXT-X-PART:DURATION=0.2,URI="seg101_part1.m4s"
#EXT-X-PART:DURATION=0.2,URI="seg101_part2.m4s"
#EXT-X-PRELOAD-HINT:TYPE=PART,URI="seg101_part3.m4s"
```

새 태그가 여러 개 추가됐다.

- `#EXT-X-SERVER-CONTROL`: 서버가 어떤 LL 기능을 지원하는지
- `#EXT-X-PART-INF`: Partial 세그먼트 목표 길이
- `#EXT-X-PART`: 개별 partial 세그먼트
- `#EXT-X-PRELOAD-HINT`: 다음에 올 partial URL 힌트

### LL-HLS의 지연 구성 — 5초가 어디에서 나오나

LL-HLS는 어떻게 25초를 5초로 줄였나?

```
[LL-HLS 지연 구성]
1. Partial 인코딩 대기: 0.2초 (6초가 아니라)
2. 즉시 배포: 0.1초
3. CDN 캐시: 0.3초 (HTTP/2 Push 활용)
4. 플레이어 즉시 발견: 0초 (Blocking Reload)
5. 다운로드: 0.5초
6. 짧은 버퍼: 3초

총: 4.1초
```

각 단계의 지연이 크게 줄었다.

핵심 변화:
- 6초 → 0.2초 (partial 단위)
- 폴링 평균 3초 → 즉시 (blocking reload)
- 버퍼 18초 → 3초 (작은 단위니 안정성 확보 가능)

---

### LL-HLS 도입 부담 — CDN의 문제

LL-HLS가 잘 동작하려면 CDN이 새 기능을 지원해야 한다.

**HTTP/2 Push 또는 Chunked Transfer Encoding 필요**

Partial 세그먼트가 완성되기 전에 클라이언트에게 흘려보내야 하기 때문이다. 일반 HTTP는 응답이 완성돼야 보내지만, LL-HLS는 만들어지는 동안 흘려보낸다.

이게 모든 CDN이 잘 지원하지 못한다. 2024년 현재:
- Akamai, CloudFront, Cloudflare: 지원
- 일부 소규모 CDN: 미지원

CDN 비용도 올라간다. partial 요청이 6배 늘어나니 CDN 부하 증가.

---

### LL-HLS vs WebRTC — 누가 더 빠르나

라이브 저지연 솔루션 비교.

| | LL-HLS | WebRTC |
|---|---|---|
| 지연 | 2-5초 | 200ms 이하 |
| CDN 활용 | 가능 | 어려움 |
| 대규모 시청자 | 100만+ | 수천 명 단위 |
| 인프라 비용 | HLS와 비슷 | 비쌈 |
| 모바일 지원 | 안정 | 변동 |

지연만 보면 WebRTC가 압도적. 하지만 CDN을 못 쓰니 대규모 라이브 방송엔 부적합.

LL-HLS는 "충분히 낮은 지연" + "기존 인프라 활용" 절충안. 치지직 같은 일반 라이브 방송에 적합.
