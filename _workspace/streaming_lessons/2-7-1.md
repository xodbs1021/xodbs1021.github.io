## 레벨 2-7 / 1강: WHIP — WebRTC 인제스트 표준의 등장

---

### 라이브 송출의 두 가지 흐름

스트리밍 산업이 두 갈래로 나뉘었다.

**갈래 1: 전통 HLS/RTMP**
```
스트리머 → RTMP → 플랫폼 → HLS → 시청자
지연: 25초
호환성: 매우 좋음
```

**갈래 2: WebRTC**
```
참가자 ↔ WebRTC ↔ 참가자
지연: 200ms
참가자 수: 제한적
```

두 갈래 사이에 빈 공간이 있다.

**저지연으로 송출하지만 대규모 시청자에게 배포**하고 싶을 때 표준이 없었다.

---

### 문제의 본질 — WebRTC가 표준화 안 된 영역

WebRTC 자체는 표준이지만, "어떻게 시그널링 하느냐"는 회사마다 다르다.

```
[WebRTC 회사별 시그널링]
Google Meet: 자체 시그널링
Zoom: 자체 시그널링
Discord: 자체 시그널링
Twitch: 자체 시그널링 (2024년부터 WHIP 도입)
유튜브: 자체 시그널링 (WHIP 도입 중)
```

OBS에서 송출 버튼을 누르면 어디로 어떻게 보낼지? 플랫폼마다 다른 SDK 필요.

RTMP는 단순했다. URL + 스트림 키 두 개면 끝. 이렇게 표준화된 게 WebRTC에 필요했다.

---

### WHIP의 발표 — 2023년 IETF RFC 9725

**WHIP (WebRTC-HTTP Ingestion Protocol)**: WebRTC를 사용한 라이브 송출의 표준 프로토콜.

```
[WHIP 핵심 아이디어]
"WebRTC 시그널링을 HTTP POST로 단순화"
```

복잡한 시그널링 프로토콜 안 만들고, HTTP의 단순함을 활용.

```
[WHIP 흐름]
1. 스트리머 → 서버: HTTP POST /whip
   Body: SDP Offer
   
2. 서버 → 스트리머: HTTP 201 Created
   Body: SDP Answer
   Location: /whip/resource_id

3. WebRTC 연결 수립 (ICE, DTLS)

4. 미디어 송출 (SRTP)

5. 종료: HTTP DELETE /whip/resource_id
```

전체 시그널링이 HTTP 한 번에 끝. 그 뒤로 표준 WebRTC.

---

### WHIP의 단순함이 큰 가치

WHIP 이전 OBS → 플랫폼 WebRTC 송출:
```
1. 플랫폼별 SDK 다운로드
2. SDK 사용해서 시그널링 처리
3. WebRTC 연결 수립
4. 미디어 송출

→ 각 플랫폼마다 OBS 플러그인 따로 만들어야 함
```

WHIP 이후:
```
[OBS 설정]
서비스: WHIP
서버: https://whip.example.com/live
Bearer Token: abc123

→ 모든 WHIP 지원 플랫폼에 동일하게 송출
```

RTMP의 단순함 + WebRTC의 저지연.

---

### WHIP과 RTMP 비교

| | RTMP | WHIP |
|---|---|---|
| 발표 | 2002 | 2023 |
| 시그널링 | RTMP Handshake | HTTP POST |
| 미디어 | TCP | UDP + SRTP |
| 지연 | 2-3초 | 200ms |
| 코덱 | H.264, AAC만 | H.264, H.265, AV1, Opus |
| 멀티 트랙 | ❌ | ✅ |
| 암호화 | RTMPS 옵션 | DTLS 필수 |
| NAT 통과 | TCP라 쉬움 | ICE |
| OBS 지원 | 표준 | OBS 30+ |
| 표준화 | 비공식 RFC | RFC 9725 |

WHIP이 모든 면에서 우수. 호환성/생태계만 RTMP가 우위.

---

### WHIP의 실제 HTTP 요청

WHIP 송출 시작 시.

```http
POST /whip HTTP/1.1
Host: whip.example.com
Content-Type: application/sdp
Authorization: Bearer abc123-xyz

v=0
o=- 4611732250362950873 2 IN IP4 127.0.0.1
s=-
t=0 0
a=group:BUNDLE 0 1
m=audio 9 UDP/TLS/RTP/SAVPF 111
a=sendonly
a=rtpmap:111 opus/48000/2
m=video 9 UDP/TLS/RTP/SAVPF 96
a=sendonly
a=rtpmap:96 H264/90000
a=fmtp:96 profile-level-id=42e01f
...
```

서버 응답:

```http
HTTP/1.1 201 Created
Content-Type: application/sdp
Location: /whip/abc-123-resource

v=0
o=- 7890 2 IN IP4 0.0.0.0
s=-
t=0 0
a=group:BUNDLE 0 1
m=audio 9 UDP/TLS/RTP/SAVPF 111
a=recvonly
a=rtpmap:111 opus/48000/2
m=video 9 UDP/TLS/RTP/SAVPF 96
a=recvonly
a=rtpmap:96 H264/90000
...
```

응답의 `Location` 헤더가 중요. 이 리소스 URL로 나중에 DELETE 요청해서 종료.

---

### ICE Trickle도 HTTP로

ICE 후보 교환도 HTTP로.

```http
PATCH /whip/abc-123-resource HTTP/1.1
Host: whip.example.com
Content-Type: application/trickle-ice-sdpfrag

a=candidate:1 1 UDP 2122252543 192.168.0.10 60001 typ host
a=candidate:2 1 UDP 1686052607 1.2.3.4 60002 typ srflx
```

서버가 자기 후보도 응답으로 전달.

WebSocket 같은 양방향 채널 불필요.

---

### 종료

```http
DELETE /whip/abc-123-resource HTTP/1.1
Host: whip.example.com
Authorization: Bearer abc123-xyz
```

서버가 200 OK 응답하면 연결 종료. WebRTC도 닫힘.

---

### Authorization — Bearer Token

WHIP은 HTTP라 표준 인증 사용 가능.

```http
Authorization: Bearer abc123-xyz
```

플랫폼이 스트리머에게 토큰 발급. RTMP의 스트림 키 역할.

토큰 안에 정보 포함:
- 스트리머 ID
- 채널 ID
- 만료 시간
- 권한
- 사용 가능 코덱

서버가 검증 후 송출 허용/거부.

---

### OBS에서 WHIP 설정

OBS 30+에서 WHIP 출력 기본 지원.

```
[OBS 설정 - 방송]
서비스: WHIP

서버: https://whip.example.com/live

Bearer Token: your_token_here
```

다른 옵션:
- 인코더: H.264 (NVENC, x264 등)
- 비트레이트: 보통 4000-8000 Kbps
- 키프레임 간격: 2초 (WebRTC 권장)

송출 시작 누르면 HTTP POST 발사.

### WHIP 서버 구현 예시

간단한 WHIP 서버를 Go로 만들면.

```go
package main

import (
  "fmt"
  "net/http"
  "github.com/pion/webrtc/v4"
)

func handleWhip(w http.ResponseWriter, r *http.Request) {
  // SDP Offer 읽기
  offer, _ := io.ReadAll(r.Body)
  
  // WebRTC 연결 생성
  pc, _ := webrtc.NewPeerConnection(webrtc.Configuration{
    ICEServers: []webrtc.ICEServer{
      {URLs: []string{"stun:stun.l.google.com:19302"}},
    },
  })
  
  // 트랜시버 추가 (수신 전용)
  pc.AddTransceiverFromKind(webrtc.RTPCodecTypeAudio, webrtc.RTPTransceiverInit{
    Direction: webrtc.RTPTransceiverDirectionRecvonly,
  })
  pc.AddTransceiverFromKind(webrtc.RTPCodecTypeVideo, webrtc.RTPTransceiverInit{
    Direction: webrtc.RTPTransceiverDirectionRecvonly,
  })
  
  // Offer 적용
  pc.SetRemoteDescription(webrtc.SessionDescription{
    Type: webrtc.SDPTypeOffer,
    SDP:  string(offer),
  })
  
  // Answer 생성
  answer, _ := pc.CreateAnswer(nil)
  pc.SetLocalDescription(answer)
  
  // 미디어 수신 처리
  pc.OnTrack(func(track *webrtc.TrackRemote, receiver *webrtc.RTPReceiver) {
    // 받은 RTP 패킷 처리
    for {
      pkt, _, _ := track.ReadRTP()
      // 다른 시청자에게 전달 또는 저장
    }
  })
  
  // Answer 응답
  w.Header().Set("Content-Type", "application/sdp")
  w.Header().Set("Location", "/whip/" + generateID())
  w.WriteHeader(http.StatusCreated)
  w.Write([]byte(answer.SDP))
}

func main() {
  http.HandleFunc("/whip", handleWhip)
  http.ListenAndServe(":8080", nil)
}
```

Pion 라이브러리 활용. 100줄 미만으로 WHIP 서버.

WHIP 채택 상황.

**완전 채택**:
- Twitch (베타에서 안정으로 전환)
- Cloudflare Stream
- Mux Live
- Dolby.io

**부분 채택**:
- YouTube Live (실험적)
- Facebook Live (테스트)

**미채택**:
- 한국 라이브 플랫폼 (치지직, SOOP) — RTMP 유지

생태계 관성으로 RTMP가 여전히 우세하지만 WHIP이 점진적 확산.

### WHIP의 영향 — 라이브 산업 변화

WHIP이 보급되면 일어날 변화.

**1. 지연 단축**
모든 라이브 방송이 25초 → 1초 수준으로.

**2. 비용 절감**
중계 서버 부하 감소 (WebRTC가 효율적).

**3. 스트리머 다양화**
모바일에서도 OBS 수준의 송출 가능 (WHIP SDK 활용).

**4. 인터랙티브 기능**
저지연으로 시청자와 즉시 상호작용.

**5. 새로운 사용 시나리오**
원격 카메라, 라이브 협업, 게임 스트리밍 등.

### WHIP과 RTMP의 공존

당분간은 둘 다 운영하는 게 현실.

```
[하이브리드 플랫폼]
스트리머 옵션:
- 옵션 A: RTMP (구버전 OBS, 호환성)
- 옵션 B: WHIP (최신 OBS, 저지연)

플랫폼 처리:
- RTMP 받음 → 트랜스코딩 → HLS 배포
- WHIP 받음 → 트랜스코딩 → HLS + LL-HLS + WHEP 배포
```

스트리머가 선택. 시청자 측은 동일.

### 1강 핵심 3줄

1. WHIP은 2023년 IETF RFC 9725로 표준화된 WebRTC 송출 프로토콜 — HTTP POST 한 번으로 시그널링 완료
2. RTMP의 단순함과 WebRTC의 저지연을 결합 — Bearer Token 인증, ICE Trickle 후 DELETE로 종료
3. OBS 30+, Twitch, Cloudflare 채택 중 — RTMP 완전 대체보다는 하이브리드 운영이 현실
