## 레벨 4-4 / 1강: CMAF가 왜 만들어졌나 — HLS/DASH의 분열을 끝내려는 시도

---

### 2010년대 중반의 분열

HLS와 DASH가 산업을 양분.

```
[2015년 즈음]
Apple 진영 (HLS):
  - iOS, Safari, Apple TV
  - 컨테이너: MPEG-TS (.ts)
  - 매니페스트: .m3u8

Google/Microsoft 진영 (DASH):
  - Android (Chrome), 스마트TV, Xbox
  - 컨테이너: fMP4 (.m4s)
  - 매니페스트: .mpd
```

플랫폼이 두 포맷 다 지원하려면.

```
[같은 콘텐츠 인코딩 2번]
HLS용: input.mp4 → .ts 세그먼트 + .m3u8
DASH용: input.mp4 → .m4s 세그먼트 + .mpd

→ 인코딩 비용 2배
→ 스토리지 2배
→ CDN 캐시 2배
```

대형 OTT (Netflix, Disney)에 매년 수억 달러 낭비.

---

### CMAF의 핵심 아이디어 — 하나의 세그먼트

2016년 Apple과 Microsoft 공동 제안.

```
[CMAF (Common Media Application Format)]
세그먼트: fMP4 통일 (.m4s)
HLS와 DASH가 같은 .m4s 공유
매니페스트만 다름 (.m3u8 vs .mpd)
```

같은 미디어 파일을 두 프로토콜이 공유.

```
[CMAF 구조]
encoding/
├── master.m3u8       ← HLS 매니페스트
├── manifest.mpd      ← DASH 매니페스트
└── segments/
    ├── init.mp4      ← 초기화 세그먼트 (공유)
    ├── seg_1.m4s     ← 미디어 세그먼트 (공유)
    ├── seg_2.m4s
    └── ...
```

같은 .m4s를 시청자의 플레이어가 선택적으로 받음.

---

### CMAF가 가능했던 이유 — 둘 다 fMP4

DASH는 처음부터 fMP4 기반. HLS는 2016년에 fMP4를 받아들였다.

```
[HLS 발전]
2009년: TS만 지원
2016년: HLS v7에서 fMP4 (fragmented MP4) 지원

이게 CMAF의 필수 조건.
```

iOS 10 (2016년 9월) 이후 모든 Apple 디바이스가 fMP4 HLS 지원.

이전 디바이스 호환성? 이제 거의 무시 가능 (8년 전 디바이스).

---

### fMP4의 구조 — MP4와 어떻게 다른가

일반 MP4와 fMP4의 차이.

```
[일반 MP4]
moov (메타데이터, 어디에 뭐가 있는지)
mdat (미디어 데이터)
→ moov가 전체 미디어 정보 가짐
→ 스트리밍에 안 좋음 (moov 받아야 재생 시작)

[fMP4]
moov (초기화 메타데이터, 작음)
moof + mdat (조각)
moof + mdat
moof + mdat
...
→ 각 조각이 독립적
→ 스트리밍/세그먼트화에 최적
```

`moof` (movie fragment): 각 조각의 메타데이터.
`mdat` (media data): 실제 미디어.

---

### 초기화 세그먼트 (init.mp4)

CMAF/fMP4에서 첫 번째로 받아야 하는 파일.

```
init.mp4 구조:
├── ftyp (file type)
└── moov (초기화 메타데이터)
    ├── 트랙 정보
    ├── 코덱 정보 (H.264 SPS/PPS)
    └── 타임스케일
```

미디어 세그먼트(.m4s)는 init.mp4 정보 없이 디코딩 못 함.

플레이어가 처음에 init.mp4를 받고, 이후 .m4s들을 받는다.

```
[HLS fMP4 플레이리스트 예시]
#EXTM3U
#EXT-X-VERSION:6
#EXT-X-MAP:URI="init.mp4"   ← 초기화 세그먼트
#EXTINF:6.000,
seg_1.m4s
#EXTINF:6.000,
seg_2.m4s
```

`#EXT-X-MAP`이 init.mp4 가리킴.

---

### 미디어 세그먼트 (.m4s)

각 조각.

```
seg_N.m4s 구조:
├── styp (segment type)
├── sidx (segment index, 선택)
├── moof (조각 메타데이터)
│   ├── 시간 정보 (PTS, DTS)
│   ├── 샘플 사이즈
│   └── 박스 오프셋
└── mdat (실제 미디어 데이터)
```

`moof`가 시간 정보 가지고 있으니 init.mp4 + 어떤 .m4s든 독립 재생 가능.

이 구조 덕분에 시킹/ABR 전환이 자연스럽다.

---

### CMAF의 코덱과 컨테이너 분리

CMAF는 컨테이너 표준일 뿐. 코덱은 별도.

```
[CMAF가 정의하는 것]
- fMP4 사용
- 박스 구조
- 초기화 세그먼트 + 미디어 세그먼트 분리

[CMAF가 정의 안 하는 것]
- 어떤 코덱 (H.264, H.265, AV1 모두 OK)
- 어떤 오디오 (AAC, Opus 모두 OK)
- 매니페스트 (HLS와 DASH 둘 다)
```

코덱은 자유. 그래서 H.264 CMAF, H.265 CMAF, AV1 CMAF 다 가능.

### CMAF 도입 효과

대형 OTT의 실제 절감.

```
[Disney+ 사례 (추정)]
- 인코딩 비용: 40% 절감 (한 번만)
- 스토리지: 50% 절감 (.ts + .m4s → .m4s만)
- CDN 캐시 히트율: 30% 향상 (한 곳에 모임)
- 신규 콘텐츠 추가 시 처리량: 2배
```

매년 수억 달러 절감.

---

### CMAF의 LL-HLS 호환

저지연 HLS의 핵심 기술 중 하나가 CMAF.

```
[일반 HLS 6초 세그먼트]
6초마다 .ts 파일 생성
플레이어 지연: 18~25초

[LL-HLS (CMAF Chunked)]
6초 .m4s 세그먼트
+ 200ms 단위 "chunk" 안에서 점진적 수신
플레이어 지연: 2~5초
```

CMAF chunked가 LL-HLS의 기반. 다음 강의에서 깊이.

---

### DRM 통합 - CMAF의 또 다른 장점

DRM(콘텐츠 보호)도 CMAF로 통일.

```
[CMAF 이전]
HLS: SAMPLE-AES 또는 FairPlay (Apple만)
DASH: PlayReady (Microsoft), Widevine (Google)

→ HLS용 암호화 + DASH용 암호화 따로

[CMAF 이후]
CENC (Common Encryption) 표준
같은 암호화된 .m4s를 모든 DRM이 해독
HLS: FairPlay 키
DASH: Widevine 또는 PlayReady 키
```

OTT의 멀티 DRM 운영이 훨씬 단순해짐.

### CMAF 매니페스트 작성 — HLS

```
#EXTM3U
#EXT-X-VERSION:7
#EXT-X-INDEPENDENT-SEGMENTS

#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="audio",NAME="English",LANGUAGE="en",DEFAULT=YES,URI="audio/playlist.m3u8"

#EXT-X-STREAM-INF:BANDWIDTH=6500000,RESOLUTION=1920x1080,CODECS="avc1.640028,mp4a.40.2",AUDIO="audio"
video_1080p/playlist.m3u8

#EXT-X-STREAM-INF:BANDWIDTH=3200000,RESOLUTION=1280x720,CODECS="avc1.64001f,mp4a.40.2",AUDIO="audio"
video_720p/playlist.m3u8
```

video_1080p/playlist.m3u8:
```
#EXTM3U
#EXT-X-VERSION:7
#EXT-X-TARGETDURATION:6
#EXT-X-MAP:URI="init.mp4"
#EXT-X-MEDIA-SEQUENCE:0
#EXTINF:6.0,
seg_1.m4s
#EXTINF:6.0,
seg_2.m4s
...
```

`CODECS="avc1.640028"`이 H.264 high profile level 4.0. 플레이어가 이걸 보고 디코딩 가능 여부 판단.

### CMAF 매니페스트 작성 — DASH (같은 미디어)

```xml
<?xml version="1.0" encoding="UTF-8"?>
<MPD xmlns="urn:mpeg:dash:schema:mpd:2011" type="static" minBufferTime="PT6S">
  <Period>
    <AdaptationSet mimeType="video/mp4" codecs="avc1.640028">
      <Representation id="1080p" bandwidth="6500000" width="1920" height="1080">
        <SegmentTemplate 
          initialization="video_1080p/init.mp4"
          media="video_1080p/seg_$Number$.m4s"
          startNumber="1"
          duration="6"
          timescale="1"/>
      </Representation>
      <Representation id="720p" bandwidth="3200000" width="1280" height="720">
        <SegmentTemplate 
          initialization="video_720p/init.mp4"
          media="video_720p/seg_$Number$.m4s"
          startNumber="1"
          duration="6"
          timescale="1"/>
      </Representation>
    </AdaptationSet>
  </Period>
</MPD>
```

같은 init.mp4와 seg_N.m4s 파일들을 가리킴.

### CMAF 도입 시 주의점

**1. 구형 디바이스 호환성**
iOS 9 이하, Android 4 이하, 구형 스마트TV는 fMP4 못 함.

```
[보완책]
- 시청자 통계로 호환성 영향 평가
- 5% 미만이면 무시
- 5% 이상이면 TS 폴백 별도 유지
```

치지직처럼 게임 방송 시청자는 거의 다 최신 디바이스. 안전한 도입.

**2. CMAF Constraint 준수**
모든 fMP4가 CMAF는 아님. 표준 따라야.

```
- 각 .m4s는 키프레임으로 시작
- moof + mdat 박스 순서
- 타임스케일 일관성
```

Shaka Packager가 자동 보장. 직접 만들 일은 거의 없음.

**3. 짧은 init.mp4 생명주기**
SPS/PPS가 바뀌면 init.mp4도 다시 받아야. 코덱 설정 도중 변경 금지.

### CMAF 대신 TS를 계속 쓰는 곳

여전히 TS 쓰는 곳들.

```
[TS 유지하는 곳]
- 매우 구형 디바이스 타겟 (피처폰)
- 방송 인프라 연동 (MPEG-TS는 방송 표준)
- 일부 RTMP/SRT 인제스트
- 라이브 분석 파이프라인 (PokeClip 같은)
```

치지직 외부 인제스트는 RTMP/SRT (MPEG-TS 기반)지만, CDN에는 CMAF로 변환해서 보낸다.

### 1강 핵심 3줄

1. CMAF는 HLS와 DASH의 미디어 통합 표준 (fMP4 .m4s 공유) — 인코딩 1번으로 양쪽 지원, 비용 40% 절감
2. fMP4 = init.mp4 (메타데이터) + 여러 .m4s (조각) 구조, 각 조각이 독립 디코딩 가능
3. CENC로 멀티 DRM 통합 + LL-HLS의 chunked CMAF 기반 — 대형 OTT의 사실상 표준
