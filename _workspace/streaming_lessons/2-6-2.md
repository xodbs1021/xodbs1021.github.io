## 레벨 2-6 / 2강: SDP와 ICE — WebRTC 연결의 핵심

---

### SDP가 뭔가

**SDP (Session Description Protocol)**: 미디어 세션의 정보를 텍스트로 표현하는 표준.

WebRTC 연결 시작 전에 양쪽이 SDP를 주고받는다. "내가 지원하는 코덱, 미디어 종류, 네트워크 정보는 이거야."

```
[SDP 교환 = Offer/Answer 모델]
A → B: SDP Offer ("나는 H.264, Opus, 비디오/오디오 지원해")
B → A: SDP Answer ("나도 H.264, Opus 가능해. 이걸로 합의")
```

이게 시그널링에서 가장 중요한 데이터.

---

### SDP 줄별 의미

**세션 레벨**:
```
v=0          → SDP 버전 0
o=- ...      → 세션 발신자 정보
s=-          → 세션 이름 (없음)
t=0 0        → 시간 (라이브)
a=group:BUNDLE 0 1  → 미디어 묶음 (BUNDLE)
```

**미디어 레벨 - 오디오**:
```
m=audio 9 UDP/TLS/RTP/SAVPF 111 103 104 ...
        ↑    ↑                  ↑
       포트  프로토콜            지원 코덱 페이로드 타입 목록

c=IN IP4 0.0.0.0  → 연결 정보 (ICE로 동적 결정)

a=ice-ufrag:F7gI      → ICE 사용자 이름
a=ice-pwd:x9cml/...   → ICE 비밀번호
a=fingerprint:...     → DTLS 인증서 지문

a=setup:actpass       → DTLS 역할
a=mid:0               → 미디어 식별자

a=sendrecv            → 양방향 (send + recv)
                       sendonly, recvonly, inactive 가능

a=rtpmap:111 opus/48000/2  → 페이로드 111 = Opus, 48kHz, 2채널
a=rtpmap:0 PCMU/8000       → 페이로드 0 = PCMU
```

**미디어 레벨 - 비디오**:
```
m=video 9 UDP/TLS/RTP/SAVPF 96 97 98 ...

a=rtpmap:96 VP8/90000       → VP8 코덱
a=rtpmap:100 H264/90000     → H.264 코덱
a=fmtp:100 profile-level-id=42e01f  → H.264 프로파일
```

### Offer/Answer 동작

연결 시작 측이 Offer, 받는 측이 Answer.

```javascript
// 시작 측 (Caller)
const pc = new RTCPeerConnection(config);
stream.getTracks().forEach(track => pc.addTrack(track, stream));

const offer = await pc.createOffer();
await pc.setLocalDescription(offer);

// offer.sdp를 시그널링 서버로 전송
signaling.send({ type: 'offer', sdp: offer.sdp });
```

```javascript
// 받는 측 (Callee)
signaling.on('offer', async (msg) => {
  await pc.setRemoteDescription({ type: 'offer', sdp: msg.sdp });
  
  const answer = await pc.createAnswer();
  await pc.setLocalDescription(answer);
  
  signaling.send({ type: 'answer', sdp: answer.sdp });
});
```

```javascript
// 시작 측이 Answer 받음
signaling.on('answer', async (msg) => {
  await pc.setRemoteDescription({ type: 'answer', sdp: msg.sdp });
  // 이제 미디어 흐름 준비 완료
});
```

이 단계가 끝나면 양쪽이 서로의 미디어 능력을 안다.

### ICE Candidate Gathering

SDP에는 IP/포트 정보가 들어가야 한다. 근데 자기 IP가 뭔지 모른다 (NAT 뒤).

ICE가 후보 IP를 수집.

```javascript
const pc = new RTCPeerConnection({
  iceServers: [
    { urls: 'stun:stun.l.google.com:19302' },
    {
      urls: 'turn:turn.example.com:3478',
      username: 'user',
      credential: 'password'
    }
  ]
});

pc.onicecandidate = (event) => {
  if (event.candidate) {
    // 새 후보 발견됨
    console.log('Candidate:', event.candidate.candidate);
    
    // 상대방에게 전달
    signaling.send({
      type: 'ice_candidate',
      candidate: event.candidate
    });
  }
};
```

수집되는 후보 종류:

**Host Candidate**: 자기 네트워크 인터페이스 IP.
```
candidate:1 1 UDP 2122252543 192.168.0.10 60001 typ host
```

**Server Reflexive Candidate (srflx)**: STUN으로 알아낸 공용 IP.
```
candidate:2 1 UDP 1686052607 1.2.3.4 60002 typ srflx
```

**Relayed Candidate (relay)**: TURN 서버 IP.
```
candidate:3 1 UDP 41819903 5.6.7.8 60003 typ relay
```

---

### ICE Candidate 형식

각 후보는 한 줄 텍스트.

```
candidate:1 1 UDP 2122252543 192.168.0.10 60001 typ host
          ↑ ↑  ↑   ↑           ↑           ↑     ↑
          │ │  │   │           │           │     타입
          │ │  │   │           │           포트
          │ │  │   │           IP
          │ │  │   우선순위
          │ │  프로토콜
          │ 컴포넌트 (1=RTP, 2=RTCP)
          후보 식별자
```

**우선순위**가 중요. Host > Server Reflexive > Relay 순서. ICE는 우선순위 높은 후보부터 시도.

---

### ICE Trickle — 점진적 후보 교환

후보 수집이 모두 끝날 때까지 기다리면 시간 낭비. ICE Trickle은 발견되는 즉시 보냄.

```
[ICE Trickle 흐름]
T=0.000: Offer 전송 (후보 0개)
T=0.050: Answer 받음
T=0.100: Host candidate 1 발견 → 즉시 전송
T=0.150: Host candidate 2 발견 → 즉시 전송
T=0.300: STUN으로 srflx 발견 → 즉시 전송
T=0.500: TURN 활성화 → relay 후보 전송
```

후보들이 비동기로 흘러간다. 양쪽 모두 받는 즉시 연결 시도.

```javascript
pc.onicecandidate = (event) => {
  if (event.candidate === null) {
    // 모든 후보 수집 완료
    console.log('ICE gathering done');
  } else {
    // 후보 발견 - 즉시 시그널링 서버로
    signaling.send({ candidate: event.candidate });
  }
};
```

---

### ICE Connectivity Check

후보들이 모이면 모든 쌍에 대해 연결 가능한지 확인.

```
[Connectivity Check]
A 후보: host_A, srflx_A, relay_A
B 후보: host_B, srflx_B, relay_B

검사할 쌍:
(host_A, host_B)     ← 같은 네트워크면 성공
(host_A, srflx_B)
(srflx_A, host_B)
(srflx_A, srflx_B)   ← P2P 성공률 높음
(relay_A, host_B)
(relay_A, srflx_B)
(relay_A, relay_B)   ← 최후 폴백
...

성공한 쌍 중 우선순위 최고로 결정
```

STUN Binding Request 패킷을 보내서 응답이 오는지 확인. 응답 오면 연결 가능.

### ICE State Machine

연결 상태가 여러 단계로 진행.

```javascript
pc.oniceconnectionstatechange = () => {
  console.log('ICE state:', pc.iceConnectionState);
};

// 상태 흐름:
// new → checking → connected → completed
//            ↓
//        disconnected → failed
//            ↑
//         재연결 시도
```

**new**: 시작
**checking**: 후보 쌍 검사 중
**connected**: 한 쌍 성공
**completed**: 모든 검사 완료
**disconnected**: 일시 끊김
**failed**: 모든 쌍 실패
**closed**: 명시적 종료

`disconnected`는 자동 복구 가능. `failed`는 새 후보 가져와야 함.

### ICE Restart — 네트워크 변경 처리

와이파이 ↔ LTE 전환 시 IP가 바뀐다. ICE Restart로 처리.

```javascript
// 네트워크 변경 감지
pc.oniceconnectionstatechange = async () => {
  if (pc.iceConnectionState === 'disconnected') {
    // ICE Restart
    const offer = await pc.createOffer({ iceRestart: true });
    await pc.setLocalDescription(offer);
    signaling.send({ type: 'offer', sdp: offer.sdp });
  }
};
```

새 후보를 다시 수집하고 연결을 재수립. 기존 연결은 잠시 끊김.

### DTLS Handshake

ICE로 연결 경로가 잡힌 후, DTLS로 암호화 키 교환.

```
[DTLS Handshake]
1. Client Hello (랜덤 + 지원 암호 슈트)
2. Server Hello (랜덤 + 선택 슈트)
3. 인증서 교환 (자체 서명 가능)
4. Pre-Master Secret 교환
5. Master Key 생성
6. Finished 메시지

이후 SRTP 키도 DTLS Master에서 파생
```

WebRTC는 자체 서명 인증서 사용. 신뢰는 SDP의 fingerprint로 검증.

```
a=fingerprint:sha-256 4A:79:6B:E5:...
```

이 fingerprint와 DTLS에서 받은 인증서의 해시가 같으면 OK.

### BUNDLE — 한 연결로 여러 미디어

```
a=group:BUNDLE 0 1
```

오디오, 비디오, 데이터를 **같은 UDP 포트로** 다중화. 한 번만 ICE, 한 번만 DTLS.

```
[BUNDLE 없으면]
- 오디오: ICE + DTLS + RTP (포트 5000)
- 비디오: ICE + DTLS + RTP (포트 5002)
- 데이터: ICE + DTLS + RTP (포트 5004)
→ 핸드셰이크 3번

[BUNDLE 있으면]
- 오디오 + 비디오 + 데이터: ICE + DTLS 한 번 (포트 5000)
- RTP 헤더의 mid 필드로 구분
→ 핸드셰이크 1번
```

연결 시간 단축. 모든 현대 WebRTC 구현이 BUNDLE 사용.

---

### RTCP-MUX — RTP와 RTCP 다중화

```
a=rtcp-mux
```

RTP(미디어 데이터)와 RTCP(제어 메시지)를 같은 포트로.

```
[전통]
RTP:  포트 5000
RTCP: 포트 5001

[RTCP-MUX]
RTP + RTCP: 포트 5000
RTP/RTCP 헤더로 구분
```

포트 사용 절약, NAT 통과 단순화.

### Perfect Negotiation 패턴

양쪽이 동시에 Offer를 만들면 충돌. 표준 해결 패턴.

```javascript
let makingOffer = false;
let ignoreOffer = false;
const polite = true;  // 한 쪽만 polite (false면 impolite)

pc.onnegotiationneeded = async () => {
  try {
    makingOffer = true;
    const offer = await pc.createOffer();
    await pc.setLocalDescription(offer);
    signaling.send({ type: 'offer', sdp: offer.sdp });
  } finally {
    makingOffer = false;
  }
};

signaling.on('offer', async (msg) => {
  const offerCollision = 
    pc.signalingState !== 'stable' || makingOffer;
  
  ignoreOffer = !polite && offerCollision;
  if (ignoreOffer) return;
  
  await pc.setRemoteDescription({ type: 'offer', sdp: msg.sdp });
  const answer = await pc.createAnswer();
  await pc.setLocalDescription(answer);
  signaling.send({ type: 'answer', sdp: answer.sdp });
});
```

복잡하지만 동시 협상 충돌 자동 처리. 표준 W3C 권장.

### 2강 핵심 3줄

1. SDP는 미디어 세션 정보(코덱, 포트, ICE/DTLS 정보) 텍스트 표현 — Offer/Answer 모델로 양쪽 능력 합의
2. ICE는 Host(자기 IP), Server Reflexive(STUN으로 알아낸 공용 IP), Relay(TURN) 3가지 후보 수집 후 모든 쌍 connectivity check
3. ICE Trickle로 후보 비동기 전송, BUNDLE + RTCP-MUX로 한 포트에 모든 미디어 다중화, DTLS-SRTP로 암호화
