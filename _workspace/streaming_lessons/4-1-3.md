## 레벨 4-1 / 3강: 비디오 인코딩 옵션 — preset, CRF, 비트레이트 제어

---

### x264 옵션이 가장 많이 쓰인다

라이브 스트리밍 인프라에서 90% 이상이 x264 (소프트웨어 H.264 인코더).

```bash
ffmpeg -i input.mp4 \
  -c:v libx264 \           # x264 인코더 사용
  -preset medium \         # 속도 vs 화질 트레이드오프
  -crf 23 \                # 품질 (낮을수록 화질 ↑)
  -profile:v high \        # H.264 프로파일
  -level 4.0 \             # 디코더 호환성 레벨
  -pix_fmt yuv420p \       # 픽셀 포맷
  -g 60 \                  # GOP 크기 (키프레임 간격)
  -keyint_min 60 \         # 최소 키프레임 간격
  -sc_threshold 0 \        # 씬 체인지 키프레임 비활성
  output.mp4
```

옵션이 많아 보이지만 각각 명확한 역할이 있다.

---

### -preset — 속도 vs 화질

같은 비트레이트로 인코딩할 때 얼마나 압축을 열심히 할지.

```
[preset 옵션 (느림 → 빠름)]
placebo > veryslow > slower > slow > medium > fast > faster > veryfast > superfast > ultrafast
```

같은 화질 기준 인코딩 시간 비교:

```
placebo:   100x (실용 불가)
veryslow:  10x
slow:      3x
medium:    1x (기준)
fast:      0.7x
veryfast:  0.4x
ultrafast: 0.1x (10배 빠름, 파일 크기 30% 큼)
```

용도별 선택:

```
VOD 아카이브:      slow / slower (한 번 인코딩, 평생 저장)
YouTube 업로드:    medium / fast (균형)
라이브 스트리밍:   veryfast / superfast (실시간 필수)
긴급 처리:         ultrafast (속도 최우선)
```

라이브에서 `medium`은 너무 느려서 못 쓴다. 1080p60을 medium으로 인코딩하면 0.5x speed 나오는 경우가 많다.

### -tune — 콘텐츠 특성에 맞춤

영상 종류에 따른 최적화.

```bash
-tune zerolatency  # 라이브 (B-frame 안 씀, look-ahead 비활성)
-tune film         # 영화
-tune animation    # 애니메이션 (큰 영역의 평면 색)
-tune grain        # 필름 그레인 보존
-tune fastdecode   # 디코딩 부하 최소화
-tune stillimage   # 정지 이미지 (썸네일 시퀀스)
```

라이브에서는 거의 항상 `zerolatency`.

```bash
# 라이브 표준 옵션
ffmpeg -re -i rtmp://input \
  -c:v libx264 -preset veryfast -tune zerolatency \
  ...
```

zerolatency가 하는 일:
- B-frame 비활성 (B는 미래 프레임 참조해야 해서 지연 발생)
- look-ahead 0 (미래 프레임 미리 보고 비트 배분 안 함)
- 슬라이스드 스레딩 (병렬 처리)

화질이 약간 떨어지지만 지연이 200ms 이하로 내려간다.

---

### CRF vs CBR vs VBR — 비트레이트 제어 모드

화질을 어떻게 결정할지.

**CRF (Constant Rate Factor)**: 화질 고정, 비트레이트 가변.

```bash
ffmpeg -i input.mp4 -c:v libx264 -crf 23 output.mp4
```

CRF 값 가이드:
- 18 이하: 시각적으로 무손실
- 23: 기본값, 일반적 화질
- 28: 보이는 품질 저하 시작
- 35 이상: 모바일/저화질

VOD에 표준. 화질이 일정하니 시청 경험이 안정적.

**CBR (Constant Bit Rate)**: 비트레이트 고정.

```bash
ffmpeg -i input.mp4 \
  -c:v libx264 \
  -b:v 5000k -maxrate 5000k -bufsize 10000k \
  output.mp4
```

라이브의 표준. 대역폭 예측 가능.

**VBR (Variable Bit Rate)**: 평균 비트레이트 기준 가변.

```bash
ffmpeg -i input.mp4 \
  -c:v libx264 \
  -b:v 5000k -maxrate 8000k -bufsize 10000k \
  output.mp4
```

VOD 다운로드에 종종 사용. 평균 5Mbps인데 액션 장면에서 8Mbps까지 허용.

---

### CBR을 진짜 CBR로 만드는 옵션 조합

`-b:v 5000k`만으로는 진짜 CBR이 아니다. x264는 약간의 변동을 허용한다.

```bash
# 진짜 CBR
ffmpeg -i input \
  -c:v libx264 \
  -b:v 5000k \
  -maxrate 5000k \      # 순간 최대도 5000k 제한
  -bufsize 10000k \     # VBV 버퍼 (보통 비트레이트의 2배)
  -minrate 5000k \      # 순간 최소도 5000k (필러로 채움)
  output.flv
```

CDN/ISP가 대역폭 예측해야 하니 진짜 CBR이 필요한 경우가 많다.

`bufsize`: VBV (Video Buffering Verifier) 버퍼. 디코더가 가질 수 있는 버퍼 크기를 가정. 작을수록 비트가 더 균일하게 분배되지만 화질이 떨어짐.

---

### -g와 keyint_min — GOP 제어

키프레임(I-frame) 간격 설정. HLS 세그먼트 경계와 직결.

```bash
# 60fps 영상에서 2초마다 키프레임
ffmpeg -i input \
  -c:v libx264 \
  -g 120 \                    # GOP 크기 = 2초 * 60fps
  -keyint_min 120 \           # 최소 간격도 같게
  -sc_threshold 0 \           # 씬 체인지 키프레임 비활성
  output.mp4
```

이 세 옵션 조합이 핵심.

- `-g 120`: 최대 GOP. 120프레임마다 강제 키프레임
- `-keyint_min 120`: 최소 GOP. 그 전엔 키프레임 안 박음
- `-sc_threshold 0`: 씬 전환 시 자동 키프레임 비활성

이 셋을 같이 설정해야 **정확히 N프레임 간격**으로 키프레임이 박힌다. 안 그러면 씬 전환 때문에 간격이 들쭉날쭉해진다.

HLS 6초 세그먼트라면:
```
30fps:  -g 180 -keyint_min 180
60fps:  -g 360 -keyint_min 360
```

ABR ladder 만들 때 모든 화질이 같은 GOP 설정이어야 세그먼트 경계가 정렬된다.

### -profile:v와 -level — 디코더 호환성

H.264 프로파일에 따라 디코딩 가능한 디바이스가 달라진다.

```
baseline:  가장 단순. 구형 디바이스 (피처폰)
main:      일반. 셋탑박스
high:      가장 효율적. 현대 디바이스
```

```bash
# 호환성 우선
ffmpeg ... -profile:v baseline -level 3.0 ...

# 효율 우선
ffmpeg ... -profile:v high -level 4.2 ...
```

라이브 플랫폼 표준: `-profile:v high -level 4.0`. 2010년 이후 디바이스 거의 다 지원.

레벨은 해상도와 비트레이트의 상한.

```
3.0: 720x480 @ 30fps
3.1: 1280x720 @ 30fps
4.0: 1920x1080 @ 30fps
4.1: 1920x1080 @ 30fps, 50Mbps
4.2: 1920x1080 @ 60fps
5.0: 4K @ 30fps
5.1: 4K @ 60fps
```

### -pix_fmt — 픽셀 포맷

10비트 인코딩, HDR, 호환성에 영향.

```bash
-pix_fmt yuv420p     # 8비트, 가장 호환성 좋음 (표준)
-pix_fmt yuv420p10le # 10비트 (HDR)
-pix_fmt yuv444p     # 풀 컬러 (호환성 낮음)
```

웹/모바일 라이브는 `yuv420p` 고정. 다른 포맷은 디바이스에서 못 디코딩하는 경우 많다.

OBS에서 종종 `yuv444p`로 인코딩하는 스트리머가 있는데, 인제스트 서버에서 강제로 `yuv420p`로 다시 인코딩하는 게 정석.

---

### -threads — 멀티스레딩

```bash
-threads 0    # 자동 (CPU 코어 수)
-threads 4    # 4스레드 명시
-threads 1    # 단일 스레드 (디버깅용)
```

기본값 `0`이 보통 최선. 컨테이너 환경에서 CPU 제한이 있으면 명시.

x264의 슬라이스드 스레딩은 약간 화질 손실이 있지만 (1% 정도) 라이브 속도 위해 감수.

### 라이브 트랜스코딩 표준 명령

지금까지 본 옵션 다 합치면:

```bash
ffmpeg \
  -re \
  -i rtmp://input/live/streamkey \
  \
  -c:v libx264 \
  -preset veryfast \
  -tune zerolatency \
  -profile:v high \
  -level 4.0 \
  -pix_fmt yuv420p \
  -g 120 -keyint_min 120 -sc_threshold 0 \
  -b:v 6000k -maxrate 6000k -bufsize 12000k \
  \
  -c:a aac \
  -b:a 128k \
  -ar 48000 \
  -ac 2 \
  \
  -f flv \
  rtmp://output/live/streamkey
```

이게 치지직/Twitch가 RTMP 1080p60 받아서 처리하는 기본형.

### 2-pass 인코딩 — VOD에서만

화질을 최대한 짜내기 위한 기법.

```bash
# Pass 1: 분석
ffmpeg -y -i input.mp4 -c:v libx264 -b:v 5000k -pass 1 \
  -an -f null /dev/null

# Pass 2: 인코딩
ffmpeg -i input.mp4 -c:v libx264 -b:v 5000k -pass 2 \
  -c:a aac -b:a 128k output.mp4
```

전체 영상을 두 번 처리한다. 1차 패스에서 각 장면 복잡도 분석하고, 2차 패스에서 비트 배분 최적화.

- 같은 비트레이트에서 화질 5~10% 향상
- 시간 2배
- 라이브에선 불가능 (전체 영상 필요)

Netflix 같은 곳에서 인기 콘텐츠 인코딩할 때 쓴다.

### 화질 측정 — VMAF

화질을 객관적으로 측정.

```bash
# 원본과 인코딩 결과 비교
ffmpeg -i original.mp4 -i encoded.mp4 \
  -lavfi libvmaf -f null -
```

출력:
```
VMAF score: 92.3
```

VMAF 점수:
- 95 이상: 시각적으로 원본과 차이 없음
- 85~95: 약간 차이 (대부분 못 느낌)
- 70 미만: 명확한 화질 저하

Netflix가 만든 지표라 OTT 업계 표준. PSNR/SSIM보다 사람 인지에 잘 맞는다.

ABR ladder 튜닝할 때 각 비트레이트 단계의 VMAF 점수를 측정해서 결정한다.

---

### 비트레이트 vs CRF — 어떻게 결정하나

라이브: CBR 고정 (대역폭 예측 위해)
VOD 일반: CRF 23
VOD 고품질: CRF 18~20
모바일 ABR: CRF + maxrate 조합

```bash
# 모바일에 친화적 (CRF 유지하되 상한 제한)
ffmpeg -i input.mp4 \
  -c:v libx264 -crf 23 \
  -maxrate 5000k -bufsize 10000k \
  output.mp4
```

평균은 CRF 23 품질, 최대는 5Mbps 제한. 모바일 다운로드에 부담 안 주면서 화질 유지.

### 면접 답변용

"라이브 트랜스코딩 옵션 어떻게 정하셨어요?" 질문.

```
"라이브는 CBR + veryfast + tune zerolatency가 기본입니다.

- CBR: ISP/CDN이 대역폭 예측해야 하니 진짜 고정 (maxrate, minrate, bufsize 같이)
- preset veryfast: 1080p60을 실시간 처리하려면 medium은 못 씀
- tune zerolatency: B-frame 비활성, 지연 200ms 이하로

GOP는 HLS 세그먼트 길이에 맞춥니다.
6초 세그먼트, 60fps라면 -g 360 -keyint_min 360 -sc_threshold 0.
이 셋 다 같이 설정해야 정확히 6초 간격으로 키프레임 박힙니다.

화질 검증은 VMAF로 했고, 85점 이상 유지를 목표로 잡았습니다."
```

### 3강 핵심 3줄

1. 라이브 표준 = `-preset veryfast -tune zerolatency -b:v X -maxrate X -bufsize 2X` (진짜 CBR)
2. GOP 정렬은 `-g N -keyint_min N -sc_threshold 0` 세 옵션 동시 설정 — ABR ladder 모든 화질이 같아야 함
3. VOD는 CRF (화질 일정), 라이브는 CBR (대역폭 고정) — 2-pass와 VMAF로 화질 검증
