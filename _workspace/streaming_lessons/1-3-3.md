## 레벨 1-3 / 3강: 모바일/와이파이/LTE 환경별 버퍼링 원인

---

### 시청자가 실제로 보는 환경

서버 엔지니어들이 종종 잊는 사실: 시청자 대부분은 **모바일**로 본다. 치지직 통계 기준 60~70%가 모바일.

모바일 환경의 특수성:
- 네트워크가 자주 바뀜 (와이파이 ↔ LTE ↔ 5G)
- 신호 강도가 위치마다 다름
- 배터리/CPU 제약
- 백그라운드 앱 영향

VOD 시청은 PC에서 가만히 있지만, 라이브 시청은 출퇴근 지하철에서 보는 경우가 많다. 환경 변화가 심하다.

---

### 와이파이 ↔ LTE 핸드오버

지하철에서 역 근처는 와이파이, 터널에선 LTE. 자동으로 전환된다. 이때:

```
[와이파이 사용 중]
  Public IP: 와이파이 공유기 IP
  세션: TCP 연결 유지
  대역폭: 80 Mbps

[LTE로 전환]
  Public IP: LTE 통신사 IP (바뀜)
  → 기존 TCP 연결 끊김
  → 새 연결 맺기 위해 핸드셰이크 (RTT 3회 = 300ms)
  → 그 사이 버퍼링
```

이래서 와이파이에서 LTE로 바뀔 때 영상이 잠깐 멈춘다. 통신 자체가 끊기는 게 아니라 **TCP 연결이 재설정**되는 시간이다.

QUIC/HTTP/3가 이 문제를 우회한다. **Connection Migration** 기능 — IP가 바뀌어도 같은 연결을 유지한다.

---

### 신호 강도와 처리량의 관계

LTE 신호 막대 4칸 vs 2칸의 차이.

```
신호 강도 ↓ → 같은 데이터 보내려면 재전송 필요 ↑
재전송 ↑ → 실효 처리량 ↓
```

신호가 약하면 통신사 기지국이 같은 데이터를 여러 번 보낸다. 시청자는 "LTE 5칸 다 차있는데 왜 느려?" 생각하지만, 실제 처리량은 1 Mbps 정도일 수 있다.

또 하나, 기지국 한 곳에 사용자가 몰리면 대역폭이 분할된다. 야구장에서 5만 명이 동시에 라이브 보면 평소 50 Mbps 나오던 LTE가 1 Mbps로 떨어진다.

### 모바일 디코딩 부하

영상 데이터를 받아도 그걸 디코딩해서 화면에 그려야 한다. 모바일 SoC의 디코더 성능에 한계가 있다.

```
저가형 안드로이드 (스냅드래곤 600 시리즈):
  H.264 1080p30: 가능
  H.264 1080p60: 약간 무리
  H.265 1080p60: 못함

플래그십 (스냅드래곤 8 시리즈):
  H.265 1080p60: 가능
  AV1 4K: 가능
```

스트리머가 H.265로 인코딩하면 고급폰만 보고 저가폰은 못 본다. 이래서 라이브 플랫폼은 보통 **H.264만 쓴다**. 호환성 100%.

치지직이 H.265를 안 쓰는 이유다. 저가폰 시청자 비율이 무시 못 한다.

---

### 배터리와 영상 품질

영상 디코딩은 CPU/GPU를 많이 쓴다. 배터리 소모가 크다.

운영체제가 이걸 자동 조절한다.

```
[배터리 100%]
  → 1080p 디코딩 풀스피드
  → 60fps 유지

[배터리 20%]
  → CPU 클럭 제한
  → 1080p 디코딩 느려짐
  → 30fps로 떨어짐 → 버퍼링 위험
```

ABR이 대역폭만 보면 안 되는 이유다. 디바이스 상태도 봐야 한다. 현대 플레이어는 CPU 사용률도 모니터링한다.

---

### 모바일 백그라운드 앱

사용자가 카톡 하면서 영상 보는 상황.

```
[카톡 알림 도착]
  → 카톡이 백그라운드에서 CPU/네트워크 사용
  → 영상 다운로드 대역폭 일시적 감소
  → 버퍼 잠깐 줄어듦

[전화 옴]
  → 영상 일시정지 or 백그라운드 재생
  → 백그라운드 재생 시 화면 디코딩은 안 하지만 오디오는 계속
```

iOS는 백그라운드 진입 시 비디오 디코딩을 멈춘다. 다시 포그라운드로 오면 라이브 엣지로 점프해야 한다.

---

### 모바일 데이터 절약 모드

시청자가 데이터 요금제 부담으로 "데이터 절약 모드" 켠 경우.

```
[데이터 절약 모드 ON]
  → 기본 화질 480p로 강제
  → 자동 화질 상향 차단
  → 와이파이 전환 시에만 1080p 허용
```

플랫폼이 강제할 수도 있다. 치지직은 LTE 환경 감지 시 기본 화질을 720p로 떨어뜨린다. 사용자가 명시적으로 1080p 선택해야 1080p를 준다.

---

### 모바일 ABR의 어려움

PC에서 ABR은 단순하다. 대역폭 측정 → 적정 화질 선택. 끝.

모바일은 변수가 너무 많다:

```
모바일 ABR 결정 요인:
- 측정 대역폭
- 신호 강도 변화 추세
- 배터리 상태
- 디바이스 디코딩 능력
- 화면 크기 (작은 화면엔 480p로 충분)
- 데이터 요금제 종류
- 백그라운드 앱 영향
- 와이파이 vs 셀룰러
```

이걸 다 고려하는 ABR 알고리즘을 짜는 게 모바일 플레이어 엔지니어의 일이다. hls.js, Shaka Player 같은 라이브러리가 이런 휴리스틱을 내장하고 있다.

---

### 3강 핵심 3줄

1. 와이파이↔LTE 전환 시 TCP 연결 재설정으로 버퍼링 — QUIC의 Connection Migration이 해결책
2. 모바일은 신호 강도, 디코딩 능력, 배터리, 화면 크기 등 다양한 변수가 영향 — 모바일 ABR이 PC보다 훨씬 복잡
3. 라이브 플랫폼이 H.265 안 쓰고 H.264 유지하는 이유는 저가폰 디코딩 호환성
