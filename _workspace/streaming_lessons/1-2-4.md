## 레벨 1-2 / 4강: 오디오 비트레이트와 멀티 트랙

---

### 오디오도 비트레이트가 있다

지금까지 영상 비트레이트만 봤는데, 오디오도 똑같이 비트레이트가 있다. 단위만 다르다 — 보통 영상은 Mbps, 오디오는 Kbps로 표기.

```
영상 1080p60: 6,000 Kbps (6 Mbps)
오디오 스테레오: 128 Kbps
오디오 모노: 64 Kbps
```

오디오는 영상보다 데이터량이 훨씬 작다. 전체 스트림에서 오디오가 차지하는 비중은 2~5% 정도.

---

### 샘플링 레이트 — 오디오의 "해상도"

영상에 픽셀과 FPS가 있다면, 오디오는 **샘플링 레이트**와 **비트 뎁스**가 있다.

**샘플링 레이트**: 1초에 소리를 몇 번 측정하는가. 단위 Hz.

```
44.1 kHz = 1초에 44,100번 측정 (CD 음질)
48 kHz   = 1초에 48,000번 측정 (방송/영상 표준)
96 kHz   = 1초에 96,000번 측정 (스튜디오 마스터링)
```

스트리밍은 거의 다 **48 kHz**다. 영상과 동기화하기 좋은 값이다.

**비트 뎁스**: 각 샘플을 몇 비트로 표현하는가.

```
16 bit: 65,536단계 (CD 음질)
24 bit: 16,777,216단계 (스튜디오)
```

스트리밍은 보통 16 bit.

---

### 오디오 코덱 — AAC, Opus, MP3

영상이 H.264로 압축되듯, 오디오도 코덱으로 압축된다.

**AAC (Advanced Audio Coding)**
- 가장 범용. 모든 기기에서 재생됨
- HLS의 사실상 표준
- 128 Kbps에서 충분히 좋은 음질

**Opus**
- WebRTC의 기본 오디오 코덱
- 같은 비트레이트에서 AAC보다 음질 우수
- 라이브 통화/저지연 스트리밍에 강함
- HLS에선 아직 지원 제한적

**MP3**
- 옛날 표준. 라이센스 이슈 없어서 아직 쓰임
- AAC보다 압축 효율 떨어짐

라이브 스트리밍 인제스트는 보통 AAC를 쓴다. 호환성 때문이다.

---

### 모노, 스테레오, 5.1 — 채널 수

같은 오디오 트랙 안에서도 채널이 여러 개일 수 있다.

```
Mono:   1채널 (왼/오 동일)
Stereo: 2채널 (왼/오 분리)
5.1:    6채널 (왼/오/중앙/서브우퍼/뒤왼/뒤오)
7.1:    8채널
```

채널이 늘면 비트레이트도 그만큼 늘어난다.

```
Stereo 128 Kbps
5.1    384 Kbps (대략 3배)
```

라이브 스트리밍은 거의 다 스테레오. 5.1 라이브는 비용 대비 효과가 적다.

---

### 멀티 오디오 트랙 — 다국어 자막의 음성 버전

HLS는 한 영상에 **여러 오디오 트랙**을 붙일 수 있다.

```
영상: 1080p H.264

오디오 트랙들:
├── 한국어 (AAC, 128 Kbps)
├── 영어 (AAC, 128 Kbps)
└── 일본어 (AAC, 128 Kbps)
```

넷플릭스에서 언어 바꿀 때 영상은 그대로고 오디오만 바뀌는 게 이거다. 영상은 한 번 받고, 오디오만 선택해서 받는다.

`.m3u8` 마스터 플레이리스트에 이렇게 표시된다:

```
#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="audio",NAME="Korean",LANGUAGE="ko",URI="audio_ko.m3u8"
#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="audio",NAME="English",LANGUAGE="en",URI="audio_en.m3u8"
```

---

### PokeClip이 멀티 오디오 트랙을 쓰는 방식 — HLS와 다른 점

여기가 중요하다. 같은 "멀티 오디오 트랙"이지만 PokeClip은 다른 목적이다.

**HLS의 멀티 오디오 트랙**: 시청자에게 "선택지"를 주기 위함. 한국어 / 영어 중 하나만 골라서 듣는다.

**PokeClip의 멀티 오디오 트랙**: 분석을 위해 "동시에 모두" 받는다. 게임 소리, 마이크, 디스코드를 각각 따로 분석.

```
HLS 시청자:
  영상 + 한국어 오디오 (1개)
  = 시청자가 선택

PokeClip 분석:
  영상 + 게임 + 마이크 + 디스코드 + 후원
  = 모두 동시에 사용
```

목적이 다르니 컨테이너 선택도 달랐던 거다. HLS의 멀티 오디오는 `.m3u8`에서 별도 플레이리스트로 관리하면 되지만, PokeClip은 **한 스트림 안에 동시에** 담아야 했다. 그래서 MPEG-TS의 PID 구조가 필요했던 거다.

---

### 오디오/비디오 동기화 — 립싱크의 원리

영상과 오디오가 어긋나면 입 모양과 소리가 안 맞는다. 이걸 막는 게 **타임스탬프**다.

각 프레임과 오디오 샘플에 시간 정보가 박혀있다.

```
영상 프레임 1: PTS = 0.000초
영상 프레임 2: PTS = 0.033초 (30fps)
영상 프레임 3: PTS = 0.067초

오디오 샘플 1: PTS = 0.000초
오디오 샘플 2: PTS = 0.021초 (48kHz, 1024 샘플)
```

**PTS (Presentation Timestamp)**: 이 데이터를 언제 보여줄지 알려주는 타임스탬프.

플레이어는 PTS를 기준으로 영상과 오디오를 맞춘다. 영상이 0.5초 빠르면 0.5초 기다렸다가 재생한다.

MPEG-TS 패킷 헤더에 PTS가 들어가 있다. 그래서 멀티 오디오 트랙이 가능하면서도 동기화가 유지된다.

---

### 라이브 송출 시 오디오 지연 문제

OBS에서 마이크를 켜고 게임을 하면, 게임 화면과 마이크 소리에 미세한 지연이 생길 수 있다. 이유:

```
[화면 캡처]
  → 게임 화면 캡처 → 즉시 인코딩 가능

[마이크 입력]
  → 마이크 → 오디오 인터페이스 (5~20ms 지연)
  → OS 오디오 드라이버 (10ms 지연)
  → OBS 처리 → 인코딩
```

OBS에서 "오디오 오프셋"을 -50ms 같은 음수로 설정하는 옵션이 이거다. 마이크 입력을 50ms 앞당겨서 영상과 맞춘다.

이게 안 맞으면 시청자 입장에선 스트리머 입과 소리가 어긋나 보인다.

---

### 4강 핵심 3줄

1. 오디오는 영상보다 데이터량 훨씬 작음 (전체의 2~5%) — 샘플링 레이트(48kHz), 코덱(AAC가 표준), 채널(스테레오)로 결정
2. 멀티 오디오 트랙 — HLS는 시청자 선택용(한국어 vs 영어), PokeClip은 동시 분석용(게임+마이크+디스코드)
3. PTS 타임스탬프로 영상-오디오 동기화 — 마이크 지연 보정을 위해 OBS에서 오프셋 설정이 필요
