## 레벨 1-1 / 3강: Progressive Download의 한계와 스트리밍의 탄생

---

### 초기 인터넷 동영상 — 그냥 파일 다운로드였다

2000년대 초반 유튜브 초창기 방식. 영상 파일 전체를 HTTP로 받아서 재생했다.

```
GET /video.mp4 HTTP/1.1

← HTTP/1.1 200 OK
← Content-Length: 500000000  (500MB)
← [데이터 계속 흘러옴...]
```

브라우저가 앞부분을 받으면서 동시에 재생을 시작할 수 있었다. 이걸 **Progressive Download**라고 한다.

언뜻 보면 괜찮아 보이는데, 치명적인 문제가 3개 있다.

---

### 문제 1: 화질이 고정된다

파일을 요청하는 순간 화질이 결정된다.

```
시청자 A: 집 와이파이 100Mbps → 1080p 요청
시청자 A: 지하철 진입, 속도 1Mbps로 감소
          → 이미 1080p 파일 받는 중 → 버퍼링 발생
```

네트워크 상태에 따라 화질을 바꾸려면 **파일 자체를 바꿔야** 한다. 그러려면 현재 받던 연결을 끊고 새로 요청해야 하는데, 그러면 재생 위치도 초기화된다.

→ ABR(Adaptive Bitrate)이 불가능하다.

---

### 문제 2: Seeking(탐색)이 느리다

타임라인에서 1시간 30분 지점을 클릭하면?

서버가 "1시간 30분 = 파일의 X번째 바이트"를 알려줘야 한다. HTTP Range Request로 가능하긴 하다.

```
GET /video.mp4 HTTP/1.1
Range: bytes=3200000000-  ← 3.2GB 지점부터 줘
```

근데 문제는 이 위치가 I-frame인지 보장이 없다. P-frame에서 시작하면 이전 프레임 없이 화면을 그릴 수 없어서 깨진다. 결국 서버가 해당 위치 근처의 I-frame을 찾아서 보내야 한다. 이게 복잡하다.

---

### 문제 3: 라이브가 불가능하다

Progressive Download는 "파일"을 전제로 한다. 아직 녹화 중인 라이브 방송은 파일이 없다.

```
GET /live.mp4 HTTP/1.1

← HTTP/1.1 ??? 
← 파일이 아직 완성 안 됐는데 어떻게 보내지?
```

---

### 해결책: 파일을 잘게 쪼개라

세 문제를 한 번에 해결하는 아이디어가 나왔다.

**영상을 작은 파일 여러 개로 쪼개고, 목차 파일을 따로 만들자.**

```
playlist.m3u8 (목차)
├── seg001.ts  (0~6초, 1080p)
├── seg002.ts  (6~12초, 1080p)
├── seg003.ts  (12~18초, 1080p)
...

playlist_720p.m3u8 (720p 목차)
├── seg001_720p.ts
├── seg002_720p.ts
...
```

이렇게 하면:

**ABR 해결**: 네트워크 느려지면 다음 세그먼트 요청 시 720p 목차로 전환. 현재 재생 중인 세그먼트는 그대로 재생.

**Seeking 해결**: 1시간 30분 = 540번째 세그먼트. 세그먼트는 반드시 I-frame으로 시작하니 바로 재생 가능.

**라이브 해결**: 인코더가 세그먼트를 만드는 족족 서버에 올리고, 목차 파일에 추가. 시청자는 목차 파일을 주기적으로 새로 받아서 새 세그먼트 확인.

```
[라이브 인코더]
  → seg001.ts 생성 → 서버 업로드
  → seg002.ts 생성 → 서버 업로드
  → seg003.ts 생성 → 서버 업로드 (현재 진행중)

[시청자 플레이어]
  → playlist.m3u8 요청 (2초마다)
  → 새 세그먼트 있으면 다운로드해서 재생
```

이게 **HLS(HTTP Live Streaming)** 다. 이름 그대로 HTTP로 라이브를 스트리밍한다.

---

### 플레이어 버퍼가 하는 일

영상이 끊기지 않으려면 재생 속도보다 빠르게 데이터를 받아야 한다. 플레이어는 미리 몇 초치를 받아두는데, 이걸 **재생 버퍼**라고 한다.

```
[현재 재생 위치]          [버퍼 끝]
      ↓                      ↓
──────●━━━━━━━━━━━━━━━━━━━━━━○──────────
      재생중    미리 받아둔 구간(30초)     아직 안 받음
```

버퍼가 비어가는 속도가 채워지는 속도보다 빠르면 → **버퍼링**.

HLS 플레이어는 이 버퍼를 유지하기 위해 끊임없이 다음 세그먼트를 요청한다.

```
플레이어: "현재 버퍼 20초 남음. seg031.ts 요청."
플레이어: "현재 버퍼 26초. 충분. 잠깐 대기."
플레이어: "현재 버퍼 14초. 빠르게 seg032.ts 요청."
```

---

### 세그먼트 크기와 지연의 트레이드오프

세그먼트를 얼마나 잘게 자르냐에 따라 달라진다.

| 세그먼트 크기 | 라이브 지연 | CDN 효율 | 재버퍼링 위험 |
|------------|-----------|---------|------------|
| 2초 | ~7초 | 낮음 | 높음 |
| 6초 | ~18초 | 높음 | 낮음 |
| 10초 | ~38초 | 매우 높음 | 매우 낮음 |

네가 이전에 공부했던 그 트레이드오프가 여기서 나온 거다. 세그먼트를 작게 하면 지연이 줄지만 CDN 캐시 효율이 떨어지고 재버퍼링 위험이 높아진다.

---

### 3강 핵심 3줄

1. Progressive Download는 ABR 불가, Seeking 어려움, 라이브 불가 — 세그먼트 방식이 이 세 문제를 한 번에 해결
2. 플레이어는 재생 버퍼를 유지하며 지속적으로 다음 세그먼트를 요청 — 버퍼가 비워지는 속도 > 채워지는 속도 = 버퍼링
3. 세그먼트 크기는 라이브 지연 vs CDN 효율의 트레이드오프 — 6초가 산업 표준인 이유
