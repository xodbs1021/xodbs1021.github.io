## 레벨 3-2 / 3강: AAC 컨테이너 — ADTS, MP4, MPEG-TS 어디에 담나

---

### 코덱과 컨테이너는 다르다

기본 개념 정리.

```
[코덱]
데이터를 압축하는 방식
AAC = 오디오 압축 알고리즘

[컨테이너]
압축된 데이터를 담는 파일/스트림 포맷
MP4, MPEG-TS, ADTS 등
```

같은 AAC 데이터를 다른 컨테이너에 담을 수 있다.

```
[AAC를 담는 컨테이너들]
.aac (ADTS): 단순 AAC 스트림
.m4a (MP4): iTunes 음원
.mp4 (MP4): 영상 + AAC
.ts (MPEG-TS): HLS, 방송
.webm: 거의 없음 (Opus가 표준)
```

---

### ADTS — 단순 AAC 스트림

가장 단순한 AAC 컨테이너.

**ADTS (Audio Data Transport Stream)**:
- 매 프레임마다 헤더 포함
- 스트리밍 친화적
- 라이브 라디오, 중간부터 듣기 가능

```
[ADTS 프레임 구조]
[헤더 7바이트][AAC 데이터][헤더 7바이트][AAC 데이터]...

헤더에 들어있는 정보:
- 프로파일 (LC/HE 등)
- 샘플링 레이트
- 채널 수
- 프레임 길이
```

각 프레임이 독립적. 중간부터 받아도 헤더 보고 디코딩 가능.

```bash
# ADTS로 저장
ffmpeg -i input.wav -c:a aac -b:a 128k output.aac
# 확장자 .aac면 자동으로 ADTS
```

---

### MP4 (.m4a, .mp4) — 파일 저장용

음악 파일이나 영상에 AAC 담을 때.

**MP4의 구조**:
- 메타데이터(moov)와 데이터(mdat) 분리
- 시킹 정확
- 챕터, 자막, 메타데이터 풍부

```
[MP4 안 AAC]
ftyp 박스
moov 박스 (메타데이터)
mdat 박스 (raw AAC 데이터, 헤더 없음)
```

ADTS와 차이: MP4 안의 AAC는 ADTS 헤더 없음. raw AAC만. 정보는 moov에.

```bash
# 음악 파일 (M4A)
ffmpeg -i input.wav -c:a aac -b:a 256k output.m4a

# 영상 + 오디오 (MP4)
ffmpeg -i input.wav -i video.mp4 \
  -c:v copy -c:a aac -b:a 128k \
  output.mp4
```

---

### MPEG-TS (.ts) — 방송/HLS

이전에 많이 본 거.

```
[MPEG-TS 안 AAC]
188바이트 패킷 안에 ADTS AAC 들어감
PID로 트랙 구분
```

HLS의 `.ts` 세그먼트, 디지털 방송, SRT 전송 모두 MPEG-TS.

```bash
# HLS용 MPEG-TS
ffmpeg -i input.mp4 \
  -c:v libx264 -c:a aac -b:a 128k \
  -f mpegts output.ts

# 또는 직접 HLS 생성
ffmpeg -i input.mp4 \
  -c:v libx264 -c:a aac -b:a 128k \
  -hls_time 6 -hls_list_size 5 \
  output.m3u8
```

---

### LATM — RTP/WebRTC 환경

**LATM (Low-overhead MPEG-4 Audio Transport Multiplex)**:
- ADTS의 RTP 버전 같은 것
- 헤더 오버헤드 적음
- WebRTC, 방송 일부

```
[LATM 사용처]
- DVB 방송
- RTP 기반 스트리밍
- 일부 IPTV
```

일반 라이브 스트리밍 개발에선 거의 안 만남. WebRTC 쓰면 Opus가 표준이라 LATM 거의 안 씀.

---

### 어떤 컨테이너 언제 쓰나 — 실무 결정

상황별 자동 결정.

```
[자동 선택 가이드]

음악 파일 저장 → .m4a (MP4)
영상 파일 → .mp4
라디오 스트림 (단순) → .aac (ADTS)
HLS 라이브 → .ts (MPEG-TS) 또는 .m4s (fMP4/CMAF)
DASH → .m4s (fMP4)
SRT 전송 → MPEG-TS
RTMP 전송 → FLV
WebRTC → Opus (AAC 거의 안 씀)
```

스트리밍 개발자는 90%가 MPEG-TS 또는 fMP4 다룬다.

---

### MPEG-TS 안 AAC — 멀티 트랙 (PokeClip)

PokeClip 같은 분석 파이프라인이 MPEG-TS 쓰는 이유.

```
[MPEG-TS 멀티 오디오]
PID 0x100: Video H.264
PID 0x101: Audio AAC 1 (게임 사운드)
PID 0x102: Audio AAC 2 (마이크)
PID 0x103: Audio AAC 3 (디스코드)
PID 0x104: Audio AAC 4 (후원)
```

각 AAC 트랙이 ADTS 형태로 PID 안에 들어있음.

FFmpeg로 분리:
```bash
ffmpeg -i "srt://0.0.0.0:9000?mode=listener" \
  -map 0:a:0 -c:a copy game.aac \
  -map 0:a:1 -c:a copy mic.aac \
  -map 0:a:2 -c:a copy discord.aac \
  -map 0:a:3 -c:a copy donation.aac
```

각 `.aac` 파일은 ADTS 포맷.

---

### fMP4 — 현대 표준 (LL-HLS, DASH)

이전 LL-HLS 강의에서 본 거. AAC를 어떻게 담는지.

```
[fMP4 안 AAC]
init.mp4: 헤더 (코덱 정보, 프로파일)
seg_001.m4s: fragment 1 (AAC raw 데이터)
seg_002.m4s: fragment 2
...
```

ADTS 헤더 없음. init 세그먼트에 모든 정보.

장점:
- 헤더 오버헤드 적음 (MPEG-TS 2% → fMP4 1%)
- 시킹 정확
- CMAF로 HLS + DASH 통합

LL-HLS, DASH는 fMP4 기본.

---

### CMAF에서 AAC

CMAF (HLS+DASH 통합)도 AAC 담음.

```bash
# CMAF로 패키징
ffmpeg -i input.mp4 \
  -c:v libx264 -c:a aac -b:a 128k \
  -hls_segment_type fmp4 \
  -hls_fmp4_init_filename "init.mp4" \
  -hls_time 6 \
  output.m3u8
```

`-hls_segment_type fmp4`로 CMAF 호환.

---

### 컨테이너 변환 — Remuxing

같은 AAC를 다른 컨테이너에 옮기기. 재인코딩 안 함.

```bash
# MP4 → MPEG-TS
ffmpeg -i input.mp4 -c copy output.ts

# MPEG-TS → MP4
ffmpeg -i input.ts -c copy output.mp4

# ADTS → MP4
ffmpeg -i input.aac -c copy output.m4a

# MP4 → ADTS
ffmpeg -i input.m4a -c copy output.aac
```

`-c copy`로 빠르게. 음질 손실 0.

---

### AAC 파일 분석 — ffprobe

실무에서 자주 함.

```bash
# 기본 정보
ffprobe input.aac

# 상세 정보
ffprobe -v error -show_streams input.m4a

# JSON 출력
ffprobe -v error -show_streams -of json input.mp4

# 오디오 정보만
ffprobe -v error -select_streams a:0 \
  -show_entries stream=codec_name,sample_rate,channels,bit_rate \
  -of csv=p=0 input.mp4
```

출력 예시:
```
codec_name=aac
codec_profile=LC
sample_rate=48000
channels=2
bit_rate=128000
```

이걸로 받은 영상의 오디오 사양 확인.

### 코덱 문자열 — MIME 표기

브라우저나 HLS 매니페스트에서.

```
mp4a.40.2

mp4a = MPEG-4 Audio
40 = Object Type Indication
2 = AAC-LC

다른 값:
mp4a.40.5 = HE-AAC
mp4a.40.29 = HE-AACv2
```

HLS 마스터 플레이리스트:
```
#EXT-X-STREAM-INF:BANDWIDTH=2000000,CODECS="avc1.4d401f,mp4a.40.2"
720p/playlist.m3u8
```

`avc1.4d401f` = H.264 Main Profile, `mp4a.40.2` = AAC-LC.

브라우저가 이거 보고 디코딩 가능한지 판단.

### 흔한 실전 문제 — 컨테이너 호환성

자주 발생하는 이슈.

**문제 1: ADTS 헤더 누락**
```
원인: MP4 → MPEG-TS 변환 시 raw AAC를 ADTS 없이 그대로 넣음
증상: 플레이어가 오디오 못 들음

해결: bitstream filter 적용
ffmpeg -i input.mp4 \
  -c:v copy -c:a copy \
  -bsf:a aac_adtstoasc \
  output.ts
```

**문제 2: 샘플링 레이트 불일치**
```
원인: 입력 44.1 kHz, 출력 48 kHz인데 변환 안 함
증상: 음악 빠르게/느리게 들림

해결: -ar 명시
ffmpeg -i input.wav -c:a aac -ar 48000 -b:a 128k output.aac
```

**문제 3: 채널 매핑 잘못됨**
```
원인: 5.1 → 스테레오 다운믹스 안 함
증상: 일부 채널만 들림

해결: -ac 2 추가
ffmpeg -i input.mkv -c:a aac -ac 2 -b:a 192k output.mp4
```

### 라이브 스트리밍 표준 흐름

전체 파이프라인에서 AAC가 어떻게 담기나.

```
[OBS]
캡처 → AAC 인코딩 → FLV 컨테이너 → RTMP 송신

[치지직 서버]
RTMP 수신 → FLV → AAC 추출
↓
트랜스코딩 (필요 시)
↓
HLS 패키징: AAC → MPEG-TS → .ts 세그먼트
또는
LL-HLS 패키징: AAC → fMP4 → .m4s 세그먼트

[CDN]
.ts 또는 .m4s 캐시 분배

[시청자]
HLS 받기 → 오디오 트랙 디코딩 → 재생
```

각 단계가 다른 컨테이너. 코덱(AAC)은 유지.

---

### PokeClip 분석 파이프라인 흐름

분석용 멀티 트랙.

```
[OBS]
4개 오디오 트랙 (게임/마이크/디스코드/후원)
→ 각각 AAC 인코딩
→ MPEG-TS에 PID별 담음
→ SRT 송신

[PokeClip 서버]
SRT 수신 → MPEG-TS
↓
FFmpeg demux로 PID별 분리
↓
각 트랙 .aac (ADTS) 파일로
↓
분석 엔진에 입력
```

스트리밍 서비스와 다른 흐름. 시청자에 안 보내고 분석.

---

### 실전 체크리스트

상황별 컨테이너 선택.

```
□ 라이브 RTMP 송출 → FLV (자동)
□ HLS 시청자 측 → MPEG-TS (.ts)
□ LL-HLS 시청자 측 → fMP4 (.m4s) + CMAF
□ DASH → fMP4 (.m4s)
□ 분석 파이프라인 (SRT) → MPEG-TS
□ 음악 다운로드 → M4A
□ 영상 파일 → MP4
□ 옛 기기 호환 → 그래도 MP4
```

이 매핑만 알면 충분.

```
"용도에 따라 다릅니다.

```

### 3강 핵심 3줄

1. AAC는 ADTS(스트리밍/라디오), MP4/M4A(파일), MPEG-TS(HLS/방송), fMP4(LL-HLS/DASH/CMAF) 등 다양한 컨테이너에 담길 수 있음 — 코덱은 같고 포장만 다름
2. PokeClip 같은 멀티 트랙 분석은 MPEG-TS의 PID 구조 활용, 일반 라이브 HLS는 .ts 또는 .m4s — `ffmpeg -c copy`로 재인코딩 없이 컨테이너 변환
3. 실전 이슈는 ADTS 헤더 누락 (`aac_adtstoasc` filter), 샘플링 레이트 (`-ar 48000`), 채널 다운믹스 (`-ac 2`) — MIME 표기 `mp4a.40.2`로 브라우저 호환성 검증
