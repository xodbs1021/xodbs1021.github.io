## 레벨 4-1 / 4강: 오디오 처리와 필터 그래프

---

### 오디오 인코딩 기본

비디오 옵션의 절반 정도만 알면 된다.

```bash
ffmpeg -i input.mp4 \
  -c:a aac \           # 코덱
  -b:a 128k \          # 비트레이트
  -ar 48000 \          # 샘플레이트
  -ac 2 \              # 채널 수
  output.mp4
```

라이브 HLS 표준:
- 코덱: AAC-LC
- 비트레이트: 128k (스테레오)
- 샘플레이트: 48000 Hz
- 채널: 2 (스테레오)

---

### -ar / -ac — 샘플레이트와 채널

샘플레이트는 1초에 몇 번 소리를 측정할지.

```
44100 Hz: CD 표준, 옛 음원
48000 Hz: 영상 표준, HLS 표준
96000 Hz: 고품질 마스터링
```

영상에선 거의 48000. 44100 들어오면 리샘플링.

```bash
# 44100 → 48000 리샘플
ffmpeg -i music.mp3 -ar 48000 -c:a aac output.m4a
```

채널.
```
1 = 모노
2 = 스테레오
6 = 5.1 서라운드
8 = 7.1 서라운드
```

라이브는 거의 스테레오. 모노로 들어오는 마이크는 스테레오로 업믹스.

```bash
# 모노 → 스테레오
ffmpeg -i mono.wav -ac 2 -c:a aac stereo.aac
```

---

### AAC 인코더 선택

FFmpeg에 AAC 인코더가 여러 개 있다.

```
aac        (FFmpeg 내장, LGPL)
libfdk_aac (Fraunhofer, 비-LGPL, 최고 화질)
libfaac    (구식, 사용 안 함)
```

```bash
# 내장 AAC (어디서나 동작)
ffmpeg -i input.mp4 -c:a aac -b:a 128k output.mp4

# fdk-aac (별도 빌드 필요, 음질 더 좋음)
ffmpeg -i input.mp4 -c:a libfdk_aac -b:a 128k output.mp4
```

fdk-aac 차이:
- 같은 비트레이트에서 음질 더 좋음
- 96k까지 떨어뜨려도 들어줄 만함
- 라이센스 문제로 jrottenberg/ffmpeg 같은 일반 도커에 안 들어가 있음

라이브 인프라는 보통 내장 aac. fdk가 좋지만 빌드 관리 부담.

---

### -af — 오디오 필터

비디오의 `-vf`와 같은 개념.

```bash
# 볼륨 조절 (50%)
ffmpeg -i input.mp4 -af "volume=0.5" output.mp4

# 노멀라이징 (다이내믹 레인지 압축)
ffmpeg -i input.mp4 -af "loudnorm=I=-16:LRA=11:TP=-1.5" output.mp4

# 페이드 인/아웃
ffmpeg -i input.mp4 -af "afade=t=in:st=0:d=2,afade=t=out:st=58:d=2" output.mp4

# 노이즈 게이트
ffmpeg -i input.mp4 -af "agate=threshold=0.05" output.mp4
```

라이브에서 가장 많이 쓰는 건 `loudnorm`. 스트리머마다 볼륨이 달라서 표준화.

---

### loudnorm — EBU R128 라우드니스 표준화

방송 표준 라우드니스로 맞춤.

```bash
ffmpeg -i input.mp4 -af "loudnorm=I=-16:LRA=11:TP=-1.5" output.mp4
```

- `I=-16`: 통합 라우드니스 -16 LUFS (스트리밍 표준)
- `LRA=11`: 라우드니스 레인지
- `TP=-1.5`: 트루 피크 -1.5 dBTP

플랫폼별 권장:
```
Spotify:     -14 LUFS
YouTube:     -14 LUFS
Twitch/치지직: -16 LUFS
방송 (한국): -23 LUFS
```

라이브 ingest에서 자동 적용하는 곳도 있고, 스트리머에게 맡기는 곳도 있다.

---

### 필터 그래프 — 복잡한 처리는 -filter_complex

`-vf`/`-af`는 단일 입력/단일 출력 필터에만 쓴다. 멀티 스트림 합성/분리는 `-filter_complex`.

```bash
# 두 영상 좌우 합성 (사이드 바이 사이드)
ffmpeg -i left.mp4 -i right.mp4 \
  -filter_complex "[0:v][1:v]hstack=inputs=2[v]" \
  -map "[v]" -map 0:a \
  output.mp4
```

문법:
- `[0:v]`: 입력 0의 비디오
- `hstack`: 가로 합성 필터
- `[v]`: 출력 라벨

---

### 멀티 오디오 트랙 다루기

PokeClip 같은 분석 파이프라인.

```bash
# 한 영상에 오디오 3개 (게임, 마이크, 디스코드)
ffmpeg \
  -i video.mp4 \
  -i game.wav \
  -i mic.wav \
  -i discord.wav \
  -map 0:v \
  -map 1:a \
  -map 2:a \
  -map 3:a \
  -c:v copy -c:a aac \
  -metadata:s:a:0 title="Game" \
  -metadata:s:a:1 title="Mic" \
  -metadata:s:a:2 title="Discord" \
  output.mp4
```

각 오디오 트랙에 라벨을 박는다. HLS multi-audio 플레이리스트 만들 때 이 라벨이 사용자에게 보인다.

---

### 오디오 추출

영상에서 오디오만 빼기.

```bash
# AAC로 추출 (재인코딩 없이)
ffmpeg -i video.mp4 -vn -c:a copy audio.aac

# WAV로 추출 (PCM)
ffmpeg -i video.mp4 -vn -c:a pcm_s16le -ar 48000 audio.wav

# MP3로 변환
ffmpeg -i video.mp4 -vn -c:a libmp3lame -b:a 192k audio.mp3
```

`-vn` = video none (비디오 비활성).

`-c:a copy`는 컨테이너만 바꾸니 무손실 + 빠름. MP3로 바꾸려면 디코딩 → 재인코딩 필요.

---

### 오디오 분리 — 채널별 추출

스테레오에서 좌/우 분리.

```bash
ffmpeg -i input.wav \
  -filter_complex "[0:a]channelsplit=channel_layout=stereo[L][R]" \
  -map "[L]" left.wav \
  -map "[R]" right.wav
```

상황: 마이크 모노가 한쪽 채널에만 녹음됨. 양쪽으로 복사.

```bash
ffmpeg -i mono_left.wav \
  -af "channelmap=channel_layout=mono" \
  output.wav
```

---

### 오디오 믹싱 — 여러 트랙 합성

여러 오디오를 하나로.

```bash
# 게임 + 마이크 동시 재생
ffmpeg -i game.wav -i mic.wav \
  -filter_complex "[0:a][1:a]amix=inputs=2:duration=longest[a]" \
  -map "[a]" mixed.wav
```

amix 옵션:
- `duration=longest`: 가장 긴 입력에 맞춤
- `dropout_transition=2`: 끝날 때 페이드 길이
- `weights="1 0.5"`: 각 입력 가중치

스트리머가 OBS에서 하는 일이 결국 이거다. 게임 + 마이크 + 디스코드 + BGM 다 amix.

---

### 오디오 동기화 — -itsoffset

오디오와 비디오가 어긋난 경우.

```bash
# 오디오를 100ms 늦추기
ffmpeg -i video.mp4 -itsoffset 0.1 -i video.mp4 \
  -map 0:v -map 1:a \
  -c copy output.mp4
```

`-itsoffset`은 그 다음 입력의 시간을 조정한다. 같은 파일을 두 번 입력하고 한쪽에만 오프셋.

OBS에서 마이크 오프셋 -50ms 설정과 같은 효과를 후처리로.

### 필터 체인 vs 그래프

체인: 직렬.
```
-vf "scale=1280:720,fps=30"
```

그래프: 병렬/분기 가능.
```
-filter_complex "[0:v]scale=1280:720[v1];[0:v]scale=854:480[v2]"
```

체인은 `,`로 구분. 그래프는 `;`로 분리하고 `[]` 라벨로 연결.

ABR ladder를 한 명령으로 만들 때.

```bash
ffmpeg -i input.mp4 \
  -filter_complex "\
    [0:v]split=3[v1080][v720][v480]; \
    [v720]scale=1280:720[v720s]; \
    [v480]scale=854:480[v480s]" \
  -map "[v1080]" -map 0:a -c:v libx264 -b:v 6000k out_1080.mp4 \
  -map "[v720s]" -map 0:a -c:v libx264 -b:v 3500k out_720.mp4 \
  -map "[v480s]" -map 0:a -c:v libx264 -b:v 1500k out_480.mp4
```

`split=3`이 디코딩 결과를 3개로 분기. 디코딩 1번 + 인코딩 3번.

### 워터마크 / 로고 오버레이

치지직 같은 플랫폼이 자체 워터마크 박을 때.

```bash
# 우상단에 로고 (반투명)
ffmpeg -i video.mp4 -i logo.png \
  -filter_complex "[1:v]format=rgba,colorchannelmixer=aa=0.5[logo]; \
                   [0:v][logo]overlay=W-w-20:20" \
  -c:a copy output.mp4
```

`overlay=W-w-20:20`: 우상단에서 20px 안쪽.
- `W`: 영상 너비
- `w`: 로고 너비

플랫폼 로고는 대개 trans-coding 단계에서 박는다. 원본은 깨끗하게 보관.

---

### 시간 표시 (drawtext)

라이브 디버깅용 타임스탬프.

```bash
ffmpeg -i video.mp4 \
  -vf "drawtext=text='%{localtime}':x=10:y=10:fontsize=24:fontcolor=white:box=1:boxcolor=black@0.5" \
  output.mp4
```

라이브 지연 측정할 때 유용. 원본에 현재 시각 박고, 시청자 화면에 표시되는 시각과 비교하면 글래스 투 글래스 지연 측정 가능.

### 라이브 인제스트 + 워터마크 + ABR 통합 예시

```bash
ffmpeg \
  -re \
  -i rtmp://input/live/streamkey \
  -i logo.png \
  \
  -filter_complex "\
    [0:v][1:v]overlay=W-w-20:20[wm]; \
    [wm]split=3[v1][v2][v3]; \
    [v2]scale=1280:720[v720]; \
    [v3]scale=854:480[v480]" \
  \
  -map "[v1]"  -map 0:a -c:v libx264 -preset veryfast -tune zerolatency \
    -b:v 6000k -maxrate 6000k -bufsize 12000k -c:a aac -b:a 128k \
    -f flv rtmp://output/1080 \
  \
  -map "[v720]" -map 0:a -c:v libx264 -preset veryfast -tune zerolatency \
    -b:v 3000k -maxrate 3000k -bufsize 6000k -c:a aac -b:a 128k \
    -f flv rtmp://output/720 \
  \
  -map "[v480]" -map 0:a -c:v libx264 -preset veryfast -tune zerolatency \
    -b:v 1500k -maxrate 1500k -bufsize 3000k -c:a aac -b:a 96k \
    -f flv rtmp://output/480
```

이게 진짜 라이브 트랜스코딩 서버 한 인스턴스가 하는 일.

근데 1080p60 + 720p + 480p를 CPU(x264)로 동시에 처리하면 32코어 서버도 부담. NVENC 같은 GPU 인코더가 필요해진다. 다음 강의 주제.

### 4강 핵심 3줄

1. 오디오 표준: AAC 128k, 48000Hz, 스테레오 — `loudnorm`으로 라우드니스 통일이 라이브 인제스트의 표준 처리
2. `-filter_complex`로 멀티 입력/출력 그래프 구성 — `split` `overlay` `amix` `channelsplit`이 핵심 필터
3. ABR ladder를 한 명령으로 (디코딩 1번 + 인코딩 N번) 가능하지만 CPU 부담 — GPU 인코더 필요해지는 지점
