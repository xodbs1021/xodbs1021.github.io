## 레벨 4-5 / 2강: 트랜스코딩 스케줄러와 큐 시스템

---

### 스케줄러의 역할

새 방송이 들어왔을 때 어느 트랜스코딩 노드로 보낼지 결정.

```
[스케줄러 입력]
- 새 스트림 ID
- 스트리머 정보 (인기도, 카테고리)
- 입력 스펙 (해상도, 비트레이트, 코덱)

[스케줄러 출력]
- 할당된 트랜스코딩 노드
- 사용할 사다리 설정
- 모니터링 설정
```

---

### 단순 스케줄러 - Round Robin

가장 단순.

```python
class RoundRobinScheduler:
    def __init__(self, nodes):
        self.nodes = nodes
        self.index = 0
    
    def assign(self, stream):
        node = self.nodes[self.index % len(self.nodes)]
        self.index += 1
        return node
```

문제:
- 노드별 부하 차이 무시
- 죽은 노드도 할당
- 인기 스트리머가 한 노드에 몰릴 수 있음

### Least-Load 스케줄러

각 노드의 현재 부하 확인.

```python
class LeastLoadScheduler:
    def assign(self, stream):
        loads = []
        for node in self.healthy_nodes():
            gpu_util = self.get_gpu_utilization(node)
            loads.append((node, gpu_util))
        
        loads.sort(key=lambda x: x[1])
        return loads[0][0]
    
    def get_gpu_utilization(self, node):
        return prometheus.query(
            f'nvidia_gpu_utilization{{instance="{node}"}}'
        )[-1]
```

GPU 사용률 가장 낮은 노드 선택.

### Bin Packing - 비용 최적화

부하 분산이 아닌 노드 패킹으로 비용 절감.

```
[부하 분산]
모든 노드 60% 사용 → 균등
스케일 다운 어려움

[Bin Packing]
일부 노드 90% 채움
나머지 노드는 비워둠
빈 노드 종료 → 비용 절감
```

```python
class BinPackingScheduler:
    def assign(self, stream):
        # 가장 부하 높은 (하지만 여유 있는) 노드 선택
        candidates = [
            (n, self.get_load(n)) 
            for n in self.healthy_nodes()
            if self.get_load(n) < 0.85  # 안전 마진
        ]
        candidates.sort(key=lambda x: -x[1])  # 부하 높은 순
        return candidates[0][0] if candidates else self.spawn_new_node()
```

이 방식으로 노드 30% 줄임. AWS 비용 절감.

### 인기 기반 우선순위 - Tier Scheduling

스트리머 인기도에 따라 다른 GPU.

```
[Tier 분리]
Tier 1 (Top 1%): 인기 스트리머
  → A10 GPU 노드
  → ABR ladder 5단계 + 1080p120
  → LL-HLS 활성

Tier 2 (Top 10%):
  → A10 또는 T4
  → ABR ladder 4단계
  → 일반 HLS

Tier 3 (일반):
  → T4 또는 CPU x264
  → ABR ladder 3단계
  → 일반 HLS
```

비용 차등으로 전체 비용 25% 절감.

```python
class TierScheduler:
    def assign(self, stream):
        tier = self.get_tier(stream.streamer_id)
        if tier == 1:
            return self.assign_in_pool("a10-premium")
        elif tier == 2:
            return self.assign_in_pool("a10-standard")
        else:
            return self.assign_in_pool("t4-basic")
```

### 큐 시스템 - 백프레셔

인제스트 → 트랜스코딩 사이의 버퍼.

```
[큐 없을 때]
인제스트 노드가 실시간 SRT로 트랜스코딩에 직접 푸시
트랜스코딩 노드 일시 부하 → 데이터 드롭

[큐 있을 때]
인제스트 → Redis Streams / Kafka
트랜스코딩 컨슈머가 큐에서 읽음
백프레셔로 인제스트가 속도 조절
```

라이브에서 큐 깊이는 작아야 함 (지연 증가). 보통 3~5초 버퍼.

---

### Redis Streams로 큐 구현

```python
# 인제스트가 큐에 푸시
import redis
r = redis.Redis()

while True:
    chunk = read_from_rtmp()
    r.xadd(
        f'stream:{stream_id}',
        {'data': chunk, 'pts': pts},
        maxlen=300  # 6초 버퍼 (50fps)
    )
```

```python
# 트랜스코딩이 큐에서 읽음
last_id = '0'
while True:
    items = r.xread(
        {f'stream:{stream_id}': last_id},
        count=10,
        block=100
    )
    for stream_key, messages in items:
        for msg_id, msg in messages:
            process(msg)
            last_id = msg_id
```

Redis Streams의 `maxlen`이 핵심. 큐가 너무 길어지면 오래된 것 자동 삭제.

### Kafka로 큐 - 대규모

수천 개 동시 스트림은 Redis 한계.

```
[Kafka 토픽 설계]
topic: live-streams
partitions: 100
replication: 3
retention: 1 hour (라이브용 짧게)

[파티션 키]
stream_id → 해시 → 파티션
한 스트림이 항상 같은 파티션 → 순서 보장
```

```python
from confluent_kafka import Producer, Consumer

# 인제스트
producer = Producer({'bootstrap.servers': 'kafka:9092'})
producer.produce(
    'live-streams',
    key=stream_id.encode(),
    value=chunk_bytes
)

# 트랜스코딩
consumer = Consumer({
    'bootstrap.servers': 'kafka:9092',
    'group.id': 'transcoder-1',
    'auto.offset.reset': 'latest'  # 라이브는 latest
})
consumer.subscribe(['live-streams'])

while True:
    msg = consumer.poll(0.1)
    if msg:
        process(msg.value())
```

Kafka는 인프라 부담 크지만 안정성/처리량 우수.

### SRT를 큐 대안으로

큐 없이 SRT 직접 연결도 가능.

```
[SRT는 자체 버퍼 가짐]
SRT_LATENCY=120ms (재전송 윈도우)
→ 패킷 손실 시 120ms 내 재전송 대기
→ 그 안에 안 오면 드롭
```

지연 보장 + 자체 버퍼 → 별도 큐 불필요.

```bash
# 인제스트 → 트랜스코딩 (SRT)
ffmpeg -i rtmp://stream \
  -c copy \
  -f mpegts srt://transcoder:9000?latency=120
```

작은 규모는 SRT 직접 연결. 대규모는 Kafka/Redis.

### 노드 헬스체크

스케줄러가 죽은 노드 할당 안 하게.

```python
class HealthChecker:
    def is_healthy(self, node):
        # 1. 마지막 heartbeat
        last_seen = self.get_last_heartbeat(node)
        if time.time() - last_seen > 30:
            return False
        
        # 2. GPU 동작 확인
        gpu_health = self.check_gpu(node)
        if not gpu_health:
            return False
        
        # 3. 최근 인코딩 에러율
        error_rate = self.get_error_rate(node, window='5m')
        if error_rate > 0.05:  # 5% 이상
            return False
        
        return True
```

비정상 노드는 자동으로 풀에서 제거. 5분 후 재진입 시도.

### Graceful Drain - 노드 종료

노드를 종료할 때 진행 중인 스트림 처리.

```python
class NodeDrainer:
    def drain(self, node):
        # 1. 새 스트림 할당 중단
        self.scheduler.mark_unavailable(node)
        
        # 2. 진행 중인 스트림 식별
        streams = self.get_streams_on_node(node)
        
        # 3. 각 스트림을 다른 노드로 마이그레이션
        for stream in streams:
            new_node = self.scheduler.assign(stream)
            self.migrate(stream, from_node=node, to_node=new_node)
            time.sleep(1)  # 부하 분산 위해 천천히
        
        # 4. 모든 스트림 이동 완료 → 노드 종료
        self.terminate(node)
```

마이그레이션 시 약 5~10초 끊김. Spot 인스턴스 회수 시 미리 알람 받아 drain 시작.

### 예측 기반 스케일링

부하 예측으로 미리 확장.

```python
class PredictiveScaler:
    def predict_load(self, target_time):
        """과거 데이터로 미래 부하 예측"""
        # 같은 요일/시간대 평균 부하
        historical = self.get_historical_load(
            day_of_week=target_time.weekday(),
            hour=target_time.hour
        )
        
        # 트렌드 보정
        recent_trend = self.get_recent_trend()
        
        return historical * recent_trend
    
    def schedule_scaling(self):
        future_5min = self.predict_load(now + 5min)
        current = self.current_capacity()
        
        if future_5min > current * 0.8:
            # 5분 후 부하 높을 예정 → 미리 확장
            self.scale_up(target=future_5min * 1.2)
```

저녁 6시 피크 5분 전부터 확장 시작. 시청자가 몰리기 전에 준비 완료.

### 우선순위 큐 - QoS

같은 노드 부하 상황에서 누구 먼저 처리할지.

```
[우선순위]
P1: Top 1% 인기 스트리머
P2: 광고/이벤트 진행 중인 스트리머
P3: 일반 인기 스트리머 (Top 10%)
P4: 일반 스트리머
P5: 베타 테스터
```

```python
class PriorityScheduler:
    def assign(self, stream):
        priority = self.get_priority(stream)
        
        # 우선순위 높을수록 좋은 노드
        if priority == 1:
            return self.assign_in_pool("premium-a10")
        elif priority <= 3:
            return self.assign_in_pool("standard-t4")
        else:
            return self.assign_in_pool("budget-x264")
```

부하 폭증 시 P5 처리 지연 또는 거부.

### Stream Migration - 라이브 마이그레이션

방송 중에 다른 노드로 옮기기.

```
[마이그레이션 시나리오]
1. 노드 헬스 악화
2. 비용 최적화 (저렴한 노드로)
3. GPU 펌웨어 업데이트
4. 노드 재시작 필요

[마이그레이션 단계]
1. 새 노드에 트랜스코딩 프로세스 시작
2. 두 노드에서 동시 인코딩 (1~2초)
3. 패키저가 새 노드 출력 받기 시작
4. 첫 GOP 경계에서 전환
5. 옛 노드 정리
```

매끄러운 전환은 어렵지만, 5~10초 끊김 정도로 가능.

### 2강 핵심 3줄

1. 스케줄러는 Least-Load + Bin Packing + Tier로 비용 최적화, 인기 스트리머만 프리미엄 GPU
2. 큐는 Redis Streams (소규모) 또는 Kafka (대규모), SRT 자체 버퍼로 큐 대체 가능
3. HPA + 예측 기반 스케일링, Graceful drain + Stream migration으로 노드 종료 시 5~10초 끊김 정도로 처리
