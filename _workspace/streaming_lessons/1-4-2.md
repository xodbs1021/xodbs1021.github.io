## 레벨 1-4 / 2강: 라이브 스트리밍을 위한 CDN 운영 — 일반 CDN과 다른 점

---

### 일반 웹 vs 라이브 스트리밍 CDN

CDN이라고 다 같은 게 아니다. 이미지 캐시랑 라이브 영상은 요구 사항이 완전히 다르다.

| | 일반 웹 (이미지/JS) | 라이브 스트리밍 |
|---|---|---|
| 파일 크기 | 작음 (KB) | 큼 (MB) |
| 요청 빈도 | 한 번 받으면 끝 | 6초마다 새 파일 |
| 캐시 수명 | 길게 (1일~) | 짧게 (수초) |
| 동시 요청 | 분산됨 | 라이브 엣지에 집중 |
| 신선도 | 별로 안 중요 | 최신 세그먼트 필수 |

라이브 스트리밍 전용 CDN이 따로 있는 이유다. Wowza, Akamai Adaptive Media Delivery, AWS CloudFront for Media, Cloudflare Stream.

---

### Thundering Herd — 라이브의 가장 큰 적

1-3에서 잠깐 봤다. 라이브의 핵심 문제다.

새 세그먼트가 생성되는 순간 100만 명이 동시에 요청한다.

```
[12:00:06.000] seg103.ts 인코딩 완료, Origin에 업로드
[12:00:06.001] 100만 시청자가 동시에 seg103.ts 요청
[12:00:06.001] CDN 엣지 100개 모두 캐시 미스
[12:00:06.002] 100개 엣지가 Origin으로 동시 요청
```

Origin 서버는 한 번에 100개 요청을 처리해야 한다. 단순한 정적 파일이라 빠르긴 하지만, 100만 명 시나리오에선 매 6초마다 100개 동시 요청이 들어온다.

해결책 두 가지:

**1. Request Coalescing (요청 병합)**
CDN 엣지에서 같은 URL 요청이 동시에 오면 Origin으로는 한 번만 보내고, 나머지 요청은 그 응답을 공유한다.

```
[엣지 서버 A]
  요청 1: seg103.ts → 캐시 미스 → Origin 호출
  요청 2: seg103.ts → "기존 요청 대기"
  요청 3: seg103.ts → "기존 요청 대기"
  ...
  요청 10,000: seg103.ts → "기존 요청 대기"

  Origin 응답 도착 → 10,000개 요청에 동시 응답
```

엣지당 Origin 호출이 1번으로 줄어든다. 100개 엣지 × 1번 = 100번. 100만 명이 와도 Origin은 100번만 처리하면 된다.

**2. Origin Shield (중간 계층)**
1강에서 본 Mid-tier. 100개 엣지가 5개 Shield로, 5개 Shield가 Origin으로.

```
시청자 100만 → 엣지 100개 → Shield 5개 → Origin 1개
```

Origin 호출 횟수: 100만 → 5번.

### Cache TTL 정책 — 라이브 특화

라이브 플레이리스트(`.m3u8`)는 짧은 캐시가 필요하지만 너무 짧으면 Origin 부하가 폭증한다. 균형이 중요.

```http
# 매 2초마다 .m3u8가 업데이트되는 라이브
Cache-Control: max-age=2, s-maxage=2

# 100만 명이 .m3u8 polling (2초마다)
# = 초당 50만 요청
# 캐시 TTL 2초 → CDN이 50만 요청 → Origin 50만/2초 = 25만 RPS
```

이걸 Stale-While-Revalidate로 줄인다.

```http
Cache-Control: max-age=2, stale-while-revalidate=5
```

- 0~2초: 캐시 응답
- 2~7초: 캐시 응답 + 백그라운드로 Origin 호출해 갱신
- 7초~: 캐시 응답 안 함, Origin 호출

Origin 호출 빈도가 확 줄어든다.

---

### CDN Purge — 캐시 강제 무효화

세그먼트가 잘못 인코딩됐을 때, 또는 저작권 이슈로 영상을 내려야 할 때.

CDN 캐시를 강제로 비우는 게 **Purge**.

```
[Purge API 호출]
POST /purge
{
  "urls": ["https://cdn.chzzk.naver.com/live/seg100.ts"]
}

→ CDN이 모든 엣지에서 해당 URL 캐시 삭제
```

Purge는 비싸다. 모든 엣지에 명령을 전파해야 하고, 다음 요청에서 캐시 미스가 일어나 Origin 부하가 튄다.

대안: **Cache Versioning**. URL에 버전을 박는다.

```
잘못된 버전: /live/v1/seg100.ts
새 버전:    /live/v2/seg100.ts (Purge 없이 새로 캐시)
```

라이브 플레이리스트만 업데이트하면 v1 캐시는 자연 만료될 때까지 두고, v2로 새 트래픽이 흐른다.

### 멀티 CDN — 한 CDN에만 의존 안 함

대형 라이브 플랫폼은 보통 **여러 CDN을 동시에** 쓴다.

```
치지직 추정 구조:
  - 1차 CDN: CloudFront
  - 2차 CDN: Cloudflare (백업/특정 지역)
  - 3차 CDN: 자체 CDN (인기 콘텐츠)
```

이유:

**1. 장애 대응**
한 CDN이 장애나도 트래픽을 다른 곳으로 돌릴 수 있다.

**2. 비용 협상력**
"우리는 다른 CDN도 쓰니까 가격 더 깎아줘."

**3. 지역별 최적 CDN 활용**
한국은 자체 CDN, 동남아는 Cloudflare, 미주는 CloudFront 같은 분리.

플레이어에서 CDN을 어떻게 고르나? Multi-CDN Switching:

```javascript
// 시작 시 여러 CDN 핑 측정
const cdnA_latency = await ping('cdn-a.chzzk.naver.com');
const cdnB_latency = await ping('cdn-b.chzzk.naver.com');

const bestCDN = cdnA_latency < cdnB_latency ? 'cdn-a' : 'cdn-b';

// 세그먼트 다운로드 중 실패 시 다른 CDN으로 폴백
hls.config.xhrSetup = (xhr, url) => {
  xhr.addEventListener('error', () => {
    const fallbackUrl = url.replace('cdn-a', 'cdn-b');
    // 재시도
  });
};
```

### Token Authentication — 무단 사용 차단

라이브 영상은 비싸다. URL이 유출되면 누구나 볼 수 있어서 비용이 폭증한다.

**Signed URL (서명된 URL)** 으로 차단한다.

```
원본 URL: /live/seg100.ts
서명 URL: /live/seg100.ts?token=abc...&expires=1700000060&user=12345

CDN이 토큰 검증:
  1. 토큰 서명이 유효한가
  2. 만료 시간 안 지났는가
  3. (선택) 같은 IP에서 온 요청인가
```

만료 시간을 60초로 짧게 잡으면, URL 유출돼도 60초 뒤엔 못 쓴다.

CloudFront, Akamai 모두 이걸 지원한다. CDN이 직접 검증해서 Origin 부하 없이 차단.

### 라이브 CDN의 새로운 도전 — Low Latency

전통적 HLS는 18~25초 지연이 정상. 근데 채팅과 동기화하려면 더 줄여야 한다.

**LL-HLS (Low Latency HLS)**: 2~5초 지연 목표.

CDN 입장에서 이게 까다롭다.

```
일반 HLS:
  세그먼트 6초 단위로 생성 → 캐시 → 배포

LL-HLS:
  Partial Segment 0.2초 단위로 생성 → 즉시 배포
  → CDN 캐시 효율 떨어짐
  → 더 많은 Origin 호출
```

CDN이 LL-HLS를 제대로 지원하려면 HTTP/2 Push, Chunked Transfer Encoding 같은 기술이 필요하다. 모든 CDN이 다 지원하지 않는다.

이게 LL-HLS 채택의 발목을 잡는 요인 중 하나다. CDN 비용 vs 지연 단축의 트레이드오프.

### 라이브 영상 CDN 비용 구조

대략적인 라이브 영상 CDN 비용 (2024년 기준):

```
CloudFront 한국 → 한국:
  $0.12 / GB

치지직 인기 스트리머 1명 4시간 방송:
  - 평균 시청자: 5만 명
  - 1080p 6 Mbps × 4시간 = 약 10 GB / 시청자
  - 총 트래픽: 5만 × 10 GB = 50만 GB
  - 비용: 50만 × $0.12 = $60,000 (약 8천만 원)
```

한 방송에 8천만 원. 인기 스트리머 10명이 매일 방송하면 월 240억. 광고 매출로 충당해야 한다.

이래서 H.265나 AV1 같은 효율적인 코덱을 도입하려는 시도가 계속된다. 같은 화질에 비트레이트가 30~50% 줄면 비용도 그만큼 줄어든다.

### 2강 핵심 3줄

1. 라이브 CDN의 핵심 과제는 Thundering Herd — Request Coalescing과 Origin Shield로 Origin 부하를 100만→100 수준으로 줄임
2. Stale-While-Revalidate, Cache Versioning 같은 라이브 특화 캐시 정책으로 짧은 TTL 한계 극복
3. 멀티 CDN, Signed URL, LL-HLS 지원 여부가 라이브 플랫폼 인프라의 핵심 운영 포인트
