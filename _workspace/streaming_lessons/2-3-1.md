## 레벨 2-3 / 1강: DASH의 탄생 — HLS에 대한 산업계의 반격

---

### 2010년 — HLS가 1년 만에 시장 장악

HLS가 2009년 나오고 1년 만에 모바일 영상 표준이 됐다. iPhone이 폭발적으로 팔리면서.

이 상황이 모든 비-Apple 진영을 불편하게 만들었다.

```
[2010년 영상 스트리밍 시장]
시청자 측:
- iOS: HLS (강제)
- Android: 표준 없음 (각자 다른 방식)
- 데스크탑: Flash + RTMP
- 스마트TV: 제각각

→ 콘텐츠 제공자는 같은 영상을 여러 방식으로 만들어야 함
```

Apple은 HLS 스펙을 IETF에 RFC로 제출했지만, 기술적으로 모두가 "Apple이 만든 것"이라는 인식. Apple이 마음대로 바꿀 수 있는 표준.

여기에 산업계가 대안을 만들기로 했다.

---

### MPEG의 개입

영상 코덱 표준을 만드는 **MPEG (Moving Picture Experts Group)**. ISO/IEC 산하 표준화 기구.

```
MPEG가 만든 표준들:
- MPEG-1: 1993년 (VCD)
- MPEG-2: 1995년 (DVD, 디지털 방송)
- MPEG-4 Part 2: 1999년
- MPEG-4 Part 10 = H.264 (2003년)
- MPEG-4 Part 14 = MP4 컨테이너
```

MPEG가 영상 코덱은 잘 만들었다. 그런데 스트리밍 프로토콜은 안 했었다. HLS에 대응할 필요가 생겼다.

2011년부터 작업 시작, 2012년 11월 **MPEG-DASH** 발표. ISO/IEC 23009-1 표준.

---

### DASH의 정식 명칭과 의미

**DASH = Dynamic Adaptive Streaming over HTTP**

이름 자체가 HLS와 거의 같은 컨셉을 선언한다.
- Dynamic: 라이브든 VOD든
- Adaptive: 화질 자동 조절 (ABR)
- HTTP: 일반 HTTP 위에서 동작

HLS도 이걸 다 하니까... 그럼 왜 따로 만들었나?

**1. 코덱 독립적**
HLS는 사실상 H.264 + AAC + MPEG-TS 컨테이너에 묶여 있었다. DASH는 어떤 코덱, 어떤 컨테이너든 OK.

**2. 컨테이너 자유**
HLS는 .ts 강제. DASH는 fMP4, WebM, MPEG-TS 다 됨.

**3. 정확한 ISO 표준**
한 회사 소유가 아닌 ISO 표준.

**4. 더 풍부한 메타데이터**
MPD가 M3U8보다 표현력이 좋음.

---

### MPD vs M3U8 — 매니페스트 비교

DASH의 매니페스트 파일은 `.mpd` (Media Presentation Description). XML 기반.

```xml
<?xml version="1.0"?>
<MPD xmlns="urn:mpeg:dash:schema:mpd:2011"
     type="static"
     mediaPresentationDuration="PT3M30S"
     minBufferTime="PT2S"
     profiles="urn:mpeg:dash:profile:isoff-on-demand:2011">
  <Period>
    <AdaptationSet mimeType="video/mp4" codecs="avc1.640028">
      <Representation id="1080p" bandwidth="6000000" width="1920" height="1080">
        <BaseURL>1080p/</BaseURL>
        <SegmentList timescale="1000" duration="6000">
          <Initialization sourceURL="init.mp4"/>
          <SegmentURL media="seg1.m4s"/>
          <SegmentURL media="seg2.m4s"/>
        </SegmentList>
      </Representation>
      <Representation id="720p" bandwidth="3000000" width="1280" height="720">
        <BaseURL>720p/</BaseURL>
        <SegmentList timescale="1000" duration="6000">
          <Initialization sourceURL="init.mp4"/>
          <SegmentURL media="seg1.m4s"/>
          <SegmentURL media="seg2.m4s"/>
        </SegmentList>
      </Representation>
    </AdaptationSet>
  </Period>
</MPD>
```

M3U8과 비교하면:

| | M3U8 (HLS) | MPD (DASH) |
|---|---|---|
| 포맷 | 텍스트 (라인 기반) | XML |
| 크기 | 작음 (~5 KB) | 큼 (~50 KB) |
| 파싱 | 단순 | 복잡 (XML 파서 필요) |
| 표현력 | 제한적 | 풍부 |
| 구조 | 평면적 | 계층적 (Period > AdaptationSet > Representation) |

---

### MPD의 계층 구조 — DASH의 표현력

MPD가 M3U8보다 복잡한 이유는 더 많은 걸 표현하려고.

```
MPD
└── Period (시간 구간 - 예: "본방송", "광고", "본방송")
    └── AdaptationSet (트랙 그룹 - 예: 비디오, 오디오)
        └── Representation (개별 트랙 - 예: 1080p, 720p)
            └── Segment (실제 데이터 조각)
```

**Period**가 흥미롭다. 한 콘텐츠 안에서 시간 구간을 나눈다.

```xml
<MPD>
  <Period id="content1" duration="PT10M">
    <!-- 본방송 -->
  </Period>
  <Period id="ad1" duration="PT30S">
    <!-- 광고 (다른 코덱, 다른 해상도 가능) -->
  </Period>
  <Period id="content2" duration="PT20M">
    <!-- 본방송 복귀 -->
  </Period>
</MPD>
```

HLS의 `#EXT-X-DISCONTINUITY`와 같은 역할이지만 더 구조화됨.

**AdaptationSet**: 같은 종류의 트랙 묶음.

```xml
<AdaptationSet mimeType="video/mp4">
  <!-- 비디오 화질들 -->
  <Representation id="1080p" .../>
  <Representation id="720p" .../>
</AdaptationSet>

<AdaptationSet mimeType="audio/mp4" lang="ko">
  <!-- 한국어 오디오들 -->
  <Representation id="ko_high" .../>
  <Representation id="ko_low" .../>
</AdaptationSet>

<AdaptationSet mimeType="audio/mp4" lang="en">
  <!-- 영어 오디오 -->
  <Representation id="en" .../>
</AdaptationSet>
```

플레이어는 시청자 환경에 맞게 각 AdaptationSet에서 하나씩 골라 조합.

---

### Segment의 종류 — 4가지 방식

DASH는 세그먼트를 표현하는 방법이 4가지나 된다.

**1. SegmentList**: 모든 세그먼트 URL을 명시

```xml
<SegmentList timescale="1000" duration="6000">
  <SegmentURL media="seg1.m4s"/>
  <SegmentURL media="seg2.m4s"/>
  <SegmentURL media="seg3.m4s"/>
  ...
</SegmentList>
```

VOD에서 가장 단순. 매니페스트가 커진다는 단점.

**2. SegmentTemplate**: URL 패턴으로 표현

```xml
<SegmentTemplate
  media="seg$Number$.m4s"
  initialization="init.mp4"
  startNumber="1"
  duration="6000"
  timescale="1000"/>
```

`$Number$`가 1, 2, 3, ...으로 치환됨. 매니페스트 크기 작음.

**3. SegmentBase**: 큰 파일을 byte range로 나눔

```xml
<SegmentBase indexRange="0-1000">
  <Initialization range="0-1000"/>
</SegmentBase>
<BaseURL>single_big_file.m4s</BaseURL>
```

HLS의 `EXT-X-BYTERANGE`와 같은 개념.

**4. SegmentTimeline**: 정확한 타임라인 명시

```xml
<SegmentTemplate media="seg$Number$.m4s">
  <SegmentTimeline>
    <S d="6000" r="50"/>  <!-- 6초 세그먼트 51개 -->
    <S d="5500"/>          <!-- 5.5초 세그먼트 1개 -->
    <S d="6000" r="20"/>   <!-- 6초 세그먼트 21개 -->
  </SegmentTimeline>
</SegmentTemplate>
```

라이브에서 세그먼트 길이가 정확히 같지 않을 때 유용.

---

### DASH의 컨테이너 자유도

HLS는 `.ts` 또는 `.fmp4`. DASH는 자유.

```
DASH 지원 컨테이너:
- ISO BMFF (.mp4, .m4s) - 가장 일반적
- MPEG-TS (.ts) - HLS와 같음
- WebM (.webm) - Google 푸시
- ISO BMFF Common Media Application Format (CMAF)
```

WebM 지원이 흥미롭다. Google이 만든 오픈 컨테이너. VP9, AV1 코덱 자연스럽게 사용.

```xml
<Representation mimeType="video/webm" codecs="vp09.00.41.08">
  ...
</Representation>
```

YouTube는 DASH + WebM 조합을 많이 쓴다.

---

### DASH 프로파일 — 사용 시나리오 분리

DASH는 너무 자유로워서 호환성 문제가 생긴다. 그래서 **프로파일**로 사용 범위를 제한.

```
DASH 주요 프로파일:
- ISO Live Profile: 라이브 스트리밍 (ISO BMFF + 동적 MPD)
- ISO On-Demand Profile: VOD (ISO BMFF + 정적 MPD)
- ISO Main Profile: 일반 ISO BMFF
- MPEG-TS Main Profile: TS 컨테이너
- DASH-IF IOP: 산업계 호환성 프로파일
```

플레이어는 프로파일 보고 "이 콘텐츠는 내가 처리 가능한가" 판단.

대부분 라이브 서비스는 **DASH-IF IOP** 프로파일 따른다. 호환성 우선.

---

### HLS와 DASH의 동시 운영 — 산업의 현실

대형 플랫폼들은 보통 둘 다 제공한다.

```
[유튜브]
- iOS Safari: HLS
- Chrome, Firefox: DASH
- Android Chrome: DASH

[넷플릭스]
- iOS: HLS
- 그 외: DASH
- 스마트TV: 자체 프로토콜
```

이유: iOS Safari가 HLS만 네이티브 지원. JavaScript로 DASH 처리도 가능은 하지만 배터리 효율이 떨어진다.

근데 같은 영상을 HLS와 DASH로 두 번 인코딩하면 비용 2배. 이걸 해결한 게 **CMAF**.

---

### CMAF — HLS와 DASH의 공통분모

이전에 LL-HLS 공부할 때 나왔다.

**CMAF (Common Media Application Format)**: HLS와 DASH가 같은 세그먼트 파일을 쓸 수 있게 만든 표준.

```
[CMAF 이전]
HLS용:  seg1.ts, seg2.ts, ... (별도 인코딩)
DASH용: seg1.m4s, seg2.m4s, ... (별도 인코딩)
저장 비용: 2배

[CMAF 이후]
공통:   seg1.cmaf, seg2.cmaf, ...
HLS m3u8가 이걸 참조
DASH MPD도 이걸 참조
저장 비용: 1배
```

CMAF는 본질적으로 fMP4 컨테이너. ISO BMFF 기반.

HLS가 .ts에서 fMP4(CMAF)로 옮겨가는 게 LL-HLS 시대의 큰 흐름. 결국 HLS와 DASH가 컨테이너 레벨에서 통합되는 방향.

### 1강 핵심 3줄

1. DASH는 HLS의 Apple 종속성을 깨려고 ISO MPEG가 2012년 만든 ISO/IEC 표준 — 코덱/컨테이너 독립적
2. MPD는 XML 기반, Period > AdaptationSet > Representation 계층 구조 — M3U8보다 표현력 풍부하지만 복잡
3. 산업 현실: iOS는 HLS, 나머지는 DASH 둘 다 운영 — CMAF로 세그먼트 통합해서 저장 비용 절감
