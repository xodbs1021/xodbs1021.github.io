## 레벨 3-1 / 3강: H.264 인코더 옵션과 실전 튜닝

---

### H.264 인코더 종류

같은 H.264를 만들어도 인코더마다 다르다.

**x264**: 오픈소스 CPU 인코더. 가장 압축률 좋음. FFmpeg에 통합.

**NVENC**: NVIDIA GPU 인코더. 빠르지만 압축률은 x264보다 떨어짐.

**Quick Sync**: Intel CPU의 내장 인코더. NVENC와 비슷.

**AMD AMF**: AMD GPU 인코더.

**Apple Video Toolbox**: macOS/iOS 하드웨어 인코더.

선택 기준:
```
[화질 우선 VOD] → x264 slow preset
[라이브 게임 방송] → NVENC (CPU 안 씀)
[모바일 송출] → Hardware 인코더 (배터리)
[서버 트랜스코딩 대량] → NVENC + 다중 GPU
```

---

### x264 Preset — 속도 vs 압축률

x264의 가장 중요한 옵션.

```
ultrafast   → 빠름, 압축률 낮음
superfast
veryfast
faster
fast
medium      ← 기본값
slow
slower
veryslow    → 느림, 압축률 높음
placebo     → 거의 의미 없는 극단
```

같은 비트레이트 6 Mbps일 때 화질:
- ultrafast: 깍두기 보임
- veryfast: 라이브에서 받아들일 수준
- medium: 일반 VOD 화질
- slow: 우수
- veryslow: 최고 (인코딩 시간 10배)

라이브는 보통 `veryfast`. 실시간 처리 한계.
VOD는 `medium` 또는 `slow`.

---

### Tune — 콘텐츠별 최적화

영상 종류에 맞춘 튜닝.

```
[Tune 옵션]
film         → 영화 (실사)
animation    → 애니메이션 (세밀한 선)
grain        → 필름 그레인 보존
stillimage   → 정지 영상
psnr         → PSNR 최적화 (수치 평가용)
ssim         → SSIM 최적화
fastdecode   → 빠른 디코딩 우선
zerolatency  → 라이브 (B-frame 안 씀)
```

라이브 스트리밍은 `zerolatency` 필수.

```bash
ffmpeg -i input \
  -c:v libx264 -preset veryfast -tune zerolatency \
  output.mp4
```

zerolatency 효과:
- B-frame 비활성화
- threadding 변경
- 인코더 버퍼 최소화

---

### 비트레이트 제어 모드

x264의 비트레이트 제어 4가지.

**1. CBR (Constant Bit Rate)**
정확히 지정 비트레이트 유지.

```bash
ffmpeg -i input \
  -c:v libx264 \
  -b:v 6000k -maxrate 6000k -minrate 6000k -bufsize 12000k \
  output.mp4
```

라이브에 적합. 네트워크 대역폭 예측 가능.

**2. VBR (Variable Bit Rate)**
평균 비트레이트 목표, 장면별 변동.

```bash
ffmpeg -i input \
  -c:v libx264 \
  -b:v 6000k -maxrate 12000k -bufsize 24000k \
  output.mp4
```

화질 일정. VOD에 적합.

**3. CRF (Constant Rate Factor)**
화질 점수 고정.

```bash
ffmpeg -i input \
  -c:v libx264 \
  -crf 23 \
  output.mp4
```

CRF 18~28 범위. 18이 거의 무손실, 28이 낮은 화질. 23이 기본.

라이브에선 못 쓴다. 출력 비트레이트 예측 불가.

**4. ABR (Average Bit Rate)**
2-pass 인코딩. 첫 패스에서 분석, 두 번째 패스에서 인코딩.

```bash
# 1-pass
ffmpeg -i input -c:v libx264 -b:v 6000k -pass 1 -f null /dev/null

# 2-pass
ffmpeg -i input -c:v libx264 -b:v 6000k -pass 2 output.mp4
```

VOD 최고 화질. 라이브 불가.

---

### GOP 설정 — 키프레임 간격

GOP 크기가 화질과 호환성에 영향.

```bash
# GOP 60프레임 (30fps에서 2초)
ffmpeg -i input -c:v libx264 -g 60 -keyint_min 60 output.mp4
```

옵션 의미:
- `-g 60`: 최대 GOP 크기
- `-keyint_min 60`: 최소 GOP 크기
- `-sc_threshold 0`: 씬 컷 키프레임 비활성화

GOP 결정 기준:
- HLS 6초 세그먼트 + 30fps → GOP 180 (또는 6의 배수)
- LL-HLS Partial 0.2초 → GOP 30 (1초)
- WebRTC → GOP 60~120 (2~4초)

GOP 짧을수록:
- 시킹 정확도 ↑
- 압축률 ↓ (IDR 자주 박힘)
- 대역폭 ↑

### B-frame 사용

B-frame은 압축률을 높이지만 지연을 만든다.

```bash
# B-frame 2개씩 (기본값)
ffmpeg -i input -c:v libx264 -bf 2 output.mp4

# B-frame 비활성화 (라이브)
ffmpeg -i input -c:v libx264 -bf 0 output.mp4
# 또는 -tune zerolatency
```

B-frame 사용 시 디코딩 순서가 표시 순서와 다름.

```
[B-frame 사용 시]
표시 순서: I B B P B B P
디코딩 순서: I P B B P B B
                ↑
            P를 먼저 디코딩
```

라이브 → B-frame 없음.
VOD → B-frame 2~3개로 압축률 향상.

---

### Profile과 Level 명시

명확한 호환성을 위해 명시.

```bash
ffmpeg -i input \
  -c:v libx264 \
  -profile:v high \
  -level 4.0 \
  output.mp4
```

프로파일 선택:
- `baseline`: iOS 옛 버전, 영상 통화
- `main`: 일반 영상
- `high`: 스트리밍, 블루레이

레벨 선택:
- `3.0`: 720p30
- `4.0`: 1080p60
- `4.1`: 1080p60 고품질
- `5.1`: 4K

---

### Pixel Format — 색상 표현

```bash
# YUV 4:2:0 (가장 일반적)
ffmpeg -i input -c:v libx264 -pix_fmt yuv420p output.mp4

# 4:2:2 (방송)
ffmpeg -i input -c:v libx264 -pix_fmt yuv422p output.mp4

# 4:4:4 (전문가용)
ffmpeg -i input -c:v libx264 -pix_fmt yuv444p output.mp4
```

YUV 4:2:0:
- Y: 휘도 (해상도 그대로)
- U, V: 색차 (가로/세로 절반 해상도)
- 사람 눈이 색 변화에 덜 민감해서 가능
- 대부분의 스트리밍 표준

`yuv420p`가 호환성 최고. 시청자 디바이스 디코더 100% 지원.

### 라이브 스트리밍 표준 명령

종합한 라이브용 H.264 인코딩.

```bash
ffmpeg -i input \
  -c:v libx264 \
  -preset veryfast \
  -tune zerolatency \
  -profile:v high \
  -level 4.0 \
  -pix_fmt yuv420p \
  -g 60 -keyint_min 60 \
  -sc_threshold 0 \
  -b:v 6000k -maxrate 6000k -minrate 6000k -bufsize 12000k \
  -bf 0 \
  -c:a aac -b:a 128k \
  -f flv rtmp://server/live/key
```

각 옵션이 라이브 환경에 맞춰져 있음.

### NVENC 명령

GPU 인코딩 동일한 결과 만들기.

```bash
ffmpeg -i input \
  -c:v h264_nvenc \
  -preset p4 \
  -tune ll \
  -profile:v high \
  -rc cbr \
  -b:v 6000k -maxrate 6000k -bufsize 12000k \
  -g 60 \
  -bf 0 \
  -c:a aac -b:a 128k \
  -f flv rtmp://server/live/key
```

NVENC 옵션:
- `-preset p4`: 균형 (p1=가장 빠름 ~ p7=가장 느림)
- `-tune ll`: low latency
- `-rc cbr`: CBR

NVENC는 x264 medium 정도의 화질을 GPU 1% 사용으로.

### ABR Ladder 인코딩

여러 화질 동시 생성.

```bash
ffmpeg -i input \
  -c:v libx264 -preset veryfast -tune zerolatency \
  -pix_fmt yuv420p -g 60 -keyint_min 60 -sc_threshold 0 \
  \
  -map 0:v:0 -s:v:0 1920x1080 -b:v:0 6000k -maxrate:v:0 6000k -bufsize:v:0 12000k \
  -map 0:v:0 -s:v:1 1280x720  -b:v:1 4500k -maxrate:v:1 4500k -bufsize:v:1 9000k \
  -map 0:v:0 -s:v:2 854x480   -b:v:2 1500k -maxrate:v:2 1500k -bufsize:v:2 3000k \
  -map 0:v:0 -s:v:3 426x240   -b:v:3 500k  -maxrate:v:3 500k  -bufsize:v:3 1000k \
  \
  -map 0:a:0 -c:a aac -b:a 128k -ac 2 \
  \
  -f hls \
  -hls_time 6 -hls_list_size 5 -hls_flags delete_segments \
  -master_pl_name master.m3u8 \
  -var_stream_map "v:0,a:0 v:1,a:0 v:2,a:0 v:3,a:0" \
  output_%v.m3u8
```

복잡하지만 1080p/720p/480p/240p 동시 생성.

### 인코딩 품질 평가 — PSNR과 SSIM

화질을 수치로 평가.

**PSNR (Peak Signal-to-Noise Ratio)**:
원본과 비교한 오차. 단위 dB.

```
PSNR 40+ dB: 거의 무손실
PSNR 35-40: 우수
PSNR 30-35: 보통
PSNR < 30: 화질 저하 인지
```

**SSIM (Structural Similarity Index)**:
구조적 유사도. 0~1.

```
SSIM 0.98+: 거의 무손실
SSIM 0.95-0.98: 우수
SSIM 0.90-0.95: 보통
SSIM < 0.90: 화질 저하
```

FFmpeg로 측정:
```bash
ffmpeg -i original.mp4 -i encoded.mp4 \
  -lavfi "ssim;[0:v][1:v]psnr" -f null /dev/null
```

---

### VMAF — Netflix의 화질 평가

PSNR/SSIM보다 사람 눈에 가까운 평가.

```bash
ffmpeg -i original.mp4 -i encoded.mp4 \
  -lavfi libvmaf=model_path=/path/to/vmaf_v0.6.1.json \
  -f null /dev/null
```

VMAF 점수:
- 100: 원본 수준
- 80+: 우수
- 60-80: 보통
- < 60: 저화질

Netflix가 만든 모델. 머신러닝 기반.

### Two-Pass 인코딩

VOD에서 최고 화질.

```bash
# 1st pass: 분석만
ffmpeg -i input \
  -c:v libx264 -preset slow -b:v 4000k \
  -pass 1 -f null /dev/null

# 2nd pass: 실제 인코딩
ffmpeg -i input \
  -c:v libx264 -preset slow -b:v 4000k \
  -pass 2 output.mp4
```

1st pass에서 영상 복잡도 분석. 2nd pass에서 비트를 효율적으로 분배.

같은 비트레이트에 1-pass보다 화질 5~10% 좋음.

### 3강 핵심 3줄

1. x264 preset (ultrafast~placebo)으로 속도/화질 트레이드오프, tune zerolatency로 라이브 최적화 — NVENC는 GPU 활용한 동급 솔루션
2. 라이브는 CBR + B-frame 없음, GOP를 세그먼트 길이에 맞춤, yuv420p + High Profile + Level 4.0이 표준
3. PSNR/SSIM/VMAF로 화질 평가, Two-pass로 VOD 최고 화질 — ABR ladder는 한 명령에 여러 해상도 동시 생성
