## 레벨 3-2 / 4강: AAC 외 오디오 코덱들 — 언제 다른 거 쓰나

---

### AAC가 표준이지만 다른 게 나은 경우

라이브 스트리밍 시청자 측은 거의 다 AAC지만, 다른 영역에선 다른 코덱이 표준.

```
[코덱별 표준 영역]
AAC: 라이브 스트리밍 (HLS), 음악 다운로드
Opus: WebRTC, 디스코드, Telegram
MP3: 옛날 음원, 호환성 우선
AC-3: 영화, 5.1 채널
FLAC: 무손실 음원
G.711: 옛 전화 시스템
```

각각 어디서 쓰는지 알아야 면접/설계할 때 답할 수 있다.

---

### Opus — WebRTC와 통화 표준

**Opus**: 2012년 IETF 표준. WebRTC 의무 코덱.

```
[Opus 특징]
- 무료, 로열티 없음
- 6 Kbps ~ 510 Kbps 광범위
- 지연 5~60ms (조절 가능)
- 음성/음악 자동 전환 (SILK + CELT)
- WebRTC 의무 지원
```

언제 쓰나:
- WebRTC 화상 회의 (Zoom, Meet)
- 디스코드 음성 채널
- Telegram, WhatsApp 통화
- WHIP/WHEP

언제 안 쓰나:
- HLS 라이브 (호환성 문제, AAC 강제)
- 일반 영상 파일 (AAC 표준)

---

### Opus vs AAC — 실무 비교

```
[같은 비트레이트 음질]
Opus 64 Kbps ≈ AAC 96 Kbps
Opus 96 Kbps ≈ AAC 128 Kbps
Opus 128 Kbps ≈ AAC 192 Kbps

[지연]
Opus: 30~40ms (WebRTC)
AAC-LC: ~100ms (HLS는 더 큼)

[호환성]
Opus: 최신 브라우저 + WebRTC만
AAC: 모든 기기
```

저비트레이트, 저지연 = Opus.
호환성, 대규모 = AAC.

### Opus FFmpeg

```bash
# 음성 통화용
ffmpeg -i voice.wav -c:a libopus -b:a 32k \
  -application voip \
  output.opus

# 음악용
ffmpeg -i music.wav -c:a libopus -b:a 128k \
  -application audio \
  output.opus

# 저지연 (게임, 통신)
ffmpeg -i input.wav -c:a libopus -b:a 64k \
  -application lowdelay \
  output.opus
```

`-application`으로 음질 모드 선택. 자동도 OK.

---

### MP3 — 여전히 살아있는 옛 표준

```
[MP3 특징]
- 1993년 표준
- 2017년 특허 만료 (완전 자유)
- 최대 320 Kbps
- 모든 기기 호환 (40년 누적)
- 다중 채널 미지원
```

언제 쓰나:
- 옛 기기 호환 필요
- 음성 팟캐스트 (관습)
- 라이센스 신경 안 쓰는 작은 프로젝트

언제 안 쓰나:
- 새 라이브 플랫폼
- 효율 중요한 곳 (AAC가 음질 우수)
- 다중 채널 (5.1)

```bash
ffmpeg -i input.wav -c:a libmp3lame -b:a 192k output.mp3
```

---

### AC-3 (Dolby Digital) — 영화/방송

**AC-3**: Dolby가 1991년 만듦.

```
[AC-3 특징]
- 5.1 채널 표준
- DVD, 블루레이, 디지털 방송
- 비트레이트 32~640 Kbps
- 라이센스 비용 있음 (Dolby)
```

언제 쓰나:
- DVD/블루레이 콘텐츠
- 케이블/위성 방송
- Netflix, Disney+의 5.1 트랙
- 영화 다운로드

언제 안 쓰나:
- 일반 라이브 스트리밍 (AAC로 충분)
- 모바일 (스테레오 위주)

```bash
# AC-3 인코딩
ffmpeg -i input.wav -c:a ac3 -b:a 384k -ac 6 output.ac3
```

5.1 채널 (6ch)에 384 Kbps가 표준.

---

### E-AC-3 (Dolby Digital Plus)

AC-3의 진화형.

```
[E-AC-3]
- 최대 15.1 채널
- 압축 효율 더 좋음
- Dolby Atmos 기반
- Netflix, Disney+ Atmos 콘텐츠
```

E-AC-3 위에 Dolby Atmos. 영화관 음향이 집까지.

일반 스트리밍 개발자가 직접 다루는 경우는 거의 없음.

---

### FLAC — 무손실 압축

**FLAC (Free Lossless Audio Codec)**.

```
[FLAC 특징]
- 무손실 (원본 그대로 복원)
- 압축률 50~60%
- 무료, 오픈소스
- 16비트 / 24비트 지원
```

언제 쓰나:
- 음원 마스터링 보관
- Tidal HiFi, Apple Music Lossless
- 오디오필 시청자

언제 안 쓰나:
- 일반 스트리밍 (대역폭 부담)
- 라이브 방송 (압축 손실 무관)

```bash
ffmpeg -i input.wav -c:a flac output.flac
# 자동 최적 설정
```

---

### ALAC — Apple Lossless

```
[ALAC 특징]
- Apple의 무손실
- FLAC과 비슷한 압축률
- 2011년 오픈소스화
- iTunes/Apple Music에서 사용
```

Apple 생태계에서만 의미 있음. 일반 스트리밍 거의 안 씀.

---

### 시나리오별 코덱 결정

실무 체크리스트.

```
[질문하면 자동 결정]

Q1: HLS/LL-HLS/DASH 라이브?
→ AAC-LC 128 Kbps 무조건

Q2: WebRTC/WHIP/WHEP?
→ Opus 64 Kbps (음성+음악)

Q3: 4K 영화 5.1 채널 VOD?
→ E-AC-3 또는 AC-3

Q4: 음악 마스터 보관?
→ FLAC

Q5: 화상 회의?
→ Opus (WebRTC 의무)

Q6: 음성 통화?
→ Opus 32 Kbps (저대역폭)

Q7: 디지털 방송 송출?
→ AAC-LC + AC-3 (지역 표준)

Q8: 모바일 데이터 절약?
→ HE-AAC 48 Kbps 또는 Opus

Q9: 옛 기기 호환?
→ MP3 또는 AAC

Q10: PokeClip 같은 분석?
→ AAC (멀티 트랙 + MPEG-TS)
```

### 라이브 게임 방송 - 치지직 예시

표준 설정.

```
[송출 측 OBS → 치지직]
오디오: AAC 128 Kbps, 48 kHz, 스테레오
컨테이너: FLV (RTMP)

[치지직 → 시청자 HLS]
오디오: AAC 128 Kbps (그대로 전달)
컨테이너: MPEG-TS

[LL-HLS 시청자]
오디오: AAC 128 Kbps
컨테이너: fMP4 (CMAF)
```

전 구간 AAC. 트랜스코딩 안 함 (오디오는).

---

### Netflix 4K HDR 영화 예시

프리미엄 OTT.

```
[Netflix 4K HDR 영화]
영상: H.265 Main 10 (HDR10) 또는 Dolby Vision
오디오 트랙들:
  - 영어 5.1: E-AC-3 (Dolby Digital Plus)
  - 영어 Atmos: E-AC-3 with Atmos
  - 한국어 5.1: E-AC-3
  - 한국어 2.0: AAC-LC 256 Kbps
  - 영어 코멘터리: AAC-LC 128 Kbps

컨테이너: fMP4 (CMAF)
배포: DASH + HLS
```

여러 오디오 트랙 + 다중 코덱. 시청자가 선택.

### Zoom 화상회의 예시

WebRTC 기반.

```
[Zoom 통화]
영상: H.264 Baseline (호환성)

```

오디오 한국어 표준. 영상보다 훨씬 더 중요.

### 코덱 변환 - 실무 흐름

스트리밍 플랫폼에서 자주 함.

```
[자주 발생하는 변환]

옛 RTMP 송출 (AAC) → HLS 시청 (AAC)
→ 변환 없음 (-c copy)

WebRTC 송출 (Opus) → HLS 배포 (AAC)
→ Opus → 디코딩 → AAC 인코딩 (트랜스코딩)

영화 DVD (AC-3 5.1) → 모바일 (AAC 2.0)
→ AC-3 → 디코딩 → 다운믹스 → AAC 인코딩

음악 (FLAC) → 스트리밍 (AAC)
→ FLAC → 디코딩 → AAC 인코딩
```

라이브 플랫폼은 송출 코덱과 시청 코덱이 같으면 트랜스코딩 회피 (CPU 절감).

---

### 다른 코덱 사이 변환 명령

```bash
# Opus → AAC (WebRTC → HLS 시청 변환)
ffmpeg -i input.opus -c:a aac -b:a 128k output.m4a

# AAC → Opus (HLS → WebRTC 변환)
ffmpeg -i input.m4a -c:a libopus -b:a 64k output.opus

# AC-3 5.1 → AAC 2.0 (영화 → 모바일)
ffmpeg -i input.ac3 -c:a aac -b:a 192k -ac 2 output.m4a

# FLAC → AAC (마스터 → 배포)
ffmpeg -i input.flac -c:a aac -b:a 256k output.m4a

# MP3 → AAC (옛 음원 마이그레이션)
ffmpeg -i input.mp3 -c:a aac -b:a 192k output.m4a
```

모든 변환은 디코딩 + 재인코딩. 미세하게 음질 손실.

원본을 유지하는 게 가장 좋음.

### 4강 핵심 3줄

1. AAC는 HLS/일반 스트리밍 표준, Opus는 WebRTC/통화 표준, AC-3/E-AC-3는 영화 5.1, FLAC은 무손실 — 코덱마다 표준 영역 다름
2. Opus 64 Kbps ≈ AAC 96 Kbps 음질 (저비트레이트일수록 Opus 유리), 지연도 Opus가 짧음 — 호환성 때문에 라이브 스트리밍은 여전히 AAC
3. 송출-시청 코덱이 같으면 `-c copy`로 트랜스코딩 회피 (CPU 절감), 다르면 디코딩+재인코딩 — Netflix 같은 OTT는 여러 코덱 트랙 동시 제공
