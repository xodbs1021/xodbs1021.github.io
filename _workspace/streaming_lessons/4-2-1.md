## 레벨 4-2 / 1강: ABR Ladder 기본 원리와 표준 사다리

---

### ABR Ladder가 뭔지 다시 정리

플랫폼이 한 원본을 여러 화질로 인코딩해서 가지고 있는 목록.

```
[치지직 ABR Ladder 예시]
1080p60 @ 6000 kbps
720p60  @ 3500 kbps
720p30  @ 2500 kbps
480p30  @ 1500 kbps
360p30  @ 800 kbps
```

플레이어가 시청자 네트워크에 맞춰 골라 받는다.

레벨 1-2에서 개념을 봤고, 이번 레벨에서는 **실제로 어떻게 설계하는지** 본다.

---

### 사다리를 잘못 짜면 일어나는 일

**너무 듬성하게 (3단계)**:
```
1080p @ 6000k
720p  @ 3000k
360p  @ 800k
```

문제:
- 1080p ↔ 720p 사이 갭이 큼. 와이파이가 흔들리면 바로 360p로 떨어짐
- 화질 전환 시 시각적 충격 큼
- 시청자가 "사용자가 화질 막 바뀌어!" 불만

**너무 촘촘하게 (10단계)**:
```
1080p, 900p, 810p, 720p, 640p, 540p, 480p, 360p, 270p, 180p
```

문제:
- 인코딩 비용 10배
- 스토리지 비용 10배
- CDN 캐시 효율 떨어짐 (요청이 분산)
- 시청자 체감 차이 거의 없음

**적절히 (4~5단계)**: 1080p, 720p, 480p, 360p가 표준.

---

### Apple HLS Authoring Spec — 업계 표준

Apple이 제시한 표준 사다리. 모든 플랫폼의 출발점.

```
[HEVC 기준 (Apple 2018 권장)]
2160p @ 11500 kbps
1080p @  6000 kbps
720p  @  3450 kbps
540p  @  2000 kbps
432p  @  1100 kbps
360p  @   730 kbps
270p  @   365 kbps
234p  @   145 kbps
```

H.264는 같은 화질에 비트레이트 2배.

```
[H.264 표준]
1080p @ 6000 kbps  (8000~10000 권장)
720p  @ 4500 kbps
480p  @ 2000 kbps
360p  @ 1100 kbps
240p  @  500 kbps
```

치지직, Twitch가 거의 이 사다리 기반.

---

### 비트레이트 결정의 원리 — Bits per Pixel

해상도와 비트레이트의 관계를 정량화.

```
Bits per Pixel = 비트레이트 / (가로 * 세로 * fps)

1080p60 @ 6000 kbps:
  6,000,000 / (1920 * 1080 * 60) = 0.048 bpp

720p30 @ 2500 kbps:
  2,500,000 / (1280 * 720 * 30) = 0.090 bpp
```

같은 화질을 유지하려면 bpp가 일정해야 한다.

```
실용 bpp 가이드:
0.04 ~ 0.06: 일반 라이브 (H.264)
0.10 ~ 0.15: 고화질 VOD (H.264)
0.02 ~ 0.04: H.265
0.015 ~ 0.025: AV1
```

새 사다리 만들 때 이 공식으로 시작점 잡는다.

---

### 콘텐츠 타입별 비트레이트 차이

같은 1080p60도 콘텐츠에 따라 필요 비트레이트가 다르다.

```
정적 콘텐츠 (토크쇼, 강의):     3000~4000 kbps OK
일반 게임 (전략, MOBA):         5000~6000 kbps
빠른 게임 (FPS, 레이싱):       6000~8000 kbps
4K HDR 영화:                  15000~25000 kbps
```

빠른 움직임 = P-frame 압축 효율 떨어짐 = 더 많은 비트 필요.

치지직이 발로란트 1080p60에 8000 kbps 권장하는 이유. 카메라가 빠르게 회전하니 일반 게임보다 비트 더 필요.

---

### 해상도 결정 — 가로세로비

표준 비율은 16:9. 사다리 안의 모든 해상도가 같은 비율을 유지해야 ABR 전환 시 자연스럽다.

```
16:9 (표준):
4320p: 7680x4320 (8K)
2160p: 3840x2160 (4K)
1440p: 2560x1440 (QHD)
1080p: 1920x1080 (FHD)
720p:  1280x720
540p:  960x540
480p:  854x480   (480÷9*16=853.33, 854로 반올림)
360p:  640x360
240p:  426x240
```

세로 영상(틱톡/쇼츠) 9:16은 별도 사다리.

```
9:16:
1080p: 1080x1920
720p:  720x1280
480p:  480x854
360p:  360x640
```

---

### 짝수 픽셀 강제

H.264/H.265 인코더는 16x16 또는 8x8 매크로블록 단위로 처리. 가로/세로가 그 배수여야 한다.

```
1280x720: OK
1281x721: x264가 자동으로 1280x720로 잘라냄 (약간 잘림)
```

ABR 사다리 짤 때 16의 배수로 맞추는 게 안전.

```bash
# FFmpeg에서 자동으로 짝수 강제
scale=-2:720
```

`-2`가 "가로는 자동인데 짝수로". `-1`은 반올림, `-2`는 강제 짝수.

---

### Bandwidth 태그 — Master Playlist의 BANDWIDTH 값

```
#EXT-X-STREAM-INF:BANDWIDTH=6500000,RESOLUTION=1920x1080
playlist_1080p.m3u8
```

여기서 BANDWIDTH는 **평균이 아닌 peak**. 세그먼트 중 가장 큰 비트레이트.

```
실제 비트레이트:  6000 kbps
peak 비트레이트:  6500 kbps (액션 장면)
오디오:           128 kbps
표시 BANDWIDTH:   6500 + 128 = 6628 → 6500000 (반올림)
```

플레이어는 이 값을 보고 "내 대역폭이 이 값보다 크면 이 화질 선택 가능"으로 판단.

VBR을 쓰면 peak가 크게 튀어서 의도보다 낮은 화질이 선택된다. 진짜 CBR을 쓰는 이유.

---

### 모바일과 데스크톱 사다리 분리

모바일은 화면이 작아서 1080p가 의미 없다.

```
[데스크톱 사다리]
1080p60 @ 6000k
720p60  @ 3500k
720p30  @ 2500k
480p30  @ 1500k

[모바일 사다리]
720p30  @ 2000k
480p30  @ 1000k
360p30  @ 600k
240p30  @ 300k
```

같은 영상을 두 사다리로 인코딩하면 비용이 2배지만, 모바일 시청자 비중이 70%면 절약 효과 큼.

Apple HLS 권장: User-Agent로 모바일 감지해서 다른 master playlist 제공.

---

### 적응형 사다리 — Per-Title vs Static

**Static (정적)**: 모든 콘텐츠에 같은 사다리 적용. 단순.

```
모든 영상에:
1080p @ 6000k
720p  @ 3000k
480p  @ 1500k
```

**Per-Title (제목별)**: 콘텐츠 복잡도에 맞춰 사다리 동적 결정.

```
"카페에서 잡담" 영상:
1080p @ 2500k  (복잡도 낮으니 비트 적게)
720p  @ 1500k
480p  @ 800k

"폭발 액션":
1080p @ 8000k  (복잡도 높으니 비트 많이)
720p  @ 4500k
480p  @ 2200k
```

같은 화질에 50%까지 비트레이트 절감. Netflix가 처음 개발 (2015년). 다음 강의 주제.

---

### 라이브에서 사다리 결정

라이브는 콘텐츠 사전 분석이 불가능. Static 사다리가 기본.

하지만 동적 조정은 가능.

```
[라이브 동적 사다리]
시작: 표준 사다리로 송출
1분 후: 첫 1분 분석 → 복잡도 추정 → 비트레이트 ±20% 조정
이후: 5분마다 재조정
```

치지직 같은 라이브 플랫폼은 이런 적응형 라이브 사다리를 일부 도입 중.

---

### 사다리 검증 — Convex Hull 그래프

각 (해상도, 비트레이트) 조합의 효율을 그래프로 본다.

```
[화질 (VMAF)]
100 ┤              ●●●●●●● 1080p ladder
 95 ┤         ●●●●●
 90 ┤    ●●●●●  720p ladder
 85 ┤●●●●
 80 ┤●●  480p ladder
 75 ┤●
    └──────────────────────────► 비트레이트
       1M      3M     6M
```

각 해상도 곡선의 가장 효율적인 점들을 잇는 게 **Convex Hull**. 이 라인 위의 점만 사다리에 포함.

예:
- 1080p @ 1500 kbps: 화질이 너무 떨어져 사다리에 안 넣음 (720p @ 1500이 더 나음)
- 720p @ 6000 kbps: 비트레이트 낭비 (1080p @ 6000이 같은 비트로 더 나음)

이 분석으로 사다리 단계를 최소화하면서 효율 극대화.

---

### 실전 사다리 — 치지직 추정

치지직 발로란트 1080p60 스트리머 기준 (시청자 분석 + 추정).

```
[원본]: 1080p60 @ 8000k (스트리머 송출)

[트랜스코딩 결과]:
1080p60 @ 8000k (원본 그대로 - source)
720p60  @ 4500k
720p30  @ 3000k
480p30  @ 1500k
360p30  @ 800k
160p30  @ 230k  (모바일 데이터 절약)
```

이게 한 명의 인기 스트리머에게 발생하는 트랜스코딩 부담. 그리고 동시 방송 스트리머가 수천 명. 인프라가 얼마나 큰지 감.

```
"Apple HLS Authoring Spec을 출발점으로 잡고
타겟 디바이스/콘텐츠에 맞춰 조정했습니다.

원칙은:
- bpp (Bits per Pixel) 일정 유지로 화질 균일
- 4~5단계가 적정 (촘촘함과 비용의 균형)
- 16:9 유지, 가로 짝수 강제 (-2:720 같이)

라이브에서는 Static 사다리 기본이지만,
프리미엄 콘텐츠는 첫 1분 분석으로 비트레이트 동적 조정.
모바일 시청자 비중 높아서 모바일 전용 사다리도 분리했습니다."
```

---

### 1강 핵심 3줄

1. ABR Ladder 설계는 bpp(Bits per Pixel) 일정 + 4~5단계 + 16:9 + 짝수 픽셀이 기본 원칙
2. Apple HLS Authoring Spec이 업계 출발점, 콘텐츠 타입에 따라 비트레이트 ±50% 조정 필요
3. BANDWIDTH는 peak 기준이라 진짜 CBR + Convex Hull 분석으로 비효율 단계 제거
