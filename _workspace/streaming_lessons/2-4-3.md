## 레벨 2-4 / 3강: Blocking Playlist Reload — 폴링을 없애는 방법

---

### 일반 HLS의 폴링 방식이 만든 지연

일반 HLS에서 시청자 플레이어가 새 세그먼트를 어떻게 찾는지부터.

플레이어는 `.m3u8` 매니페스트를 주기적으로 다시 받는다. 6초마다.

```
[12:00:00] 플레이어가 .m3u8 받음 → seg100까지 있네
[12:00:06] 플레이어가 .m3u8 다시 받음 → seg101 추가됐다!
[12:00:12] 플레이어가 .m3u8 다시 받음 → seg102 추가됐다!
```

문제: **새 세그먼트가 만들어진 순간을 정확히 모른다.**

```
[12:00:03] 서버에서 seg101 만들어짐
[12:00:06] 플레이어가 .m3u8 다시 받음 → seg101 발견
        ↑
   3초 지연 발생 (평균 폴링 주기 / 2)
```

매번 평균 3초씩 손해. 이게 일반 HLS 지연의 큰 부분이다.

---

### Blocking Playlist Reload의 아이디어

거꾸로 뒤집자.

**플레이어가 폴링하는 게 아니라, 서버가 새 정보 생기면 응답한다.**

```
[12:00:03] 플레이어: "seg101이 나오면 알려줘"
        서버: "아직 없어. 잠깐 기다려..."
[12:00:06] 서버에서 seg101 만들어짐
        서버: "seg101 나왔어! 여기 매니페스트 보내" → 응답
        플레이어: 즉시 받음 (지연 0초)
```

플레이어가 미리 "다음 거 나오면 줘"라고 요청을 걸어두는 거다. 서버는 새 정보가 생길 때까지 응답을 잡고 있다가, 생기면 즉시 응답한다.

이걸 **Blocking Playlist Reload** 또는 **HTTP Long Polling**이라고 부른다.

---

### HLS_MSN과 HLS_PART 쿼리 파라미터

플레이어가 어떻게 "다음 거 줘"라고 요청하나? URL 쿼리 파라미터로 표현한다.

```
GET /playlist.m3u8?_HLS_msn=102&_HLS_part=5
```

**`_HLS_msn=102`**: Media Sequence Number 102까지는 안다. 다음 정보 줘.

**`_HLS_part=5`**: 102번 세그먼트의 partial 5번까지 안다. 다음 partial 정보 줘.

서버는 두 조건을 보고 응답을 잡고 있는다:
- seg102가 만들어지거나
- seg101의 partial 6번이 만들어지면

뭔가 나오면 그때 응답.

---

### 서버 측 동작 — Long Polling 처리

서버 입장에서 응답을 어떻게 잡고 있나.

```
[일반 HTTP 요청]
요청 받음 → 즉시 응답 → 끝

[Blocking Reload]
요청 받음
  → 조건 검사: _HLS_msn=102 도달했나?
    → 아직 (seg102 안 만들어짐)
  → 응답 보류
  → 인코더가 새 세그먼트 만들 때까지 대기
  → seg102 만들어짐 알림 받음
  → 갱신된 매니페스트 응답
```

이런 비동기 처리를 서버가 지원해야 한다. 일반 nginx 설정으론 안 되고, 전용 LL-HLS 서버가 필요하다.

같은 시간에 1000명이 같은 채널을 본다면, 1000개의 응답이 대기 중인 상태. 서버가 메모리에 1000개의 요청을 들고 있다.

---

### CAN-BLOCK-RELOAD 선언

서버가 Blocking Reload를 지원한다는 걸 클라이언트에 알리는 방법.

```
#EXTM3U
#EXT-X-VERSION:9
#EXT-X-SERVER-CONTROL:CAN-BLOCK-RELOAD=YES,PART-HOLD-BACK=1.5
```

`CAN-BLOCK-RELOAD=YES`가 표시되면 플레이어는 폴링 대신 blocking reload를 쓴다.

플레이어가 이걸 못 보면 옛날 방식으로 폴링한다. 하위 호환성 유지.

---

### Timeout 처리 — 무한정 잡을 순 없다

서버가 응답을 영원히 잡고 있을 수는 없다. 네트워크 끊김, 클라이언트 종료, 시간 초과 등.

타임아웃 정책:

```
[서버 측 타임아웃]
응답 대기 최대 시간: 보통 TARGETDURATION × 1.5 = 9초

9초 동안 새 세그먼트 안 만들어지면:
→ 서버가 현재 상태로 응답
→ 클라이언트가 다시 요청
```

스트리머가 일시정지하거나 인코더에 문제가 생기면 9초 후 응답이 와서 클라이언트가 알 수 있다.

---

### 클라이언트의 재시도 패턴

클라이언트는 응답을 받으면 즉시 다음 요청을 건다.

```javascript
// 의사 코드
async function pollPlaylist() {
  while (true) {
    const url = `/playlist.m3u8?_HLS_msn=${lastMsn + 1}&_HLS_part=${nextPart}`;
    
    const response = await fetch(url);
    const m3u8 = await response.text();
    
    // 새 세그먼트/partial 처리
    processNewSegments(m3u8);
    
    // 즉시 다음 요청
    // (await가 위에서 새 정보 올 때까지 기다리니 폴링 간격 없음)
  }
}
```

while 루프지만 fetch가 응답을 대기하니 사실상 이벤트 기반.

---

### Blocking Reload의 효과

폴링 평균 3초가 사라진다.

```
[일반 HLS 지연 - 폴링 포함]
- 세그먼트 인코딩 대기: 6초
- 폴링 발견: 평균 3초
- 기타: 16초
총: 25초

[LL-HLS - Blocking Reload 사용]
- Partial 인코딩 대기: 0.2초
- 폴링 발견: 0초 ← Blocking으로 즉시
- 기타: 2.8초
총: 3초
```

폴링이 사라진 게 LL-HLS 지연 단축의 큰 축이다.

---

### CDN과 Blocking Reload의 충돌

여기 큰 문제가 있다. CDN은 보통 캐시된 응답을 즉시 반환한다.

```
[일반 CDN 동작]
요청 받음 → 캐시 확인 → 있으면 즉시 응답
```

Blocking Reload는 다르다. 응답을 잡고 있어야 한다.

```
[Blocking Reload]
요청 받음 → 캐시에 _HLS_msn=102 응답이 있는지 확인
  → 없으면: 캐시된 응답 주면 안 됨. Origin에서 대기 응답 받아야 함
  → Origin이 응답할 때까지 CDN도 대기
```

CDN이 long polling을 지원하지 않으면 LL-HLS 동작 안 한다.

지원 CDN:
- Akamai (Adaptive Media Delivery)
- CloudFront (한국 리전 지원)
- Cloudflare Stream
- Fastly

미지원 CDN이나 자체 nginx 캐시 서버에선 LL-HLS 구현 어려움.

---

### Blocking Reload + Partial 결합

Partial과 Blocking Reload는 같이 쓰여야 진가가 나온다.

```
[전통 HLS - 6초 단위]
시청 지연: 25초

[Partial만 사용 - 폴링 유지]
시청 지연: 0.2 + 폴링 대기 3 + 버퍼 3 = 6.2초
폴링 때문에 partial의 의미 약함

[Partial + Blocking Reload]
시청 지연: 0.2 + 0 + 버퍼 3 = 3.2초
폴링 제거로 partial 진가 발휘
```

둘 다 같이 써야 LL-HLS 약속한 3~5초 지연이 가능.

---

### Blocking Reload 비용 증가

서버 부하가 늘어난다.

```
[일반 HLS]
시청자 100만 명 × 6초마다 폴링 1번 = 16만 RPS
서버: 16만 RPS 처리

[LL-HLS - Blocking Reload]
시청자 100만 명 × 항상 대기 1개 = 100만 동시 connection
서버: 100만 connection 유지 + 갱신마다 100만 응답
```

연결 수가 6배 증가. CPU와 메모리 부하 증가.

해결책:
- HTTP/2 / HTTP/3 활용 (멀티플렉싱)
- 효율적인 long polling 서버 (Node.js, Go의 비동기 I/O)
- Origin Shield 강화

### Skip 메커니즘 — 빠른 갱신

매니페스트가 길어지면 클라이언트가 매번 전체를 받는 게 부담.

LL-HLS는 **Skip** 기능 제공.

```
GET /playlist.m3u8?_HLS_msn=200&_HLS_skip=YES
```

`_HLS_skip=YES`: 알고 있는 부분은 건너뛰고, 새 정보만 줘.

응답:
```
#EXTM3U
#EXT-X-VERSION:9
#EXT-X-SKIP:SKIPPED-SEGMENTS=195
#EXTINF:6.000,
seg200.m4s
#EXT-X-PART:DURATION=0.2,URI="seg201_part0.m4s",INDEPENDENT=YES
```

`#EXT-X-SKIP:SKIPPED-SEGMENTS=195`: 앞의 195개 세그먼트는 클라이언트가 알고 있으니 생략.

전체 매니페스트 50 KB → 압축된 응답 2 KB. 네트워크 비용 절감.

---

### Cache-Control 처리

Blocking Reload 응답의 캐시 헤더는 어떻게 잡나.

```http
[일반 HLS 매니페스트]
Cache-Control: max-age=2

[LL-HLS Blocking Reload 응답]
Cache-Control: max-age=0, must-revalidate
```

`max-age=0`: 즉시 만료. 다음 요청 시 반드시 서버 확인.
`must-revalidate`: 캐시 만료 후 무조건 재검증.

캐시 의미가 거의 없지만, 명시적으로 표시.

대안: ETag로 조건부 요청.
```http
If-None-Match: "abc123"
```
서버가 같은 ETag면 304 응답.

### 모바일에서의 Blocking Reload 문제

모바일 환경에서 long polling은 까다롭다.

**1. 백그라운드 진입**
앱이 백그라운드 가면 OS가 네트워크 연결 종료. blocking 요청 끊김.
→ 포그라운드 복귀 시 재요청.

**2. 슬립 모드**
화면 꺼지면 네트워크 절전. 응답 지연.
→ 영상 재생 중엔 슬립 방지.

**3. 셀룰러 → 와이파이 전환**
IP 바뀌면 TCP 연결 끊김.
→ 새 연결로 재요청.

LL-HLS 플레이어가 모바일에서 잘 동작하려면 이런 상황 처리 코드가 추가로 필요하다.

---

### 3강 핵심 3줄

1. Blocking Playlist Reload는 폴링을 long polling으로 바꿔서 평균 3초 지연을 0으로 제거 — `_HLS_msn`, `_HLS_part` 쿼리 파라미터로 다음 정보 요청
2. CAN-BLOCK-RELOAD 표시로 서버 지원 알림, 미지원 CDN(폴링만 가능)에선 LL-HLS 동작 안 함
3. 100만 동시 connection 부담 — HTTP/2 멀티플렉싱, Skip 응답으로 비용 절감, 모바일 백그라운드 처리 필요
