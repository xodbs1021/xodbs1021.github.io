## 레벨 3-4 / 1강: Opus가 WebRTC의 표준이 된 이유

---

### Opus의 위치 정리

오디오 코덱 지도에서 Opus는 특별한 위치.

```
[오디오 코덱 영역별 표준]
HLS/일반 스트리밍: AAC
WebRTC/통화: Opus
영화 5.1: AC-3/E-AC-3
무손실 보관: FLAC
```

WebRTC가 표준화된 게 Opus 보급의 결정적 사건.

거의 모든 화상 회의, 음성 통화, 디스코드 음성 채널이 Opus.

---

### 누가 만들었나

```
[Opus 개발]
2012년 IETF RFC 6716 표준화
2017년 RFC 8251로 업데이트

만든 곳:
- Xiph.Org Foundation (CELT 알고리즘)
- Skype (SILK 알고리즘, Microsoft에 인수 후 공개)
- Mozilla
- Google
- Broadcom
```

두 다른 회사가 만든 코덱을 합침. 흔치 않은 사례.

오픈소스. 완전 무료. 로열티 없음.

---

### Opus가 통화에 적합한 이유

WebRTC 표준이 된 핵심 이유.

```
[Opus의 강점]
1. 매우 넓은 비트레이트 범위 (6 ~ 510 Kbps)
   → 같은 코덱으로 음성부터 음악까지

2. 매우 짧은 지연 (5~60ms 조절 가능)
   → 실시간 통화에 필수

3. 패킷 손실에 강함
   → 인터넷 네트워크 환경

4. 무료, 로열티 없음
   → 모든 회사가 부담 없이 사용

5. 음성/음악 자동 적응
   → 통화 중 BGM 깔려도 자연스러움
```

다른 코덱은 이 중 일부만 만족.

---

### AAC와 직접 비교

```
[같은 비트레이트 음질]
6 Kbps:
  Opus: 음성 명료
  AAC: 거의 불가능

24 Kbps:
  Opus: 음성 + 약간 음악
  AAC: 음성도 거침

64 Kbps:
  Opus: 음악 양호
  AAC: 음성 우수

128 Kbps:
  Opus ≈ AAC (둘 다 우수)

256 Kbps:
  Opus ≈ AAC (구분 불가)
```

저비트레이트일수록 Opus 우위가 큼.

---

### 지연 비교

```
[코덱별 알고리즘 지연]
G.711: 0.125ms (전화 표준)
Opus 2.5ms 프레임: 약 5ms
Opus 20ms 프레임: 약 25ms
AAC-LC 23ms 프레임: 약 100ms+
AAC-LD: 20ms (저지연 특화)
MP3: 100ms+
```

Opus는 짧은 프레임 옵션이 핵심. 게임/통화에 필수.

---

### 두 알고리즘의 결합 — SILK + CELT

내부 구조 이해는 안 해도 OK. 결과만.

```
[Opus 모드]
저비트레이트 + 음성: SILK 사용
  - Skype에서 가져온 음성 통화 알고리즘
  
고비트레이트 + 음악: CELT 사용
  - Xiph가 만든 저지연 음악 코덱
  
중간 + 혼합: 하이브리드
  - 두 알고리즘 동시
```

인코더가 자동 선택. 사용자는 신경 안 써도 됨.

```
[강제 모드 옵션]
-application voip: SILK 우선 (음성)
-application audio: CELT 우선 (음악)
-application lowdelay: CELT (저지연)
```

---

### 손실 복구 — FEC

WebRTC 환경에서 핵심.

```
[FEC (Forward Error Correction)]
각 패킷에 다음 패킷 정보 일부 포함
패킷 손실 시 다음 패킷에서 복구

예시:
패킷 1: 자기 데이터 + 패킷 2의 일부
패킷 2: 자기 데이터 + 패킷 3의 일부
패킷 3: 자기 데이터 + 패킷 4의 일부

패킷 2 손실:
→ 패킷 3에 있던 패킷 2 정보로 복구
```

손실율 5%까지 자연스러운 음질 유지.

SDP에서 활성화:
```
a=fmtp:111 useinbandfec=1
```

WebRTC는 보통 기본 활성화.

---

### DTX — 침묵 시 절약

말 안 할 때 데이터 안 보냄.

```
[DTX (Discontinuous Transmission)]
침묵 감지 → 전송 중단
가끔 배경음 메타데이터만 전송

[효과 - 3자 화상 회의]
보통 한 명만 말함
나머지 2명: DTX로 거의 0 Kbps
전체 대역폭 70% 절감
```

WebRTC SFU에서 자동 활용.

```
[SDP 활성화]
a=fmtp:111 usedtx=1
```

---

### 프레임 크기 선택

WebRTC 표준은 20ms 프레임.

```
[프레임 크기별 사용처]
2.5ms: 게임, 초저지연 (드물게)
5ms: 게임
10ms: 화상 회의 일부
20ms: WebRTC 표준 (대부분)
40ms: 음성 위주, 저대역폭
60ms: 매우 저대역폭
```

20ms가 지연/오버헤드의 균형점.

---

### WebRTC SDP 협상

Opus 협상 모습.

```
[WebRTC SDP - 오디오 부분]
m=audio 9 UDP/TLS/RTP/SAVPF 111 0 8
a=rtpmap:111 opus/48000/2
a=fmtp:111 minptime=10;useinbandfec=1;usedtx=1
a=rtpmap:0 PCMU/8000
a=rtpmap:8 PCMA/8000
```

페이로드 111이 Opus.
0, 8은 G.711 (옛 전화 호환용 폴백).

`fmtp` 옵션:
- `minptime=10`: 최소 프레임 10ms
- `useinbandfec=1`: FEC 활성화
- `usedtx=1`: DTX 활성화

### 라이브 스트리밍에서 Opus

HLS는 AAC가 강제지만 일부 라이브 환경에 Opus.

```
[Opus 라이브 환경]
WHEP: 시청자 측 Opus 가능
LL-HLS: 일부 지원
DASH: 지원
WebM 컨테이너: 표준

[Opus 라이브 안 쓰는 곳]
일반 HLS (iOS 호환 위해 AAC)
방송 인제스트 (RTMP는 AAC)
대부분의 라이브 플랫폼
```

WebRTC 기반 라이브가 늘면 Opus 시청자 측 보급될 것.

### Opus FFmpeg 명령 - 종합

```bash
# 기본 (자동 모드 선택)
ffmpeg -i input.wav -c:a libopus -b:a 128k output.opus

# 음성 통화
ffmpeg -i voice.wav -c:a libopus -b:a 32k \
  -application voip \
  -frame_duration 20 \
  output.opus

# 음악 (스튜디오 품질)
ffmpeg -i music.wav -c:a libopus -b:a 256k \
  -application audio \
  output.opus

# 게임 (저지연)
ffmpeg -i game.wav -c:a libopus -b:a 64k \
  -application lowdelay \
  -frame_duration 10 \
  output.opus

# FEC 활성화 (손실 환경)
ffmpeg -i input.wav -c:a libopus -b:a 64k \
  -application voip \
  -fec true \
  -packet_loss 5 \
  output.opus
```

### WebM 컨테이너 — Opus 표준

Opus는 WebM에 잘 맞음.

```bash
# WebM (VP9 + Opus)
ffmpeg -i input.mp4 \
  -c:v libvpx-vp9 -b:v 2000k \
  -c:a libopus -b:a 128k \
  output.webm
```

YouTube가 채택한 형식.

브라우저 호환:
- Chrome, Firefox, Edge: 완전 지원
- Safari: 부분 지원 (최근)

### Opus의 한계

```
[Opus 약점]
1. HLS 표준 아님 (iOS 호환성 문제)
2. 5.1+ 채널 가능하지만 잘 안 씀
3. AAC만큼의 디바이스 보급률 아님
4. 일부 옛 디바이스 미지원

[극복 방법]
WebRTC 환경에선 표준
HLS 시청자 측엔 AAC 그대로 사용
하이브리드 운영
```

라이브 스트리밍이 Opus 못 옮기는 이유 = iOS Safari.

```
HLS는 호환성 때문에 여전히 AAC지만,

```

### 1강 핵심 3줄

1. Opus는 2012년 IETF 표준, Xiph (CELT) + Skype (SILK) 합작 — 6~510 Kbps 넓은 범위, 5~60ms 가변 지연, FEC/DTX로 통화 환경에 최적화
2. WebRTC 의무 코덱 (Discord, Zoom, WhatsApp, Meet) — AAC 대비 저비트레이트에 우수 (Opus 64 Kbps ≈ AAC 96 Kbps), 지연 짧음
3. HLS는 iOS 호환성 때문에 AAC 유지, WebRTC/WHIP/WHEP은 Opus 표준 — `-application voip/audio/lowdelay`로 모드 선택, WebM 컨테이너가 자연스러운 짝
