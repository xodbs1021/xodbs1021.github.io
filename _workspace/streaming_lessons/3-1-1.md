## 레벨 3-1 / 1강: H.264가 어떻게 영상을 압축하나

---

### 영상 압축이 왜 필요한지 다시

레벨 1에서 봤다. 1080p30 영상을 압축 없이 저장하면 1시간에 648GB. 인터넷으로 보낼 수 없다.

코덱이 압축을 맡는다. 그 중 가장 널리 쓰이는 게 H.264.

```
[1080p30 영상 압축 전후]
원본:      648 GB / 시간
H.264:    약 4 GB / 시간  (160배 압축)
```

이 정도 압축이 어떻게 가능한가? 이번 강의에서 다룬다.

---

### H.264의 정식 명칭

여러 이름으로 불린다.

```
정식 명칭:
- H.264 (ITU-T 명명법)
- MPEG-4 Part 10 (ISO/IEC 명명법)
- AVC (Advanced Video Coding)
```

같은 코덱의 다른 이름. 두 표준화 기구가 공동 개발해서.

2003년 표준 완성. 20년 이상 산업 표준 자리 지키는 중.

---

### 누가 만들었나

**JVT (Joint Video Team)**: ITU-T VCEG + ISO/IEC MPEG의 공동 작업팀.

이전엔 따로 표준 만들었다.
- ITU-T: H.261, H.263 (영상 통화용)
- MPEG: MPEG-1, MPEG-2 (방송/저장용)

H.264부터 합쳐서 만들기 시작. 합쳐진 게 큰 의미.

대표 기여자:
- Iain Richardson (영국, 기술 문서)
- Gary Sullivan (Microsoft)
- Thomas Wiegand (독일 HHI)

### 압축의 3가지 원리

영상 압축은 본질적으로 3가지 중복을 제거.

**1. 공간적 중복 (Spatial Redundancy)**
한 프레임 안에서 비슷한 픽셀들.

```
[하늘 사진]
픽셀: 파란색 1000개 연속
→ "파란색이 1000개 있다"로 압축
```

**2. 시간적 중복 (Temporal Redundancy)**
연속 프레임 간 비슷한 부분.

```
[고정 카메라, 사람이 약간 움직임]
프레임 1: 전체 화면 저장
프레임 2: "전체 똑같은데 사람만 5픽셀 오른쪽으로"
```

**3. 통계적 중복 (Statistical Redundancy)**
자주 나오는 값에 짧은 코드 할당.

```
[데이터에 0이 많이 나오면]
0 → 1비트로 표현
드물게 나오는 큰 값 → 긴 비트로 표현
```

H.264는 이 3가지를 모두 정교하게 활용.

---

### Macroblock — H.264의 기본 단위

영상을 16×16 픽셀 블록으로 나눈다. 이게 **macroblock**.

```
[1920×1080 영상]
가로: 1920 / 16 = 120 macroblock
세로: 1080 / 16 = 68 macroblock (마지막 줄은 8픽셀 패딩)
총: 120 × 68 = 8160 macroblock per frame
```

각 macroblock을 독립적으로 처리. 압축, 디코딩이 블록 단위.

작은 블록의 장점:
- 병렬 처리 가능
- 한 영역의 변화가 다른 영역에 영향 안 줌
- 오류가 발생해도 한 블록만 깨짐

---

### 프레임 종류 — I, P, B

H.264는 세 종류의 프레임.

**I-frame (Intra-coded)**:
혼자서 그릴 수 있는 프레임. 다른 프레임 참조 없음.
- 압축률: 낮음 (큰 파일)
- 독립적
- HLS 세그먼트 시작에 필수

**P-frame (Predicted)**:
이전 프레임을 참조해서 차이만 저장.
- 압축률: 높음
- 이전 P/I 프레임 필요

**B-frame (Bi-directional)**:
이전과 이후 프레임 모두 참조.
- 압축률: 최고
- 이전과 이후 P/I 프레임 모두 필요
- 디코딩 지연 발생

```
[GOP 구조 예시]
I B B P B B P B B P B B P I B B P...
↑                          ↑
새 GOP 시작                새 GOP
```

라이브 스트리밍은 보통 B-frame 안 씀. 디코딩 지연 줄이려고.

---

### Inter Prediction — 움직임 예측

P-frame의 핵심. 이전 프레임에서 비슷한 블록 찾기.

```
[프레임 N의 블록 (16,16)]
이전 프레임에서 비슷한 블록 찾기...

위치 (14,18)에 거의 똑같은 블록 발견!

저장 내용:
- 움직임 벡터: (-2, +2)  ← 2픽셀 왼쪽, 2픽셀 아래
- 차이값: 거의 0
```

전체 블록 데이터 대신 움직임 벡터 + 차이만 저장. 압축 효과 큼.

**Motion Estimation**: 이전 프레임에서 가장 비슷한 블록 찾는 과정. 인코딩 시간의 80% 차지.

인코딩 옵션 `preset` 차이가 여기서 나온다.
- `ultrafast`: 작은 영역만 검색
- `veryslow`: 큰 영역 정밀 검색

---

### Intra Prediction — 같은 프레임 내 예측

I-frame과 P/B-frame의 일부 블록은 이전 프레임 참조 없이 같은 프레임 내 주변 블록으로 예측.

```
[Intra Prediction 모드]
위쪽 블록과 비슷
왼쪽 블록과 비슷
대각선과 비슷
...

9가지 예측 모드 중 선택
```

```
[예시]
하늘 영역의 블록들:
- 위쪽 블록과 비슷 → "위쪽 블록 복사" 모드
- 차이만 저장
```

H.264는 인접 블록과의 관계도 활용.

---

### 변환 — DCT 비슷한 정수 변환

압축의 핵심 단계. 픽셀 데이터를 주파수 영역으로 변환.

```
[픽셀 데이터]
8×8 블록의 픽셀 값
[120, 122, 125, ...]

↓ DCT 비슷한 정수 변환

[주파수 데이터]
8×8 계수
[1000, 5, 2, 0, 0, ...]
  ↑    ↑    
저주파 고주파
```

사람 눈은 저주파(큰 변화)에 민감, 고주파(세부)에 둔감. 고주파를 버려도 화질 차이 적음.

---

### 양자화 — 손실 압축의 본질

변환된 계수를 정수로 반올림.

```
[양자화 전 계수]
[1000.0, 5.3, 2.1, 0.7, 0.2, 0.05, ...]

↓ 양자화 (나누고 반올림)

[양자화 후]
[100, 1, 0, 0, 0, 0, ...]
```

0이 많아진다. 압축 잘 됨.

**QP (Quantization Parameter)**: 양자화 강도.
- QP 낮음 (예: 18): 정밀한 양자화, 화질 좋음, 파일 큼
- QP 높음 (예: 35): 거친 양자화, 화질 나쁨, 파일 작음

FFmpeg의 `-crf` 옵션이 본질적으로 QP 제어.

```bash
ffmpeg -i input -c:v libx264 -crf 23 output.mp4
# CRF 23이 기본. QP 23 정도 의미
```

---

### 엔트로피 코딩 — 마지막 압축

양자화된 계수를 비트스트림으로 변환.

H.264는 두 가지 방식.

**CAVLC (Context-Adaptive Variable-Length Coding)**:
간단하고 빠름. 모든 프로파일 지원.

**CABAC (Context-Adaptive Binary Arithmetic Coding)**:
복잡하지만 압축률 10% 좋음. High Profile에서 사용.

```
[엔트로피 코딩 원리]
자주 나오는 값 → 짧은 비트
드문 값 → 긴 비트

예:
0 → 1비트
1 → 3비트
큰 값 → 10+비트
```

라이브 스트리밍은 보통 CABAC. 같은 비트레이트에 더 좋은 화질.

---

### 프로파일 — 사용 시나리오별 기능 집합

H.264는 사용처에 따라 프로파일이 다르다.

**Baseline Profile**:
- 가장 단순. CAVLC만, B-frame 없음.
- 옛날 모바일, 영상 통화

**Main Profile**:
- CABAC, B-frame 지원
- 일반 영상

**High Profile**:
- 모든 기능
- 8×8 변환 추가
- 블루레이, 스트리밍

**High 10 Profile**:
- 10비트 색심도
- 전문 방송

대부분의 라이브 스트리밍은 **High Profile**.

```bash
ffmpeg -i input \
  -c:v libx264 \
  -profile:v high \
  -level 4.0 \
  output.mp4
```

---

### 레벨 — 처리 부담 제한

같은 프로파일이라도 해상도/비트레이트에 따라 레벨이 다르다.

```
[H.264 주요 레벨]
Level 3.0: 720p30
Level 3.1: 720p60, 1080p30
Level 4.0: 1080p60
Level 4.1: 1080p60 + 더 높은 비트레이트
Level 5.0: 2K
Level 5.1: 4K
Level 6.0: 8K
```

디코더가 자기 처리 능력에 맞는 레벨만 지원. 저가폰은 Level 4.0까지 지원.

스트리머가 4K 영상을 만들어도 시청자 폰이 디코딩 못 함.

---

### 코덱 문자열 표기

SDP나 MIME 타입에서 H.264 표기.

```
avc1.640028
└─┬─┘ └──┬──┘
 코덱   파라미터

avc1 = H.264
6400 = High Profile (64 = profile_idc 100)
28 = Level 4.0 (0x28 = 40)
```

브라우저가 `MediaSource.isTypeSupported('video/mp4; codecs="avc1.640028"')` 호출해서 디코딩 가능한지 확인.

### 1강 핵심 3줄

1. H.264 (AVC, MPEG-4 Part 10)는 2003년 ITU-T + ISO/IEC 공동 표준 — 공간/시간/통계 중복을 모두 활용해 약 160배 압축
2. 16×16 macroblock 단위로 처리, I/P/B-frame 조합으로 GOP 구성 — Inter Prediction(움직임 벡터)과 Intra Prediction(주변 블록)으로 예측
3. DCT 비슷한 정수 변환 + 양자화 + 엔트로피 코딩(CABAC/CAVLC) — 프로파일(Baseline/Main/High)과 레벨(1080p60=4.0)로 사용 시나리오 구분
