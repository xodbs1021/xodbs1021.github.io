## 레벨 2-4 / 5강: LL-HLS 실전 구축 — 인코더, CDN, 플레이어 통합

---

### LL-HLS 구축에 필요한 4가지 컴포넌트

전체 파이프라인을 다시 보자.

```
[스트리머 OBS]
   │ RTMP
   ▼
[1. LL-HLS 인코더/패키저]
   │ partial 생성, 매니페스트 갱신
   ▼
[2. Origin 서버]
   │ Long polling, Chunked Transfer
   ▼
[3. CDN]
   │ LL-HLS 지원 CDN 필수
   ▼
[4. LL-HLS 플레이어]
   │ Preload Hint, Blocking Reload 처리
```

각 단계에서 도구 선택과 설정이 중요하다.

---

### 컴포넌트 1: LL-HLS 인코더 선택

LL-HLS 지원 인코더는 제한적이다.

**무료/오픈소스:**

```
FFmpeg
- LL-HLS 부분 지원
- hls_flags lhls 활용
- 안정성: 베타 수준
- 비용: 무료
- 단점: 일부 LL-HLS 기능 미구현

OvenMediaEngine
- 한국 회사 AirenSoft 제품
- LL-HLS 완전 지원
- 비용: 무료 (오픈소스)
- 장점: 한국어 문서, 활발한 개발

MistServer
- 오픈소스 (Community Edition)
- LL-HLS 지원
- 비용: 무료 (상용은 유료)
```

**상용 제품:**

```
Wowza Streaming Engine
- 업계 표준
- 비용: 연 $5000 이상
- 장점: 안정성, 기술 지원

AWS MediaLive + MediaPackage
- 클라우드 매니지드
- 비용: 시간당 $1~ + 트래픽
- 장점: 운영 부담 0, AWS 통합

Bitmovin Live
- 클라우드 SaaS
- 비용: 시간당 $5~
- 장점: 엔터프라이즈 기능
```

치지직 같은 규모는 자체 인코더 개발. 일반 라이브 서비스는 OvenMediaEngine 또는 Wowza가 무난.

### OvenMediaEngine으로 LL-HLS 구축 예시

가장 접근성 좋은 무료 옵션. 실제 설정 보자.

`Server.xml` 설정:

```xml
<Server version="8">
  <Name>OvenMediaEngine</Name>
  
  <VirtualHosts>
    <VirtualHost>
      <Name>default</Name>
      
      <!-- 입력: RTMP -->
      <Providers>
        <RTMP>
          <Port>1935</Port>
        </RTMP>
      </Providers>
      
      <!-- 출력: LL-HLS -->
      <Publishers>
        <LLHLS>
          <OriginMode>true</OriginMode>
          <ChunkDuration>0.2</ChunkDuration>       <!-- Partial 길이 -->
          <PartHoldBack>1.5</PartHoldBack>         <!-- partial 유지 시간 -->
          <SegmentDuration>4</SegmentDuration>     <!-- 세그먼트 길이 -->
          <SegmentCount>5</SegmentCount>           <!-- 매니페스트에 유지할 세그먼트 수 -->
          <CrossDomains>
            <Url>*</Url>
          </CrossDomains>
        </LLHLS>
      </Publishers>
      
      <!-- 트랜스코딩 ABR ladder -->
      <Applications>
        <Application>
          <Name>live</Name>
          <Type>live</Type>
          <OutputProfiles>
            <OutputProfile>
              <Name>720p</Name>
              <OutputStreamName>${OriginStreamName}_720p</OutputStreamName>
              <Encodes>
                <Video>
                  <Codec>h264</Codec>
                  <Bitrate>3000000</Bitrate>
                  <Width>1280</Width>
                  <Height>720</Height>
                  <Framerate>30</Framerate>
                  <KeyFrameInterval>30</KeyFrameInterval>  <!-- 1초 -->
                </Video>
                <Audio>
                  <Codec>aac</Codec>
                  <Bitrate>128000</Bitrate>
                  <Samplerate>48000</Samplerate>
                  <Channel>2</Channel>
                </Audio>
              </Encodes>
            </OutputProfile>
          </OutputProfiles>
        </Application>
      </Applications>
    </VirtualHost>
  </VirtualHosts>
</Server>
```

실행:

```bash
docker run -d \
  -p 1935:1935 \
  -p 3333:3333 \
  -v $(pwd)/Server.xml:/opt/ovenmediaengine/bin/origin_conf/Server.xml \
  airensoft/ovenmediaengine:latest
```

스트리머가 `rtmp://localhost:1935/live/stream_key`로 송출하면 시청자는 `https://localhost:3333/live/stream_key_720p/llhls.m3u8`로 시청.

### Key Frame Interval 설정의 중요성

LL-HLS에서 가장 중요한 인코더 설정.

```xml
<KeyFrameInterval>30</KeyFrameInterval>  <!-- 30fps에서 1초마다 IDR -->
```

LL-HLS는 모든 INDEPENDENT partial이 IDR로 시작해야 한다. PART-TARGET 0.2초인데 KeyFrameInterval이 2초면, 매 partial이 IDR이 아니다.

```
[잘못된 설정]
KeyFrameInterval = 2초 (GOP 60 프레임)
PART-TARGET = 0.2초

→ 매 10개 partial 중 1개만 INDEPENDENT
→ 첫 재생 시 평균 1초 대기
```

권장 설정:
```
PART-TARGET = 0.2초 (0.2 ~ 1.0초 사이)
KeyFrameInterval = 1초 (GOP 30 프레임 @ 30fps)

→ 매 5개 partial 중 1개 INDEPENDENT
→ 첫 재생 시 평균 0.5초 대기
```

GOP를 짧게 하면 압축 효율이 떨어진다. 같은 화질에 비트레이트 약 10~15% 증가. LL-HLS의 추가 비용.

### 컴포넌트 2: Origin 서버 — Long Polling 처리

OvenMediaEngine 같은 인코더가 Origin도 겸하는 경우가 많다.

자체 nginx로 LL-HLS Origin 구축 시 주의점:

```nginx
# nginx.conf
http {
    # LL-HLS partial 응답을 위한 chunked transfer
    chunked_transfer_encoding on;
    
    # Long polling 대기 시간 설정
    proxy_read_timeout 30s;
    keepalive_timeout 65s;
    
    # HTTP/2 활성화 (multiplexing)
    listen 443 ssl http2;
    
    location ~ \.m3u8$ {
        # 캐시 안 함 (long polling)
        add_header Cache-Control "no-cache, must-revalidate";
        
        # CORS
        add_header Access-Control-Allow-Origin *;
        
        # Origin 서버로 프록시
        proxy_pass http://llhls_backend;
        proxy_buffering off;  # Chunked transfer 통과
    }
    
    location ~ \.m4s$ {
        # 완성된 partial/세그먼트는 캐시 가능
        add_header Cache-Control "public, max-age=10";
        
        proxy_pass http://llhls_backend;
        proxy_buffering off;
    }
}
```

`proxy_buffering off`가 중요. nginx가 응답을 모아서 보내려는 기본 동작을 끈다. Chunk가 그대로 흘러가야 LL-HLS가 동작.

### Origin 서버 부하 측정

LL-HLS Origin은 부하 패턴이 일반 HLS와 다르다.

```
[일반 HLS - 100만 시청자]
요청률: 100만 / 6초 = 167K RPS
동시 연결: 평균 0.5초 × 167K = 84K
서버 메모리: 약 10 GB

[LL-HLS - 100만 시청자]
Blocking Reload: 100만 동시 대기 connection
Preload Hint: 100만 동시 대기 connection
서버 메모리: 약 60 GB
```

메모리 6배. 그러나 CPU는 비슷. 대기만 하니까.

해결책:
- Go, Rust로 만든 효율적 서버 (Node.js도 가능)
- HTTP/2 multiplexing으로 connection 1개에 여러 요청
- Origin Shield 계층 추가

### 컴포넌트 3: CDN 설정 — Akamai, Cloudflare 등

LL-HLS 지원 CDN과 미지원 CDN을 구분해야 한다.

**Akamai AMD (Adaptive Media Delivery)**:
LL-HLS 전용 옵션 활성화.

```
[Akamai 설정]
Behavior: Adaptive Media Delivery
- Format: HLS
- Low Latency Mode: Enable
- Chunked Transfer: Pass through
- Long Polling Support: Enable
```

요금: 일반 HLS 대비 약 20% 추가.

**CloudFront**:

```json
// CloudFront Distribution 설정
{
  "DefaultCacheBehavior": {
    "Compress": true,
    "AllowedMethods": ["GET", "HEAD", "OPTIONS"],
    "CachedMethods": ["GET", "HEAD"],
    
    // LL-HLS 매니페스트는 캐시 짧게
    "CachePolicyId": "managed-caching-disabled",
    
    // Origin Shield 활성화
    "OriginShield": {
      "Enabled": true,
      "OriginShieldRegion": "ap-northeast-2"
    }
  }
}
```

**Cloudflare Stream**:
완전 매니지드 서비스. 자체 인코더 + CDN + 플레이어.

```javascript
// Cloudflare Stream Live API
const response = await fetch('https://api.cloudflare.com/client/v4/accounts/{account_id}/stream/live_inputs', {
  method: 'POST',
  headers: { 'Authorization': 'Bearer {token}' },
  body: JSON.stringify({
    meta: { name: 'My Live Stream' },
    recording: { mode: 'automatic' },
    deleteRecordingAfterDays: 7
  })
});

const { data } = await response.json();
// data.rtmps.url, data.rtmps.streamKey 받음
// 시청자 URL: https://customer-{id}.cloudflarestream.com/{video_id}/manifest/video.m3u8
```

Cloudflare Stream은 가장 쉽지만 가장 비싸다. 분당 $5~.

### 컴포넌트 4: hls.js로 LL-HLS 플레이어

브라우저 측 LL-HLS 재생. hls.js 1.4 이상 필요.

```html
<!DOCTYPE html>
<html>
<head>
  <script src="https://cdn.jsdelivr.net/npm/hls.js@latest"></script>
</head>
<body>
  <video id="video" controls></video>
  
  <script>
    if (Hls.isSupported()) {
      const video = document.getElementById('video');
      const hls = new Hls({
        // LL-HLS 활성화
        lowLatencyMode: true,
        
        // 라이브 엣지 가까이
        liveSyncDuration: 3,
        liveMaxLatencyDuration: 5,
        
        // 백업 옵션
        backBufferLength: 60,
        maxBufferLength: 30,
        
        // 디버깅
        debug: false,
      });
      
      hls.loadSource('https://example.com/live/stream/llhls.m3u8');
      hls.attachMedia(video);
      
      // 라이브 엣지 추적
      hls.on(Hls.Events.LEVEL_UPDATED, (event, data) => {
        const latency = hls.latency;
        console.log(`현재 지연: ${latency}초`);
      });
    } else if (video.canPlayType('application/vnd.apple.mpegurl')) {
      // iOS Safari 네이티브
      video.src = 'https://example.com/live/stream/llhls.m3u8';
    }
  </script>
</body>
</html>
```

`lowLatencyMode: true`만 추가하면 hls.js가 알아서 LL-HLS 모드 동작.

`hls.latency` 속성으로 실시간 지연 측정 가능.

### iOS Safari 네이티브 LL-HLS

iOS 14 이상은 네이티브로 LL-HLS 지원.

```html
<video controls>
  <source src="https://example.com/live/stream/llhls.m3u8" type="application/vnd.apple.mpegurl">
</video>
```

별도 JavaScript 없음. `<video>` 태그가 자동으로 LL-HLS 인식.

iOS Safari가 LL-HLS 우선 지원해서 어떤 면에선 hls.js보다 안정적.

### 실제 지연 측정 방법

LL-HLS가 잘 동작하는지 어떻게 확인하나.

**방법 1: 화면 시계 사용**

스트리머 화면에 ms 단위 시계 띄움.
- 시청자 영상 시간 - 시청자 시청 시각 = 지연

```
스트리머 화면 시계: 12:00:00.000
시청자 디바이스 시계: 12:00:03.200
→ 지연 3.2초
```

가장 직관적. NDI 워크플로우에서 표준.

**방법 2: ID3 메타데이터 삽입**

세그먼트마다 인코딩 시각 ID3 태그로 박는다.

```javascript
hls.on(Hls.Events.FRAG_PARSING_METADATA, (event, data) => {
  for (const sample of data.samples) {
    const encodedTime = parseID3(sample.data);
    const currentTime = Date.now();
    const latency = (currentTime - encodedTime) / 1000;
    console.log(`지연: ${latency}초`);
  }
});
```

자동 측정. 대시보드 구축 가능.

**방법 3: WebRTC 채팅과 비교**

같은 페이지에 WebRTC 채팅이 있으면, 채팅 도착 시각과 영상 도착 시각 차이가 지연.

### 실전 트러블슈팅

자주 발생하는 문제들.

**문제 1: 지연이 줄지 않음**

원인: CDN이 LL-HLS 미지원 또는 설정 안 됨.

확인:
```bash
curl -v "https://cdn.example.com/live/playlist.m3u8?_HLS_msn=100&_HLS_part=5"
```

응답이 즉시 오면 long polling 미지원. 새 partial 만들 때까지 응답이 늦게 오면 정상.

**문제 2: 첫 재생 시 까만 화면**

원인: INDEPENDENT partial이 너무 드물어서 IDR 대기.

해결: KeyFrameInterval 줄임 (예: 30프레임 = 1초).

**문제 3: 화질 전환 시 깨짐**

원인: 화질별 partial 경계 불일치.

해결: 인코더에서 모든 화질의 GOP 동기화 강제.

**문제 4: Origin 서버 메모리 폭증**

원인: Long polling connection이 너무 많이 쌓임.

해결:
- HTTP/2 활성화
- Origin Shield 추가
- 타임아웃 짧게 (9초 → 5초)

### 5강 핵심 3줄

1. LL-HLS는 OvenMediaEngine(무료)부터 Wowza/AWS MediaLive(유료)까지 다양한 인코더 — KeyFrameInterval 1초가 핵심 설정
2. nginx Origin은 `proxy_buffering off`, CDN은 Akamai/CloudFront/Cloudflare 같은 LL-HLS 지원 서비스 필요
3. hls.js 1.4+ `lowLatencyMode: true`로 활성화, iOS Safari 14+는 네이티브 지원 — `hls.latency`로 실시간 지연 측정
