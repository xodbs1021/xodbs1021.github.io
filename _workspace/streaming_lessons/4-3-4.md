## 레벨 4-3 / 4강: GPU 트랜스코딩 인프라 설계

---

### 전체 그림 다시 보기

라이브 트랜스코딩 클러스터의 컴포넌트.

```
[스트리머] → RTMP/SRT → [인제스트 클러스터]
                              ↓ (내부 SRT/RTMP)
                         [트랜스코딩 클러스터 (GPU)]
                              ↓
                         [패키저 클러스터]
                              ↓
                         [Origin Storage]
                              ↓
                         [CDN]

[관제]
- Prometheus + Grafana
- 알람: PagerDuty/Slack
- 로그: ELK / Loki
```

각 단계 자세히.

### 인제스트 ↔ 트랜스코딩 분리 이유

같은 노드에서 인제스트 + 트랜스코딩 하면 안 되나?

```
[통합 아키텍처]
스트리머 → [인제스트 + 트랜스코딩 한 노드] → CDN

문제:
- RTMP 받는 노드와 GPU 노드가 같은 곳
- 인제스트는 네트워크 바운드, 트랜스코딩은 GPU 바운드
- 자원 사용 패턴 다름
- 스케일링 불가능 (한쪽만 늘릴 수 없음)
```

분리해야 하는 이유:

```
[분리 아키텍처]
스트리머 → [인제스트 노드 (네트워크 중심)]
              ↓ 내부 SRT
          [트랜스코딩 노드 (GPU 중심)]
              ↓
          [패키저]

장점:
- 인제스트는 네트워크 큰 노드, 트랜스코딩은 GPU 노드
- 독립 스케일링
- 인제스트 노드 장애 시 트랜스코딩 살아있음
```

대형 플랫폼 다 이 구조.

---

### 인제스트 노드 설계

스트리머가 처음 접속하는 곳.

```
[인제스트 노드 역할]
1. RTMP/SRT 연결 수락
2. 스트림 키 인증 (DB 쿼리)
3. 입력 검증 (코덱, 비트레이트, 해상도)
4. 트랜스코딩 노드로 SRT 포워딩
5. 백업 노드로 동시 송신 (active-active)
```

스펙:
```
CPU: 16~32코어 (인증, 검증, 포워딩)
RAM: 32~64GB
NIC: 10Gbps+ (대용량 스트림 받기)
GPU: 없음
```

소프트웨어 스택:
- Nginx RTMP 모듈
- 또는 SRS (Simple Realtime Server)
- 자체 Go 서버 (커스텀)

### Nginx RTMP 예시

```nginx
rtmp {
    server {
        listen 1935;
        chunk_size 4096;
        
        application live {
            live on;
            
            # 인증
            on_publish http://auth-service/verify;
            
            # 트랜스코딩 노드로 푸시
            push rtmp://transcoder-a/live;
            push rtmp://transcoder-b/live;  # 백업
            
            # 로컬 저장 안 함
            record off;
        }
    }
}
```

`on_publish`로 스트림 키 검증. `push`로 트랜스코딩 노드에 포워딩.

### 트랜스코딩 노드 설계

GPU가 핵심.

```
[트랜스코딩 노드 스펙]
CPU: 16~32코어 (FFmpeg 파이프라인, 디코딩 보조)
GPU: NVIDIA T4 (대당 16GB)
RAM: 64GB
NIC: 10Gbps+
스토리지: NVMe (세그먼트 임시 보관)
```

```
[T4 한 대 처리량]
1080p60 H.264: 15~20 채널
720p60:        30~40 채널
ABR ladder 통합 채널: 5~7 채널 (1080p+720p+480p)
```

채널 패킹 계산:
```
방송 1개 = 5단 ABR ladder
= 1080p60 1 + 720p60 1 + 720p30 1 + 480p30 1 + 360p30 1
≈ T4 NVENC 처리량 5/15 = 33% 사용

T4 1대당 방송 3개 동시 (안전 마진 포함)
```

### 채널 스케줄러

새 방송이 들어오면 어느 노드로 보낼지 결정.

```python
class TranscodingScheduler:
    def assign_node(self, stream_id):
        nodes = self.get_available_nodes()
        
        # 각 노드의 현재 부하 확인
        loads = [(n, self.get_load(n)) for n in nodes]
        
        # 가장 부하 낮은 노드 선택
        loads.sort(key=lambda x: x[1])
        
        if loads[0][1] < 0.8:  # 80% 미만
            return loads[0][0]
        else:
            # 모든 노드 부하 높음 → 새 노드 스폰
            return self.spawn_new_node()
    
    def get_load(self, node):
        return prometheus.query(
            f'avg(nvidia_gpu_utilization{{node="{node}"}})'
        )
```

부하 기반 + 자동 스케일링.

### Kubernetes 운영

GPU 트랜스코딩 워크로드의 K8s 배포.

```yaml
# transcoder-statefulset.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: transcoder
spec:
  serviceName: transcoder
  replicas: 5
  template:
    spec:
      nodeSelector:
        gpu-type: nvidia-t4
      containers:
      - name: ffmpeg
        image: registry/nvenc-transcoder:v1.2
        resources:
          limits:
            nvidia.com/gpu: 1
            cpu: 8
            memory: 16Gi
        env:
        - name: INPUT_RTMP
          value: "rtmp://ingest/live/$(STREAM_ID)"
        - name: OUTPUT_BASE
          value: "/output/$(STREAM_ID)"
        volumeMounts:
        - name: segments
          mountPath: /output
      volumes:
      - name: segments
        emptyDir: {}
```

NVIDIA device plugin 설치 후 `nvidia.com/gpu: 1`로 GPU 할당.

### 오토스케일링

방송 시작 시점에 부하 급증. HPA로 대응.

```yaml
# transcoder-hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: transcoder-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: StatefulSet
    name: transcoder
  minReplicas: 5
  maxReplicas: 100
  metrics:
  - type: External
    external:
      metric:
        name: nvidia_gpu_utilization
      target:
        type: AverageValue
        averageValue: "70"
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 30  # 빨리 늘림
    scaleDown:
      stabilizationWindowSeconds: 300  # 천천히 줄임
```

GPU 사용률 70% 기준으로 스케일. 늘릴 땐 빨리, 줄일 땐 천천히 (방송 중에 못 꺼짐).

### 노드 풀 — 인스턴스 다양화

상시 부하 + 피크 부하를 다른 인스턴스로.

```
[상시 풀 (Reserved)]
- AWS T4 인스턴스 예약
- 80% 할인
- 평소 부하 처리

[피크 풀 (Spot)]
- AWS Spot 인스턴스
- 90% 할인
- 갑작스러운 부하 처리
- 중단 가능성 (라이브 잠깐 끊김)
```

Spot 인스턴스 중단 시 graceful drain.

```python
# spot 중단 알림 받으면 새 노드로 마이그레이션
def handle_spot_termination():
    streams = get_active_streams_on_this_node()
    for stream in streams:
        new_node = scheduler.assign_node(stream.id)
        migrate_stream(stream, new_node)
        # 마이그레이션 시 5~10초 끊김 발생
```

Spot 사용으로 비용 50% 추가 절감. 약간의 끊김 감수.

### 패키저 노드

트랜스코딩 결과를 HLS/DASH로.

```
[패키저 노드]
CPU: 8~16코어
GPU: 없음 (CPU 작업)
RAM: 32GB
스토리지: 빠른 NVMe (세그먼트 빠른 IO)
```

```bash

  'in=tcp://transcoder:8080/1080p,stream=video,...' \
  --hls_master_playlist_output /output/master.m3u8 \
  ...
```

패키저는 트랜스코딩보다 가벼움. 인스턴스 적게.

### Origin Storage — 패키저 다음

세그먼트가 저장되는 곳.

```
[옵션]
1. 파일시스템 (NFS, EFS)
   - 단순
   - 성능 한계 (HPS, IOPS)

2. 객체 스토리지 (S3, GCS)
   - 무한 확장
   - 라이브엔 약간 지연

3. 메모리 캐시 + 비동기 영구화 (Redis + S3)
   - 빠름
   - 복잡
```

라이브 패키저 → Origin Storage는 보통 객체 스토리지 + 빠른 캐시.

```
패키저 → Redis 캐시 (최근 1분 세그먼트)
       → S3 비동기 업로드 (영구 저장)
```

CDN은 패키저 또는 캐시에서 가져감.

---

### CDN 연동

CDN과 Origin 사이.

```
[연동 패턴]
1. CDN이 Origin 풀
   - CDN이 시청자 요청 시 Origin에서 가져옴
   - 단순, 대부분 CDN의 기본

2. Origin이 CDN 푸시
   - 세그먼트 생성 즉시 CDN에 미리 푸시
   - 라이브 초기 지연 최소화
   - 인기 콘텐츠에 효과적
```

Pre-positioning이 푸시 방식의 일종.

### 장애 대응 — 시나리오별

**1. 트랜스코딩 노드 1대 죽음**
```
스케줄러 감지 → 영향 받는 스트림 식별 → 다른 노드로 재할당
실제 지연: 10~30초 끊김
```

**2. 트랜스코딩 데이터센터 전체 죽음**
```
인제스트가 다른 region 트랜스코딩으로 SRT 포워딩 전환
실제 지연: 30~60초 끊김
```

**3. GPU 장애 (드라이버 hang)**
```
헬스체크가 출력 .ts 파일 mtime 모니터링
10초간 mtime 갱신 없으면 NVIDIA 드라이버 재시작
실패 시 노드 자체 교체
```

**4. 패키저 죽음**
```
트랜스코딩은 살아있는데 패키저만 죽음
세그먼트 임시 큐에 쌓임
패키저 복구 시 catch-up
시청자는 짧은 버퍼링 후 정상
```

### 모니터링 - 핵심 알람

```
[Critical (즉시 호출)]
- 트랜스코딩 노드 NVENC 처리량 < 1.0x speed (인코딩 뒤처짐)
- 인제스트 노드 연결 실패율 > 1%

[Warning (업무 시간 대응)]
- GPU 사용률 > 90% (확장 검토)
- 트랜스코딩 에러율 > 0.1%
- VMAF 회귀 > 2점

[Info (대시보드)]
- 전체 동시 채널 수
- 비용 트렌드
- 채널당 평균 비용
```

### 트래픽 예측 — Capacity Planning

평일 vs 주말, 시간대별 패턴.

```
[치지직 트래픽 패턴 (가상)]
새벽 3~7시:  50채널 (오프피크)
오전 8~12시: 200채널
점심 12~2시: 400채널
오후 2~6시:  300채널
저녁 6~10시: 1000채널 (피크)
밤 10~3시:   800채널

피크/오프피크 비: 20:1
```

피크 기준 인프라 = 80% 시간 낭비.

해결책: 예약 인스턴스 (상시 부하 만큼) + Spot/Burst (피크 만큼).

```
[Capacity 전략]
Reserved: 200채널 분 (오프피크 처리)
On-demand: 300채널 분 (평균)
Spot: 500채널 분 (피크)
```

비용 30~40% 절감.

### Multi-region 배포

지역별 트랜스코딩 클러스터.

```
[Region 구성]
서울:    국내 트래픽 90%
도쿄:    일본/대만/홍콩
싱가포르: 동남아
미국 서부: 북미 일부
```

스트리머는 가장 가까운 region에 송출. 시청자는 글로벌 CDN으로 분산.

장점:
- 지연 최소
- 한 region 장애 시 fallback
- 지역별 컴플라이언스

단점:
- 운영 복잡도 증가
- 인프라 비용 증가

### 4강 핵심 3줄

1. 인제스트/트랜스코딩/패키저 분리가 정석 — 자원 사용 패턴 다르고 독립 스케일링 필요
2. K8s + NVIDIA device plugin + HPA로 GPU 사용률 70% 기준 자동 스케일링이 표준
3. Reserved + Spot 혼합 + Multi-region + 카테고리별 비트레이트로 비용 30~40% 절감 가능
