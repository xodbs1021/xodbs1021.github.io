## 레벨 3-2 / 2강: AAC 인코더 선택 — 음질 차이가 왜 나나

---

### 같은 AAC라도 인코더마다 음질이 다르다

이게 개발자가 알아야 할 첫 번째 사실.

```
[같은 128 Kbps AAC인데 음질 차이]
Fraunhofer FDK AAC: 최고 (상용 기준)
Apple AAC: 매우 좋음 (iTunes 인코딩)
libfdk_aac (오픈소스 포트): 매우 좋음
FFmpeg 기본 aac: 보통
옛날 libavcodec aac: 거침
```

같은 규격을 따르지만 인코더 안의 "사람 청각 분석 로직"이 다름. 비트 어디 쓸지 결정하는 정교함의 차이.

핵심: **저비트레이트일수록 인코더 차이가 크게 들린다.**

---

### FFmpeg에서 인코더 선택

실무에서 가장 자주 만나는 결정.

```bash
# 기본 (FFmpeg 내장 AAC)
ffmpeg -i input.wav -c:a aac -b:a 128k output.aac

# Fraunhofer (libfdk_aac, 더 좋은 음질)
ffmpeg -i input.wav -c:a libfdk_aac -b:a 128k output.aac
```

`libfdk_aac` 쓰려면:
- 기본 FFmpeg에 안 들어있음 (라이센스 이슈)
- 직접 빌드하거나 `--enable-libfdk-aac` 포함된 빌드 받아야 함
- Homebrew: `brew install ffmpeg --with-fdk-aac`

언제 `libfdk_aac` 쓰나:
- 96 Kbps 이하 (저비트레이트): 무조건
- 128 Kbps: 가능하면
- 192 Kbps 이상: 어느 거나 거의 비슷

---

### 비트레이트 결정 가이드

실무에서 결정할 때 기준.

```
[용도별 권장 비트레이트]

음악 스트리밍:
- 일반: 128 Kbps (Spotify 기본)
- 고음질: 256 Kbps (Apple Music)
- HiFi: FLAC 무손실

라이브 방송 오디오:
- 게임 방송: 128 Kbps 스테레오
- 토크쇼: 96 Kbps 스테레오 또는 64 Kbps 모노
- 라디오: 64 Kbps 모노

영상의 오디오 트랙:
- HD 영상: 128 Kbps
- 4K 영상: 192~256 Kbps
- 라이브: 128 Kbps

저대역폭 모바일:
- HE-AAC 64 Kbps (음질 양호)
- HE-AACv2 32 Kbps (음성 + 약간 음악)
```

치지직 같은 게임 방송 = 128 Kbps가 사실상 표준.

---

### AAC 프로파일 — 언제 뭘 쓰나

세 가지만 알면 충분.

**AAC-LC (Low Complexity)**:
- 가장 일반적
- 64~320 Kbps에 적합
- 모든 디바이스 호환
- **거의 모든 경우 이거 쓰면 됨**

```bash
ffmpeg -i input.wav -c:a aac -profile:a aac_low -b:a 128k output.aac
```

**HE-AAC**:
- 32~64 Kbps에 강함
- SBR (Spectral Band Replication)로 저비트레이트에 고주파 살림
- 모바일 라디오, 저대역폭

```bash
ffmpeg -i input.wav -c:a libfdk_aac -profile:a aac_he -b:a 48k output.aac
```

**HE-AACv2**:
- 16~32 Kbps 초저비트레이트
- Parametric Stereo 추가
- 거의 안 쓰임 (Opus가 대체)

라이브 스트리밍은 90% AAC-LC.

---

### 샘플링 레이트 선택

```
[샘플링 레이트 선택]
48 kHz: 영상 표준 (라이브 스트리밍, 방송)
44.1 kHz: 음악 표준 (CD, Spotify)
32 kHz: 저대역폭 (FM 라디오 품질)
22.05 kHz, 16 kHz: 음성 전용
```

영상과 같이 쓸 땐 **48 kHz**. 동기화 편함.

음악만 다룰 땐 44.1 kHz도 OK.

FFmpeg:
```bash
ffmpeg -i input.wav -c:a aac -b:a 128k -ar 48000 output.aac
```

---

### 채널 수 결정

```
[채널 수]
모노 (1ch): 음성 위주, 라디오, 팟캐스트
스테레오 (2ch): 99% 경우의 기본값
5.1 (6ch): 영화, 프리미엄 OTT
7.1 (8ch): 영화관, 최고급
```

라이브 게임 방송은 스테레오. 5.1은 콘서트나 영화관 라이브 정도.

```bash
# 5.1 → 스테레오 다운믹스
ffmpeg -i input.mkv -c:a aac -ac 2 -b:a 192k output.mp4
```

`-ac 2`: 채널 수 2개로.

---

### CBR vs VBR — 라이브에선 CBR

음악 다운로드 같은 VOD는 VBR이 좋지만, 라이브는 CBR.

```bash
# CBR (라이브)
ffmpeg -i input.wav -c:a aac -b:a 128k -minrate:a 128k -maxrate:a 128k output.aac

# VBR (음악 다운로드)
ffmpeg -i input.wav -c:a libfdk_aac -vbr 4 output.aac
```

CBR 이유:
- 대역폭 예측 가능
- 스트리밍 안정성
- 동기화 쉬움

VBR 이유:
- 같은 평균 비트레이트에 음질 좋음
- 파일 단위에서 의미 있음

---

### 오디오 비트레이트가 영상 비트레이트 대비 무시할 수준

실무에서 자주 잊는 것.

```
[1080p 라이브 방송 비트레이트 비교]
영상: 6,000 Kbps (6 Mbps)
오디오 128 Kbps

오디오 비중: 약 2%
```

오디오 비트레이트 줄여봐야 전체 대역폭 거의 안 줄어.

→ 굳이 64 Kbps로 줄이지 말고 128 Kbps 유지하는 게 음질에 이득.

---

### 라이브 송출 표준 명령

치지직 같은 데 RTMP 송출.

```bash
ffmpeg -i input.mp4 \
  -c:v libx264 -preset veryfast -tune zerolatency -b:v 6000k \
  -c:a aac -b:a 128k -ar 48000 -ac 2 \
  -f flv rtmp://server/live/key
```

오디오 부분:
- `-c:a aac`: AAC 코덱
- `-b:a 128k`: 128 Kbps
- `-ar 48000`: 48 kHz 샘플링
- `-ac 2`: 스테레오

이게 표준. 안 바꿔도 됨.

---

### 멀티 오디오 트랙 처리 — PokeClip 예시

여러 오디오 트랙을 따로 인코딩.

```bash
ffmpeg -i game.wav -i mic.wav -i discord.wav -i donation.wav \
  -map 0:a -c:a:0 aac -b:a:0 128k \
  -map 1:a -c:a:1 aac -b:a:1 128k \
  -map 2:a -c:a:2 aac -b:a:2 64k \
  -map 3:a -c:a:3 aac -b:a:3 64k \
  -f mpegts output.ts
```

각 트랙별 비트레이트 다르게:
- 게임 사운드, 마이크: 128 Kbps (음악/음성 모두)
- 디스코드, 후원: 64 Kbps (음성 위주)

이 MPEG-TS를 SRT로 보내면 분석 서버가 트랙별 분리해서 분석.

---

### 오디오 동기화 — PTS 맞추기

영상과 오디오 시작이 안 맞을 때.

```bash
# 오디오를 0.1초 늦추기
ffmpeg -i input.mp4 -itsoffset 0.1 -i input.mp4 \
  -map 0:v -map 1:a \
  -c copy output.mp4

# 오디오를 0.1초 앞당기기
ffmpeg -i input.mp4 -itsoffset -0.1 -i input.mp4 \
  -map 0:v -map 1:a \
  -c copy output.mp4
```

OBS에서 마이크 입력 지연 보정도 같은 원리.

---

### 볼륨 노멀라이제이션

라이브 방송에서 자주 쓰는 처리.

```bash
# 표준 -16 LUFS (방송 표준)
ffmpeg -i input.mp4 \
  -af "loudnorm=I=-16:LRA=11:TP=-1.5" \
  -c:v copy \
  output.mp4

# 간단히 1.5배 키우기
ffmpeg -i input.mp4 -af "volume=1.5" output.mp4

# 너무 큰 소리 자동 제한
ffmpeg -i input.mp4 -af "alimiter=limit=0.9" output.mp4
```

스트리머가 갑자기 크게 외쳐도 시청자 귀 안 터지게.

### AAC의 한계 — 언제 다른 코덱 쓰나

AAC가 표준이지만 안 맞는 경우.

**Opus가 나은 경우**:
- WebRTC (의무 코덱)
- 저지연 통신 (40ms 이하)
- 64 Kbps 이하 저비트레이트
- 음성 + 음악 혼합 (디스코드 같은 곳)

**AC-3가 나은 경우**:
- 5.1 채널 영화 (호환성)
- 케이블 방송

**FLAC/ALAC이 나은 경우**:
- 음질 보관용
- 무손실 필요

**MP3가 나은 경우**:
- 옛날 기기 호환 필요
- 라이센스 자유 (이젠 AAC도 비슷)

---

### 실전 체크리스트

오디오 인코딩할 때 확인할 것.

```
□ 코덱: AAC-LC (95% 경우)
□ 비트레이트: 128 Kbps (스테레오 기본)
□ 샘플링: 48 kHz (영상과 같이 쓸 때)
□ 채널: 2 (스테레오)
□ 인코더: libfdk_aac (저비트레이트면)
□ CBR (라이브) / VBR (VOD)
□ 영상과 동기화 확인
□ 볼륨 노멀라이제이션 (필요 시)
```

이거 다 맞추면 음질 문제 거의 없음.

### 2강 핵심 3줄

1. 같은 AAC라도 인코더가 음질을 80% 결정 — 저비트레이트(96 Kbps 이하)일수록 차이 큼, `libfdk_aac`가 FFmpeg 기본 `aac`보다 우수
2. 라이브 스트리밍 표준은 AAC-LC + 128 Kbps + 48 kHz + 스테레오 + CBR — 영상 대비 오디오 비트레이트는 2% 수준, 굳이 줄일 필요 없음
3. WebRTC는 Opus, 5.1 영화는 AC-3, 무손실은 FLAC, 그 외 거의 모두 AAC — 실무는 표준 명령 그대로 쓰면 음질 문제 거의 없음
