## 레벨 4-1 / 5강: HLS / DASH 패키징 — 인코딩 결과를 배포 가능한 형태로

---

### 패키징이란 인코딩과 별개의 단계다

많은 사람이 헷갈리는 부분.

```
[전체 파이프라인]
인코딩 (Encoding):  raw → H.264 비트스트림
패키징 (Packaging): H.264 → HLS .ts 세그먼트 + .m3u8 / DASH 세그먼트 + MPD
```

x264가 만든 H.264 비트스트림을 그대로 시청자에게 줄 수는 없다. HLS 규격에 맞게 잘라서 컨테이너에 담아야 한다.

FFmpeg는 이 두 단계를 한 번에 해주지만, 대규모 인프라에서는 보통 분리한다.

```
[인코딩 서버]: FFmpeg로 H.264 인코딩 → MP4 출력
[패키저]:      Shaka Packager, Bento4가 HLS/DASH로 패키징
```

분리 이유: 인코딩은 무거운 GPU 작업, 패키징은 가벼운 CPU 작업. 인프라 분리하면 스케일링 효율적.

근데 FFmpeg로 통합하는 곳도 많아서 일단 FFmpeg HLS 패키징부터.

---

### FFmpeg HLS 옵션 총정리

```bash
ffmpeg -i input.mp4 \
  -c:v libx264 -preset veryfast \
  -c:a aac -b:a 128k \
  \
  -f hls \
  -hls_time 6 \
  -hls_list_size 5 \
  -hls_segment_type mpegts \
  -hls_segment_filename 'seg_%05d.ts' \
  -hls_flags delete_segments+independent_segments+program_date_time \
  -hls_playlist_type event \
  -master_pl_name master.m3u8 \
  -hls_allow_cache 0 \
  \
  output.m3u8
```

옵션별 정리.

---

### -hls_time — 세그먼트 길이

```bash
-hls_time 6    # 6초 세그먼트
```

실제로는 정확히 6초가 안 된다. 키프레임 위치에서 자르니까. GOP 정렬이 6초면 정확히 6초.

```
GOP 6초 (-g 360 for 60fps), -hls_time 6 → 정확히 6초 세그먼트
GOP 2초, -hls_time 6 → 6초 정도 (다음 키프레임까지 기다림)
```

이래서 인코딩 단계의 `-g`와 `-hls_time`을 같이 설정해야 한다.

라이브 표준은 6초. 일부 LL-HLS는 2초 + Partial Segment.

---

### -hls_list_size — 플레이리스트 길이

```bash
-hls_list_size 5    # m3u8에 5개 세그먼트 유지 (sliding window)
-hls_list_size 0    # 무제한 (VOD)
```

라이브: 3~6개 (sliding window).  
VOD: 0 (전부 유지).

플레이리스트 길이 = 시청자가 뒤로 감을 수 있는 범위.

```
6초 * 5개 = 30초 backward seek 가능
```

치지직 라이브는 보통 3분 정도 (30개 세그먼트) 유지. "되감기" 기능을 위해.

---

### -hls_segment_type — 컨테이너 선택

```
mpegts: .ts 파일 (HLS 전통)
fmp4:   fragmented MP4 (CMAF, LL-HLS 표준)
```

```bash
# 전통적 HLS
-hls_segment_type mpegts

# CMAF (DASH와 호환, LL-HLS)
-hls_segment_type fmp4
```

CMAF (fMP4)의 장점:
- HLS와 DASH가 같은 세그먼트 파일 공유 (스토리지 절반)
- LL-HLS Partial Segment 지원
- 오버헤드 작음 (TS는 188바이트 패킷 헤더가 많음)

새로 만드는 라이브 인프라는 거의 CMAF.

---

### -hls_flags — 동작 옵션 모음

```bash
-hls_flags delete_segments+independent_segments+program_date_time
```

주요 플래그:

- `delete_segments`: 플레이리스트에서 빠진 .ts 자동 삭제 (디스크 용량 관리)
- `independent_segments`: 각 세그먼트가 키프레임으로 시작 (시킹/ABR 보장)
- `program_date_time`: `#EXT-X-PROGRAM-DATE-TIME` 태그 삽입 (시청자 측 시간 동기화)
- `append_list`: 기존 플레이리스트에 이어 쓰기 (재시작 시)
- `discont_start`: 첫 세그먼트에 discontinuity 마커
- `omit_endlist`: VOD 모드여도 ENDLIST 안 박음

라이브 표준: `delete_segments+independent_segments+program_date_time`

---

### -hls_playlist_type — 플레이리스트 타입

```bash
-hls_playlist_type event  # 이벤트 (라이브 → VOD 변환 가능)
-hls_playlist_type vod    # VOD (#EXT-X-ENDLIST 추가)
# 미지정: 라이브 (sliding window)
```

`event` 타입:
- 세그먼트가 계속 추가됨 (라이브처럼)
- 오래된 세그먼트가 안 빠짐 (VOD처럼)
- 방송 종료 시 ENDLIST 추가하면 그대로 VOD로 보존

다시보기 자동 생성하는 라이브 플랫폼이 자주 쓴다. 라이브 보다가 끊겨도 뒤로 감으면 처음부터 다 보임.

### Master Playlist — ABR Ladder

여러 화질을 묶는 마스터 플레이리스트.

```bash
ffmpeg -i input.mp4 \
  -filter_complex "[0:v]split=3[v1][v2][v3];[v2]scale=1280:720[v720];[v3]scale=854:480[v480]" \
  \
  -map "[v1]"   -map 0:a -c:v libx264 -b:v 6000k -c:a aac -b:a 128k \
  -map "[v720]" -map 0:a -c:v libx264 -b:v 3000k -c:a aac -b:a 128k \
  -map "[v480]" -map 0:a -c:v libx264 -b:v 1500k -c:a aac -b:a 96k \
  \
  -f hls \
  -hls_time 6 \
  -hls_segment_filename 'stream_%v/seg_%05d.ts' \
  -master_pl_name master.m3u8 \
  -var_stream_map "v:0,a:0,name:1080p v:1,a:1,name:720p v:2,a:2,name:480p" \
  \
  'stream_%v/playlist.m3u8'
```

`-var_stream_map`이 핵심. 각 출력을 어떤 화질로 묶을지 지정.

결과 파일 구조:
```
master.m3u8              ← 시청자가 받는 마스터
stream_1080p/
  playlist.m3u8
  seg_00000.ts, seg_00001.ts, ...
stream_720p/
  playlist.m3u8
  seg_00000.ts, ...
stream_480p/
  ...
```

master.m3u8 내용:
```
#EXTM3U
#EXT-X-VERSION:3
#EXT-X-STREAM-INF:BANDWIDTH=6500000,RESOLUTION=1920x1080
stream_1080p/playlist.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=3200000,RESOLUTION=1280x720
stream_720p/playlist.m3u8
#EXT-X-STREAM-INF:BANDWIDTH=1700000,RESOLUTION=854x480
stream_480p/playlist.m3u8
```

플레이어가 master 받아서 현재 대역폭에 맞는 sub playlist 선택.

### DASH 패키징

DASH도 비슷한 옵션.

```bash
ffmpeg -i input.mp4 \
  -c:v libx264 -b:v 5000k \
  -c:a aac -b:a 128k \
  \
  -f dash \
  -seg_duration 6 \
  -use_template 1 \
  -use_timeline 1 \
  -window_size 5 \
  -extra_window_size 10 \
  -remove_at_exit 1 \
  \
  output.mpd
```

차이점:
- `.mpd` 파일이 매니페스트 (HLS의 `.m3u8`)
- 세그먼트는 `.m4s` (fMP4) 또는 `.ts`
- `use_template` + `use_timeline`이 거의 항상 1

DASH는 XML 기반이라 사람이 읽기 어렵지만 표현력이 풍부.

### CMAF 통합 패키징 — HLS와 DASH 동시 생성

CMAF의 진짜 가치.

```bash
ffmpeg -i input.mp4 \
  -c:v libx264 -b:v 5000k \
  -c:a aac -b:a 128k \
  \
  # HLS (fMP4)
  -f hls \
  -hls_segment_type fmp4 \
  -hls_segment_filename 'cmaf/seg_%05d.m4s' \
  -hls_fmp4_init_filename 'cmaf/init.mp4' \
  cmaf/playlist.m3u8 \
  \
  # DASH (같은 .m4s 파일 사용)
  -f dash \
  -seg_duration 6 \
  -init_seg_name 'cmaf/init.mp4' \
  -media_seg_name 'cmaf/seg_$Number%05d$.m4s' \
  cmaf/manifest.mpd
```

같은 `.m4s` 파일을 HLS와 DASH가 공유. 스토리지/CDN 비용 절반.

이게 CMAF (Common Media Application Format)의 핵심 아이디어.

### LL-HLS Partial Segment

저지연 HLS의 핵심.

```bash
ffmpeg -i rtmp://input \
  -c:v libx264 -preset veryfast -tune zerolatency \
  -c:a aac -b:a 128k \
  \
  -f hls \
  -hls_time 4 \
  -hls_segment_type fmp4 \
  -hls_flags independent_segments \
  -hls_playlist_type event \
  \
  # LL-HLS 옵션
  -hls_init_time 0.2 \
  -hls_segment_options "movflags=+frag_keyframe+empty_moov+default_base_moof" \
  \
  ll_hls.m3u8
```

근데 FFmpeg의 LL-HLS 지원은 아직 완전하지 않다. 진짜 LL-HLS는 Shaka Packager나 Apple 자체 도구 쓰는 게 정석.

```bash
# Shaka Packager 예시
packager \
  'in=video.mp4,stream=video,init_segment=init.mp4,segment_template=seg_$Number$.m4s' \
  --hls_master_playlist_output master.m3u8 \
  --segment_duration 4 \
  --hls_playlist_type LIVE \
  --low_latency_dash_mode
```

### 세그먼트 파일명 패턴

```
seg_%05d.ts        # 5자리 숫자 (seg_00000.ts)
seg_%v_%05d.ts     # 스트림 + 숫자
seg_$Number$.m4s   # DASH 템플릿
seg_$Time$.m4s     # 시간 기반
```

```
%05d: 00000, 00001, 00002, ...
%v:   스트림 이름 (var_stream_map의 name)
```

DASH는 `$...$` 문법. HLS는 `%...` 문법. 헷갈리는 부분.

### 자막 처리 — WebVTT

HLS에 자막 붙이기.

```bash
ffmpeg -i input.mp4 -i subs.srt \
  -map 0:v -map 0:a -map 1:0 \
  -c:v libx264 -c:a aac -c:s webvtt \
  \
  -f hls \
  -hls_time 6 \
  -hls_segment_type mpegts \
  \
  master.m3u8
```

자막은 별도 `.m3u8`로 분리되고 마스터에서 참조.

```
#EXT-X-MEDIA:TYPE=SUBTITLES,GROUP-ID="subs",NAME="Korean",LANGUAGE="ko",DEFAULT=YES,URI="subs/playlist.m3u8"
```

### 멀티 오디오 트랙 HLS

다국어 더빙.

```bash
ffmpeg -i input.mkv \
  -map 0:v -map 0:a:0 -map 0:a:1 -map 0:a:2 \
  -c:v libx264 -c:a aac \
  \
  -f hls \
  -hls_time 6 \
  -var_stream_map "v:0,name:video a:0,agroup:audio,name:ko,language:ko a:1,agroup:audio,name:en,language:en a:2,agroup:audio,name:ja,language:ja" \
  -master_pl_name master.m3u8 \
  \
  '%v/playlist.m3u8'
```

`agroup`이 핵심. 같은 그룹 안에서 사용자가 오디오 선택.

마스터 결과:
```
#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="audio",NAME="ko",LANGUAGE="ko",DEFAULT=YES,URI="ko/playlist.m3u8"
#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="audio",NAME="en",LANGUAGE="en",URI="en/playlist.m3u8"
#EXT-X-MEDIA:TYPE=AUDIO,GROUP-ID="audio",NAME="ja",LANGUAGE="ja",URI="ja/playlist.m3u8"
#EXT-X-STREAM-INF:BANDWIDTH=5000000,AUDIO="audio",RESOLUTION=1920x1080
video/playlist.m3u8
```

Netflix 같은 OTT의 다국어 처리가 이 구조.

### 라이브 → VOD 자동 변환

방송 종료 시 자동으로 VOD 매니페스트 생성.

```bash
# 1. event 타입으로 라이브 시작
ffmpeg -re -i rtmp://input \
  -c:v libx264 ... \
  -hls_playlist_type event \
  -hls_list_size 0 \           # 무제한 (다 보존)
  -hls_flags omit_endlist \    # 라이브 동안은 ENDLIST 없음
  /var/www/hls/live.m3u8

# 2. 방송 종료 후 ENDLIST만 추가
echo "#EXT-X-ENDLIST" >> /var/www/hls/live.m3u8
```

이러면 같은 매니페스트가 라이브에서 VOD로 자연스럽게 전환. 시청자는 다시보기로 그대로 본다.

### 패키저 분리의 가치

대규모 인프라에서 FFmpeg 패키징을 안 쓰는 이유.

```
[FFmpeg 통합]
인코딩 + 패키징 한 프로세스
장점: 단순
단점: 인코딩 끝나야 패키징 시작, 스케일링 어려움

[Shaka Packager 분리]
인코딩: FFmpeg → MP4 출력
패키징: Shaka가 .m4s + .m3u8 + .mpd 생성
장점: 인코딩과 독립 스케일링, DRM 통합 쉬움
단점: 파이프라인 복잡
```

```bash
# 분리 패턴
ffmpeg -i rtmp://input ... -f mp4 - | \
  packager 'in=-,stream=video' \
    --hls_master_playlist_output master.m3u8 \
    --segment_duration 6
```

치지직/Twitch 같은 대형 플랫폼은 분리. 작은 라이브 서비스는 FFmpeg 통합.

```
라이브 → VOD 자동화는 event 타입으로 두고

```

### 5강 핵심 3줄

1. HLS 패키징 표준: `-hls_time 6 -hls_segment_type fmp4 -hls_flags delete_segments+independent_segments+program_date_time`
2. ABR Ladder는 `-var_stream_map` + `-master_pl_name`으로 한 명령에 통합, 멀티 오디오는 `agroup` 사용
3. 대규모는 FFmpeg(인코딩) + Shaka Packager(패키징) 분리 — 독립 스케일링, DRM, LL-HLS 완전 지원
