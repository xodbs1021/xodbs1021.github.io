## 레벨 1-4 / 3강: CDN의 한계와 차세대 배포 — P2P, Edge Compute, 자체 CDN

---

### 기존 CDN의 한계 — 비용과 지연

2강 마지막에 본 라이브 트래픽 비용. 인기 스트리머 한 명에 8천만 원. 이게 플랫폼이 매일 안고 가는 비용이다.

CDN 회사들이 가격을 안 내리는 이유:

```
CDN 회사 입장:
- 전세계 POP 운영비
- 트래픽 피크 대비 여유 용량
- 24/7 운영팀
- 마진
```

대형 플랫폼들은 이 비용을 줄이는 방향으로 움직인다. 세 가지 전략이 있다.

---

### 전략 1: 자체 CDN 구축

규모가 커지면 외부 CDN보다 직접 운영이 싸다.

```
[연간 트래픽 1 Exabyte 기준]
외부 CDN (CloudFront): 약 $50M/년
자체 CDN 운영:        약 $20M/년 (서버/네트워크/팀)
                                
→ 손익분기점: 약 100 PB/월
```

넷플릭스의 **Open Connect**, 페이스북의 자체 CDN, 카카오의 자체 CDN이 다 같은 맥락이다.

자체 CDN의 어려움:
- POP 부지 확보 (ISP와 협상)
- 24/7 운영팀 필수
- BGP 라우팅 직접 관리
- 장애 대응 인력

치지직 같은 곳은 네이버 자체 CDN 인프라를 활용한다고 추정된다. 외부 CDN을 보조로 쓰면서.

---

### 넷플릭스 Open Connect — 자체 CDN의 모범 사례

넷플릭스는 자체 CDN을 **ISP 안에 직접 설치**한다.

```
[일반 CDN]
시청자 → ISP → 인터넷 백본 → CDN POP → Origin

[Open Connect]
시청자 → ISP 내부 OCA → 끝
```

ISP 안에 Open Connect Appliance(OCA) 서버를 두고, 인기 영상을 미리 채워둔다. ISP 입장에선 자기 네트워크 트래픽이 줄어드니 환영. 넷플릭스 입장에선 인터넷 백본 비용이 0.

이런 모델은 VOD에 최적화돼 있다. 라이브는 미리 콘텐츠를 채울 수 없어서 적용이 어렵다.

---

### 전략 2: P2P 보조 배포

**P2P CDN**: 시청자끼리 세그먼트를 공유.

```
[일반 CDN]
시청자 A: CDN에서 seg100.ts 다운로드
시청자 B: CDN에서 seg100.ts 다운로드 (같은 파일 또 받음)
시청자 C: CDN에서 seg100.ts 다운로드

[P2P CDN]
시청자 A: CDN에서 seg100.ts 다운로드
시청자 B: A에게 seg100.ts 받음 (CDN 안 거침)
시청자 C: B에게 seg100.ts 받음
```

A가 한 번만 CDN에 부담을 주고, B/C는 P2P로 받는다. CDN 비용 30~70% 절감 가능.

WebRTC DataChannel을 활용한 구조다. 시청자 브라우저끼리 직접 데이터 주고받는다.

**현실적 문제:**
- 모바일은 P2P 부담이 큼 (배터리, 데이터)
- NAT/방화벽 통과 어려움 (STUN/TURN 필요)
- 콘텐츠 정합성 검증 필요 (악의적 P2P 차단)
- 라이브는 지연이 추가됨

상용 P2P CDN: Peer5 (Microsoft 인수), CDNBye, Streamroot. 트위치가 일부 도입했다는 보고가 있다.

### 전략 3: Edge Compute — CDN에서 직접 처리

CDN 엣지에서 정적 캐시만 제공하지 않고, **코드를 실행**한다.

```
[일반 CDN]
시청자: GET /api/stream-info
  → CDN 미스
  → Origin 호출 (지연 100ms)
  → 응답

[Edge Compute]
시청자: GET /api/stream-info
  → CDN 엣지에서 직접 계산
  → 즉시 응답 (지연 5ms)
```

대표 서비스:
- Cloudflare Workers
- AWS Lambda@Edge
- Fastly Compute@Edge

라이브 스트리밍 활용 예시:

**1. Token 검증**
서명된 URL을 엣지에서 검증. Origin 안 거침.

**2. ABR 동적 조정**
시청자 IP의 ISP를 보고 마스터 플레이리스트를 동적으로 생성. KT 사용자에겐 KT 캐시가 잘 된 화질만 노출.

**3. 광고 삽입**
세그먼트 사이에 광고를 엣지에서 끼워 넣음 (Server-Side Ad Insertion, SSAI). 시청자별로 다른 광고를 보여줘도 캐시 가능.

**4. 실시간 트랜스코딩**
드물지만, 일부 회사는 엣지에서 즉석 트랜스코딩까지 한다. AWS MediaLive Edge.

### CDN의 미래 — Multi-CDN + Edge + P2P 하이브리드

대형 플랫폼은 한 가지에 의존하지 않는다. 시청자 환경에 따라 동적으로 결정.

```javascript
// 시청자 컨텍스트
const context = {
  network: '5G',
  device: 'Premium Phone',
  region: 'KR-Seoul',
  contentType: 'live-popular'
};

// 라우팅 결정
if (context.region === 'KR-Seoul' && context.contentType === 'live-popular') {
  // 자체 CDN + P2P 활용
  primaryCDN = 'naver-cdn';
  enableP2P = true;
} else if (context.region === 'US') {
  // CloudFront 사용
  primaryCDN = 'cloudfront';
}
```

이런 정책을 Edge Compute에서 실행하면 추가 지연 없이 최적 경로 선택 가능.

### CDN 의존 위험 — 단일 장애점

2021년 Fastly 장애로 트위치, 레딧, NYT가 한 시간 동안 다운됐다. CDN 장애 = 서비스 장애.

```
[2021 Fastly 장애 사례]
원인: 한 고객의 잘못된 설정이 전체 시스템에 전파
영향: 전세계 Fastly 사용 서비스 모두 다운 (1시간)
```

대비책:

**1. 멀티 CDN 자동 페일오버**
주 CDN 장애 감지 시 보조 CDN으로 트래픽 전환.

**2. DNS 기반 빠른 전환**
TTL을 짧게 (1분) 잡아서 빠르게 다른 CDN으로 라우팅.

**3. Origin 직접 노출 (최후의 수단)**
극단적 상황에선 Origin이 직접 트래픽 받음 (성능 떨어지지만 동작은 함).

라이브 스트리밍 플랫폼은 이 페일오버를 자동화해야 한다. 인기 스트리머 방송 중 다운되면 시청자 수십만이 즉시 이탈한다.

### 한국 라이브 시장의 특수성

치지직, SOOP 같은 한국 라이브 플랫폼이 글로벌 플랫폼과 다른 점:

**1. ISP가 적음 (KT, SKT, LGU+)**
3개 ISP에 시청자 거의 다. ISP 내 캐시 협력이 가능하다.

**2. 국토가 작음**
서울 한 곳에 Origin 둬도 전국 지연이 50ms 이내. CDN 의존도가 낮을 수 있다.

**3. 모바일 비중 높음 (70%+)**
5G 보급률 세계 1위. 모바일 CDN 최적화가 핵심.

**4. 인터넷 종량제 없음**
시청자가 데이터 걱정 없이 고화질 시청. 1080p 보급률 매우 높음.

이런 특성에 맞춰 한국 라이브 플랫폼은 자체 CDN + ISP 협력 모델을 강하게 추구한다.

### 3강 핵심 3줄

1. 자체 CDN(넷플릭스 Open Connect)은 트래픽 임계점 넘으면 외부 CDN보다 싸지만 운영 부담 큼
2. P2P CDN으로 비용 30~70% 절감 가능 — WebRTC 기반, 트위치 일부 도입
3. Edge Compute는 토큰 검증, ABR 조정, 광고 삽입을 엣지에서 처리해 Origin 지연 제거
