## 레벨 1-3 / 2강: 라이브 vs VOD — 버퍼링이 일어나는 양상이 다르다

---

### VOD와 라이브의 근본 차이

**VOD (Video On Demand)**: 미리 만들어진 영상 (유튜브 영상, 넷플릭스 콘텐츠)
**라이브 (Live)**: 실시간으로 만들어지고 있는 영상 (치지직 방송, 스포츠 중계)

같은 HLS를 써도 동작 방식이 완전히 다르다.

---

### VOD — 끝이 정해진 영상

VOD는 모든 세그먼트가 이미 서버에 존재한다.

```
playlist.m3u8 (VOD)
#EXT-X-VERSION:3
#EXT-X-TARGETDURATION:6
#EXT-X-PLAYLIST-TYPE:VOD
#EXTINF:6.000, seg001.ts
#EXTINF:6.000, seg002.ts
#EXTINF:6.000, seg003.ts
...
#EXTINF:6.000, seg600.ts  (1시간 영상)
#EXT-X-ENDLIST              ← 끝났다는 표시
```

`#EXT-X-ENDLIST`가 핵심이다. "더 이상 추가 안 됨" 신호.

플레이어는 이걸 보면:
- 처음부터 끝까지 자유롭게 시킹(seek) 가능
- 버퍼를 마음껏 많이 잡아도 됨
- 다운로드를 미리 끝내도 됨

---

### 라이브 — 끝나지 않는 플레이리스트

라이브 플레이리스트는 계속 업데이트된다.

```
[12:00:00 시점]
playlist.m3u8
#EXT-X-TARGETDURATION:6
#EXT-X-MEDIA-SEQUENCE:100
#EXTINF:6.000, seg100.ts
#EXTINF:6.000, seg101.ts
#EXTINF:6.000, seg102.ts
                            ← ENDLIST 없음. 계속 추가됨

[12:00:06 시점]
#EXT-X-MEDIA-SEQUENCE:101
#EXTINF:6.000, seg101.ts
#EXTINF:6.000, seg102.ts
#EXTINF:6.000, seg103.ts    ← 새로 추가됨
                            ← seg100은 빠짐 (창이 이동)
```

**Sliding Window**: 라이브 플레이리스트는 보통 3~5개 세그먼트만 유지한다. 오래된 세그먼트는 목록에서 빠진다.

플레이어는 주기적으로 플레이리스트를 새로 받아서 새 세그먼트가 있나 확인한다. 이걸 **playlist polling**이라고 한다.

```
플레이어가 6초마다 playlist.m3u8 재요청
  → 새 세그먼트 있으면 다운로드
  → 없으면 다시 6초 후 재요청
```

---

### 라이브의 "라이브 엣지"

라이브 방송에서 시청자가 가장 최신 시점을 보는 위치를 **라이브 엣지(Live Edge)** 라고 한다.

```
[방송 진행 중]
시간:    ─────────────────────────●(실제 현재 시간)
세그먼트:                       [seg103 인코딩 중]
                          [seg102 업로드 완료]
                    [seg101 업로드 완료]
              [seg100 업로드 완료]
                  ↑
            라이브 엣지 (시청 가능한 최신)
```

세그먼트가 6초 단위로 생성되니, **시청자는 최소 6초 전 영상을 본다**. 여기에 플레이어 버퍼 12~18초가 더해진다. 합치면 18~25초 지연이 일반적이다.

이래서 채팅이 영상보다 빠르다. "지금 죽었어!" 채팅이 떴는데 영상은 18초 뒤에 죽는 장면이 나온다.

---

### 라이브에서 버퍼링이 발생하면 어떻게 되나

VOD에서 버퍼링이 일어나면 멈추고 채워질 때까지 기다리면 된다. 영상 끝은 정해져 있으니 시청자는 결국 끝까지 본다.

라이브는 다르다. 멈추는 동안에도 방송은 계속된다.

```
[12:00:00] 시청자가 라이브 엣지(seg100) 보고 있음
[12:00:05] 네트워크 끊김. 버퍼링 시작
[12:00:30] 네트워크 복구. 버퍼 채워야 함
            → 그 사이 방송은 30초 더 진행됨
            → seg105까지 생성됨
            → 시청자는 어디부터 봐야 하나?
```

선택지가 두 가지다.

**1. 끊긴 시점부터 이어보기**
- 라이브 엣지에서 30초 뒤로 처짐
- 채팅과 영상 더 어긋남

**2. 라이브 엣지로 점프**
- 끊긴 30초는 못 봄
- 다시 실시간 따라잡음

플레이어 정책에 따라 다르다. 치지직 같은 라이브 플랫폼은 보통 **자동으로 라이브 엣지로 점프**시킨다. 채팅과 영상 동기화가 더 중요하기 때문이다.

---

### VOD는 버퍼를 크게, 라이브는 작게

```
VOD 플레이어:
  버퍼 30~60초 미리 받음
  → 안정성 최우선
  → 한 번 받으면 끝까지 안 끊김

라이브 플레이어:
  버퍼 6~18초
  → 지연 최소화
  → 네트워크 흔들리면 버퍼링 위험
```

같은 hls.js 라이브러리를 써도 VOD 모드와 라이브 모드의 설정이 다르다.

```javascript
// VOD 모드
hls.config = {
  maxBufferLength: 60,  // 60초까지 버퍼
};

// 라이브 모드
hls.config = {
  liveSyncDuration: 18,  // 라이브 엣지에서 18초 뒤
  liveMaxLatencyDuration: 30,  // 30초 이상 처지면 점프
};
```

---

### 라이브 멀티 뷰어 환경 — 100만 명이 동시에 보면

치지직 인기 스트리머 방송은 동시 시청자 10만~100만 명이 같은 라이브 엣지에서 본다.

```
[라이브 엣지 = seg100]
  └── 100만 명이 동시에 seg100.ts 요청
```

이 100만 요청이 어떻게 처리되나? 1강에서 본 CDN 캐시가 여기서 결정적이다.

```
[CDN 엣지 서버 A]
  첫 요청: seg100.ts를 원본 서버에서 받아옴
  나머지 99,999 요청: 캐시에서 바로 응답

→ 원본 서버 부하: 100만 → CDN 노드 수만큼만 (예: 50개)
```

근데 여기서 문제가 있다. 라이브는 **세그먼트가 만들어지자마자 요청이 몰린다**. CDN 엣지에 캐시가 없는 시점이 0초간이라도 존재한다.

이 짧은 순간에 99,999개 요청이 원본 서버로 가면? **Thundering Herd 문제**. CDN이 이걸 막기 위해 "request coalescing" (요청 병합) 기능을 쓴다. 같은 URL 요청이 동시에 오면 하나만 원본 서버로 보내고 나머지는 그 응답을 기다리게 한다.

---

### 2강 핵심 3줄

1. VOD는 모든 세그먼트가 미리 존재(`#EXT-X-ENDLIST`), 라이브는 sliding window로 새 세그먼트가 계속 추가됨
2. 라이브 엣지 지연은 세그먼트 크기 + 플레이어 버퍼의 합 (보통 18~25초) — 채팅이 영상보다 빨라 보이는 이유
3. 라이브 버퍼링 후 복구 시 라이브 엣지로 자동 점프하는 게 일반적 — 채팅 동기화가 더 중요해서
