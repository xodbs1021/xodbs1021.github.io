## 레벨 2-1 / 2강: RTMP 연결과 메시지 흐름 — 실제 동작 분석

---

### RTMP URL 구조

OBS에서 본 적 있는 그 주소.

```
rtmp://rtmp.chzzk.naver.com/live/abcd-1234-efgh
└┬─┘ └─────────┬─────────┘ └┬┘ └────────┬────────┘
프로토콜    호스트         앱     스트림 키
```

각 부분의 역할:

**프로토콜**: `rtmp` / `rtmps` / `rtmpt`

**호스트**: 서버 주소. 포트 생략 시 1935.

**앱 (Application)**: 서버 안의 논리적 공간. `live`, `vod`, `chat` 같은 이름. 같은 서버에서 여러 용도를 구분.

**스트림 키**: 어떤 스트림인지 식별. 동시에 같은 키로 송출하면 충돌. 보통 인증 토큰 역할도 함.

치지직의 경우 방송 시작할 때 발급받은 스트림 키가 곧 인증 수단이다. 다른 사람에게 알려주면 누구나 내 채널에 송출할 수 있다.

---

### 연결 전체 흐름

OBS가 방송 시작 누를 때 일어나는 일:

```
[1단계: TCP 연결]
OBS → 서버: TCP SYN (포트 1935)
서버 → OBS: TCP SYN-ACK
OBS → 서버: TCP ACK
                                        (RTT 1번 = ~30ms)

[2단계: RTMP Handshake]
OBS → 서버: C0+C1 (1537 byte)
서버 → OBS: S0+S1+S2 (3073 byte)
OBS → 서버: C2 (1536 byte)
                                        (RTT 1.5번 = ~45ms)

[3단계: NetConnection]
OBS → 서버: connect("live")
서버 → OBS: _result (Window Ack Size, Set Peer BW)
                                        (RTT 1번 = ~30ms)

[4단계: NetStream]
OBS → 서버: createStream()
서버 → OBS: _result (streamId = 1)
OBS → 서버: publish("abcd-1234-efgh", "live")
서버 → OBS: onStatus (NetStream.Publish.Start)
                                        (RTT 1번 = ~30ms)

[5단계: 메타데이터]
OBS → 서버: @setDataFrame (onMetaData)
  - width, height, framerate, videocodecid, audiocodecid

[6단계: 영상 송출 시작]
OBS → 서버: Video Message (계속)
OBS → 서버: Audio Message (계속)
```

연결 시작에 약 135ms 걸린다. OBS의 "방송 시작" 버튼 누르고 실제 데이터가 흐르기 시작할 때까지.

---

### NetConnection vs NetStream — 두 단계로 나뉜 이유

RTMP는 연결을 두 계층으로 나눈다.

**NetConnection**: 서버와의 논리적 연결. 한 OBS에서 한 서버에 하나만.

**NetStream**: 그 위에서 실제 데이터를 주고받는 채널. 여러 개 가능.

```
[NetConnection: rtmp://chzzk/live]
    │
    ├── NetStream 1: 메인 방송 (1080p60)
    ├── NetStream 2: 백업 (720p30)
    └── NetStream 3: 채팅 메타데이터
```

이론적으로 한 연결에서 여러 스트림을 동시에 보낼 수 있다. 실제로는 OBS가 1 NetConnection : 1 NetStream으로 단순화해서 쓴다.

---

### publish vs play — 송출과 시청

RTMP는 양방향 프로토콜이라 같은 구조로 송출과 시청을 한다.

```
[송출 (OBS)]
NetStream.publish("abcd-1234", "live")
  → 클라이언트가 서버로 데이터 전송

[시청 (Flash Player 시절)]
NetStream.play("abcd-1234")
  → 서버가 클라이언트로 데이터 전송
```

publish는 명령형: "이 스트림 이름으로 송출 시작할게".
play는 요청형: "이 스트림 이름으로 보내줘".

지금은 시청자 측 RTMP가 사실상 없어졌다. play는 거의 안 쓰이고 publish만 남았다.

### Video Message의 실제 내용

RTMP의 Video Message 1개에 들어가는 내용:

```
[Video Message 헤더 - 1 byte]
- 첫 4비트: Frame Type
    1 = Key Frame (I-frame)
    2 = Inter Frame (P/B-frame)
- 다음 4비트: Codec ID
    7 = H.264 (AVC)
    12 = H.265 (HEVC) - 일부 지원

[H.264인 경우 추가 헤더 - 4 byte]
- AVCPacketType (1 byte)
    0 = Sequence Header (SPS/PPS)
    1 = NALU
- Composition Time Offset (3 byte)

[Payload - 가변]
- 실제 H.264 NAL Unit 데이터
```

이 구조가 RTMP가 H.264만 잘 지원하는 이유다. H.265는 Adobe가 공식 스펙에 추가하지 않아서 비표준 확장으로만 가능.

### 송출 중 발생하는 메시지들

영상 데이터만 흐르는 게 아니다. 제어 메시지가 계속 오간다.

```
[OBS → 서버 - 주기적]
- Video Message (30fps면 초당 30개)
- Audio Message (AAC 1024 샘플 단위)
- Acknowledgement (받은 데이터 양 알림)

[서버 → OBS - 가끔]
- Window Acknowledgement Size (받기 윈도우 크기)
- Set Peer Bandwidth (대역폭 조절 요청)
- Ping (연결 유지 확인)
```

**Acknowledgement**가 중요하다. 일정량(보통 2.5 MB) 데이터를 보낼 때마다 OBS는 "여기까지 보냈어"를 서버에 알린다. 서버는 자기 윈도우 안에서 적절히 처리하는지 확인.

---

### Send Buffer — RTMP가 사용하는 송신 큐

OBS는 인코딩된 데이터를 즉시 보내지 않는다. **Send Buffer**에 쌓아두고 TCP가 처리할 수 있는 만큼만 보낸다.

```
[정상 상태]
인코더 → Send Buffer → TCP 전송 (잘 흐름)
       Buffer 점유율: 1% 이하

[네트워크 느려짐]
인코더 → Send Buffer → TCP 전송 (느림)
       Buffer 점유율: 50% (계속 증가)

[네트워크 매우 느림 / 끊김]
인코더 → Send Buffer → TCP 전송 (못 보냄)
       Buffer 점유율: 100% → Drop Frame
```

OBS 상태바에 빨간색으로 "네트워크가 느려서 프레임 드롭됨"이 뜨는 게 이거다. Send Buffer가 가득 차서 인코더가 만든 프레임을 버린다.

방송 중 OBS 우하단 통계 보면 "건너뛴 프레임"이 카운트되는데, 그게 이거다.

---

### RTMP 한 연결에 여러 오디오 트랙 못 담는 이유 — 깊이 보기

스펙을 보면 RTMP는 음성 메시지가 하나의 Audio Message 타입만 정의한다.

```
RTMP Audio Message:
- Codec, Sample Rate, Sample Size, Stereo/Mono 정보만
- 트랙 번호 같은 필드 없음
```

여러 오디오 트랙을 보내려면 Audio Message에 "트랙 ID" 필드가 있어야 한다. 그게 없다.

이론적으로 NetStream을 여러 개 만들면 가능하지만, 그러면 동기화가 안 맞다. 영상-오디오 동기화는 한 NetStream 안에서만 보장된다.

이게 PokeClip에서 RTMP 대신 MPEG-TS를 쓴 핵심 이유다.

```
RTMP Audio Message:
  [헤더] [오디오 데이터]
       ↑
  트랙 번호 필드 없음

MPEG-TS Audio Packet:
  [헤더 (PID 포함)] [오디오 데이터]
                ↑
  PID로 트랙 구분 가능
```

### 연결 끊김 처리

라이브 송출 중 RTMP 연결이 끊어지면?

```
[정상]
OBS ──RTMP──▶ 서버

[네트워크 일시 단절]
OBS    X    서버
OBS의 TCP는 끊김 감지 (수 초 ~ 수십 초)
  → OBS가 자동 재연결 시도
  → 재연결 성공 시 새 RTMP 핸드셰이크부터 다시
  → 그 사이 영상 끊김
```

OBS의 "스트림 자동 재시도" 옵션이 이 동작을 제어한다. 기본값 10초 간격 무한 재시도.

치지직 같은 플랫폼은 OBS 재연결 동안 시청자에게 "잠시 연결 중..." 화면을 보여준다. 30초 이상 끊기면 방송 종료 처리.

### 2강 핵심 3줄

1. RTMP는 TCP 1935 기반, Handshake + NetConnection + NetStream 단계로 연결 — 시작에 약 135ms
2. Video/Audio/제어 메시지가 한 TCP 연결을 인터리빙해서 흐름 — Send Buffer 가득 차면 프레임 드롭
3. Audio Message에 트랙 ID 필드가 없어서 멀티 오디오 트랙 불가 — 이게 PokeClip이 MPEG-TS 쓴 진짜 이유
