## 레벨 2-2 / 5강: HLS 운영 — CDN, 비용, 보안, 실전 이슈

---

### Origin 서버의 역할

라이브 HLS 파이프라인의 시작점.

```
[Origin이 하는 일]
1. RTMP/SRT 인제스트 받음
2. 트랜스코딩 (1080p → 720p, 480p, 360p ABR ladder 생성)
3. HLS 패키징 (.ts 세그먼트 + .m3u8 플레이리스트 생성)
4. CDN에 푸시 또는 CDN이 pull
```

Origin은 보통 강력한 GPU 서버. 트랜스코딩이 CPU/GPU 집약적.

```
[Origin 서버 사양 예시]
- CPU: AMD EPYC 64코어
- GPU: NVIDIA L4 × 2 (NVENC 트랜스코딩)
- RAM: 256 GB
- Network: 25 Gbps
- 동시 처리: 라이브 채널 10~20개
```

치지직 같은 곳은 동시 라이브 채널이 수천~수만 개. Origin 서버를 수백 대 운영한다.

---

### CDN으로 전달하는 방식 — Push vs Pull

Origin이 CDN과 통신하는 두 가지 방식.

**Push 방식**:
```
Origin이 직접 CDN에 세그먼트 업로드
Origin → "새 seg100.ts 만들었어" → CDN
```

장점: 빠름 (CDN이 요청 기다릴 필요 없음).
단점: Origin이 모든 CDN 노드에 보내야 해서 복잡.

**Pull 방식**:
```
CDN이 필요할 때 Origin에 요청
시청자 → CDN → "seg100.ts 없는데?" → Origin → 전달
```

장점: Origin이 단순. 시청자 없으면 데이터 안 보냄.
단점: 첫 요청 시 지연 발생 (Cache Miss).

대부분 Pull 방식. CDN이 알아서 캐시 관리.

### Origin Shield 운영

1-4강에서 본 중간 캐시 레이어.

```
[전세계 시청자 1000만]
        ↓
[CDN Edge 1000개]
        ↓ (cache miss 시)
[Origin Shield 5~10개]
        ↓ (shield도 cache miss)
[Origin 서버]
```

Shield 없으면: Edge 1000개가 동시에 Origin에 요청. Origin 폭발.

Shield 있으면: Edge 1000개 → Shield 10개 → Origin 1대. 부하 100배 감소.

Cloudflare, CloudFront, Akamai 모두 Origin Shield 기능 제공. 활성화 권장.

### Cache 정책의 실전 설계

이전에 본 Cache-Control을 실제 라이브 환경에서 어떻게 잡나.

```http
# .ts 세그먼트 - 영원불변
Cache-Control: public, max-age=86400, immutable
ETag: "abc123"

# .m3u8 라이브 플레이리스트 - 2초마다 갱신
Cache-Control: public, max-age=2, stale-while-revalidate=4

# .m3u8 VOD 플레이리스트 - 변하지 않음
Cache-Control: public, max-age=3600
```

**immutable** 디렉티브가 중요. 브라우저가 "이 파일은 절대 안 바뀐다"고 알면 재검증조차 안 한다. 304 응답조차 안 생김.

세그먼트는 한 번 만들어지면 안 바뀌니 immutable이 안전.

### Signed URL — 무단 사용 차단

라이브 영상이 인기 있으면 URL이 SNS에 퍼진다. 누구나 보면 CDN 비용 폭증.

```
시청자가 사이트에서 .m3u8 요청
  ↓
백엔드가 토큰 발급
  ↓
플레이어가 .m3u8?token=abc&expires=1700000060 요청
  ↓
CDN이 토큰 검증
  ↓
유효하면 응답, 아니면 401
```

토큰 구조:

```python
import hmac, hashlib, time

def sign_url(url, secret, expires_in=60):
    expires = int(time.time()) + expires_in
    message = f"{url}?expires={expires}"
    signature = hmac.new(
        secret.encode(),
        message.encode(),
        hashlib.sha256
    ).hexdigest()
    return f"{url}?expires={expires}&signature={signature}"
```

CDN에서 검증:
1. expires 안 지났는지
2. signature가 secret로 다시 만든 값과 같은지
3. (선택) 같은 IP인지

CloudFront의 Signed URL, Akamai Token Authentication 모두 같은 원리.

### 토큰 만료 처리 — 라이브 방송의 어려움

VOD는 토큰을 길게 잡아도 된다 (1시간). 한 번 받으면 끝.

라이브는 까다롭다. 방송이 3시간 이어지는데, 그동안 계속 새 세그먼트를 받아야 한다.

```
[방송 시작 - 토큰 발급]
플레이어가 token1 받음 (60분 유효)
  ↓
60분 후 만료
  ↓
세그먼트 다운로드 실패
  ↓
플레이어 멈춤?
```

해결책 두 가지:

**1. 길게 잡기**
방송 평균 시간 + 여유 (예: 12시간)
단점: 유출되면 12시간 동안 도용 가능

**2. Refresh Token**
플레이어가 만료 전에 새 토큰 요청
복잡하지만 보안 우수

대부분 플랫폼이 1번. 단순함이 이김.

### DRM — 유료 콘텐츠 보호

라이브 스포츠 중계, 유료 콘서트 같은 건 DRM이 필요.

```
[DRM 흐름]
1. 영상 인코딩 시 키로 암호화
2. 시청자가 결제
3. 백엔드가 라이센스 발급
4. 플레이어가 라이센스로 영상 복호화 → 재생
```

주요 DRM 시스템:

| DRM | 지원 브라우저 | 회사 |
|-----|------------|-----|
| Widevine | Chrome, Firefox, Edge, Android | Google |
| FairPlay | Safari, iOS, tvOS | Apple |
| PlayReady | Edge, Xbox | Microsoft |

세 가지 모두 지원해야 모든 시청자가 본다. **Multi-DRM** 구성.

HLS는 **AES-128 암호화**나 **Sample-AES** 방식으로 세그먼트를 암호화.

```
#EXTM3U
#EXT-X-VERSION:3
#EXT-X-TARGETDURATION:6
#EXT-X-KEY:METHOD=SAMPLE-AES,URI="https://license.example.com/key",KEYFORMAT="com.apple.streamingkeydelivery"
#EXTINF:6.000,
seg100.ts
```

`#EXT-X-KEY`가 플레이어에게 "라이센스 서버 가서 키 받아와"를 알린다.

라이브 스트리밍에서 DRM은 비용 증가 요인. 라이센스 서버 부하 + Origin Shield 친화성 떨어짐.

치지직 같은 광고 모델은 DRM 없음. 유료 콘텐츠 플랫폼만 사용.

### 모니터링 — Beacon과 QoE

1-3강에서 본 QoE 측정을 HLS 환경에 적용.

```javascript
hls.on(Hls.Events.FRAG_LOADED, (event, data) => {
  beacon.send({
    type: 'fragment_loaded',
    url: data.frag.url,
    duration: data.frag.duration,
    loadTime: data.stats.tload - data.stats.trequest,
    size: data.stats.total,
  });
});

hls.on(Hls.Events.LEVEL_SWITCHED, (event, data) => {
  beacon.send({
    type: 'level_switched',
    newLevel: data.level,
    bandwidth: hls.bandwidthEstimate,
  });
});

hls.on(Hls.Events.BUFFER_STALLED, () => {
  beacon.send({
    type: 'rebuffer_start',
    timestamp: Date.now(),
  });
});
```

이 Beacon들이 Prometheus/Grafana로 모임.

```
[실시간 대시보드]
지역별 ABR 분포:
  서울: 1080p 70%, 720p 25%, 480p 5%
  부산: 1080p 60%, 720p 30%, 480p 10%
  ↑ 부산에서 1080p 비율 낮음 → 지역 네트워크 점검 필요

Rebuffering Ratio:
  KT: 0.4%
  SKT: 0.5%
  LGU+: 1.2%
  ↑ LGU+ 시청자가 더 자주 버퍼링 → ISP 페어링 점검
```

### 비용 구조 — 라이브의 현실

1-4 3강에서 본 비용 예시 다시. 더 구체적으로.

```
[치지직 인기 스트리머 1명, 4시간 방송]
시청자: 5만 명 (피크 10만)
평균 화질: 720p 4 Mbps

총 트래픽:
5만 × 4 Mbps × 4시간 × 60 × 60 / 8 ÷ 1024
= 약 360,000 GB
= 360 TB

CDN 비용 (CloudFront, $0.085/GB 한국):
360,000 × $0.085 = $30,600 (약 4천만원)

자체 CDN이라면 절반 이하: 약 1.5천만원
```

월 단위로 가면:
- 인기 스트리머 100명 × 매일 4시간 × 30일
- 월 트래픽: 약 1 EB (엑사바이트)
- 월 CDN 비용: 수십~수백억

이래서 자체 CDN, P2P, 트랜스코딩 효율화에 투자한다.

### 흔한 실전 이슈 — 5가지

**1. M3U8과 .ts 호스트 분리 시 CORS**
```
.m3u8: a.cdn.com
.ts:   b.cdn.com
→ 다른 도메인이라 CORS 차단
```

해결: 같은 도메인 사용 또는 CORS 헤더 설정.

**2. 첫 세그먼트 시작 지연**
```
플레이어 시작 → m3u8 요청 → 파싱 → 세그먼트 다운로드 → 트랜스먹싱 → 재생
이 모든 게 끝나야 첫 프레임
```

해결: 작은 세그먼트, 마스터 플레이리스트 인라인 화질 정보.

**3. 라이브 방송 종료 처리**
```
스트리머가 방송 끝냄
→ Origin이 .m3u8에 #EXT-X-ENDLIST 추가
→ CDN 캐시에 옛날 버전 있음
→ 플레이어는 계속 폴링
```

해결: ENDLIST 즉시 push, 캐시 짧게.

**4. 인기 스트리머 동접 폭증**
```
일반 시간: 1만 명
인기 스트리머 시작: 5분 만에 10만 명
→ CDN 캐시 워밍업 안 됨 → Origin 부하
```

해결: Pre-positioning, Origin Shield 강화.

**5. 모바일 백그라운드 진입**
```
시청자가 카톡 켜면 영상 백그라운드로
→ iOS는 영상 디코딩 중단, 오디오만 재생
→ 다시 포그라운드 시 라이브 엣지 못 따라잡음
```

해결: 백그라운드 복귀 감지 → 라이브 엣지 점프.

---

### 5강 핵심 3줄

1. Origin → Origin Shield → CDN Edge 다층 구조 — Pull 방식이 표준, immutable 캐시 정책으로 효율 극대화
2. Signed URL로 무단 사용 차단, DRM은 유료 콘텐츠에만 — 라이브 토큰 만료는 길게 잡거나 Refresh 패턴
3. 인기 스트리머 한 명 4시간 방송 CDN 비용 수천만원 — 자체 CDN, P2P, 효율적 코덱이 핵심 비용 절감 수단
