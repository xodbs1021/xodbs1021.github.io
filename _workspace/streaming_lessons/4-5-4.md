## 레벨 4-5 / 4강: 비용 최적화 종합 — 라이브 인프라의 진짜 ROI

### 라이브 플랫폼의 비용 구조

전체 비용을 세부 분해.

```
[월 100만 시청자 시간 기준 - 가상 산정]
CDN 트래픽:       $45,000 (45%)
트랜스코딩 GPU:   $25,000 (25%)
인제스트:         $5,000  (5%)
패키저:           $5,000  (5%)
스토리지:         $10,000 (10%)
관제/네트워크:    $10,000 (10%)
─────────────────────────
합계:           $100,000/월
```

CDN과 트랜스코딩이 70%. 절감 우선순위.

### CDN 비용 최적화 - 가장 큰 ROI

CDN 비용 구조.

```
[CDN 비용 = 트래픽 GB × 단가]
GB 단가 (대량 협상 시):
- $0.02/GB (Akamai 대형 고객)
- $0.05/GB (CloudFront 표준)
- $0.08/GB (소규모 약정)
```

절감 방법.

**1. Multi-CDN 비용 협상**
한 CDN에 의존하면 협상력 없음.

```
[전략]
3개 CDN과 계약: Akamai, CloudFront, Cloudflare
지역/시간대별 가장 싼 곳으로 라우팅
경쟁시켜 단가 인하
```

협상으로 30~50% 절감 가능.

**2. Origin Shield 효율**
CDN → Origin 트래픽 최소화.

```
[Shield 없을 때]
CDN Edge 100개 × 각각 Origin Pull = 100배 부하

[Shield 있을 때]
CDN Edge 100개 → Shield 5개 → Origin
Shield의 hit rate 99%
Origin 트래픽 100배 → 1배
```

Origin 트래픽 비용 90% 절감.

**3. 캐시 TTL 최적화**
```
[권장 TTL]
세그먼트 (.m4s): 24시간
매니페스트 (라이브): 1~2초
매니페스트 (VOD): 30분
init.mp4: 24시간
```

매니페스트 TTL이 너무 길면 라이브 업데이트 안 됨.
너무 짧으면 Origin 부하 증가. 1~2초가 균형점.

### 트랜스코딩 비용 최적화

**1. 인스턴스 타입 다양화**
```
[Reserved (1년 약정)]
80% 할인, 평소 부하만큼
약 50% 비중

[On-demand]
기준 가격, 평균 부하 변동분
약 30% 비중

[Spot]
80~90% 할인, 갑작스러운 부하
약 20% 비중 (중단 감수)
```

전체 트랜스코딩 비용 40~50% 절감.

**2. GPU 활용률 극대화**
```
목표: GPU NVENC 사용률 80%
현재: 평균 50% → 30% 낭비

개선:
- 채널 패킹 알고리즘 개선
- MIG (Multi-Instance GPU)로 분할
- 동적 사다리 (필요한 채널만 인코딩)
```

**3. 카테고리별 사다리**
정적 콘텐츠는 비트레이트 적게.

```
[저스트 채팅]
1080p @ 3000k (일반 6000k 대비 50% 절감)

[발로란트]
1080p @ 8000k (필요한 만큼)

전체 평균 30% 비트레이트 절감
→ 트랜스코딩 부하 및 CDN 트래픽 같이 절감
```

**4. Per-Title (인기 스트리머만)**
```
인기 스트리머: Per-Title 적용 (분석 비용 < 절감 효과)
일반: 표준 사다리
```

### 코덱 비용 분석

각 코덱의 라이센스 비용.

```
[코덱 비용 (100만 디바이스 기준)]
H.264: ~$250,000/년
H.265: ~$500,000~1,500,000/년 (복잡)
VP9:   $0 (Google 오픈소스)
AV1:   $0 (AOMedia)
```

치지직 규모 (시청자 수백만):
```
H.264 라이센스: 약 $10M/년 (디바이스 다 합쳐서)
AV1로 전환: $0
```

근데 AV1은 디코더 보급률 30% → 일부만 적용.

```
[AV1 전환 시나리오]
프리미엄 사다리: AV1 (보급된 디바이스만)
일반 사다리: H.264 (호환성)
점진적 AV1 전환
```

### 스토리지 비용

세그먼트 저장.

```
[스토리지 계층]
Hot (즉시 접근): Redis 또는 SSD
  - 라이브 + 최근 1시간 VOD
  - $0.1/GB/월

Warm (자주 접근): S3 Standard
  - 7일 내 VOD
  - $0.023/GB/월

Cold (드물게 접근): S3 IA + Glacier
  - 30일 후
  - $0.0125/GB/월 (IA), $0.004/GB/월 (Glacier)
```

라이프사이클 정책으로 자동 이전.

```yaml
# S3 Lifecycle
Rules:
  - Transition Hot → Standard after 1 hour
  - Transition Standard → IA after 7 days
  - Transition IA → Glacier after 30 days
  - Expiration after 365 days (or 다시보기 정책)
```

### 비용 모니터링 - 일별 추적

```

어제 비용: $32,800
전주 평균: $30,500
```

증가율 10% 이상 시 자동 알람. 누가 어떤 설정 변경했는지 추적.

### 비용 vs 매출 비교

라이브 플랫폼 비즈니스 모델.

```
[수익 구조]
광고 매출: 시청자당 $0.50/시간 (CPM 기반)
구독 매출: 구독자당 $5/월
후원 수수료: 후원 금액의 30%

[손익]
수익: $1.5/시청자/시간 (모든 소스 합계, 추정)
비용: $0.10/시청자/시간
마진: $1.4/시청자/시간 (93%)
```

라이브 플랫폼은 마진이 매우 좋음. 비용 절감보다 시청 시간 증가가 더 중요.

근데 비용을 무한정 늘리면 안 됨. 채널당 단가 모니터링.

### A/B 테스트로 비용/품질 트레이드오프

새 사다리 도입 시 매출 영향 검증.

```
[실험 설계]
A: 현재 사다리 (1080p @ 6000k, ...)
B: 절감 사다리 (1080p @ 4500k, ...)

50% 사용자 각각

[1주일 측정]
A 그룹:
  - 평균 시청 시간: 38분
  - Rebuffering: 0.4%
  - 채널당 비용: $0.12/시간

B 그룹:
  - 평균 시청 시간: 37분 (-3%)
  - Rebuffering: 0.6% (+0.2%p)
  - 채널당 비용: $0.09/시간 (-25%)

[결정]
시청 시간 3% 감소 = 매출 3% 감소
비용 25% 감소

수익성 계산:
A: 매출 $1.5 × 0.633시간 = $0.95, 마진 $0.83
B: 매출 $1.5 × 0.617시간 = $0.93, 마진 $0.84

→ B가 마진 약간 우수 → 도입
```

데이터 기반 결정의 표본.

### 캐시 워밍 - Pre-positioning

인기 방송 시작 전 미리.

```python
class PrePositioner:
    def predict_popular_stream(self):
        """5분 후 시작될 인기 방송 예측"""
        scheduled = self.get_scheduled_streams(in_minutes=5)
        return [s for s in scheduled if s.expected_viewers > 10000]
    
    def warm_cache(self, stream):
        """CDN 엣지에 미리 캐시 워밍"""
        for region in self.regions:
            for edge in region.edges:
                # 가짜 요청으로 캐시 채우기
                edge.preload(stream.expected_segments)
```

방송 시작 직후 Thundering Herd 방지. Origin 부하 90% 감소.

### 비용 거버넌스

큰 조직에서 비용 통제.

```
[승인 제도]
$1,000/월 미만 변경: 엔지니어 자율
$1,000~10,000: 팀 리드 승인
$10,000+: 디렉터 승인

[예산 관리]
월별 예산 설정
80% 도달 시 알람
100% 초과 시 추가 비용 발생 자동 거부 (가능한 경우)
```

```python
class CostGovernor:
    def check_change(self, change):
        """변경 사항의 예상 비용 변화"""
        delta = self.estimate_cost_impact(change)
        
        if delta < 1000:
            return "AUTO_APPROVED"
        elif delta < 10000:
            return "NEEDS_TEAM_LEAD"
        else:
            return "NEEDS_DIRECTOR"
```

설정 변경 PR마다 비용 영향 분석 자동 첨부.

### FinOps - 클라우드 비용 관리

FinOps Foundation 표준 프랙티스.

```
[3단계]
1. Inform: 비용 가시화
   - 팀별/서비스별 비용 분배
   - 단위 비용 (채널당, 시청자당)
   
2. Optimize: 절감
   - Reserved/Spot 비율
   - 사용 안 하는 리소스 정리
   - 인스턴스 라이트사이징

3. Operate: 거버넌스
   - 예산 자동 알람
   - 변경 승인 프로세스
   - 정기 비용 리뷰
```

대형 플랫폼은 FinOps 팀 따로 운영.

### 비용 절감 vs 화질 - 최종 트레이드오프

```
[극단적 비용 절감]
모든 채널 480p @ 800k
시청 시간 50% 감소 → 매출 50% 감소
비용 80% 감소
→ 손실

[극단적 화질 우선]
모든 채널 1080p60 @ 10000k + Per-Title + 5단 사다리
시청 시간 5% 증가 → 매출 5% 증가
비용 100% 증가
→ 손실

[최적점]
인기 콘텐츠: 풀 화질
일반 콘텐츠: 표준
모바일: 축소 사다리
A/B 테스트로 데이터 기반 결정
```

엔지니어링은 결국 트레이드오프. 데이터로 결정.

### 4강 핵심 3줄

1. CDN(45%) + 트랜스코딩(25%) 절감이 ROI 큼 — Multi-CDN 협상 + Origin Shield + Reserved/Spot 혼합
2. 카테고리별 사다리, Per-Title (인기 스트리머만), AV1 점진 도입으로 30~50% 비용 절감
3. A/B 테스트로 비용 절감의 매출 영향 검증 필수, FinOps + 채널당 단위 비용 일별 추적이 거버넌스 표준
